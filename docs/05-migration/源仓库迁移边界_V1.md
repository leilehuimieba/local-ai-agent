## 1. 文档目的

本文件用于回答以下问题：

1. 原仓库哪些内容可以直接参考
2. 原仓库哪些内容可以复制后重构
3. 原仓库哪些内容只适合借鉴设计，不适合直接迁移
4. 原仓库哪些内容必须明确排除
5. 迁移工作应该遵循什么减法原则

本文件的目标不是“尽可能多搬”，而是“只搬对新项目第一版真正有价值的部分”。

## 2. 迁移总原则

### 2.1 新项目不是旧项目的延续开发分支

新项目是独立重构仓库，不是原仓库的直接延续。

因此：

1. 不要求目录结构完全一致
2. 不要求接口命名完全一致
3. 不要求实现行为逐项等价

### 2.2 迁移目标是吸收价值，不是复制历史

迁移应优先吸收：

1. 成熟边界
2. 有价值的架构经验
3. 已验证的控制面思路
4. 适合第一版的正式主链路能力

不应复制：

1. 历史复杂性
2. 研究性残留
3. 比赛/临时工作流
4. 与第一版主链路无关的功能区

### 2.3 迁移必须服务于第一版 MVP

只有满足以下条件的内容才有资格进入迁移候选：

1. 服务于首版正式产品边界
2. 服务于 MVP 闭环
3. 不会明显拖高实现复杂度
4. 不会把原项目包袱直接带入新仓库

---

## 3. 迁移关系分类

建议把源仓库内容分成四类关系。

### 3.1 `参考`

定义：

1. 只用于理解需求
2. 只用于理解架构思路
3. 不直接复制代码

### 3.2 `复用后重构`

定义：

1. 可复制到新项目
2. 但必须经过裁剪、重命名或结构性调整
3. 不允许盲目原样保留

### 3.3 `设计借鉴但重写`

定义：

1. 核心思想可保留
2. 代码实现不适合直接迁移
3. 新项目必须按新架构重写

### 3.4 `明确排除`

定义：

1. 不复制
2. 不并入新仓库主链路
3. 即使保留参考，也不能误导为正式首版功能来源

## 4. 源仓库目录级迁移判断

以下判断基于源仓库当前结构。

### 4.1 `gateway/`

迁移关系：

1. `复用后重构`

原因：

1. Go 控制面角色与新项目方向一致
2. 有较高参考价值
3. 可减少重复造轮子

迁移建议：

1. 保留控制面定位
2. 筛选对首版有价值的 handler、router、session、event 等结构
3. 去除与首版无关的面板、能力和历史接口
4. 不要求 API 完全保持旧命名

### 4.2 `frontend/`

迁移关系：

1. `参考`
2. `设计借鉴但重写`

原因：

1. 前端第一版允许重写
2. 新前端只做聊天页、日志页、设置页
3. 旧前端面板较多，不适合整包带入

迁移建议：

1. 借鉴信息架构、状态流、事件显示方式
2. 不直接复制旧前端作为新前端主体
3. 如果局部组件或样式思路成熟，可后续选择性借鉴

### 4.3 `browser-extension/`

迁移关系：

1. `明确排除`

原因：

1. 第一版不纳入浏览器扩展
2. 不属于首版正式产品入口

迁移建议：

1. 不复制到新项目主仓库
2. 如需保留资料，仅作为后续专题参考

### 4.4 `src/local_agent_control/`

迁移关系：

1. `设计借鉴但重写`

原因：

1. 这里主要承载旧 Python runtime 和 agent 逻辑
2. 新项目要用 Rust 替代 Python 正式主链路
3. 旧实现复杂度高，不适合直接复制

迁移建议：

1. 只拆“职责”，不搬“实现”
2. 把任务分析、执行调度、记忆、风险、输出等职责抽象出来
3. 由 Rust 新运行时重新承接

### 4.5 `workers/`

迁移关系：

1. `参考`
2. `设计借鉴但重写`

原因：

1. workers 中有不少正式能力原型
2. 但第一版不应把所有 worker 一股脑搬过去
3. Rust 时代应重新定义能力来源

迁移建议：

1. 只参考首版需要的命令、文件、知识、记忆相关能力
2. 不默认迁入浏览器和桌面相关 worker
3. 对 Python worker 逻辑做 Rust 化重写，而非直接保留

### 4.6 `shared/contracts/`

迁移关系：

1. `复用后重构`

原因：

1. 共享合同思路很有价值
2. 有助于减少前端、Go、Rust 三层对齐成本

迁移建议：

1. 可以复制思路或部分文件
2. 但必须按新项目 API 合同重新整理
3. 不要求完全沿用旧 schema

### 4.7 `config/`

迁移关系：

1. `复用后重构`

原因：

1. 配置模板与策略模板可复用价值较高
2. 但新项目端口、模式、模型和权限边界与旧项目不同

迁移建议：

1. 筛选模式、工具、策略、provider 相关模板
2. 去除与旧项目历史功能强耦合的配置项
3. 重新命名和整理首版可见配置

### 4.8 `docs/`

迁移关系：

1. `参考`

原因：

1. 原项目文档是重要经验来源
2. 但新项目已重新建立文档体系
3. 不应将旧文档直接当成新项目正式设计文档

迁移建议：

1. 只作为设计参考资料
2. 新项目正式文档一律在新仓库重写

### 4.9 `scripts/`

迁移关系：

1. `参考`
2. `局部复用后重构`

原因：

1. 可能存在有价值的启动、预检、构建脚本思路
2. 但脚本区历史包袱和临时用途内容较多

迁移建议：

1. 只保留首版真正需要的启动和维护脚本
2. 比赛、临时、实验脚本一律不进入首版主链路

### 4.10 `skills/`

迁移关系：

1. `参考`

原因：

1. 技能思想对新项目有价值
2. 但旧技能目录包含 staging / imported / quarantine 等复杂层级
3. 第一版不应原样继承

迁移建议：

1. 只借鉴“模块化能力包”的思想
2. 不整体复制旧技能库
3. 后续在新项目中重新定义 Skill 包规范

### 4.11 `deliverables/`、`output/`、`outputs/`

迁移关系：

1. `明确排除`

原因：

1. 属于产物、归档或过程文件
2. 不属于新项目第一版正式代码与文档边界

### 4.12 `node_modules/`、缓存目录、编译产物

迁移关系：

1. `明确排除`

包括但不限于：

1. `node_modules/`
2. `.pytest_cache/`
3. `.ruff_cache/`
4. `gateway-server.exe`
5. `lach-launcher.exe`

原因：

1. 都不应进入新项目迁移范围

## 5. 文件级重点迁移策略

以下不是穷举，而是第一版最值得重点关注的文件级方向。

### 5.1 可重点参考或复用后重构的 Go 层文件

重点关注：

1. `gateway/internal/api/router.go`
2. `gateway/internal/session/`
3. `gateway/internal/api/handler/` 中与首版相关的部分
4. `gateway/cmd/server/`
5. `gateway/cmd/launcher/`

原则：

1. 优先保留控制面边界
2. 不把旧项目所有 handler 直接带入

### 5.2 仅做职责拆解的 Python 运行时文件

重点关注：

1. `src/local_agent_control/agent/loop.py`
2. `src/local_agent_control/runtime/`
3. `src/local_agent_control/agents/`

原则：

1. 只拆职责
2. 不原样迁移实现
3. 不追求 Python -> Rust 一一对照

### 5.3 可参考的 worker 方向

重点关注首版需要的：

1. 命令执行
2. 文件系统
3. 知识检索
4. 记忆相关

暂不优先：

1. 浏览器
2. 桌面
3. Doubao 专项
4. 复杂扩展 worker

## 6. 旧复杂性排除清单

以下复杂性不应进入新项目首版：

1. 研究型打分器和大量微策略模块
2. 比赛模式相关逻辑
3. 复杂多面板 UI 历史结构
4. 浏览器扩展主路径
5. 浏览器/桌面自动化能力主链路
6. staging/imported/quarantine 相关直接继承
7. 历史归档文档和交付产物

## 7. 新项目建议保留的“思想资产”

虽然很多代码不直接迁，但以下思想值得保留：

1. Go 作为控制面
2. 前后端通过统一控制面交互
3. 执行层独立于前端
4. 风险与确认是主链路组成部分
5. 能力层可插拔
6. 会话、事件、日志是正式系统概念

## 8. 新项目建议重写的“核心资产”

以下内容建议在新项目中重写：

1. Python runtime -> Rust runtime
2. Python 执行层 -> Rust 能力执行层
3. 旧前端 -> 新前端
4. 技能体系 -> 新的模块化 Skill 包体系

## 9. 迁移减法原则

迁移时必须坚持以下减法原则：

### 9.1 先问“为什么要带”

每迁一个目录、模块、文件，都必须先回答：

1. 它是否服务于第一版 MVP
2. 它是否服务于正式产品边界
3. 它是否能减少实现成本而不是增加复杂度

### 9.2 先带边界，不带历史

优先带：

1. 边界定义
2. 结构经验
3. 控制面思路
4. 合同思路

不优先带：

1. 历史功能堆叠
2. 研究残留
3. 未收口能力

### 9.3 先带主链路，不带枝叶

迁移优先顺序应是：

1. 启动入口
2. 控制面
3. 运行时主链路
4. 文件/命令/工作区/知识/记忆能力
5. 日志与确认

而不是：

1. 先带浏览器/桌面花哨能力
2. 先带扩展入口
3. 先带大量辅助面板

## 11. 建议的迁移工作顺序

第一版建议先做“迁移分析”，再做“代码迁移”。

推荐顺序：

1. 明确正式主链路
2. 识别 Go 可复用边界
3. 识别 Python 运行时职责
4. 识别首版能力实现来源
5. 明确不迁移项
6. 再开始复制和重构

不建议：

1. 先整仓复制再慢慢删
2. 先搬 Python 运行时再想 Rust 怎么做

---

## 12. 首版迁移时的目录处理建议

建议在新项目中按“新结构”落地，而不是直接复制旧结构。

例如：

1. Go 代码进入新项目的 `gateway/`
2. 前端进入新项目的 `frontend/`
3. Rust 运行时进入 `crates/`
4. 合同进入 `shared/`
5. 新文档全部进入新项目 `docs/`

旧项目的目录路径只是参考，不应成为新项目的结构束缚。

---

## 13. 成功迁移的判断标准

迁移成功不等于“搬了很多文件”，而应满足：

1. 新项目主链路更清晰
2. 旧复杂性没有被原样继承
3. Go 保留了控制面优势
4. Rust 运行时边界明确
5. 前端更聚焦首版产品
6. 新旧项目运行互不干扰

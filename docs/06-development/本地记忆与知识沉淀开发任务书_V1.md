## 1. 任务定位

本任务书只解决一件事：

> 把本地智能体的短期工作记忆、长期记忆、本地知识沉淀主链路正式落到当前 Go + Rust 架构中。

当前任务不是做：

1. 复杂记忆评分算法
2. 重型向量数据库
3. 多智能体共享记忆网络
4. 自动知识图谱
5. 高级自治记忆编排

当前任务要做的是：

1. 三层结构落地
2. 写入边界落地
3. 按需召回落地
4. Finish 阶段沉淀落地
5. 事件合同和前端可见结果落地

---

## 2. 当前阶段目标

本阶段的唯一目标是：

> 让记忆系统从“概念存在”升级为“运行时主链路正式能力”。

做到这一步后，系统应具备：

1. 当前任务可维护短期工作记忆
2. 跨任务可读写长期记忆
3. 高价值结论可沉淀到本地知识库
4. 所有召回按需进行，不做全量加载
5. 前端和日志能看到记忆写入与命中摘要

---

## 3. 本阶段硬约束

开始实现前，必须先满足：

1. 后端主链路仍以 `Rust + Go` 为主
2. 记忆主逻辑优先放在 `Rust runtime`
3. `Go` 只负责桥接、事件回流、会话和接口组织
4. 第一版不引入重型数据库
5. 第一版不做全量 embedding 依赖
6. 第一版不把记忆系统做成聊天记录堆积

---

## 4. 本阶段必须完成的 6 个模块

### 4.1 `working-memory`

职责：

1. 保存当前任务运行态摘要
2. 保存当前目标、计划、阻塞、最近观察
3. 为主循环提供最小工作状态

### 4.2 `long-term-memory`

职责：

1. 保存跨任务可复用的结构化记忆
2. 按类型存放偏好、规则、教训、决策
3. 支持按任务特征召回

### 4.3 `knowledge-store`

职责：

1. 保存经过总结后的稳定知识
2. 保持知识摘要、来源、标签、验证状态
3. 支持先摘要再展开的检索方式

### 4.4 `memory-router`

职责：

1. 在运行时按阶段决定读什么
2. 决定写入哪一层
3. 控制召回粒度和注入范围

### 4.5 `memory-persistence`

职责：

1. 负责本地落盘
2. 负责结构化读写
3. 负责基础去重与更新

### 4.6 `memory-events`

职责：

1. 输出记忆命中事件
2. 输出记忆写入事件
3. 输出知识沉淀事件
4. 输出跳过原因摘要

---

## 5. 推荐实现顺序

### Phase A：三层结构先立起来

先做：

1. `working-memory`
2. `long-term-memory`
3. `knowledge-store`

通过标准：

1. 三层结构有正式类型
2. 三层数据不会混存
3. 基础落盘可用

### Phase B：读写路由接入运行时

再做：

1. `memory-router`
2. `memory digest / knowledge digest`
3. `Finish` 阶段写入评估

通过标准：

1. 主循环可读取短期记忆
2. 长期记忆和知识库只按需召回
3. 任务结束后可评估沉淀候选

### Phase C：事件与三端合同补齐

最后做：

1. `memory-events`
2. Rust / Go / Frontend 合同补齐
3. UI 最小展示

通过标准：

1. 前端能看到记忆命中与写入摘要
2. 日志能区分记忆和知识沉淀事件
3. 不需要新页面也能消费结果

## 7. 文件级任务清单

### 7.1 `crates/runtime-core/src/memory.rs`

任务：

1. 定义短期工作记忆和长期记忆核心结构
2. 定义记忆类型枚举
3. 定义写入候选对象
4. 提供最小读写接口

要求：

1. 不把知识库结构混进长期记忆结构
2. 不直接存原始长文本日志

### 7.2 `crates/runtime-core/src/knowledge.rs`

任务：

1. 定义知识条目结构
2. 定义知识类型枚举
3. 定义知识摘要与详细内容分层
4. 提供最小检索接口

要求：

1. 支持摘要优先返回
2. 支持来源和验证状态字段

### 7.3 `crates/runtime-core/src/planner.rs`

任务：

1. 接入任务类型到记忆召回的选择逻辑
2. 让 planner 感知可用的记忆摘要和知识摘要
3. 控制召回粒度

要求：

1. 只传入摘要化结果
2. 不允许全量拼接长期记忆

### 7.4 `crates/runtime-core/src/lib.rs`

任务：

1. 在主循环中接入短期工作记忆更新
2. 在 `Analyze / Plan / Observe / Verify / Finish` 节点调用 memory router
3. 在 `Finish` 阶段触发沉淀评估

要求：

1. 写入动作必须在统一主线内完成
2. 不能做成旁路临时逻辑

### 7.5 `crates/runtime-core/src/contracts.rs`

任务：

1. 定义记忆命中事件结构
2. 定义记忆写入事件结构
3. 定义知识沉淀事件结构
4. 定义跳过写入原因结构

要求：

1. 字段稳定
2. 不依赖临时字符串拼接

### 7.6 `crates/runtime-core/src/events.rs`

任务：

1. 输出 `memory_recalled`
2. 输出 `memory_written`
3. 输出 `knowledge_written`
4. 输出 `memory_write_skipped`

要求：

1. 事件有摘要
2. 事件有类型
3. 事件有来源层级

### 7.7 `crates/runtime-core/src/storage.rs`

任务：

1. 提供记忆和知识的基础落盘能力
2. 提供按层级分目录或分表存储能力
3. 提供基础更新与去重入口

要求：

1. 第一版优先轻量
2. 不先引入重型基础设施

### 7.8 `crates/runtime-core/src/paths.rs`

任务：

1. 建立 `working_memory / long_term_memory / knowledge_base` 路径约定
2. 保证不同工作区可分区存储

### 7.9 `gateway/internal/contracts/contracts.go`

任务：

1. 对齐 Rust 侧新增记忆事件结构
2. 保证 Go 能稳定解析和转发

### 7.10 `gateway/internal/runtime/client.go`

任务：

1. 接收运行时新增事件
2. 回流到会话事件总线

### 7.11 `gateway/internal/session/bus.go`

任务：

1. 补齐记忆与知识事件分发
2. 保证前端能消费新增事件

### 7.12 `frontend/src/shared/contracts.ts`

任务：

1. 对齐记忆事件合同
2. 让前端类型系统能消费

### 8.3 知识条目至少包含

1. `id`

6. `tags`

## 9. 召回要求

### 9.1 短期工作记忆

默认随当前会话加载，但必须裁剪为最小工作摘要。

### 9.2 长期记忆

召回过滤条件至少包括：

1. 任务类型
2. 工作区
3. 模块路径
4. 记忆类型
5. 最近有效性

### 9.3 知识库

召回顺序固定为：

1. 先找摘要
2. 再按需展开正文
3. 再决定是否注入模型上下文

---

## 10. 写入要求

### 10.1 短期工作记忆

在以下节点更新：

1. `Analyze`
2. `Plan`
3. `Observe`
4. `Verify`

### 10.2 长期记忆

在以下条件成立时写入：

1. 用户长期偏好被明确确认
2. 失败教训形成明确结论
3. 项目规则被稳定识别
4. 任务完成后形成可复用经验

### 10.3 知识库

只有满足以下条件才写：

1. 结论稳定
2. 内容已摘要化
3. 可复用价值明确
4. 不是日志和聊天原文

---

## 11. 事件合同要求

第一版至少补齐以下事件：

1. `memory_recalled`
2. `memory_written`
3. `knowledge_written`
4. `memory_write_skipped`

每个事件至少应包含：

1. `run_id`
2. `layer`
3. `record_type`
4. `title`
5. `summary`
6. `source`
7. `reason`

---

## 12. 存储实现建议

### 12.1 第一版建议方案

优先采用：

1. `working memory`：会话态文件或轻量结构缓存
2. `long term memory`：`SQLite` 或 `JSONL`
3. `knowledge base`：`SQLite + 摘要文本`

### 12.2 去重要求

第一版至少做到：

1. 同标题同类型基础去重
2. 同来源重复写入拦截
3. 同一任务内重复候选折叠

### 12.3 不做的事

第一版不做：

1. 全量向量化
2. 重型外部数据库
3. 复杂语义排序模型

---

## 13. 验收标准

### 13.1 必须通过

1. 三层结构已正式存在
2. 主循环能读取短期工作记忆
3. 长期记忆和知识库按需召回
4. `Finish` 阶段能评估并写入长期记忆或知识库
5. 事件流能看到记忆命中和写入摘要
6. Go 和前端能消费新增合同

### 13.2 不通过情形

1. 仍把三层内容混在一个文件或一个结构里
2. 仍把聊天记录直接当长期记忆
3. 知识沉淀没有正式落点
4. 召回仍是全量读取
5. 没有事件合同，只能靠日志硬猜

---

## 14. 第二版增强边界

本任务书通过后，下一阶段才允许继续：

1. 记忆质量评分
2. 知识版本管理
3. 工作区级知识分区增强
4. embedding 检索
5. 更强去重和排序

---

## 15. 给实现型 AI 的明确要求

1. 先在现有 `runtime-core` 模块上收口
2. 必要时允许新增少量记忆相关模块，但必须职责单一
3. 不允许为了快把知识库做成“聊天记录备份”
4. 不允许先引入重型依赖再说
5. 完成后必须补对应验收文档

一句话要求：

> 当前任务不是做聪明的记忆算法，而是把本地记忆与知识沉淀主链路做稳。

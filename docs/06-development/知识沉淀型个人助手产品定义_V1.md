## 1. 文档目的

本文用于正式明确当前项目的主产品定义、适用场景与后续文档口径。

当前项目在产品层面首先定义为一个本地知识沉淀型个人助手，而不是多 Agent 平台、单一领域工具或能力拼装系统。该产品面向用户本人长期使用，核心目标是持续承接用户在学习、工作与思考过程中的输入，并将其沉淀为可复用的知识资产与可追踪的个人状态，在此基础上提供理解、建议与有限执行能力。

在上述主定义之上，产品可通过 Skill、MCP 及其他受控能力扩展适配不同场景，例如工程开发、通用学习、CTF / 安全分析等。但这些场景能力属于对核心助手的增强，不构成对产品本体的重新定义。

本文重点回答以下问题：

1. 当前主产品的正式定位是什么
2. 当前主产品的核心用户价值是什么
3. 哪些能力属于产品主干
4. 哪些能力属于场景增强，而不属于产品本体
5. 当前版本的边界应收在哪一层
6. 后续路线、开发与验收文档应以哪条主线为准

本文不展开以下内容：

1. 多智能体编排
2. 浏览器高权限自动化
3. 自动 fallback
4. 重型记忆评分机制
5. 大范围架构重写
6. 各类场景能力的具体工程实现细节

本文的作用，是先稳定“知识沉淀型个人助手”这一核心产品定义，再在不偏离主线的前提下逐步扩展场景能力，而不是在当前阶段将项目定义为能力平台或多产品集合。

## 2. 当前产品定位

当前项目不定义为单纯的本地项目助手、纯研发副驾、多 Agent 平台、全能桌面自动化 Agent 或单一领域专项工具。

当前更准确的正式定义应是：

> 一个面向用户本人长期使用、以知识沉淀与状态理解为主干、以下一步建议为高价值输出、并在明确授权下具备有限执行能力的本地个人助手。

这一定位包含五个核心含义：

1. `以人而不是以单一任务为中心`
   产品服务的对象是用户在学习、工作与长期成长中的连续过程，而不是某一次提问、某一个项目或某一类短期任务。

2. `以知识沉淀而不是一次性回答为核心价值`
   产品不仅负责回应当前问题，更负责将输入、过程、结果与经验沉淀为后续可复用的长期资产。

3. `以状态理解和下一步建议作为关键输出`
   产品需要持续理解用户当前做到哪里、缺什么、卡在哪里，并给出贴近真实情况的下一步建议。

4. `以场景增强扩展能力，而不改变产品本体`
   工程开发、通用学习、CTF / 安全分析等属于产品可支持的使用场景；Skill、MCP 等能力用于增强特定场景下的表现，而不构成多个并列产品定义。

5. `以有限执行作为辅助桥接能力`
   执行能力用于帮助用户推进任务、整理结果和完成少量明确动作，但不将产品定义为高权限自动化系统。

---

## 3. 当前服务对象

当前产品首先服务于：

1. `用户本人`

当前重点受益对象包括：

1. `有长期学习、工作推进、知识积累与复盘需求的个人用户`
2. `需要持续理解自身状态、沉淀过程并获得下一步建议的人`
3. `在当前阶段，重点包括工程师、技术学习者、CTF / 安全方向学习者，以及其他有明确成长目标的学习型用户`

当前产品持续承接的重复工作包括：

1. 接住用户每天产生的学习、工作、项目与思考输入
2. 将这些输入沉淀为可复用的知识、经验、阶段结论与个人状态
3. 在不同场景下持续回答“我现在做到哪了、还差什么、下一步该做什么”
4. 在明确授权下帮助完成少量与当前任务直接相关的辅助动作
5. 让用户能够回看自己的学习轨迹、工作推进过程与阶段性成长变化

---

## 4. 核心价值

当前产品的核心价值，不应表述为“什么都能做”，而应收束为一条明确主线：

> 持续接住用户在不同场景中的输入，将其沉淀为长期可复用的知识与状态，并据此给出贴近个人真实情况的下一步建议。

围绕这条主线，当前产品的核心价值固定为以下四项：

1. `知识沉淀`
   将聊天、学习记录、项目过程、网页内容、阶段总结以及问题处理过程与结果，逐步沉淀为后续可持续使用的个人知识资产，而不是停留在一次性记录或一次性回答。

2. `状态理解`
   基于用户近期输入、历史沉淀、当前任务与学习过程，持续判断用户当前做到哪里、掌握到什么程度、卡在哪里、处于什么阶段，并维护可追踪的个人状态上下文。

3. `下一步建议`
   结合用户当前状态、历史过程与具体场景，给出当前最值得采取的下一步，而不是脱离上下文的泛泛建议。这是产品最重要的高价值输出之一。

4. `有限执行`
   在清晰授权下完成少量与当前目标直接相关的动作，例如整理内容、读写文件、调用受控工具或执行明确任务，用于帮助用户将建议和结论更顺畅地推进到实际行动中。

这四项价值之间的关系明确如下：

1. `知识沉淀` 是基础
2. `状态理解` 是核心中层能力
3. `下一步建议` 是关键高价值输出
4. `有限执行` 是辅助能力，而不是产品主体

## 5. 默认工作流

当前产品第一优先级不是单项能力清单，而是一条能够在不同场景中持续成立的默认工作流。

当前默认工作流定义如下：

1. 用户告诉助手自己当前在做什么、学什么、看什么，或遇到了什么问题
2. 助手记录当前目标、当前主题、已有背景、当前进度与已知阻塞
3. 用户在过程中持续补充材料、网页内容、笔记、结果、问题或阶段总结
4. 助手将这些输入沉淀为知识更新、状态更新、经验记录或阶段结论
5. 用户可以随时追问“现在做到哪了、还差什么、下一步做什么”
6. 助手结合当前状态与历史沉淀给出贴近实际情况的建议
7. 在明确授权下，助手完成少量与当前目标直接相关的辅助动作
8. 一轮结束后，助手保留交接信息、阶段结论、后续建议，以及可回看的知识与状态记录

这条工作流构成当前产品的主线。

该主线既适用于工程开发，也适用于通用学习、浏览学习、CTF / 安全分析等场景；变化的是具体内容与所用 Skill，主线本身不变。

凡是不能自然接入这条主线的新增能力，当前版本均不应优先投入。

---

## 6. 当前主线能力

当前版本优先围绕以下主线能力展开：

1. `输入接入`
   持续接住用户在不同场景中的输入，包括自然语言描述、学习笔记、网页内容、工作区结果、阶段总结、问题记录，以及用户主动补充的线下学习内容整理结果。

2. `知识沉淀`
   将输入逐步整理为可复用、可回看、可关联的长期知识资产，包括主题知识、阶段结论、经验记录、常见问题、个人偏好与教训沉淀。

3. `状态维护`
   持续维护用户当前的目标、主题、阶段、进度、阻塞、近期结果与掌握情况，形成可追踪的个人状态上下文，而不是仅保留零散对话记录。

4. `下一步建议`
   结合当前状态、历史沉淀与具体场景，识别当前缺口、判断优先级，并给出贴近用户真实情况的下一步动作建议。

5. `有限执行`
   在明确授权下完成少量与当前目标直接相关的辅助动作，例如整理结果、读写文件、调用受控工具或执行明确任务，用于帮助用户更顺畅地推进当前工作。

以上能力构成当前产品的通用主干。

工程开发、通用学习、CTF / 安全分析等具体场景，可通过 Skill、MCP 或其他受控扩展能力增强体验；但这些能力属于主干上的场景增强层，不单独构成当前版本的产品主线定义。

## 7. 明确后置能力

以下能力保留设想，但当前不进入第一优先级：

1. `全电脑高权限自动操作`
2. `浏览器高复杂度自动化控制`
3. `QQ / 微信等即时通讯深接入`
4. `手机远程下发指令并直接控制电脑`
5. `全局行为监控`
6. `多智能体编排`
7. `自动 fallback`
8. `重型记忆评分系统`
9. `以平台化扩张为目标的大范围能力拼装`

这些能力并非被否定，而是明确后置。

后置原因主要包括以下四类：

1. `风险高`
   高权限自动化、远程控制、全局行为监控等能力天然带来更高的安全、授权与误操作风险。

2. `落地复杂`
   多智能体编排、重型记忆评分与复杂自动化链路会显著提高系统复杂度与资源消耗。

3. `容易稀释主线`
   这些能力容易将产品从“知识沉淀、状态理解、下一步建议、有限执行”的主线带偏到平台化、全能化或高自动化方向。

4. `不决定当前产品是否成立`
   在当前阶段，只要知识沉淀、状态理解、下一步建议与有限执行尚未稳定成立，就不应优先投入这些后置能力。

同时需要明确：

1. 浏览学习场景中的内容接入、理解、总结与知识沉淀，仍属于当前主线范围
2. 工程开发、通用学习、CTF / 安全分析等场景能力，仍可通过 Skill 等方式增强
3. 当前后置的是高复杂度自动化、平台化扩张与高风险控制能力，而不是所有场景能力本身

## 8. 产品边界

### 8.1 当前应该做的

1. 持续接住用户在学习、工作、项目与思考中的输入，并沉淀为可复用的知识资产
2. 持续维护用户当前目标、主题、进度、阻塞、阶段与掌握情况
3. 稳定回答“做到哪、还差什么、下一步做什么”
4. 在不同场景下复用同一主线能力，包括工程开发、通用学习、浏览学习、CTF / 安全分析等
5. 通过 Skill、MCP 或其他受控扩展能力增强特定场景下的使用体验
6. 在授权边界内完成少量与当前目标直接相关的辅助动作
7. 让知识更新、状态变化、阶段结论、验证结果与交接信息都有留痕，并支持后续回看

### 8.2 当前不应该做的

1. 将产品重新定义为“全能操作系统助手”
2. 将当前主线重新切回“纯研发副驾”或“单一领域工具”
3. 提前将产品改写为“多智能体平台”或“能力拼装平台”
4. 为了远程控制、高权限自动化或工具广度而牺牲当前知识沉淀主线
5. 将浏览学习场景扩张为高复杂度浏览器自动化控制
6. 将场景增强能力误写为产品本体，导致主定义失焦
7. 为了接入更多外部项目或能力而无边界扩张系统复杂度

### 8.3 当前边界判断原则

新增能力只有在满足以下条件时，才应优先进入当前版本：

1. 能自然接入“输入接入 -> 知识沉淀 -> 状态理解 -> 下一步建议 -> 有限执行”这条主线
2. 能提升用户在真实场景中的长期使用价值
3. 不明显增加高风险自动化、平台复杂度或资源消耗

如果一个新增能力主要强化的是平台化、全能化、高权限自动化或多智能体协作，而不是强化知识沉淀型个人助手的主线价值，则当前不应优先投入。

## 9. 当前与 Codex 的对标方式

当前不应以“工具数量、接入服务数量或自动化覆盖面”作为与 Codex 对标的主要标准。

更合理的对标方式应分为两层：

第一层是通用运行能力，重点看：

1. `上下文壳是否稳定`
2. `权限边界是否清晰`
3. `工具调用合同是否统一`
4. `验证与收口是否真实`
5. `长任务过程是否可追踪、可交接、可恢复`

第二层是当前产品更重要的差异化能力，重点看：

1. `是否能持续沉淀用户的长期知识资产`
2. `是否能维护并理解用户当前状态与阶段变化`
3. `是否能在不同场景下给出贴近个人真实情况的下一步建议`
4. `是否能将一次次输入转化为长期可复用的个人成长上下文`

在这一意义上，当前项目可以借鉴 Codex 在任务执行、上下文管理、权限控制与验证收口方面的优点，但不应将自身重新定义为“本地版 Codex”。

一句话：

> Codex 更偏执行型、任务型协作助手；当前项目的目标则是一个以知识沉淀、状态理解、下一步建议为主干，并辅以有限执行能力的本地个人助手。

因此，当前与 Codex 的关系更适合定义为：

1. `在工程协作场景中，可参考其优点并承担部分替代价值`
2. `在产品本体上，不以执行能力覆盖面作为核心判断标准`
3. `在长期价值上，应比 Codex 更强调个人知识沉淀与成长支持`

## 10. 第一版最小成功标准

第一版最小成功标准定义如下：

1. 用户可以持续将学习、工作、项目与思考输入交给系统，而不是只能进行一次性问答
2. 系统能够将这些输入沉淀为可追溯、可回看、可继续使用的知识与阶段记录
3. 系统能够稳定回答“现在做到哪了、还差什么、下一步做什么”，并体现对用户当前状态的持续理解
4. 系统能够在明确授权下完成少量与当前目标直接相关的辅助动作
5. 系统能够为多轮任务或长期学习过程保留交接信息、阶段结论与后续建议
6. 用户能够在连续使用过程中感受到知识积累、状态维护与建议质量的持续增强
7. 上述主线能力能够在至少两类真实场景中成立，例如工程开发与通用学习，或学习场景与 CTF / 安全分析场景

只要以上标准尚未稳定成立，就不应宣称产品已经真正落地。

第一版是否成立，不以多智能体、浏览器自动化、自动 fallback、重型记忆评分或大范围平台化能力为判断标准，而以“知识沉淀、状态理解、下一步建议、有限执行”这条主线是否稳定成立为判断标准。

## 11. 当前非目标

以下内容不属于本文当前拍板范围，也不作为当前版本成立与否的判断重点：

1. `QQ / 微信等即时通讯入口的具体接入方案`
2. `手机远程下发指令与远程控制的具体实现细节`
3. `全局行为监控的采集范围、策略与实现方案`
4. `浏览器高权限自动化与复杂控制链路`
5. `多智能体编排或平台化扩张方案`
6. `自动 fallback 的机制设计`
7. `重型记忆评分体系的设计与实现`
8. `Skill、MCP 与各类场景增强能力的具体文件结构、协议与工程组织方式`
9. `各专项场景下的完整自动化流程细节，例如高风险安全操作链路`
10. `为支持上述内容而进行的大范围架构重写`

上述内容并非永久否定，而是明确不在当前产品定义文档中展开，也不作为当前版本的主线交付目标。

本文当前只负责稳定以下口径：

1. 产品的核心定位
2. 产品的核心价值
3. 默认工作流与主线能力
4. 当前边界、后置项与最小成功标准

## 12. 当前待商议项

以下内容暂不在本文中定死，但会直接影响后续路线规划、开发顺序与验收重点：

1. `第一版优先深做哪些场景`
   当前已明确可支持工程开发、通用学习、浏览学习、CTF / 安全分析等场景，但第一版应优先在哪两类场景中做深，仍需后续收口。

2. `哪些输入渠道应优先接入`
   例如自然语言输入、网页内容输入、工作区结果、学习笔记、人工补充的线下学习总结等，哪些应优先形成稳定入口，仍需进一步明确。

3. `状态维护应收在什么粒度`
   当前已明确需要维护目标、阶段、进度、阻塞与掌握情况，但第一版的状态粒度应收在多细，仍需后续明确。

4. `下一步建议的“够用标准”如何定义`
   当前已明确下一步建议是高价值输出，但在第一版中，建议达到什么程度才算真正可用，仍需进一步判断。

5. `场景增强能力应如何分层接入`
   工程开发、通用学习、CTF / 安全分析等场景中的 Skill、MCP 与其他增强能力，后续应如何分层接入而不破坏主线，仍需进一步设计。

6. `自用版本与未来公开版本的边界如何划分`
   当前已明确产品先以自用有效为优先，但未来公开版本应保留哪些主干能力、裁剪哪些个性化或高复杂度能力，仍需后续明确。

7. `长期知识资产中应优先沉淀哪些内容`
   例如主题知识、阶段结论、方法经验、错误记录、个人偏好、学习轨迹等，哪些应作为第一优先级长期资产，仍需进一步取舍。

---

## 13. 当前结论

当前阶段的产品方向正式固定为：

1. `知识沉淀型个人助手`
2. `以知识沉淀、状态理解、下一步建议、有限执行为主线`
3. `以用户本人长期使用与长期成长为核心服务对象`
4. `以工程开发、通用学习、浏览学习、CTF / 安全分析等作为可支持场景，而不是并列产品定义`
5. `以 Skill、MCP 等能力作为场景增强方式，而不是产品本体`

一句话：

> 当前项目后续不再围绕“做一个什么都能干的助手”或“做一个能力平台”发散，而是围绕“知识沉淀、状态理解、下一步建议、有限执行”这条主线，构建一个可在不同场景中持续使用的本地个人助手。

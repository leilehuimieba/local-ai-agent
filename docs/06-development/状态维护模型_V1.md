# 状态维护模型 V1

## 1. 文档目的与作用

本文用于在《知识沉淀型个人助手产品定义_V1》《真实用户场景与任务脚本_V1》《第一版场景优先级与范围_V1》与《知识资产模型_V1》的基础上，进一步明确第一版应持续维护哪些状态、这些状态分别承载什么价值，以及哪些内容当前不作为重点状态维护对象。

本文关注的不是底层状态存储实现，也不是复杂状态机工程方案，而是产品层面的状态维护定义问题，重点回答以下问题：

1. 在第一版中，哪些内容应被视为需要持续维护的当前状态
2. 这些状态分别服务于哪些场景与主线能力
3. 哪些内容属于状态信息，哪些内容属于知识资产、过程材料或建议结果
4. 第一版应优先维护哪些状态类型，暂不深做哪些状态类型
5. 后续建议能力、验收标准与开发拆解应围绕哪些状态类型展开

本文作为“状态理解”主线的核心建模文档，直接服务于以下后续工作：

1. 下一步建议能力定义与质量标准制定
2. 有限执行与权限边界收口
3. 输入渠道设计与结果留痕设计
4. 验收标准与评估用例设计
5. 第一版开发任务拆解

本文不展开以下内容：

1. 数据库存储结构与底层同步机制
2. 复杂状态图、复杂调度流转或多智能体状态协作
3. 重型用户画像、复杂能力评分与复杂情绪推断体系
4. 具体事件合同、检索优化与工程实现细节

本文的作用，是把“状态理解”从抽象方向进一步收口为“第一版到底维护什么状态、优先维护什么、哪些先不重做”的明确建模基础。

## 2. 状态维护模型的总体原则

第一版状态维护模型应围绕“知识沉淀、状态理解、下一步建议、有限执行”这条产品主线展开，不追求一次性覆盖所有用户变化，而优先维护最能反映当前阶段、最能支持下一步建议、最适合在高频场景中稳定更新的状态类型。

第一版状态维护模型应遵循以下原则：

1. `当前有效优先`
   只有能够直接反映用户当前阶段、当前推进位置与当前理解状态的内容，才应优先进入状态维护模型，而不是把所有历史信息都当作当前状态。

2. `直接支撑建议优先`
   优先维护那些能够直接影响“现在做到哪了、还差什么、下一步做什么”的状态，而不是先维护边缘型、展示型或低频使用的状态。

3. `高频场景优先`
   优先服务工程开发场景与通用学习场景中的高频状态维护需求，再逐步兼容浏览学习与 CTF / 安全学习等差异化场景。

4. `轻量清晰优先`
   第一版优先采用边界清晰、含义明确、便于更新的状态类型，不引入复杂评分、复杂推断或重型画像机制。

5. `可更新优先于静态归档`
   状态的价值在于能够随着用户输入、阶段结果与环境变化持续更新，而不是被保存为一次性结论后不再变化。

6. `与知识资产适度分离`
   状态维护主要关注当前目标、当前阶段、当前进度、当前阻塞、当前掌握情况与最近结果；长期可复用的结论、经验与主题内容，原则上优先归入知识资产模型，而不是全部混入状态维护模型。

基于上述原则，第一版不以“维护得更细”为目标，而以“维护得更有用、更稳定、更能支撑建议与阶段收口”为目标。

## 3. 第一优先级状态类型

结合第一版产品主线、场景优先级与状态维护模型原则，第一版建议优先维护以下八类状态。

### 3.1 当前场景

`当前场景` 指用户当前主要处于哪类使用场景，例如工程开发、通用学习、浏览学习或 CTF / 安全学习。

优先原因：

1. 它决定系统对当前输入的理解语境
2. 它直接影响建议方式、状态解释方式与结果留痕方式
3. 它能够帮助系统在多场景间保持主线一致但输出有差异

### 3.2 当前主题

`当前主题` 指用户当前正在集中处理或学习的主题对象，例如某个任务主题、学习主题、问题域或分析主题。

优先原因：

1. 它是多轮连续使用中最核心的聚焦点之一
2. 它能连接知识沉淀与当前状态
3. 它能够帮助系统避免对话在多主题之间频繁失焦

### 3.3 当前目标

`当前目标` 指用户在当前阶段最直接希望完成的结果或推进方向。

优先原因：

1. 它是判断“当前是否在推进”的基础
2. 它直接影响下一步建议与有限执行的方向
3. 它对工程开发与通用学习两个第一优先级场景都具有直接价值

### 3.4 当前阶段

`当前阶段` 指用户当前处于该主题或任务的哪个阶段，例如初始理解、信息收集、推进执行、阶段收口、回顾巩固等。

优先原因：

1. 它能够帮助系统判断当前更适合提供解释、推进建议还是收口建议
2. 它比单纯的“完成 / 未完成”更能反映真实过程
3. 它是连接状态理解与建议质量的重要中层状态

### 3.5 当前进度

`当前进度` 指相对于当前目标，已经推进到什么程度，哪些部分已完成，哪些部分仍待推进。

优先原因：

1. 它直接支撑“做到哪了”的回答
2. 它能够帮助系统识别当前的推进缺口
3. 它是阶段收口与后续交接的基础状态之一

### 3.6 当前阻塞

`当前阻塞` 指当前最影响推进的缺口、疑问、条件不足或判断未定点。

优先原因：

1. 它直接支撑“还差什么”的回答
2. 它能够帮助系统避免给出脱离阻塞实际情况的建议
3. 它对工程开发与 CTF / 安全学习等强步骤场景尤其重要

### 3.7 最近结果

`最近结果` 指最近一轮输入、操作、分析或学习之后形成的最新结果、最新发现或最新阶段性产出。

优先原因：

1. 它帮助系统理解“刚刚发生了什么变化”
2. 它是状态更新与阶段判断的重要触发依据
3. 它能够帮助用户快速恢复最近上下文

### 3.8 当前掌握情况

`当前掌握情况` 指用户对当前主题或当前问题的理解程度、稳定程度与待巩固情况。

优先原因：

1. 它是通用学习场景中的关键状态
2. 它能帮助系统区分“已理解、待巩固、未理解”等不同建议方向
3. 它也能在工程开发和 CTF / 安全学习场景中反映用户对当前问题的把握程度

### 3.9 第一优先级状态的共同特点

上述状态类型被列为第一优先级，主要因为它们共同满足以下条件：

1. 能直接支撑“做到哪了、还差什么、下一步做什么”这组核心问题
2. 能直接服务工程开发与通用学习两个第一优先级场景
3. 边界相对清晰，适合第一版轻量维护
4. 能与知识资产形成稳定协同，而不依赖复杂推断体系

## 4. 第二优先级状态类型

除第一优先级状态外，以下状态类型也具有明确价值，但在第一版中更适合作为保留方向或轻量兼容对象，而不作为重点维护对象。

### 4.1 连续轨迹状态

`连续轨迹状态` 指用户在一段时间内围绕同一主题或同一方向的连续推进轨迹，例如持续推进节奏、阶段变化链路与跨轮次演变情况。

保留原因：

1. 它有助于系统理解长期推进趋势
2. 它能够增强回顾与阶段判断能力
3. 它在学习场景与长期任务场景中具有潜在价值

当前不作为第一优先级的原因：

1. 第一版更应优先保证单轮状态和近期状态的稳定维护
2. 长周期轨迹维护容易与知识资产、回顾机制和复杂推荐混合，增加复杂度

### 4.2 用户节奏与习惯状态

`用户节奏与习惯状态` 指用户在长期使用中表现出的推进节奏、信息处理习惯与常用工作方式。

保留原因：

1. 它有助于提升建议方式的贴合度
2. 它能够增强产品对“用户本人”的长期适应性
3. 它在多场景中都可能产生辅助价值

当前不作为第一优先级的原因：

1. 第一版更应优先维护显性、可验证的任务状态与学习状态
2. 节奏与习惯往往需要更长时间积累后才具备稳定价值

### 4.3 薄弱点状态

`薄弱点状态` 指用户当前反复暴露出的知识缺口、常见误判或理解薄弱区。

保留原因：

1. 它能够增强后续建议与回顾提示的针对性
2. 它对通用学习和 CTF / 安全学习场景尤其有价值
3. 它有助于形成更长期的成长支持能力

当前不作为第一优先级的原因：

1. 第一版应先保证基础状态维护成立，再逐步形成对薄弱点的稳定识别
2. 过早引入薄弱点评估容易滑向复杂画像与重型评分体系

### 4.4 外部环境上下文状态

`外部环境上下文状态` 指当前页面、当前资料来源、当前工作区结果或当前分析环境等与场景相关的环境性状态。

保留原因：

1. 它有助于增强浏览学习与工程开发场景中的上下文理解
2. 它能够为建议生成提供更贴近当下环境的依据
3. 它在差异化场景中具有较高实用价值

当前不作为第一优先级的原因：

1. 第一版更应优先维护用户自身任务状态与学习状态
2. 外部环境状态在不同场景中的边界差异较大，当前不宜重做统一体系

### 4.5 第二优先级状态的定位

上述状态类型在第一版中的定位是：

1. 可保留轻量维护能力
2. 可作为后续建议能力与回顾能力的增强来源
3. 不作为第一版核心建模重点
4. 不在当前阶段引入复杂推断、复杂评分或重型画像机制

---

## 5. 不作为第一版重点维护的状态类型

以下状态类型虽然在长期演进中可能具备价值，但当前不作为第一版重点维护对象。

### 5.1 重型能力评估状态

`重型能力评估状态` 指对用户能力水平、成长曲线、阶段评分与多维表现进行复杂刻画的状态体系。

当前不重点维护的原因：

1. 第一版已明确不优先引入重型评分与复杂画像机制
2. 这类状态对数据积累周期与判断稳定性要求较高
3. 过早建设容易让状态理解主线滑向复杂评估系统

### 5.2 自动推断型长期偏好状态

`自动推断型长期偏好状态` 指系统在缺乏充分显式确认时，基于长期行为自动推断出的复杂偏好、决策倾向与风格模型。

当前不重点维护的原因：

1. 第一版更应优先维护显性、低歧义、可验证的状态
2. 自动推断型偏好容易引入误判与解释不清的问题
3. 这类状态更适合作为后续阶段逐步增强方向，而不是第一版核心维护对象

### 5.3 高噪声行为监测状态

`高噪声行为监测状态` 指对大量细碎操作、全局行为、弱相关过程信号进行持续追踪后形成的动态状态集合。

当前不重点维护的原因：

1. 第一版不应把“监测得更多”误当成“理解得更好”
2. 高噪声行为状态容易稀释真正高价值的任务与学习状态
3. 这类建设通常伴随更高的边界、隐私与复杂度问题

### 5.4 情绪与动机推断状态

`情绪与动机推断状态` 指系统基于长期行为或语言线索，对用户情绪、动机、投入度与内在心理变化进行推断后维护的状态。

当前不重点维护的原因：

1. 第一版更应聚焦任务状态、学习状态与推进状态
2. 这类状态边界模糊、歧义较高、验证困难
3. 过早建设容易偏离“知识沉淀型个人助手”的主线收口

### 5.5 当前排除原则

上述状态类型当前不作为重点维护对象，核心原因在于：

1. 对第一版主线的直接支撑不足
2. 价值边界不够稳定，容易引入高歧义与高噪声
3. 容易显著增加复杂度并分散当前状态维护主线的收口能力

这些状态类型并非永久排除，而是在第一版阶段不作为主要建模与实现投入方向。

## 6. 状态之间的关系与边界

为保证第一版状态维护模型可落地、可维护、可与知识资产模型配合使用，当前需要对各类状态之间的关系与边界作出明确区分。

### 6.1 状态信息与知识资产的边界

状态信息与知识资产不是同一层内容。

1. `状态信息`
   主要用于描述用户当前阶段的目标、进度、阻塞、掌握情况与当前任务位置，强调当前轮次与近期过程中的动态判断。

2. `知识资产`
   主要用于沉淀长期可复用的内容、结论、方法、经验与可回看的过程结果，强调后续仍可被参考、复用与继续生长。

当前判断原则是：

1. 如果某项内容的核心价值在于“反映当前阶段与当前推进”，优先归入状态信息
2. 如果某项内容的核心价值在于“后续仍值得复用与回看”，优先归入知识资产
3. 如果某项内容同时具备两种属性，则可由状态维护引用对应知识资产，而不是重复建设两套内容

### 6.2 当前状态与建议结果的边界

当前状态与下一步建议也不是同一层内容。

1. `当前状态`
   用于描述“现在处于什么位置、缺什么、卡在哪里”

2. `建议结果`
   用于回答“基于当前状态与已有沉淀，下一步最值得做什么”

当前判断原则是：

1. 状态维护负责提供判断依据
2. 建议能力负责基于状态与资产生成行动建议
3. 不将“建议本身”直接替代“状态本身”

### 6.3 各类状态之间的边界

当前第一版状态之间的区分可按以下方式理解：

1. `当前场景`
   侧重“用户当前处在哪类使用语境中”

2. `当前主题`
   侧重“用户当前主要围绕什么对象展开输入与推进”

3. `当前目标`
   侧重“用户当前最想达成的结果是什么”

4. `当前阶段`
   侧重“用户当前处于该目标推进或学习过程的哪个阶段”

5. `当前进度`
   侧重“相对于目标已经推进到什么程度”

6. `当前阻塞`
   侧重“当前最影响推进的缺口与未解决问题是什么”

7. `最近结果`
   侧重“最近一轮变化后形成了什么新结果”

8. `当前掌握情况`
   侧重“用户对当前主题或问题的理解与把握程度如何”

这些状态类型之间允许存在引用关系，但当前不要求在第一版中建设复杂状态图或复杂推断链；重点是保证每类状态的含义清晰、用途明确。

### 6.4 当前边界控制原则

第一版状态维护模型的边界控制原则如下：

1. 不将所有输入默认视为需要长期维护的状态
2. 不将知识资产内容全部混入状态维护模型
3. 不将建议结果直接替代状态本身
4. 不为状态之间的关系提前建设复杂推断体系
5. 不因为未来可能有用，就在第一版中提前重建所有长期状态类型

当前状态维护模型应优先保证“有用、清晰、可更新”，而不是追求“更细、更全、更复杂”。

## 7. 第一版状态维护优先级建议

结合产品主线、场景优先级与状态维护模型原则，第一版状态维护优先级建议分为三层。

### 7.1 第一层：重点维护的状态

第一版应优先重点维护以下状态类型：

1. `当前场景`
2. `当前主题`
3. `当前目标`
4. `当前阶段`
5. `当前进度`
6. `当前阻塞`
7. `最近结果`
8. `当前掌握情况`

这类状态优先维护的原因在于：

1. 它们直接支撑工程开发与通用学习两个第一优先级场景
2. 它们能直接支持“做到哪了、还差什么、下一步做什么”这组核心问题
3. 它们具有清晰、稳定、可更新的当前价值
4. 它们适合第一版进行轻量但有效的建模与维护

### 7.2 第二层：轻量保留的状态

第一版可轻量保留，但不作为重点维护对象的状态类型包括：

1. `连续轨迹状态`
2. `用户节奏与习惯状态`
3. `薄弱点状态`
4. `外部环境上下文状态`

这类状态当前的定位是：

1. 可以保留基础维护能力
2. 可以为后续建议能力与回顾能力提供补充参考
3. 当前不投入过多建模资源，不引入复杂推断与复杂评分

### 7.3 第三层：当前不重点维护的状态

第一版当前不重点维护的状态类型包括：

1. `重型能力评估状态`
2. `自动推断型长期偏好状态`
3. `高噪声行为监测状态`
4. `情绪与动机推断状态`

这类状态当前不重点维护的原因在于：

1. 与第一版主线的直接关系不够强
2. 边界不够稳定，容易引入高歧义与高噪声
3. 容易显著增加复杂度并分散当前主线收口能力

### 7.4 当前维护优先级的核心判断

第一版状态维护的核心目标，不是尽可能维护更多状态，而是优先维护：

1. 最能反映当前阶段的状态
2. 最能支撑下一步建议生成的状态
3. 最适合在高频场景中形成稳定更新闭环的状态

因此，第一版状态维护优先追求“维护得更有用”，而不是“维护得更全面”。

## 8. 文档结论与后续衔接

基于《知识沉淀型个人助手产品定义_V1》《真实用户场景与任务脚本_V1》《第一版场景优先级与范围_V1》与《知识资产模型_V1》的上游结论，第一版状态维护模型当前正式收口如下：

1. 第一版优先重点维护 `当前场景`、`当前主题`、`当前目标`、`当前阶段`、`当前进度`、`当前阻塞`、`最近结果` 与 `当前掌握情况`
2. 第一版对 `连续轨迹状态`、`用户节奏与习惯状态`、`薄弱点状态`、`外部环境上下文状态` 采取轻量保留策略
3. 第一版当前不重点维护重型能力评估状态、自动推断型长期偏好状态、高噪声行为监测状态与情绪动机推断状态
4. 第一版状态维护的核心目标，是优先维护最能反映当前阶段、最能支持下一步建议、最适合在高频场景中形成稳定更新闭环的状态

基于上述结论，后续设计与开发应优先围绕以下方向展开：

1. 为第一优先级状态建立清晰、稳定、可更新、可回看的维护方式
2. 在知识资产模型中明确与状态维护的边界及引用关系
3. 在下一步建议能力中优先使用高价值状态作为建议依据
4. 在验收标准中重点验证第一优先级状态是否真正形成连续使用价值

2. 有限执行与权限边界

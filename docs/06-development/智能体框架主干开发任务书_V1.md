## 0. 本地适配强约束

开始实现前，必须先满足以下约束：

1. 后端长期主链路以 `Rust + Go` 为主
2. `Rust` 负责高频运行时核心，`Go` 负责网关、会话、事件桥接与服务编排
3. 不把 `Python` 拉回 Agent 主循环、正式网关、记忆主链路、工具调度主链路
4. 复刻的是 Claude Code 的运行时模式，不是原始语言栈和原始目录结构
5. 本地记忆、长期记忆、知识沉淀、知识库扩展位必须保留
6. 性能目标优先级高于“像参考仓库”

如果与历史实现思路冲突，以 [本地适配架构原则_V1](D:/newwork/本地智能体/docs/02-architecture/本地适配架构原则_V1.md) 为准。

## 1. 任务定位

本任务书只解决一件事：

> 在当前 Go + Rust + React 架构上，完成本地单智能体运行时的主干框架。

这里说的“主干框架”不是：

1. 做更多前端页面
2. 做多智能体编排
3. 做浏览器自动化界面
4. 做研究型复杂 heuristics
5. 复刻旧项目的超大 Python loop

这里说的“主干框架”是：

1. 有稳定的 Agent 主循环
2. 有统一工具协议
3. 有上下文封装层
4. 有权限与风险闸门
5. 有记忆与知识路由
6. 有事件、审计、产物回流

---

## 2. 当前阶段目标

当前阶段的唯一目标是：

> 让系统从“已有运行时骨架”升级为“有正式运行时主干的本地智能体系统”。

做到这一步后，系统应具备：

1. 稳定的任务推进主线
2. 稳定的工具调度主线
3. 稳定的上下文输入与结果回填主线
4. 稳定的风险确认与事件回流主线

## 3. 核心设计原则

### 3.1 总原则

1. 不逐文件复刻 Claude Code。
2. 不直接迁移旧项目 Python `loop.py`。
3. 复刻的是“运行时骨架”和“能力组织方式”。
4. 现有 Go / Rust / 前端分层不推翻。
5. 运行时主干优先，枝叶能力后置。

### 3.2 Claude Code 应学习的点

当前必须吸收的不是 UI，而是以下运行时能力：

1. Query Engine / Agent Loop
2. Tool Protocol / Tool Registry
3. Context Assembly / Context Compression
4. Permission / Approval Gate
5. Structured Memory
6. Artifact Externalization
7. Event / Audit / Transcript

### 3.3 claw-code 应学习的点

重点学习它的基础设施分层：

1. query engine
2. session store
3. execution registry
4. transcript / context
5. permissions
6. tool pool

### 3.4 旧项目应学习的点

旧项目 `D:\newwork\local-agent-control-hub` 中，建议保留的不是实现形态，而是经验：

1. planner 与 execution 分离
2. session / event / memory 有正式落点
3. tool registry 独立存在
4. security broker / policy profile 有边界意识
5. worker 能力与主运行时解耦

## 4. 当前必须完成的 7 个主干模块

### 4.1 `runtime-query-engine`

职责：

1. 接收一次任务运行请求
2. 驱动最小主循环
3. 在各阶段调用 planner、tool registry、risk gate、memory router
4. 生成阶段事件和最终结果

本阶段要求的主循环固定为：

1. `Analyze`
2. `Plan`
3. `Execute`
4. `Observe`
5. `Verify`
6. `Finish`

### 4.2 `runtime-tool-registry`

职责：

1. 统一定义工具协议
2. 工具注册与发现
3. 根据当前模式和策略过滤可见工具
4. 统一工具执行结果结构

工具不再被视为零散函数，而是能力对象。

### 4.3 `runtime-context-envelope`

职责：

1. 把任务输入、repo context、workspace、memory digest、tool preview 组装成统一上下文
2. 对大结果做外置和摘要化
3. 控制模型可见上下文的密度

### 4.4 `runtime-permission-gate`

职责：

1. 接入运行模式
2. 拦截高风险动作
3. 生成确认请求
4. 处理确认结果并回到主循环

### 4.5 `runtime-memory-router`

职责：

1. 按需读取记忆
2. 按需写入记忆
3. 生成 memory digest
4. 避免全量记忆注入

### 4.6 `runtime-artifact-store`

职责：

1. 把大输出、日志、摘要、工具结果外置到磁盘
2. 返回 artifact path 给运行时和前端
3. 避免上下文和事件被大文本淹没

### 4.7 `runtime-event-audit`

职责：

1. 统一运行时阶段事件
2. 统一工具调用事件
3. 统一错误与风险事件
4. 统一最终结果与摘要回流

## 5. 当前明确不做

本阶段禁止进入以下内容：

1. 多智能体编排
2. 复杂 worker 并发系统
3. 全量 LSP 深度能力
4. 浏览器自动化主链路
5. 桌面自动化主链路
6. 研究型记忆评分体系
7. 复杂自动 fallback 策略

一句话：

> 当前先把单智能体运行时主干做稳，不做高级枝叶。

## 7. 模块落点与文件级任务

### 7.1 Query Engine 主干

目标：

把 `lib.rs` 从当前“可运行入口”提升为“正式主循环调度入口”。

文件落点：

1. `crates/runtime-core/src/lib.rs`
2. `crates/runtime-core/src/planner.rs`
3. `crates/runtime-core/src/execution.rs`
4. `crates/runtime-core/src/session.rs`

任务：

1. 明确 `RunRequest -> Analyze -> Plan -> Execute -> Observe -> Verify -> Finish` 主线。
2. 每一阶段输出统一 metadata。
3. 阶段间传递统一的运行态对象，而不是零散临时字段。
4. 失败和等待确认都必须回到统一主线中。

### 7.2 Tool Protocol / Tool Registry

目标：

建立运行时内统一工具协议。

建议新增或重构落点：

1. `crates/runtime-core/src/execution.rs`
2. `crates/runtime-core/src/contracts.rs`
3. 如有必要，可新增 `crates/runtime-core/src/tools.rs`

任务：

1. 定义工具对象的最小字段：
   - `tool_name`
   - `display_name`
   - `category`
   - `risk_level`
   - `input_schema`
   - `output_kind`
   - `requires_confirmation`
2. 统一工具调用结果结构：
   - `summary`
   - `detail_preview`
   - `artifact_path`
   - `error_code`
   - `retryable`
3. 执行前可进行工具可见性过滤。

### 7.3 Context Envelope

目标：

把上下文组装从零散 metadata 升级为统一上下文封装。

建议新增或重构落点：

1. `crates/runtime-core/src/repo_context.rs`
2. `crates/runtime-core/src/memory.rs`
3. `crates/runtime-core/src/knowledge.rs`
4. 如有必要，可新增 `crates/runtime-core/src/context.rs`

任务：

1. 统一组装：
   - 用户输入
   - 当前模式
   - 当前工作区
   - repo context
   - memory digest
   - 相关 knowledge 摘要
   - 最近工具结果摘要
2. 对超长结果进行 preview 化。
3. 给 planner 和 execution 提供统一上下文入口。

### 7.4 Permission / Risk Gate

目标：

把风险控制从局部判断升级为运行时内建能力。

文件落点：

1. `crates/runtime-core/src/risk.rs`
2. `crates/runtime-core/src/contracts.rs`
3. `gateway/internal/state/confirmation_store.go`
4. `gateway/internal/api/chat.go`

任务：

1. 明确模式到权限策略的映射。
2. 对高风险工具调用生成统一确认请求。
3. 确认结果能恢复主循环继续执行。
4. 事件流能明确显示“为何等待确认”。

### 7.5 Memory / Knowledge Router

目标：

记忆与知识进入主链路，但保持按需调用。

文件落点：

1. `crates/runtime-core/src/memory.rs`
2. `crates/runtime-core/src/knowledge.rs`
3. `crates/runtime-core/src/planner.rs`

任务：

1. 先读“最相关摘要”，不读全量。
2. 本轮关键结果可写入结构化记忆。
3. planner 可感知 memory / knowledge 是否可用。
4. 当前阶段只做轻量路由，不做复杂评分系统。

### 7.6 Artifact / Storage

目标：

大结果外置，主链路只保留摘要和引用。

文件落点：

1. `crates/runtime-core/src/storage.rs`
2. `crates/runtime-core/src/paths.rs`
3. `crates/runtime-core/src/events.rs`

任务：

1. 建立运行产物目录约定。
2. 工具结果过大时写入 artifact。
3. 事件只回流：
   - 摘要
   - 预览
   - artifact path

### 7.7 Event / Audit / Contracts

目标：

统一三端理解的事件合同。

文件落点：

1. `crates/runtime-core/src/events.rs`
2. `crates/runtime-core/src/contracts.rs`
3. `gateway/internal/contracts/contracts.go`
4. `gateway/internal/runtime/client.go`
5. `gateway/internal/session/bus.go`
6. `frontend/src/shared/contracts.ts`

任务：

1. 明确阶段事件类型。
2. 明确工具调用事件类型。
3. 明确风险确认事件类型。
4. 明确最终结果事件类型。
5. 所有事件尽量依靠稳定字段和 metadata，而不是临时字符串拼接。

## 8. 复刻与重写边界

### 8.1 可以直接借思路

1. `claw-code` 的 query engine / permissions / transcript / session store 分层
2. 旧项目的 planner / event bus / tool registry / memory / session 结构意识
3. Claude Code 的工具协议、上下文压缩、artifact 外置、权限前置

### 8.2 不能直接照搬

1. 旧项目 Python 超大 loop
2. 旧项目实验性 heuristics
3. 与当前 Go + Rust 边界冲突的设计
4. 把 UI 逻辑塞回运行时

### 8.3 必须重写

1. Rust 主循环
2. Go <-> Rust 正式合同桥接
3. 工具协议落点
4. Context Envelope
5. Artifact 外置主线

## 9. 推荐实现顺序

### Phase A：主循环打通

先做：

1. Query Engine 主线
2. Planner / Execution 串接
3. 基础阶段事件

通过标准：

1. 一次运行能稳定走完主线
2. 失败与完成都有统一收口

### Phase B：工具协议与风险闸门

再做：

1. Tool Protocol
2. Tool Registry
3. Permission / Risk Gate

通过标准：

1. 工具调用有统一结构
2. 高风险动作可稳定确认

### Phase C：上下文、记忆、产物

最后做：

1. Context Envelope
2. Memory / Knowledge Router
3. Artifact Store
4. 事件与合同补齐

通过标准：

1. 上下文可控
2. 大结果不压爆主链路
3. 前端和日志能消费关键结果

## 10. 当前验收标准

本任务书对应的验收目标不是“功能很多”，而是“主干成立”。

### 10.1 必须通过

1. Rust 运行时存在清晰主循环。
2. 工具调用不再是零散执行，而有统一协议。
3. 高风险动作进入统一确认主线。
4. 记忆和知识进入主链路，但保持按需读写。
5. 大结果可外置为 artifact。
6. 运行事件能稳定回流 Go 和前端。
7. 现有三页前端不需要新增页面也能消费主干结果。

### 10.2 不通过情形

1. 继续把逻辑堆回单一大文件。
2. 为了快而把 Go / Rust 边界打乱。
3. 工具结果仍然没有统一结构。
4. 风险确认依旧是旁路逻辑。
5. 上下文继续无边界膨胀。
6. 大输出仍直接塞进事件和模型上下文。

## 11. 给实现型 AI 的明确要求

1. 不推翻现有 Go + Rust + React 分层。
2. 优先在 `runtime-core` 现有模块上继续收口。
3. 必要时允许新增少量 Rust 模块，但必须说明职责。
4. 不直接迁移旧项目 Python 大文件。
5. 不把多代理、浏览器自动化等高级能力提前带入当前阶段。
6. 完成后必须补一份对应验收记录文档。

一句话要求：

> 当前任务不是做更多能力，而是把单智能体运行时主干真正立起来。

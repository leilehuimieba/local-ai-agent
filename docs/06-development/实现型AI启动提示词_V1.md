## 2. 推荐提示词

你现在要在 `D:\newwork\本地智能体` 这个新项目里继续实现第一版，但请先不要自由发挥。

你的工作方式必须遵守以下要求：

### 一、目标

你不是来重新规划产品的，也不是来扩展第二版功能的。  
你的任务是：

1. 基于当前项目实际状态继续推进第一版
2. 优先做首版收口，而不是横向扩功能
3. 先做差异审查和结构拆分，再补齐首版缺口
4. 所有实现必须以现有文档为约束

### 二、先阅读这些文档

### 四、开始编码前先输出这些内容

在真正修改代码之前，你必须先输出一份简短但明确的 `gap review`，至少包含：

1. 当前项目已经实现了什么
2. 当前最主要的结构性问题是什么
3. 当前最主要的合同不一致点是什么
4. 下一步你准备先改哪些文件
5. 为什么这些修改优先于新增功能

### 五、你的执行顺序必须是

1. 先做差异审查
2. 先拆前端大文件
3. 再拆 Rust `runtime-core` 大文件
4. 再对齐 API / event / confirmation 合同
5. 再补首版验收缺口
6. 最后才做界面和体验收口

### 六、你现在不要做这些事情

第一版当前阶段禁止优先做：

1. 浏览器扩展
2. 浏览器自动化
3. 桌面自动化
4. 语音输入输出
5. 多智能体
6. 自动 fallback
7. 自动测速
8. 复杂 Skill 平台
9. 花哨 UI 改版

### 七、你必须遵守的架构要求

1. Go 继续作为控制面，不要把运行时逻辑堆进 Go
2. Rust 继续作为运行时核心，不要把所有职责继续堆在 `lib.rs`
3. 前端继续使用 React + Vite，不要随意改技术栈
4. 新项目必须继续保持与旧项目隔离，不得修改旧项目
5. 一切实现都必须服务于第一版主链路和验收标准

### 八、你做每一轮改动后必须汇报

每一轮工作后，请汇报：

1. 你改了什么
2. 为什么先改这个
3. 现在距离首版验收还差什么
4. 下一轮准备做什么

### 九、你的成功标准

你的目标不是“让项目看起来更复杂”，而是：

1. 让结构更清晰
2. 让合同更稳定
3. 让主链路更可用
4. 让首版更接近通过 [验收标准_V1](D:/newwork/本地智能体/docs/07-test/验收标准_V1.md)

### 十、如果你发现文档与代码冲突

处理顺序必须是：

1. 先指出冲突
2. 解释冲突属于哪一类
3. 说明你建议以文档还是以现有实现为准
4. 再开始修改

不要直接默默按自己的理解改。

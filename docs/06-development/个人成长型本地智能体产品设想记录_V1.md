## 1. 当前产品定位补充

当前产品不应只定义为“本地单智能体工作台”。

更准确的定位应是：

1. `一个面向长期学习、长期成长、长期积累的本地个人智能体平台`
2. `一个以本地知识库、个性化规划、专业技能接入、长期记忆为核心的个人成长型智能体`
3. `一个可以长期陪伴用户学习、实践、复盘、沉淀经验的本地 AI 系统`

核心价值不是单次聊天，而是：

1. 用户知识持续沉淀
2. 用户目标持续规划
3. 用户专业能力持续接入
4. 用户偏好、习惯、失败教训持续积累

---

## 2. 当前明确优势方向

### 2.1 本地知识库

目标不是“做一个普通 RAG”。

目标是：

1. 在用户浏览网页、学习软件、阅读文档时，帮助提炼重点
2. 将学习记录、总结、理解结果持续写入本地知识库
3. 让智能体后续回答不只依赖联网模型，而是结合本地知识资产
4. 让知识库成为“和用户一起成长”的长期资产

### 2.2 个性化规划能力

规划能力不是网上模板拼接，而是围绕用户本人动态调整。

重点场景包括：

1. 学习规划
2. 考研规划
3. 阶段目标变更后的重新排程
4. 根据用户学习习惯、投入时长、执行情况做提醒和修正

### 2.3 专业能力接入

产品应预留“可专精”的方向，而不是所有能力都做成泛能力。

首个明确方向：

1. `网络安全 / CTF / 渗透相关 skill`

后续能力形式：

1. 工具调用
2. 脚本执行
3. 题型分析
4. 技能推荐
5. 解题复盘
6. 专业知识沉淀

### 2.4 记忆与灵魂

智能体不仅要有结构化记忆，还要具备长期陪伴感。

可预留的正式记忆载体包括：

1. `soul.md`
2. `user.md`
3. 失败教训沉淀
4. 用户偏好沉淀
5. 长期知识总结

目标不是做“拟人表演”，而是让智能体具备：

1. 连续人格
2. 长期关系
3. 稳定语言风格
4. 对用户成长轨迹的理解

## 3. 双智能体协作设想

该设想当前记录为第二阶段或更后阶段规划输入，不进入首版必做范围。

### 3.1 角色划分

1. `执行智能体`
   负责对话域内的任务推进、工具执行、结果回流
2. `全局语言智能体`
   负责全局观察、语音理解、意图总结、任务分发、跨界面协调

### 3.2 全局语言智能体职责

1. 看到用户与执行智能体的对话内容
2. 在未来条件满足时，观察浏览器或桌面层行为
3. 接收用户语音或自然表达
4. 将自然表达总结成更精准的任务意图
5. 向执行智能体派发任务
6. 汇总执行结果并以文字或语音回报

### 3.3 潜在价值

1. 降低打字负担
2. 提高表达容错率
3. 强化陪伴感和人格感
4. 形成“全局理解 + 局部执行”的协作结构

### 3.4 当前风险

1. Token 消耗会明显升高
2. 架构复杂度会提高
3. 全局监听和跨界面观察会带来更高安全风险
4. 语音链路、浏览器观察链路、全局调度链路都需要单独收口

当前结论：

1. `值得预留架构接口`
2. `不适合立即重做当前主链路`

## 4. 本地小模型兜底设想

### 4.1 定位

本地小模型不是为了替代云端大模型的深度思考能力。

它的目标是：

1. 防止系统在无 API、无联网、模型失效时完全瘫痪
2. 承担简单调度、简单执行、简单总结
3. 作为低成本、短上下文、强执行的兜底模型

### 4.2 适合承担的能力

1. 查看和总结最新日志
2. 切换模型 / 切换 provider / 修改本地配置
3. 按规则写入知识库或记忆库
4. 执行明确、短链路、低歧义的本地操作
5. 调用已经写好的 skill 或脚本

### 4.3 不应强调的能力

1. 长链路复杂规划
2. 高质量深度推理
3. 长上下文持续思考
4. 复杂浏览器理解
5. 重型多工具自主探索

### 4.4 当前架构启示

后续应该把模型层拆成两层：

1. `强思考模型`
2. `兜底执行模型`

## 5. Token 优化方向

后续优化不应只停留在“减少提示词长度”。

重点应是：

1. 本地缓存稳定上下文
2. 本地缓存高频知识
3. 本地缓存工具结果摘要
4. 按需召回，而不是全量注入
5. 稳定规则、偏好、教训结构化
6. skill / script 优先替代长文本推理
7. 小模型承担简单执行，大模型只负责高价值思考

目标是：

1. 降低重复推理
2. 降低重复检索
3. 降低重复解释
4. 把 Token 消耗集中在真正值得花的地方

---

## 6. 平台化方向

长期目标不是只做一个聊天页。

长期目标应是：

1. 用一个本地平台统一管理用户的个人智能体能力
2. 统一管理 skill、知识、记忆、模型、工具和策略
3. 允许后续继续开发新的专精智能体或专精技能
4. 让当前智能体具备“帮助开发其他智能体技能”的能力

平台的核心价值：

1. 不同 AI 不再彼此割裂
2. 用户不必在不同平台重复操作
3. 本地可以形成统一入口、统一调度、统一记忆

---

## 7. 风险问题

后续需要专项评估：

1. 联网状态下的 Prompt Injection 防护
2. 浏览器内容进入本地智能体前的清洗策略
3. 本地 AI 在联网情况下的攻击面
4. 全局观察能力带来的权限与隐私边界
5. 语音监听 / 桌面观察 / 浏览器观察的授权模式

---

## 8. 第二阶段建议预留的架构接口

当前建议只预留，不急于全部实现：

1. `skill registry`
2. `global observer interface`
3. `voice input / output adapter`
4. `local fallback model adapter`
5. `memory soul layer`
6. `browser selection handoff interface`
7. `global task dispatch interface`

---

## 9. 当前结论

这轮新增想法的核心不是“继续堆功能”，而是把产品方向明确成：

1. `长期个人成长型智能体`
2. `可专精、可进化、可接入技能的本地平台`
3. `在线强思考 + 离线不瘫痪 + 本地长期沉淀`

当前阶段建议：

1. 先继续把主链路和产品化收口做稳
2. 架构上为 skill、全局语言智能体、本地兜底模型预留位置
3. 不在当前阶段直接跳到重型双智能体、语音、桌面全局观察

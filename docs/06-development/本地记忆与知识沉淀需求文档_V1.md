## 1. 文档目的

本文只解决一件事：

> 明确本地智能体的记忆系统和知识沉淀系统该存什么、何时写、如何取、落到哪里。

后续实现不允许再把“聊天历史”“项目记忆”“知识库”“日志”“工具输出”混成一个概念。

---

## 2. 当前正式结论

本项目的本地信息体系固定分为 3 层：

1. `短期工作记忆`
2. `长期记忆`
3. `本地知识库`

三层的角色不同：

1. 短期工作记忆服务当前任务推进
2. 长期记忆服务跨任务复用
3. 本地知识库服务稳定知识沉淀与检索

一句话：

> 记忆是让智能体“记住如何继续做事”，知识库是让智能体“记住哪些内容值得长期复用”。

---

## 3. 为什么必须单独做这套系统

我们的项目是本地智能体，不是普通聊天助手。

因此必须满足：

1. 能连续推进长任务
2. 能跨天复用经验
3. 能把稳定结论沉淀成本地知识资产
4. 能按需召回，而不是每轮全量注入
5. 能在低内存、低 token 浪费前提下长期运行

---

## 4. 三层体系定义

### 4.1 短期工作记忆

定义：

当前会话、当前任务、当前运行周期内必须保留的最小工作状态。

典型内容：

1. 当前目标
2. 当前计划
3. 最近关键动作
4. 最近关键观察
5. 当前阻塞点
6. 当前未完成子任务
7. 最近一次工具结果摘要
8. 当前确认等待状态

特点：

1. 时效短
2. 更新频繁
3. 必须轻量
4. 主要服务当前任务主循环

### 4.2 长期记忆

定义：

跨任务、跨会话仍然有复用价值的结构化经验和偏好。

典型内容：

1. 用户稳定偏好
2. 项目长期规则
3. 常见失败教训
4. 多次任务验证过的经验
5. 工作区高层摘要
6. 阶段性项目状态

特点：

1. 时效中长
2. 不频繁改写
3. 必须结构化
4. 只能按需召回

### 4.3 本地知识库

定义：

经过整理、归纳、去噪后的稳定知识集合，作为本地知识资产长期保存。

典型内容：

1. 项目架构知识
2. 模块职责说明
3. 环境排障知识
4. 命令与工具经验
5. 领域规则总结
6. 文档摘要与索引
7. 已验证的实现模式

特点：

1. 时效长
2. 需要去重和版本意识
3. 允许被检索系统使用
4. 允许后续扩展到数据库或索引系统

---

## 5. 哪些内容写记忆，哪些内容写知识库

### 5.1 写入短期工作记忆

满足以下任一条件即可写入短期工作记忆：

1. 对当前任务下一步有直接影响
2. 是当前循环刚产生的关键结论
3. 是当前用户刚确认的限制条件
4. 是当前工具执行后的高价值摘要
5. 是当前失败或阻塞原因

### 5.2 写入长期记忆

满足以下条件时才写长期记忆：

1. 跨任务仍可能复用
2. 不是一次性中间过程
3. 已经过至少一次明确验证或稳定确认
4. 可以结构化表达

优先写入的长期记忆类型：

1. `user_preference`
2. `project_rule`
3. `lesson_learned`
4. `workspace_summary`
5. `workflow_preference`

### 5.3 写入本地知识库

只有满足下面全部条件，才进入知识库：

1. 内容足够稳定
2. 内容对后续任务有较高复用价值
3. 内容已经经过总结和去噪
4. 内容不是单纯聊天片段
5. 内容不是可快速重新获取的临时结果

优先进入知识库的内容：

1. 项目结构说明
2. 模块边界说明
3. 环境安装与排障结论
4. 高价值实现方案总结
5. 经过验证的命令与流程模板
6. 用户或项目明确确认的长期规则

---

## 6. 明确不写入的内容

以下内容默认不进入长期记忆和知识库：

1. 代码全文
2. 大段原始日志
3. 全量终端输出
4. 一次性 grep 结果
5. 一次性调试过程
6. 很快失效的临时推理
7. 没有验证过的猜测
8. 纯情绪化表达

这些内容允许：

1. 留在会话记录
2. 写入 artifact
3. 作为日志保留

但不应污染长期沉淀层。

## 7. 记忆类型模型

第一版建议固定以下类型。

### 7.1 短期工作记忆类型

1. `current_goal`
2. `current_plan`
3. `open_issue`
4. `recent_observation`
5. `recent_tool_result`
6. `pending_confirmation`

### 7.2 长期记忆类型

1. `user_preference`
2. `project_rule`
3. `workspace_summary`
4. `lesson_learned`
5. `decision_record`
6. `workflow_pattern`

### 7.3 知识库类型

1. `architecture_note`
2. `module_knowledge`
3. `environment_guide`
4. `debug_guide`
5. `command_pattern`
6. `document_digest`
7. `domain_knowledge`

---

## 8. 触发写入规则

### 8.1 短期工作记忆触发

以下时机应自动评估是否写入短期工作记忆：

1. `Analyze` 结束后
2. `Plan` 生成后
3. `Execute` 完成关键动作后
4. `Observe` 得到关键发现后
5. `Verify` 发现阻塞或确认需求后

### 8.2 长期记忆触发

以下时机应自动评估是否写入长期记忆：

1. 任务完成后
2. 用户明确纠正并确认长期偏好后
3. 同类问题重复出现并总结出稳定规则后
4. 失败复盘得到可复用教训后
5. 工作区高层信息发生重要变化后

### 8.3 知识库触发

以下时机应评估是否沉淀到知识库：

1. 形成稳定项目知识后
2. 文档或代码分析产出高质量摘要后
3. 排障过程形成可复用解决方案后
4. 某类实现模式已经验证可用后
5. 用户明确要求“沉淀为知识”时

---

## 9. 召回规则

### 9.1 总原则

记忆与知识一律按需召回，不允许全量加载。

### 9.2 短期工作记忆召回

默认随当前运行一起加载，但只加载当前任务所需最小块。

### 9.3 长期记忆召回

召回依据至少包括：

1. 当前任务类型
2. 当前工作区
3. 当前模块或目录
4. 当前用户偏好类型
5. 当前错误或目标特征

### 9.4 知识库召回

知识库召回应先查摘要，再按需展开正文。

要求：

1. 先返回标题、摘要、来源
2. 再按需返回详细内容
3. 优先返回高相关、近期有效、已验证条目

---

## 10. 存储落点建议

### 10.1 第一阶段建议

第一阶段先采用轻量本地存储：

1. `短期工作记忆`：会话文件或会话数据库表
2. `长期记忆`：结构化 `JSONL` 或 `SQLite`
3. `知识库`：`SQLite + 文本文件索引`

原因：

1. 实现简单
2. 资源占用低
3. 易于本地调试
4. 方便后续迁移

### 10.2 第二阶段建议

当知识量增长后，再增加：

1. 倒排索引
2. 标签过滤
3. 简单 embedding 检索
4. 去重与版本管理

### 10.3 第三阶段建议

后续如果知识体量明显增大，再引入：

1. 独立知识索引层
2. 更强检索排序
3. 知识版本快照
4. 工作区级知识分区

---

## 11. 存储结构建议

第一版建议至少保留以下逻辑结构：

1. `sessions`
2. `working_memory`
3. `long_term_memory`
4. `knowledge_base`
5. `artifacts`

建议字段方向：

1. `id`
2. `workspace_id`
3. `session_id`
4. `memory_type` 或 `knowledge_type`
5. `title`
6. `summary`
7. `content`
8. `tags`
9. `source`
10. `confidence`
11. `verified`
12. `created_at`
13. `updated_at`
14. `expires_at`

## 12. 写入流程要求

后续实现必须把写入流程拆成 4 步：

1. `候选产生`
2. `结构化整理`
3. `去重与分类`
4. `正式写入`

### 12.1 候选产生

来源可以是：

1. 当前运行时阶段结果
2. 工具执行结果摘要
3. 用户明确确认的信息
4. 任务完成后的复盘摘要

### 12.2 结构化整理

必须至少提炼：

1. 标题
2. 类型
3. 摘要
4. 来源
5. 适用范围

### 12.3 去重与分类

必须避免：

1. 同一结论反复写入
2. 同一问题的低质量重复摘要
3. 把工作记忆误写进知识库

### 12.4 正式写入

写入后必须可追踪：

1. 何时写入
2. 从哪次任务写入
3. 写入的是哪一层

---

## 13. 与运行时主链路的关系

### 13.1 运行时使用顺序

后续主链路建议固定为：

1. 先装配短期工作记忆
2. 再按需召回长期记忆
3. 再按需检索知识库
4. 最后构建 memory digest / knowledge digest

### 13.2 Finish 阶段要求

`Finish` 阶段必须额外完成：

1. 评估是否产生新的长期记忆
2. 评估是否产生新的知识沉淀候选
3. 输出写入结果事件或跳过原因

---

## 14. 与日志、artifact、会话记录的边界

### 14.1 日志

日志用于过程追踪，不等于记忆。

### 14.2 Artifact

artifact 用于保存大文本、大输出、详细原始结果，不等于知识库。

### 14.3 会话记录

会话记录用于复盘，不等于长期记忆。

边界要求：

1. 原始内容先归日志或 artifact
2. 有价值结论再抽成记忆或知识
3. 不允许用“保存聊天全文”代替知识沉淀

---

## 15. 第一版验收要求

### 15.1 必须做到

1. 能区分短期工作记忆、长期记忆、知识库三层
2. 能按类型写入长期记忆
3. 能按任务相关性召回长期记忆
4. 能把高价值总结写入知识沉淀层
5. 能避免全量加载所有记忆
6. 能避免把日志和工具原文直接写进知识库

### 15.2 不通过情形

1. 仍把所有内容塞进一个统一大记忆池
2. 仍把聊天历史直接当知识库
3. 没有明确写入触发规则
4. 没有知识沉淀层，只做会话记忆
5. 每次任务都全量加载长期记忆

---

## 16. 第二版增强要求

在第一版通过后，再进入第二版增强：

1. 增加知识去重
2. 增加知识版本管理
3. 增加工作区级知识分区
4. 增加记忆质量评分
5. 增加更强检索排序

---

## 17. 给后续代码 AI 的实现约束

1. 不允许把记忆系统只做成普通聊天历史存档
2. 不允许跳过知识沉淀层
3. 不允许先做复杂算法而没有基础三层结构
4. 不允许让记忆系统变成高内存常驻大对象
5. 不允许先上重型数据库再验证基础链路

一句话要求：

> 先把三层结构、写入边界、召回规则、知识沉淀主链路做对，再谈高级检索和评分。

## 1. 任务定位

本阶段只做三件事：

> 提升主链路成功率、收口默认使用路径、增强记忆与知识召回质量。

本任务书不是新能力扩张计划。
本任务书要求实现型 AI 直接围绕当前首版主链路做稳定性和完成度收口。

---

## 2. 当前阶段目标

本阶段必须完成以下目标：

1. 自然语言任务更稳定地命中 `project_answer / context_answer / explain` 正确路径。
2. `project_answer` 与 `context_answer` 在单次失败后具备受控恢复能力。
3. 系统默认入口更接近真实使用路径，优先把用户带入独立对话页。
4. 工作台退回概览层，不再承担大段主线程内容。
5. 记忆与知识召回结果从“只有摘要”提升为“摘要 + 依据 + 来源解释”。
6. 建立最小真实验收样本，后续优化必须可复跑、可对照、可留证。

---

## 3. 本阶段硬约束

1. 不新增一级页面。
2. 不新增多智能体主链路。
3. 不新增浏览器自动化或桌面自动化深链路。
4. 不把当前工作区、日志、设置页重新推翻重做。
5. 不引入新的重型存储方案或新的知识库类型。
6. 不把“成功率优化”扩写成新一代运行时重构工程。
7. 不把“记忆质量优化”扩写成复杂评分平台或大规模算法工程。
8. 所有改动都必须能明确归类到：
   1. `成功率优化`
   2. `默认路径优化`
   3. `记忆与知识质量优化`
   4. `最小验收基线`

---

## 4. 当前阶段实现原则

### 4.1 成功率优先

1. 先提升任务命中正确路径的概率，再谈体验细节。
2. 先补轻量受控恢复，再谈复杂自修复。
3. 验证链不只负责收口，也要参与判定是否进入恢复。

### 4.2 默认路径优先

1. 首次进入系统时，优先引导用户直接进入对话主链路。
2. 工作台只保留任务概览和调查入口，不再重复承载聊天线程。
3. 空状态、失败态必须给出下一步，而不是只展示状态名。

### 4.3 召回质量优先

1. 记忆与知识必须继续按需召回，不允许全量注入。
2. 召回结果必须逐步可解释。
3. 必须继续压制低价值 runtime 内容污染长期记忆和知识层。
4. 必须继续维持 `README / 06-development 优先、07-test 降权` 的知识排序口径。

### 4.4 验收先行

1. 所有优化都必须可以通过固定样本复跑。
2. 不允许只靠“感觉更好了”作为完成依据。
3. 本阶段结束时必须形成真实样本集和固定留证方式。

---

## 5. 推荐实现顺序

1. 收紧运行时意图路由。
2. 补 `project_answer / context_answer` 的单次受控恢复。
3. 调整默认入口与工作台结构。
4. 把召回依据接入事件、日志和前端详情区。
5. 建立固定样本集与最小验收脚本。

## 7. 文件级任务清单

### 7.1 运行时意图路由

1. 收紧 `planner.rs` 中自然语言路由规则。
2. 显式区分：
   1. 项目说明类任务
   2. 延续当前任务类任务
   3. 能力说明类任务
3. 降低本应进入 `project_answer` 却落到 `Explain` 的概率。
4. 降低本应延续会话却误判为纯说明任务的概率。

### 7.2 单次受控恢复

1. 在 `execution.rs` 中给 `project_answer` 增加一次保底恢复。
2. 在 `execution.rs` 中给 `context_answer` 增加一次保底恢复。
3. 恢复只允许单次，不允许循环重试。
4. 恢复链路必须在事件与日志中可见。
5. 恢复应优先基于现有 `repo_context / session_summary / docs 摘要` 完成，不新增重型外部依赖。

### 7.3 默认路径优化

1. `App.tsx` 默认视图切到 `agent`。
2. `WorkbenchOverview.tsx` 收口为概览页，只保留：
   1. 当前任务状态
   2. 进入对话页
   3. 进入日志页
3. `ChatPanel.tsx` 空状态升级为黄金路径空态。
4. 失败态输出“错误分类 + 下一步建议”，而不是只有通用失败提示。

### 7.4 记忆与知识召回解释

1. `memory_recall.rs` 输出召回摘要与至少一条召回依据。
2. `knowledge.rs / knowledge_store.rs` 输出知识摘要与来源依据。
3. 事件与日志中显式带出：
   1. 来源路径或标题
   2. 来源类型
   3. 命中理由
   4. 命中顺序或优先级线索
4. `EventTimeline.tsx / LogsPanel.tsx / ContextSidebar.tsx` 必须直接展示这些正式字段。

### 7.5 最小验收样本

1. 建立固定样本集。
2. 样本至少覆盖：
   1. 项目说明问答
   2. 会话延续问答
   3. 文件读取任务
   4. 命令执行任务
   5. 风险确认任务
   6. 记忆召回可见性
   7. 知识召回优先级
3. 每条样本都必须定义明确通过条件。
4. 每条样本都必须能留出真实运行证据。

## 8. 分工作包要求

### 8.1 工作包 A：主链路成功率收口

完成标准：

1. 自然语言路由误判下降。
2. `project_answer / context_answer` 出现模型或执行异常时，存在单次受控恢复。
3. 验证结果能参与恢复决策。
4. 恢复链路不会破坏原有完成判定逻辑。

直接判不通过：

1. 引入循环恢复。
2. 引入明显新的误路由。
3. 主链路事件结构被改坏。

### 8.2 工作包 B：默认路径收口

完成标准：

1. 默认进入独立对话页。
2. 工作台明显退回概览层。
3. 对话页空态能把用户带入真实任务。
4. 错误态不再泛泛而谈。

直接判不通过：

1. 首页、日志页、设置页可用性被破坏。
2. 工作台退化为无信息空壳。
3. 对话页空态仍无法引导真实使用。

### 8.3 工作包 C：记忆与知识质量收口

完成标准：

1. 召回结果不再只有摘要。
2. 前端详情区能看到来源依据。
3. 低价值 runtime 内容污染进一步下降。
4. 现有排序治理口径不回退。

直接判不通过：

1. 召回结果解释仍完全缺失。
2. 记忆与知识边界再次混乱。
3. README / 06-development 优先级被破坏。

### 8.4 工作包 D：最小验收基线

完成标准：

1. 有固定样本。
2. 有固定通过条件。
3. 有固定留证格式。
4. 能重复复跑。

直接判不通过：

1. 没有形成固定样本集。
2. 验收仍靠人工口头描述。

---

## 9. 验收标准

做到以下程度，才算本阶段完成：

1. 自然语言主链路更稳定地落到正确任务类型。
2. `project_answer / context_answer` 出现异常时有受控恢复。
3. 默认入口已经切到对话主线程。
4. 工作台只保留概览与调查入口。
5. 记忆与知识召回结果已具备摘要与依据。
6. 日志页与事件流详情区可直接查看召回解释。
7. 已形成固定样本与留证方式。

以下情况直接判不通过：

1. 误路由明显增加。
2. 恢复逻辑造成主链路混乱。
3. 默认路径优化破坏现有页面结构。
4. 召回解释仍停留在 metadata 黑盒。
5. 没有建立固定样本基线。

当前最值得肯定的地方：

1. 文档与代码方向基本一致
2. React / Go / Rust 三层都已起步
3. 聊天、事件、确认、日志、记忆、知识这些核心概念都已具备雏形

### 2.4 当前最大的风险

当前最大的风险不是“功能太少”，而是“原型代码开始集中堆积”。

重点风险：

1. 前端顶层文件承担过多职责
2. Rust `runtime-core` 开始承担过多职责
3. 代码正在接近“能跑，但越长越重”的边缘
4. 如果继续直接加功能，首版会很快失去清晰边界

工程结论：

> 当前阶段最重要的不是继续加很多新功能，而是对现有原型做一次工程化收口。

## 3. 下一阶段总目标

下一阶段的总目标不是做“第二版能力扩展”，而是：

> 把当前已经跑起来的首版纵向原型，收口成一个可按文档验收的、结构清晰的、边界稳定的第一版候选系统。

这一阶段要完成三件大事：

1. 收口结构
2. 对齐合同
3. 对照验收标准补齐主链路缺口

---

## 4. 下一阶段的核心原则

### 4.1 不继续横向扩功能

下一阶段不应优先新增：

1. 浏览器自动化
2. 桌面自动化
3. 语音能力
4. 多智能体
5. 自动 fallback
6. 复杂 Skill 平台

### 4.2 先拆重，再补缺

如果不先拆掉当前集中式实现，后续补功能只会让系统越来越重。

### 4.3 先对齐合同，再追求体验

当前前端、Go、Rust 虽已打通，但必须先检查实现是否真正遵守合同文档。

### 4.4 一切围绕首版验收

新增工作必须能回答：

1. 它是否让首版更接近通过验收
2. 它是否只是看起来更强，但并不提高首版通过率

---

## 5. 下一阶段执行重点

### 5.1 重点一：做当前状态与文档的差异审查

目标：

1. 查当前代码与文档是否一致
2. 找出哪些地方偏离了文档
3. 找出哪些首版能力缺口还没补齐

输出物：

1. `gap review` 清单
2. 按严重程度排序的整改项

建议优先审查：

1. 前端是否只围绕聊天、日志、设置
2. Go API 是否与合同文档一致
3. Rust 运行时对象与状态机是否已偏离设计草案
4. 模式、确认、事件、日志、记忆字段是否一致

通过标准：

1. 能列出明确差异清单
2. 能判断哪些差异必须先改

---

### 5.2 重点二：拆分前端顶层结构

目标：

把当前偏集中式的前端顶层逻辑拆成模块化结构。

建议拆分方向：

1. 页面壳层
2. 聊天控制
3. 事件订阅
4. 设置读取与变更
5. 工作区切换
6. 模型切换
7. 确认流展示
8. 日志数据展示

说明：

1. 下一阶段不要求前端做复杂重构，但必须避免继续堆到顶层文件。

通过标准：

1. 前端主文件明显瘦身
2. 核心逻辑进入独立模块
3. 页面职责更加单一

---

### 5.3 重点三：拆分 Rust runtime-core

目标：

把当前 Rust 运行时从“集中式原型实现”推进到“首版可维护运行时结构”。

建议优先拆分：

1. 状态机与主循环
2. 风险判断
3. 文件与命令执行
4. 记忆读写
5. 知识检索
6. 事件构造
7. 输出摘要

说明：

1. 不要求一次拆成最终理想 crate 结构。
2. 但必须先把明显多职责的大文件拆开。

通过标准：

1. `runtime-core` 顶层不再承载所有职责
2. 核心功能能按模块被理解和替换
3. 行为不因拆分而丢失主链路能力

---

### 5.4 重点四：对齐 API、事件、确认合同

目标：

让前端、Go、Rust 的实现与合同文档真正一致。

重点检查：

1. `chat/run`
2. `chat/confirm`
3. `events/stream`
4. `settings`
5. `logs`
6. `RunRequest`
7. `RunResult`
8. `RunEvent`
9. `ConfirmationRequest`

说明：

1. 现在已经有合同文档，下一阶段要把实现跟合同逐条对齐。
2. 如果发现合同文档不合理，应先修文档，再修实现。

通过标准：

1. 关键对象字段语义稳定
2. 前端无需猜后端字段含义
3. 确认流和事件流可稳定使用

---

### 5.5 重点五：对照验收标准补首版缺口

目标：

逐项检查首版验收标准中哪些项还没真正稳定实现。

建议优先补的点：

1. 模式切换是否真正影响行为
2. 高危动作确认是否完整
3. 新目录首次接触提示是否稳定
4. 会话连续与压缩是否真正生效
5. 日志页是否真正可用于复盘
6. 错误提示是否统一和清楚

通过标准：

1. 核心验收项从“有雏形”变成“可稳定演示”
2. 至少能稳定跑通首版核心场景

---

## 7. 下一阶段的推荐执行顺序

建议严格按以下顺序推进：

1. `当前代码 vs 文档` 差异审查
2. 前端顶层拆分

## 11. 下一阶段的验收标准

下一阶段不要求“首版最终通过”，但至少要满足以下阶段目标：

### 11.1 结构验收

1. 前端和 Rust 不再继续向单一大文件堆逻辑
2. 关键模块边界开始明确

### 11.2 合同验收

1. 前端、Go、Rust 对关键对象的理解一致
2. 事件、确认、结果对象可稳定使用

### 11.3 主链路验收

1. 聊天任务可发起
2. 过程事件可见
3. 文件/命令类任务能工作
4. 高危动作能确认

### 11.4 记忆与知识验收

1. 会话上下文连续有效
2. 重要信息能写入
3. 基础知识检索可用

### 11.5 风险与模式验收

1. 模式切换真实影响行为
2. 观察模式下不会随便动手
3. 高危动作不会绕过确认

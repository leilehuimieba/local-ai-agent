## 1. 文档定位

本文件不是愿景文档，也不是仅供阅读的架构介绍。

本文件的定位是：

> 给另一个实现对话提供可直接执行的工程规范，要求其在不突破现有产品边界的前提下，按统一技术路线完成运行时主干补强。

本文件重点解决四类问题：

1. 每个优先能力到底该用什么技术与方法落地
2. 落地时应遵守哪些工程边界
3. 每一批次完成后如何验收
4. V2 与 V3 分别应达到什么交付口径

## 3. 当前工程约束

实现对话必须默认以下约束已经生效。

### 3.1 技术栈约束

当前正式主链路技术栈固定为：

1. Frontend：TypeScript + React + Vite
2. Gateway：Go
3. Runtime：Rust

本约束不是“当前建议”，而是“正式实现要求”。

后续实现对话必须默认：

1. 前端正式重构继续落在 React 现有 V2 形态上
2. 控制面正式实现只能落在 Go
3. 运行时、执行层、模型接入层、上下文层、工具层正式实现只能落在 Rust

不允许在本轮实现中引入新的主技术栈，例如：

1. 不引入 Python 回到正式主链路
2. 不引入 Node 作为第二后端
3. 不引入额外桌面主框架改写现有入口
4. 不允许把 Python 做成“临时 runtime 过渡层”
5. 不允许把 Python worker 保留为首版正式依赖

技术目标补充：

本项目后端路线之所以固定为 Go + Rust，不只是为了语言偏好，而是为了服务以下正式目标：

1. 更低内存占用
2. 更高并发承载能力
3. 更快冷启动和响应速度
4. 更低运行时抖动
5. 更适合长期本地常驻

因此，实现对话在所有设计判断中，都必须优先考虑：

1. 少常驻进程
2. 少冗余中间层
3. 少大对象常驻内存
4. 少重复上下文构造
5. 少非必要解释器依赖

### 3.2 架构边界约束

必须保持：

1. Frontend 只面向控制面
2. Go 只做控制面和桥接
3. Rust 只做运行时与执行主循环

明确禁止：

1. Frontend 直接调用 Rust runtime
2. Go 吞入 planner / verifier 主逻辑
3. Rust 直接承载 HTTP 路由和前端状态管理

### 3.3 产品边界约束

当前只围绕单智能体 V2 主形态推进。

前端形态约束补充：

1. 首页、日志、设置三视图继续作为正式 V2 形态
2. 首页继续采用你当前已确定的 `Top Bar + Main Chat + Context Sidebar + Bottom Panel` 结构
3. 前端本轮重构目标是收口、清晰、可读，不是继续扩展更多视图和面板
4. 任何后端字段设计都不应破坏当前前端 V2 形态

这意味着：

1. 后端新增数据应优先通过 metadata、summary、artifact reference 回流
2. 不应为了后端实现方便，把前端重新变成运维控制台式结构
3. 不应引入新的一级导航或复杂产品对象模型

明确不进入本轮范围：

1. 多智能体编排界面
2. 浏览器自动化主链路
3. 桌面自动化主链路
4. 语音
5. 大量研究型评分器
6. 新增复杂产品视图

### 3.4 合同约束

实现中若改动合同，必须同步至少三处：

1. `crates/runtime-core/src/contracts.rs`
2. `gateway/internal/contracts/contracts.go`
3. `frontend/src/shared/contracts.ts`

任何一处不同步都视为未完成。

## 4. 参考实现吸收原则

参考仓库只用于吸收方法，不用于直接复制实现。

这里要特别强调：

> 我们学习的是思路、边界和架构模式，不是技术手段复用，更不是语言级迁移。

### 4.1 应吸收的部分

应重点吸收：

1. Agent Runtime 的分层方式
2. 上下文装配方式
3. 工具协议设计思想
4. 压缩与记忆的系统化处理方式
5. 验证与结果外置思想

### 4.2 不应直接继承的部分

不应直接继承：

1. 具体 prompt 文案
2. 任意仓库中的全部工具集合
3. 未与本项目边界对齐的多代理流程
4. 研究型试验机制
5. Python 版 runtime 的具体实现手法
6. Python 解释器时代常见的多进程拼接式组织方式
7. 任何不利于低内存、低延迟、高并发目标的临时桥接层

### 4.2.1 对旧 Python 项目的正式处理原则

旧项目 `D:\newwork\local-agent-control-hub` 的作用仅限于：

1. 提供边界经验
2. 提供模块职责参考
3. 提供可吸收的系统概念

旧项目不提供：

1. 正式技术路线
2. 正式实现方式
3. 正式运行时基础设施

因此，实现对话必须遵守：

1. 不得把旧 Python 代码“翻译式迁移”为 Rust
2. 不得把旧 Python 的流程细节视为必须保留的行为合同
3. 不得为了省事保留 Python 组件作为正式后端组成部分
4. 只能重新设计成适合 Go + Rust 的实现

### 4.3 当前规划引用结论

基于当前公开参考资料与现有项目判断，以下工程结论可以直接采用：

1. 上下文必须分层构造
2. 工具必须协议化
3. 长输出必须外置，不能持续灌回主上下文
4. 验证应是主链路步骤，不是后置补丁
5. 模型适配层应晚于上下文和工具主干落地

#### M3：Tool Registry 与 Tool Contract

## 6. 工程通用要求

以下要求对所有里程碑都生效。

### 6.1 代码组织要求

1. 优先新增独立模块，不继续向 `lib.rs` 堆积主逻辑
2. 单个新增模块必须有明确单一职责
3. 每个模块都应有明确输入、输出、失败语义
4. 不允许把多个能力混在一个大文件中一次性硬编码

### 6.1.1 面向性能的组织要求

由于本项目明确追求低内存占用、高并发和快速响应，实现对话必须额外遵守：

1. 优先使用静态结构和有限对象层级
2. 避免引入重型依赖和复杂运行时框架
3. 避免让一次 run 长时间持有大块原始文本
4. 避免复制大对象多份驻留内存
5. 优先把大结果转为摘要 + 引用，而不是长文本常驻
6. 避免为了抽象而引入过深调用链和过多包装对象

### 6.2 数据结构要求

1. 所有主链路对象优先使用结构化对象，而不是字符串拼接协议
2. 事件 metadata 只放摘要与引用，不放超长原文
3. 所有枚举状态必须有限可控
4. 所有对象应优先支持序列化与日志记录

### 6.3 错误处理要求

1. 所有新模块都要定义失败时的降级行为
2. 不允许因 repo context 获取失败直接让整个 run 崩溃
3. 不允许因 artifact 写盘失败就让主链路完全中断
4. 所有错误都必须能回流到 `RunEvent` 或 `RunResult`

### 6.4 性能与资源要求

1. 不引入重型后台常驻索引进程作为首选方案
2. 不在每轮都全量扫描全仓库
3. 不把大输出原样保存在会话摘要中
4. 尽量复用已有 `storage.rs`、`paths.rs`、`text.rs` 基础设施
5. 不引入 Python runtime、Python worker 或嵌入式解释器作为正式依赖
6. 不通过增加额外桥接服务来“模拟”多语言协作
7. 优先使用 Go 处理控制面并发，Rust 处理执行与上下文主干
8. 所有新增能力都应回答其对内存占用和响应延迟的影响

### 6.5 实现文档要求

每完成一个里程碑，实现对话必须同步提交：

1. 修改清单
2. 合同变更说明
3. 手工测试清单
4. 未完成项
5. 风险项

## 7. M1 规范：实时仓库上下文加载

### 7.1 目标定义

M1 的目标不是“做完整索引器”，而是建立任务开始前的仓库运行态快照。

### 7.2 必须使用的技术与方法

推荐技术：

1. Rust 标准库文件系统 API
2. `std::process::Command` 调用 Git
3. 现有 `paths.rs` 与 `text.rs`

推荐方法：

1. 快照式采集
2. 小成本读取
3. 失败降级
4. 摘要优先

不推荐方法：

1. 全仓库递归深度扫描
2. 一上来做复杂 AST 或 LSP 集成
3. 每轮都读大量文件原文

### 7.3 必须产出的结构

M1 至少应定义：

1. `RepoContextSnapshot`
2. `GitSnapshot`
3. `WorkspaceDocSummary`
4. `RepoContextLoadResult`

建议字段：

#### `RepoContextSnapshot`

1. `workspace_root`
2. `repo_root`
3. `git_available`
4. `git`
5. `docs`
6. `warnings`

#### `GitSnapshot`

1. `current_branch`
2. `default_branch`
3. `is_dirty`
4. `recent_commits`

#### `WorkspaceDocSummary`

1. `path`
2. `kind`
3. `summary`
4. `exists`

### 7.4 推荐实现步骤

1. 先判断工作区是否位于 Git 仓库内
2. 再采集 Git 基本状态
3. 再查找项目说明文件
4. 再对说明文件做轻量摘要
5. 最后把所有结果装配成 `RepoContextSnapshot`

### 7.5 文件落点要求

必须新增：

1. `crates/runtime-core/src/repo_context.rs`

必须修改：

1. `crates/runtime-core/src/lib.rs`
2. `crates/runtime-core/src/planner.rs`
3. `crates/runtime-core/src/contracts.rs`

如前端展示需要，可修改：

1. `frontend/src/workspace/ContextSidebar.tsx`
2. `frontend/src/shared/contracts.ts`

### 7.6 验收条件

M1 通过的最低条件：

1. 每次运行前都能形成 repo context
2. Git 不可用时能降级
3. 当前分支与 dirty 状态至少能进入事件 metadata
4. planner 入口能拿到 repo context 摘要

M1 不通过情形：

1. 获取失败就中断 run
2. 快照结构不稳定
3. 事件里只有原始字符串没有结构化信息

### 7.7 交付物要求

实现对话完成 M1 后，必须补一份文档：

建议文档名：

`docs/07-test/里程碑验收与首版候选汇总_V1.md`

文档至少包含：

1. 数据结构说明
2. 触发样例
3. 成功案例
4. 降级案例
5. 已知限制

## 8. M2 规范：Prompt 分层与上下文装配

### 8.1 目标定义

M2 的目标是建立运行时统一上下文入口，而不是立即接入复杂模型编排。

### 8.2 必须使用的技术与方法

推荐技术：

1. Rust 结构体组合
2. 独立上下文装配模块
3. 轻量字符串渲染函数

推荐方法：

1. 分层上下文
2. 结构先于文本
3. 先形成 `ContextEnvelope`，再渲染 prompt

不推荐方法：

1. 在 `planner.rs` 中到处拼接上下文
2. 每个模块自己临时追加 prompt 文本
3. 直接把所有信息做成一个超长字符串对象

### 8.3 必须产出的结构

M2 至少应定义：

1. `RuntimeContextEnvelope`
2. `StaticPromptBlock`
3. `ProjectPromptBlock`
4. `DynamicPromptBlock`
5. `PromptRenderResult`

### 8.4 推荐实现步骤

1. 定义上下文 envelope 数据结构
2. 接入 M1 的 repo context
3. 接入 session summary
4. 预留 memory / knowledge recall 槽位
5. 最后渲染成 prompt layers

### 8.5 文件落点要求

必须新增：

1. `crates/runtime-core/src/context_builder.rs`
2. `crates/runtime-core/src/prompt.rs`

必须修改：

1. `crates/runtime-core/src/lib.rs`
2. `crates/runtime-core/src/planner.rs`
3. `crates/runtime-core/src/session.rs`

### 8.6 验收条件

M2 通过的最低条件：

1. runtime 上下文有统一入口
2. planner 改为消费 `RuntimeContextEnvelope`
3. prompt 至少分成三层
4. 后续 recall / compaction 能无缝挂接

M2 不通过情形：

1. 只是新增了几个 helper，但上下文入口仍分散
2. planner 仍大量直接读取各模块
3. prompt 分层只是命名变化，没有结构变化

### 8.7 交付物要求

实现对话完成 M2 后，必须补一份文档：

`docs/07-test/里程碑验收与首版候选汇总_V1.md`

内容至少包含：

1. Envelope 结构
2. Prompt 层定义
3. 一个自然语言任务的上下文装配示例
4. 上下文缺失时的降级逻辑

## 9. M3 规范：Tool Registry / Tool Contract

### 9.1 目标定义

M3 的目标是把“动作分支”提升为“工具协议”。

### 9.2 必须使用的技术与方法

推荐技术：

1. Rust 枚举 + 结构体
2. registry + dispatch 模式
3. 粗粒度工具抽象

推荐方法：

1. 工具先定义合同
2. 再定义风险语义
3. 最后做 dispatch

不推荐方法：

1. 继续在 `planner.rs` 里无限追加前缀分支
2. 继续让 `execution.rs` 直接承载所有工具选择语义
3. 为每个细小动作建一套独立主流程

### 9.3 必须产出的结构

至少应定义：

1. `ToolName`
2. `ToolCall`
3. `ToolResult`
4. `ToolSpec`
5. `ToolRegistry`

### 9.4 必须支持的首批工具

第一批正式工具至少包括：

1. `run_command`
2. `workspace_read`
3. `workspace_write`
4. `workspace_delete`
5. `workspace_list`
6. `memory_write`
7. `memory_recall`
8. `knowledge_search`

### 9.5 风险绑定要求

每个工具至少要能表达：

1. 是否只读
2. 是否修改工作区
3. 是否高风险
4. 是否可能不可逆

### 9.6 文件落点要求

必须新增：

1. `crates/runtime-core/src/tools.rs`
2. `crates/runtime-core/src/tool_registry.rs`

必须修改：

1. `crates/runtime-core/src/planner.rs`
2. `crates/runtime-core/src/execution.rs`
3. `crates/runtime-core/src/risk.rs`
4. `crates/runtime-core/src/contracts.rs`

### 9.7 验收条件

M3 通过的最低条件：

1. planner 输出的是统一 `ToolCall`
2. execution 按 registry dispatch
3. risk 可对 `ToolCall` 做判断
4. 新增工具无需大面积改主循环

M3 不通过情形：

1. 只是重命名，没有实际协议化
2. 新增一个工具仍要改很多主逻辑
3. 风险仍然只能靠字符串规则拼凑

## 10. M4 规范：结构化记忆与输出压缩

### 10.1 目标定义

M4 由两部分构成，但必须一起落地：

1. 结构化记忆 schema
2. 长输出 artifact 外置与 compaction

原因：

1. 两者共同决定上下文是否可持续
2. 只做记忆不做压缩，长任务仍会膨胀
3. 只做压缩不做结构化记忆，长期沉淀仍混乱

### 10.2 必须使用的技术与方法

推荐技术：

1. 结构化 JSON 序列化
2. 本地文件存储或 JSONL
3. 轻量索引而不是重型向量基础设施

推荐方法：

1. Memory Schema 先行
2. Recall 按需触发
3. 长输出写 artifact，主链路只保留摘要和引用

不推荐方法：

1. 每轮把所有记忆重新载入上下文
2. 每个工具都把完整原文写入 session summary
3. 为首版引入外部数据库和复杂 embedding 系统

### 10.3 记忆 schema 要求

至少支持以下 kind：

1. `preference`
2. `project_knowledge`
3. `lesson_learned`
4. `daily_note`
5. `task_outcome`

每条记忆至少包含：

1. `id`
2. `kind`
3. `summary`
4. `content`
5. `workspace_scope`
6. `source`
7. `created_at`

### 10.4 压缩与 artifact 要求

每个大输出至少支持：

1. 摘要
2. 原始内容引用
3. 内容类型
4. 产生时间

建议 artifact 类型：

1. `command_output`
2. `file_snapshot`
3. `knowledge_result`
4. `verification_report`

### 10.5 文件落点要求

必须新增：

1. `crates/runtime-core/src/memory_schema.rs`
2. `crates/runtime-core/src/memory_recall.rs`
3. `crates/runtime-core/src/artifacts.rs`
4. `crates/runtime-core/src/compaction.rs`

必须修改：

1. `crates/runtime-core/src/memory.rs`
2. `crates/runtime-core/src/session.rs`
3. `crates/runtime-core/src/storage.rs`
4. `crates/runtime-core/src/execution.rs`
5. `crates/runtime-core/src/events.rs`
6. `crates/runtime-core/src/text.rs`

如需详情入口，可新增：

1. `gateway/internal/api/artifacts.go`

### 10.6 验收条件

M4 通过的最低条件：

1. 长输出不会直接污染会话摘要
2. 记忆区分类型与作用域
3. recall 是按需而不是全量
4. 事件与日志中能看到 artifact 摘要和引用

M4 不通过情形：

1. 只是加了字符串标签，没有真正 schema
2. 只是截断输出，没有原始内容引用
3. 只是换了存储位置，但上下文仍持续膨胀

### 10.7 交付物要求

实现对话完成 M4 后，必须补两份文档：

1. `docs/07-test/里程碑验收与首版候选汇总_V1.md`
2. `docs/07-test/里程碑验收与首版候选汇总_V1.md`

## 11. M5 规范：验证链与完成判定

### 11.1 目标定义

M5 的目标是把 `Verify` 从“阶段名”变成“真实能力”。

### 11.2 必须使用的技术与方法

推荐技术：

1. Rust 枚举表示验证结果
2. 独立 completion 判定模块
3. 结构化 verification report

推荐方法：

1. 先判断动作是否成功
2. 再判断子目标是否完成
3. 再判断任务是否可以收口

不推荐方法：

1. 成功执行工具就默认任务完成
2. 在 `lib.rs` 里继续混写大量完成逻辑
3. 引入研究型 verifier 组合系统

### 11.3 必须产出的结构

至少应定义：

1. `VerificationOutcome`
2. `VerificationNeed`
3. `CompletionDecision`
4. `VerificationReport`

### 11.4 文件落点要求

必须新增：

1. `crates/runtime-core/src/verify.rs`
2. `crates/runtime-core/src/completion.rs`

必须修改：

1. `crates/runtime-core/src/lib.rs`
2. `crates/runtime-core/src/execution.rs`
3. `crates/runtime-core/src/planner.rs`
4. `crates/runtime-core/src/events.rs`

### 11.5 验收条件

M5 通过的最低条件：

1. 能明确区分“动作完成”和“任务完成”
2. `Verify` 阶段产出真实验证结果
3. `Finish` 阶段建立在 `CompletionDecision` 之上
4. 前端可以看到验证相关状态

M5 不通过情形：

1. 只是补了几个事件文案
2. 主循环仍无法解释为何结束
3. 无法说明下一步为什么继续或停止

### 11.6 交付物要求

实现对话完成 M5 后，必须补一份文档：

`docs/07-test/里程碑验收与首版候选汇总_V1.md`

## 12. M6 规范：统一模型适配层

### 12.1 目标定义

M6 的目标是正式建立模型适配层，为后续接本地模型和多模型打基础。

### 12.2 必须使用的技术与方法

推荐技术：

1. Rust trait 或等价抽象表示 adapter
2. 本地 HTTP client
3. OpenAI 兼容接口优先

推荐方法：

1. 先做单 provider 接入
2. 先做非流式稳定调用
3. 再预留流式接口

不推荐方法：

1. 一开始接多个 provider 和复杂路由
2. 让 planner 直接依赖模型实现
3. 把模型错误直接暴露为不可理解的底层异常

### 12.3 必须产出的结构

至少应定义：

1. `ModelAdapter`
2. `ModelRequest`
3. `ModelResponse`
4. `ModelError`
5. `ProviderConfig`

### 12.4 文件落点要求

必须新增：

1. `crates/runtime-core/src/model_adapter.rs`
2. `crates/runtime-core/src/model_client.rs`

必须修改：

1. `crates/runtime-core/src/contracts.rs`
2. `crates/runtime-core/src/lib.rs`
3. `crates/runtime-core/src/prompt.rs`
4. `gateway/internal/config/config.go`
5. `gateway/internal/contracts/contracts.go`
6. `frontend/src/shared/contracts.ts`
7. `frontend/src/settings/useSettings.ts`

### 12.5 验收条件

M6 通过的最低条件：

1. runtime 存在统一模型接入点
2. 配置层、合同层、前端层语义一致
3. 模型不可用时错误能统一回流
4. 后续接本地模型不需重构主循环

M6 不通过情形：

1. 模型调用仍散落在多个模块
2. 配置与合同语义不一致
3. 前端无法稳定展示模型不可用状态

### 12.6 交付物要求

实现对话完成 M6 后，必须补一份文档：

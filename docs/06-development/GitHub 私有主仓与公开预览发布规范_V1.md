/# GitHub 私有主仓与公开预览发布规范_V1

更新时间：2026-04-10
状态：当前有效
适用范围：`本仓库` 与对应 GitHub 仓库发布流程

## 1. 目标与边界

1. 目标：主代码仓库保持私有，外部通过公开预览仓查看最新前端效果。
2. 目标：通过版本标签触发自动构建，并同步发布到公开仓库指定分支。
3. 边界：公开仓库仅用于承载构建产物，不接收业务源代码开发。
4. 边界：完整后端联调与敏感日志排查仍在私有主仓完成。

## 2. 仓库角色定义

1. 私有主仓：
   - 用途：开发、评审、测试、缺陷修复。
   - 可见性：`Private`。
   - 唯一可信源码来源。
2. 公开发布仓：
   - 用途：托管发布产物与在线预览页面。
   - 可见性：`Public`。
   - 默认不直接手工提交，优先由 CI 自动写入。

## 3. 分支与触发策略

1. 私有主仓默认开发分支：`main`。
2. 发布目标分支默认：`gh-pages`。
3. 触发方式：
   - 自动触发：`push tag` 且标签名匹配 `v*`（示例：`v1.0.0`）。
   - 手动触发：`workflow_dispatch`，可临时覆盖目标仓库参数。
4. 当前默认不采用“每次 push main 自动发布”，避免频繁覆盖线上预览。

## 4. 权限与凭据规范

1. 统一使用 `DEPLOY_TOKEN` 作为发布令牌（GitHub Actions Secret）。
2. `DEPLOY_TOKEN` 权限最小化：
   - 仅授权目标公开仓。
   - 仅授予 `Contents: Read and write`。
3. 严禁把 PAT 明文写入代码、文档截图、日志。
4. 令牌轮换建议：至少每 90 天轮换一次，仓库人员变动后立即轮换。

## 5. Actions 参数规范

在私有主仓 `Settings -> Secrets and variables -> Actions` 配置：

1. Secrets：
   - `DEPLOY_TOKEN`：发布到公开仓的访问令牌。
2. Variables：
   - `DEPLOY_REPO`：目标仓库，格式 `owner/repo`。
   - `DEPLOY_BRANCH`：目标分支，建议 `gh-pages`。
   - `DEPLOY_DIR`：目标目录，默认留空表示仓库根目录。

## 6. 产物目录规范

当前发布结构：

1. 站点根目录：前端 `dist` 内容（用于 GitHub Pages 在线访问）。
2. `build/gateway/`：Go 二进制产物。
3. `build/runtime/`：Rust 二进制产物。
4. `build/BUILD_INFO.txt`：构建来源与追溯信息。

要求：

1. 公开预览站点必须保证根目录存在 `index.html`。
2. 二进制统一放入 `build/` 子目录，不污染站点根目录。
3. 每次发布需可通过 `BUILD_INFO.txt` 追溯到主仓 commit。

## 7. 发布操作流程（标准）

1. 在私有主仓将待发布内容合并到 `main`。
2. 打版本标签并推送标签：
   - `git tag v1.0.0`
   - `git push origin v1.0.0`
3. CI 自动构建并发布到公开仓目标分支。
4. 公开仓 `Settings -> Pages` 选择 `gh-pages` + `/ (root)`。
5. 访问 GitHub Pages 地址验证页面。
6. 若发现问题，在私有主仓修复后打新标签再次发布。

## 8. 在线排障规范

1. 构建失败：优先查看私有主仓 Actions 日志。
2. 页面异常：在公开预览站点打开浏览器控制台定位前端错误。
3. 发布未更新：检查标签是否匹配 `v*`、`DEPLOY_TOKEN`、`DEPLOY_REPO`、目标分支权限。
4. 产物回溯：通过 `build/BUILD_INFO.txt` 对照主仓 SHA 与 workflow run。

## 9. 回滚规范

1. 快速回滚：在公开仓把发布分支回退到上一个可用 commit。
2. 根因修复：在私有主仓修复后重新打标签触发发布。
3. 回滚与重发都必须保留可审计记录（commit、run id、时间）。

## 10. 安全红线

1. 禁止将私有配置、密钥、用户数据发布到公开仓。
2. 禁止在公开仓保留调试日志、测试数据快照、内部文档。
3. 发布脚本改动必须经代码评审，重点审查“令牌使用与目标仓写入范围”。

## 11. 当前工作流文件

1. 私有主仓工作流：
   - `.github/workflows/build-and-publish-frontend.yml`
2. 本规范优先约束该工作流的发布行为。

## 12. 变更控制

1. 修改发布目标、触发策略、权限模型前，先更新本规范。
2. 规范更新后再提交 workflow 变更，避免“实现先行、口径滞后”。

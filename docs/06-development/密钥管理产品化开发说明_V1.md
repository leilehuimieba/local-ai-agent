1. 当前项目为什么需要单独做密钥管理产品化
2. 参考 `cc-switch` 后，本项目应采用什么产品原则

5. 前端设置页应该如何组织交互与页面结构
6. 当前项目是否支持热更新，以及应该按什么路径实施

## 2. 当前项目现状

当前项目的 Provider 配置主要来自 [app.json](D:/newwork/本地智能体/config/app.json)。

当前链路特点如下：

1. `gateway` 启动时通过 [main.go](D:/newwork/本地智能体/gateway/cmd/server/main.go) 读取 `config/app.json`
2. 聊天请求在 [chat.go](D:/newwork/本地智能体/gateway/internal/api/chat.go) 中通过 `h.appConfig` 生成 `ProviderRef`
3. 设置页当前仅管理 UI 层状态与运行偏好，核心落点在 [store.go](D:/newwork/本地智能体/gateway/internal/state/store.go)
4. 前端设置接口目前不支持新增、修改、切换 provider 凭据，入口在 [api.ts](D:/newwork/本地智能体/frontend/src/settings/api.ts)
5. 当前修改 `config/app.json` 后，运行中的 `gateway` 不会自动刷新内存配置，因此默认需要重启进程才能吃到新 key

这意味着当前产品存在 4 个问题：

1. 凭据和静态配置混在一起，产品层无法安全管理
2. 前端无法完成“新增 key / 修改 key / 验证 key / 应用 key”
3. 用户会误以为“保存即生效”，但实际运行中仍可能使用旧内存配置
4. 出错时缺少“是 key 错、接口错、模型错、还是未应用”的明确反馈

## 3. 参考 cc-switch 后的正式产品原则

参考 `cc-switch`，本项目建议吸收 4 个核心思想：

1. 主数据和设备本地偏好分层
2. “保存”与“应用”拆成两个动作
3. 当前激活项是独立状态，不直接等于主数据本身
4. 配置写入应采用稳定、可回滚、尽量原子的方式

结合本项目的常驻 `gateway/runtime` 结构，正式原则调整为：

1. `ProviderCatalog`
   负责保存 provider 的静态定义，例如 `provider_id`、显示名、基础 URL、路径、模型兼容信息
2. `ProviderCredentialStore`
   负责保存凭据本身和脱敏状态，是产品中的敏感数据层
3. `RuntimeProviderState`
   负责记录当前运行时正在使用哪套 provider 配置，以及是否已应用
4. `UI Settings`
   继续只负责模式、工作区、风险开关、当前模型等普通设置，不直接承载敏感明文

正式产品动作固定为三步：

1. `测试连接`
2. `保存凭据`
3. `应用到运行时`

其中：

1. `测试连接` 解决“这把 key 本身能不能用”
2. `保存凭据` 解决“数据先安全持久化”
3. `应用到运行时` 解决“当前进程真正开始使用它”

## 4. 产品范围定义

本轮只做密钥管理的最小可用版，不追求一次做成完全自动热更新平台。

### 4.1 本轮必须实现

1. 前端设置页可查看 provider 列表和凭据状态
2. 前端可为已有 provider 新增或更新 API key
3. 前端可先测试连接，再保存
4. 前端可显式执行“应用到运行时”
5. 页面能清楚告诉用户“已保存但未应用”还是“已应用”
6. 错误反馈能区分认证失败、接口不可达、模型不可用、未应用

### 4.2 本轮明确不做

1. 不做多端同步
2. 不做系统级加密仓库集成
3. 不做复杂凭据版本历史
4. 不做自动推送刷新所有在运行中的会话
5. 不把所有模型配置改造成数据库主数据

---

## 5. 产品方案总览

本项目建议采用：

> `静态 Provider 目录 + 本地凭据存储 + 显式应用到运行时`

对应产品表现为：

1. `app.json` 继续保存默认 provider 元信息和默认模型
2. 新增本地凭据文件，保存用户更新后的 key
3. 新增运行时激活状态文件，记录当前已经应用的 provider 版本
4. `gateway` 在处理聊天请求前，优先读取运行中的 provider store，而不是只看启动时的 `cfg.Providers`
5. 设置页为用户展示三个状态：`未配置`、`已保存未应用`、`已应用`

## 6. 数据结构设计

### 6.1 分层结构

建议新增以下三层数据：

1. `ProviderCatalog`
2. `ProviderCredentialStore`
3. `RuntimeProviderState`

### 6.2 ProviderCatalog

来源：

1. 继续以 [app.json](D:/newwork/本地智能体/config/app.json) 为基础

职责：

1. 描述 provider 基本信息
2. 描述哪些 provider 可以出现在设置页
3. 描述默认模型和模型到 provider 的映射

建议结构：

```json
{
  "provider_id": "scnet",
  "display_name": "SCNet LLM",
  "base_url": "https://api.scnet.cn/api/llm/v1",
  "chat_completions_path": "/chat/completions",
  "models_path": "/models",
  "credential_kind": "api_key",
  "supports_test": true,
  "editable": true
}
```

说明：

1. `api_key` 不再作为设置页响应中的可写主字段返回
2. `editable` 用于标识该 provider 是否允许前端改写
3. `supports_test` 用于控制“测试连接”按钮是否可用

### 6.3 ProviderCredentialStore

建议新增文件：

1. `data/settings/provider-credentials.json`

职责：

1. 保存用户显式保存过的凭据
2. 保存凭据状态，但不回显明文
3. 保存最后一次测试结果与时间

建议结构：

```json
{
  "providers": [
    {
      "provider_id": "scnet",
      "credential_kind": "api_key",
      "api_key_ciphertext": "",
      "api_key_masked": "sk-MjQ0...MTY=",

      "last_test_message": "连接校验成功",
      "last_test_at": "1744099300000"
    }
  ]
}
```

最小可用实现建议：

1. 第一版允许先明文写本地文件，但接口和前端永远不回显明文
2. 文件权限和展示逻辑先收紧，后续再替换为系统加密

若要继续增强，下一阶段可替换为：

1. Windows DPAPI
2. OS keychain
3. 加密后落盘

### 6.4 RuntimeProviderState

建议新增文件：

## 7. 后端接口设计

### 7.1 接口设计原则

本项目不建议把所有动作都塞进现有 `/api/v1/settings` 的单个 `POST`。

原因：

1. 普通 UI 设置和敏感凭据操作风险等级不同
2. 测试连接、保存凭据、应用运行时是三个语义不同的动作
3. 后续若支持审计和权限控制，分接口更稳

### 7.2 建议接口列表

建议新增以下接口：

1. `GET /api/v1/settings/providers`
2. `POST /api/v1/settings/providers/test`
3. `POST /api/v1/settings/providers/save`
4. `POST /api/v1/settings/providers/apply`
5. `POST /api/v1/settings/providers/remove`

### 7.4 POST /api/v1/settings/providers/test

用途：

1. 使用用户当前输入的 key 做即时校验
2. 可选校验模型列表或默认模型
3. 不落盘

请求示例：

```json
{
  "provider_id": "scnet",
  "base_url": "https://api.scnet.cn/api/llm/v1",
  "api_key": "sk-xxxx"

```

返回示例：

```json
{
  "ok": true,
  "provider_id": "scnet",
  "message": "连接校验成功",
  "checked_at": "1744099300000"
}
```

错误口径建议固定为：

1. `authentication_failed`
2. `provider_unreachable`
3. `invalid_provider_response`
4. `model_unavailable`

用途：

1. 将凭据保存到本地凭据仓
2. 更新测试状态
3. 不直接改变正在运行的 `gateway` 生效配置

请求示例：

```json

  "provider_id": "scnet",
  "api_key": "sk-xxxx"
}
```

返回示例：

```json

  "provider_id": "scnet",
```

### 7.6 POST /api/v1/settings/providers/apply

用途：

1. 将已保存的 provider 凭据装载到运行中 store
2. 更新运行时激活状态
3. 返回是否需要重启

请求示例：

```json
{
  "provider_id": "scnet"
}
```

返回示例：

```json
{
  "ok": true,
  "provider_id": "scnet",
  "message": "已应用到运行时",
  "apply_mode": "hot_reload",
  "applied_at": "1744099400000",
  "restart_required": false
}
```

若当前阶段仍走重启路径，允许返回：

```json
{
  "ok": true,
  "provider_id": "scnet",
  "message": "凭据已保存，请重启 gateway 后生效",
  "apply_mode": "restart_required",
  "restart_required": true
}
```

### 7.7 POST /api/v1/settings/providers/remove

用途：

1. 移除某个 provider 已保存凭据
2. 对已应用 provider 给出阻断提醒

请求示例：

```json
{
  "provider_id": "scnet"
}
```

返回规则：

1. 若 provider 正在被当前模型使用，应要求先切走模型或再次确认
2. 若已删除但运行中仍保留旧配置，应返回 `saved_removed_but_runtime_still_active`

## 11. 热更新实施路径

### 11.1 当前结论

当前项目默认不支持真正意义上的运行中热更新。

原因：

1. `gateway` 启动时加载 `config/app.json`
2. 聊天请求生成 `ProviderRef` 时读取的是内存中的 `appConfig`
3. 当前没有运行态 provider store，也没有配置变更广播

因此当前真实口径是：

1. 修改配置文件后，`gateway` 默认需要重启才会完全生效

### 11.2 建议实施路径

建议分三步推进。

#### 第一步：显式应用，但底层仍允许重启生效

目标：

1. 先把产品动作闭环做出来

实现方式：

1. 前端提供 `保存` 和 `应用`
2. 后端 `apply` 接口更新状态文件
3. 若当前版本无法热切换，则明确返回 `restart_required=true`

好处：

1. 产品语义先稳定
2. 用户知道为什么保存后没立即生效
3. 后续底层升级时，前端无需重做

#### 第二步：gateway 内存级热更新

目标：

1. 不重启进程即可让新任务使用新凭据

实现方式：

1. `gateway` 启动时初始化 `RuntimeProviderStore`
2. `apply` 接口直接刷新该 store
3. 新的聊天请求读取 store，而不是固定读取 `h.appConfig`

效果：

1. 新发起的任务立即使用新配置
2. 已在运行中的旧任务不强制中断

这是本项目最推荐的热更新实现终点。

#### 第三步：跨组件广播与更细粒度刷新

目标：

1. 设置页、首页、任务页都能立即感知 provider 生效状态变化

实现方式：

1. 设置接口返回最新 provider 状态
2. 首页和任务页在进入时重新拉取设置
3. 如有需要，再增加轻量事件广播

当前不建议优先做：

1. 运行中任务强制切换 provider
2. 多会话即时重绑
3. 自动重放失败请求

---

## 12. 推荐实施方案

当前项目最推荐采用：

> `先做“测试连接 + 保存 + 应用到运行时”，apply 接口优先支持 gateway 内存热更新；若部分 provider 暂不支持，则返回需要重启。`

原因：

1. 这条路径和当前项目结构最兼容
2. 能直接解决“改 key 后用户不知道是否生效”的核心问题
3. 能复用设置页现有控制面板形态
4. diff 范围可控，不需要一上来重构整体配置体系
5. 与 `cc-switch` 的“保存 / 应用分离”产品逻辑一致

---

## 13. 开发步骤建议

建议按以下顺序推进：

1. 新增 provider 凭据状态数据结构与本地存储
2. 新增 provider 相关接口：列表、测试、保存、应用、移除
3. 改造聊天链路，使新请求优先读取运行时 provider store
4. 扩展设置页 contracts 与 API
5. 在设置页新增“模型服务与密钥”模块
6. 补充错误态与状态文案
7. 再做回归验证

---

## 14. 验收标准

本主题至少必须满足以下验收口径：

1. 设置页能显示每个 provider 的密钥状态，且不会回显明文
2. 用户可为已有 provider 输入 API key 并测试连接
3. 测试失败时，页面能区分认证失败、服务不可达、响应异常
4. 用户保存后，页面明确显示“已保存未应用”或“已应用”
5. 点击“应用到运行时”后，新任务会按最新配置执行，或明确提示需要重启
6. 聊天链路不再只依赖启动时 `appConfig` 中的旧 key
7. 旧设置功能不回归，包括模型切换、模式切换、工作区切换、风险开关、诊断导出

---

## 15. 验证建议

建议至少做以下验证：

1. 使用错误 key 测试连接，应返回认证失败
2. 使用正确 key 测试连接，应返回成功
3. 保存后不应用，执行任务时应显示“仍使用旧配置”或明确未应用状态
4. 应用后执行任务，新请求应使用新 key
5. 移除正在使用的 provider 凭据时，应有阻断或明确提示
6. 重启 `gateway` 后，已保存状态和应用状态不丢失

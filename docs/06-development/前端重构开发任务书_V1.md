## 1. 任务定位

本任务书只解决一件事：

> 把当前前端从“能运行的三视图页面壳”重构为“围绕任务运行组织的本地智能体工作界面”。

当前任务不是：

1. 重做一套全新产品视觉稿
2. 引入新前端框架或大规模更换技术栈
3. 新增多智能体页、记忆页、插件中心等新产品域
4. 为了抽象而重写全部前端逻辑

当前任务要做的是：

1. 重建状态模型
2. 重建页面职责边界
3. 建立全局层
4. 降低 `App.tsx` 总装压力
5. 让用户更稳定地理解当前任务状态

---

## 2. 当前阶段目标

本阶段唯一目标：

> 让前端的产品体验骨架从“页面切换驱动”升级为“任务状态驱动”。

做到这一步后，前端应具备：

1. 当前任务状态与连接状态可明确表达
2. 风险确认、错误、连接提示具备全局可见性
3. 工作台真正成为当前任务现场
4. 日志页与工作台的边界清晰
5. 设置页回归系统配置入口
6. 前端状态与模块边界开始围绕领域收口

---

## 3. 本阶段硬约束

开始实现前必须满足：

1. 正式产品主结构仍为工作台、日志页、设置页三类信息域
2. 不新增顶层 API 作为本阶段依赖前提
3. 不允许破坏聊天、确认、事件流、日志页、模型切换、模式切换、工作区切换主链路
4. 不允许把前端重构成重运维控制台或复杂 IDE 壳
5. 不允许为了“先抽象好”而中断当前真实可用功能

---

## 4. 本阶段必须完成的 6 个模块

### 4.1 `runtime-state-model`

职责：

1. 建立正式任务运行状态模型
2. 建立正式连接状态模型
3. 将运行状态与页面状态拆开

### 4.2 `global-overlays`

职责：

1. 提升风险确认为全局层
2. 提升严重错误为全局层
3. 提升连接状态提示为全局层

### 4.3 `workspace-shell`

职责：

1. 将工作台固定为统一任务页
2. 重组中央任务区、右侧任务中枢、底部调查层的职责关系
3. 让当前任务信息在视觉和结构上获得最高优先级

### 4.4 `history-review`

职责：

1. 让日志页明确承接历史复盘和排障
2. 保持日志筛查、详情联动和调查能力真实可用
3. 避免当前任务实时信息与历史信息混写

### 4.5 `settings-scope`

职责：

1. 收口设置页职责
2. 把系统配置、工作区与授权、风险策略保留在设置页
3. 减少设置页承担工作台解释职责

### 4.6 `module-boundary`

职责：

1. 按领域拆分前端状态与 API 消费层
2. 逐步降低单一入口文件的总装负担
3. 为后续 `capabilities / connectors / jobs` 预留稳定挂位

---

## 5. 推荐实现顺序

### Phase A：状态模型收口

先做：

1. 正式定义 `runState`
2. 正式定义 `connectionState`
3. 梳理事件到状态的映射规则
4. 梳理确认、失败、完成、恢复的状态迁移

通过标准：

1. UI 不再通过中文文案判断状态
2. 运行态和连接态能够独立表达

### Phase B：全局层收口

再做：

1. 确认层提升为全局层
2. 错误层提升为全局层
3. 连接提示提升为全局层

通过标准：

1. 风险确认不再依赖页面位置可见
2. 连接中断与恢复有统一提示位置

### Phase C：工作台重构

再做：

1. 工作台中央任务区收口
2. 右侧任务中枢重组
3. 底部调查层与主任务区重新分工

通过标准：

1. 用户进入工作台后能快速理解当前任务、当前状态、下一步
2. 主任务区不再被次级信息干扰

### Phase D：日志页与设置页收口

最后做：

1. 日志页强化复盘定位
2. 设置页回归配置入口
3. 同步清理跨页重复信息

通过标准：

1. 历史信息和当前现场边界清晰
2. 设置页不再继续膨胀成混合控制面

## 7. 文件级任务清单

### 7.1 `frontend/src/App.tsx`

任务：

1. 从总装入口逐步退化为应用壳与视图装配层
2. 将运行态、连接态、全局层挂载逻辑与页面切换逻辑分离
3. 不再在页面层散落状态迁移规则

### 7.2 `frontend/src/events/useSessionEventStream.ts`

任务：

1. 补齐连接状态表达
2. 预留重连和断开体验入口
3. 不再把“流关闭”当作无感知内部行为

### 7.3 `frontend/src/chat/ChatPanel.tsx`

任务：

1. 强化当前任务线程的主区定位
2. 减少展示层对状态文案的反向判断
3. 让任务输入、执行结果、失败提示、确认入口层级更清晰

### 7.4 `frontend/src/workspace/ContextSidebar.tsx`

任务：

1. 固定任务中枢结构
2. 聚焦“当前任务、下一步、最近动作、风险、上下文、记忆痕迹”
3. 不再继续漂成无主次卡片集合

### 7.5 `frontend/src/workspace/BottomPanel.tsx`

任务：

1. 收口为当前任务调查层
2. 明确主任务区与调查区的关系
3. 为后续工具动作、验证结果、系统细节预留位置

### 7.6 `frontend/src/logs/useLogs.ts`

任务：

1. 解除“当前事件数驱动全局日志刷新”的耦合
2. 让日志数据围绕历史复盘语义组织
3. 为后续分页、筛选、任务级聚合预留结构

### 7.7 `frontend/src/settings/useSettings.ts`

任务：

1. 梳理配置数据与资源数据边界
2. 避免设置与记忆资源相互污染
3. 为后续 `capabilities / connectors` 消费预留位置

### 7.8 `frontend/src/index.css`

任务：

1. 保持现有统一视觉基础
2. 增强全局状态带、全局层、任务现场与调查层的层级表达
3. 保障桌面与窄屏下布局退化顺序合理

## 8. 功能要求

### 8.1 任务状态

必须满足：

1. 有正式 `idle / submitting / streaming / awaiting_confirmation / resuming / completed / failed` 状态
2. UI 判断依赖稳定状态值，而不是展示文案

### 8.2 连接状态

必须满足：

1. 有正式 `connecting / connected / reconnecting / disconnected / closed` 状态
2. 用户能感知连接异常和恢复过程

### 8.3 全局确认

必须满足：

1. 确认请求不受页面位置影响
2. 用户能在任何页面看到高优先级确认
3. 当前任务上下文能支持确认判断

### 8.4 工作台

必须满足：

1. 工作台是当前任务第一现场
2. 当前任务、当前状态、下一步建议、调查入口清晰
3. 当前任务输入和结果收口不被次级信息挤压

### 8.5 日志页

必须满足：

1. 历史记录和当前任务现场有明确边界
2. 日志页继续服务复盘、排障、验收观察
3. 现有真实筛查和详情联动能力不能退化

### 8.6 设置页

必须满足：

1. 设置页只承接系统配置与策略入口
2. 模型、模式、工作区、授权、风险策略仍完整可用
3. 不继续把运行现场解释塞入设置页

---

## 9. UI 要求

必须符合：

1. 重要状态固定落位
2. 高优先级事件有明显接管机制
3. 当前任务区、右侧任务中枢、底部调查层形成明确主辅关系
4. 日志页和设置页不抢当前任务的认知焦点

不允许：

1. 为了抽象重构破坏现有视觉秩序
2. 做成重后台或复杂 IDE 样式
3. 让用户需要频繁切页才能搞清系统状态

---

## 10. 验收标准

### 10.1 必须通过

1. 用户可以稳定判断当前任务状态和连接状态
2. 风险确认可在全局层稳定出现
3. 工作台明显成为当前任务现场
4. 日志页与工作台边界清晰
5. 设置页职责更收敛
6. 当前真实主链路未被改坏

### 10.2 不通过情形

1. 只是换了结构或样式，但状态判断仍然依赖文案
2. 确认、错误、断连仍然只是局部卡片提示
3. 工作台仍然像页面拼装，而不是任务现场
4. 日志刷新和当前事件流继续互相耦合
5. 设置页继续混写配置、资源和运行现场

---

## 11. 构建与验证要求

实现完成后至少提交以下验证：

1. `frontend`: `npm run build`
2. `gateway`: `go build ./...`
3. `workspace`: `cargo build`

手工验证至少覆盖：

1. 发起任务后的状态切换
2. SSE 中断或异常时的连接提示
3. 风险确认出现、确认、拒绝、取消链路
4. 日志页筛查与详情联动
5. 设置页模型、模式、工作区、授权、风险设置

---

## 12. 交付要求

实现完成后必须同步产出：

1. 前端重构验收文档

验收文档必须逐项填写：

1. 做了什么
2. 没做什么
3. 状态模型如何落地
4. 模块边界如何调整
5. 主链路如何验证
6. 最终结论是 `通过 / 有条件通过 / 不通过`

---

## 13. 给实现型 AI 的明确要求

1. 先做状态真实收口，再做页面结构收口
2. 先保证全局层稳定，再优化局部体验
3. 先降低总装耦合，再考虑更多展示层细化
4. 不做一次性大重构
5. 每一步都必须保住当前真实主链路

一句话要求：

> 当前任务不是重做前端，而是让前端真正围绕任务运行组织起来。

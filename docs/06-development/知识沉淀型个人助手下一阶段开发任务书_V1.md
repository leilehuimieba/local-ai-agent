## 1. 文档目的

本文只解决一件事：

> 基于 `V1` 首轮真实使用内测结果，开启“知识沉淀型个人助手”下一阶段的最小正式治理，不重新扩能力域，只把已经暴露出的关键缺口补成下一轮可继续内测的正式输入。

本文不是：

1. 新一轮运行时架构重写
2. 新一轮多能力发散规划
3. 远程入口、浏览器自动化、多智能体的立项文档

本文只回答：

1. 下一阶段为什么可以开题
2. 下一阶段只做什么
3. 下一阶段按什么工作包推进
4. 做到什么程度才允许进入下一轮真实使用

---

## 2. 开题依据

下一阶段允许开启，依据固定为：

1. `V1` 已冻结通过。
2. 运行时 Harness 四项收口已完成并留证。
3. 可发布候选、真实场景脚本和首轮真实使用内测均已完成。
4. 首轮内测结论为 `通过`，且已明确允许开启下一阶段任务书。

当前开题不依据“想做更多能力”，只依据真实内测暴露出的 3 个问题：

1. `S1` 还缺一份“学习资料摘要写回”独立样本。
2. 续推与下一步建议的措辞仍偏“项目治理 / 验收口径”。
3. 日志与证据页对普通学习用户仍偏技术复盘视角。

---

## 3. 当前阶段判断

一句话判断：

> 当前阶段不是继续补运行时主干，而是把 `V1` 已成立的知识沉淀、状态理解和下一步建议能力，收束成更像“知识沉淀型个人助手”的真实使用体验。

当前已经成立：

1. 学习资料整理与知识沉淀场景的最小闭环已成立。
2. 知识沉淀、记忆可见、续推和下一步建议已进入正式主链路。
3. 真实样本、构建验证和交接文档已经具备下一阶段治理基础。

当前还没成立：

1. 学习资料摘要写回的独立留证闭环。
2. 面向普通学习用户的自然学习指导语言。
3. 更偏学习复盘视角的日志与证据呈现。

---

## 4. 本阶段总目标

下一阶段总目标固定为 3 件事：

1. `补齐学习资料摘要独立样本`
2. `把续推与下一步建议收成学习型表达`
3. `把知识沉淀型个人助手产品定义收成正式执行输入`

只有这 3 件事，不新增第 4 件大事。

---

## 5. 本阶段明确不做

下一阶段明确禁止进入：

1. 多智能体编排
2. 浏览器自动化深链路
3. 自动 fallback
4. 重型记忆评分
5. 运行时主干重写
6. 新增一级产品能力域
7. QQ / 微信接入
8. 手机远程下发执行指令
9. 高权限桌面自动化

一句话：

> 下一阶段只收口“知识沉淀型个人助手”主线表达与样本，不重新扩大产品边界。

## 6. 本阶段工作包

下一阶段只按 4 个工作包推进。

### 6.1 工作包 A：学习资料摘要独立样本补齐

目标：

1. 为 `S1` 学习资料摘要整理补一份独立正式样本。
2. 样本能体现“读取真实学习材料 -> 生成摘要 -> 写回整理结果 -> 留证”的完整链路。

本工作包应完成：

1. 固定一份学习资料输入源。
2. 固定摘要写回产物路径。
3. 固定对应日志或事件留证。
4. 在证据目录中形成可追踪样本。

通过标准：

1. `S1` 不再只靠旁证成立。
2. 样本能直接展示摘要写回结果。
3. 样本可被后续内测和验收重复引用。

### 6.2 工作包 B：学习型表达收口

目标：

1. 把当前续推与下一步建议，从“项目治理 / 验收口径”收成更自然的学习指导语言。

本工作包应完成：

1. 固定学习场景下的默认表达口径。
2. 区分“学习指导表达”和“研发治理表达”的使用边界。
3. 为续推、下一步建议、阶段判断补对应样本。

通过标准：

1. 回答仍然基于真实状态，不退化成空泛鼓励。
2. 同样的事实能用更自然的学习语言表达出来。
3. 新口径有独立样本和人工核对记录。

### 6.3 工作包 C：日志与证据页学习视角收口

目标：

1. 让日志与证据页对普通学习用户更容易理解。

本工作包应完成：

1. 收口“当前结论、为什么这样判断、下一步建议”的首屏表述。
2. 弱化纯技术复盘口吻。
3. 补一份普通用户视角可读性样本。

通过标准：

1. 日志与证据页仍保留研发留证能力。
2. 同时对非研发用户不再主要表现为技术事件堆叠。
3. 存在对照样本或页面留证。

### 6.4 工作包 D：产品定义与路线输入定稿

目标：

1. 把现有两份讨论稿收成下一阶段正式上游输入。

本工作包应完成：

1. 固定产品定义文档状态。
2. 固定路线图文档状态。
3. 明确这两份文档是“下一阶段上游输入”，不是替换当前 `V1` 唯一执行口径。

通过标准：

1. 后续相关实现可直接引用这两份文档。
2. 顶层入口不会因此产生新的并列主口径。

---

## 7. 推荐执行顺序

严格按以下顺序推进：

1. 先做工作包 A，补齐 `S1` 独立样本。
2. 再做工作包 B，收口学习型表达。
3. 再做工作包 C，收口日志与证据页表达。
4. 最后完成工作包 D，把产品定义和路线图定稿并回写入口。

原因：

1. 先补样本，避免后面继续只靠推断说能力成立。
2. 先收语言，再收界面，更容易形成一致用户感知。
3. 产品定义与路线图应建立在真实使用观察之上，而不是先行空定。

## 12. 两周执行清单（默认下发模板）

本节用于解决一个具体问题：

> 当后续需要给实现型 AI 连续下发“知识沉淀主线”任务时，直接使用哪套默认节奏、任务卡和验收口径。

本节不新增能力域，只把既定主线拆成一个可直接执行的两周模板。

默认前提：

1. 当前仍遵守 [知识沉淀型个人助手开发路线图_V1](D:/newwork/本地智能体/docs/06-development/知识沉淀型个人助手开发路线图_V1.md) 的阶段顺序。
2. 两周模板只覆盖 `Stage B`、`Stage C` 和最小 `Stage D`。
3. 多智能体、浏览器深链自动化、IM 接入、cron 调度、远程入口仍不进入本轮。

### 12.1 两周总目标

第一周目标固定为：

1. `Stage B：知识沉淀闭环`
2. 让系统稳定完成“导入资料 -> 形成结构化沉淀 -> 可检索 -> 回答时能引用沉淀”。

第二周目标固定为：

1. `Stage C：状态理解与下一步建议`
2. `Stage D：最小执行桥接`
3. 让系统稳定完成“更新状态 -> 回答做到哪/还差什么 -> 给出下一步建议 -> 执行一个低风险动作并回写”。

两周统一主线固定为：

1. `导入资料`
2. `形成沉淀`
3. `可检索回答`
4. `更新状态`
5. `给下一步建议`
6. `必要时执行小动作并回写`

### 12.2 第 1 周执行清单（Stage B）

Day 1：

1. 冻结输入类型分层，明确哪些输入进入长期沉淀，哪些只进入短期状态。
2. 冻结最小沉淀字段，建议固定为：`主题 / 类型 / 结论 / 依据 / 来源 / 时间 / 下一步 / 标签`。
3. 产出物：`输入类型表`、`沉淀字段表`。

Day 2：

1. 选择 `2-3` 份真实资料样本完成导入。
2. 确认导入后可见、可追溯、可区分主题。
3. 产出物：`导入样本清单`、`导入前后对照记录`。

Day 3：

1. 把导入样本整理成结构化沉淀，而不是只保存原文。
2. 补最小去重规则和噪音过滤规则。
3. 产出物：`结构化沉淀样本`、`去重/过滤规则说明`。

Day 4：

1. 围绕当前样本设计 `5` 条检索问句。
2. 记录命中 / 部分命中 / 未命中及原因。
3. 产出物：`5 条检索问句结果表`、`未命中原因表`。

Day 5：

1. 围绕同一批资料设计 `5` 条用户真实会问的问题。
2. 观察回答是否显式使用沉淀内容。
3. 产出物：`5 条真实问句回答记录`、`回答复用沉淀标记表`。

Day 6：

1. 从 Day 5 的回答中选择 `3` 条回写为学习记录或阶段结论。
2. 检查来源追溯是否成立。
3. 产出物：`3 条回写记录`、`来源追溯检查`。

Day 7：

1. 做一次端到端演练：`导入新资料 -> 生成沉淀 -> 设计问句 -> 观察回答 -> 回写记录`。
2. 只允许提出 `1-2` 个最小修补项，不并行开新方向。
3. 产出物：`第 1 周端到端演练记录`、`本周问题 Top 2`。

第 1 周通过标准：

1. 至少 `2` 份真实资料已完成导入和追溯。
2. 至少 `5` 条结构化沉淀样本已形成。
3. `5` 条检索问句中至少 `3` 条稳定命中且结果可用。
4. `5` 条真实问句中至少 `3` 条可见历史沉淀复用痕迹。
5. 至少 `1` 条完整端到端链路已留证。

### 12.3 第 2 周执行清单（Stage C + 最小 Stage D）

Day 8：

1. 冻结最小状态字段，建议固定为：`当前目标 / 当前主题 / 当前阶段 / 最近进展 / 当前阻塞 / 最近结果 / 下一步候选`。
2. 明确每个字段由什么输入更新。
3. 产出物：`状态字段表`、`状态更新规则表`。

Day 9：

1. 让前一周沉淀结果能够更新状态字段。
2. 补“最近状态快照”结果或调试输出。
3. 产出物：`状态更新样本`、`状态快照结果`。

Day 10：

1. 设计 `6` 条续推问题，至少覆盖“做到哪了 / 还差什么 / 下一步做什么”。
2. 记录失败类型：`状态缺失 / 判断不准 / 建议太泛 / 没引用依据`。
3. 产出物：`续推问题专项结果表`、`失败类型统计`。

Day 11：

1. 固定下一步建议最小格式：`当前判断 / 缺口 / 最该做的一步 / 为什么是这一步`。
2. 让建议尽量绑定具体资料、文件或记录。
3. 产出物：`下一步建议模板`、`建议样本`。

Day 12：

1. 只选择 `2` 类低风险动作接入建议链路。
2. 优先动作固定为：`工作区内读写文件`、`受控命令执行`。
3. 产出物：`桥接动作样本`、`执行后结果摘要`。

Day 13：

1. 让执行结果自动进入：`近期结果 / 阶段结论 / 下一步候选`。
2. 检查失败时是否也形成收口记录。
3. 产出物：`执行回写记录`、`失败收口记录`。

Day 14：

1. 完整跑一条真实链路：`导入资料 -> 形成沉淀 -> 更新状态 -> 回答续推问题 -> 给出下一步建议 -> 执行一个低风险动作 -> 回写结果`。
2. 只列 `3` 个下阶段问题，不新增大范围任务。
3. 产出物：`两周总演练记录`、`下阶段问题 Top 3`。

第 2 周通过标准：

1. 状态字段和更新规则已固定。
2. `6` 条续推问题中至少 `4` 条达到“可直接推进下一步”的程度。
3. 下一步建议不再是泛泛 checklist，而是绑定具体对象的动作句。
4. 至少 `1-2` 类低风险动作已接入，且执行结果能回写。
5. 两周总演练已形成完整留证。

### 12.4 任务卡拆分

1. `B1 输入类型分层表`
2. `B2 沉淀字段结构冻结`

4. `B4 结构化沉淀生成`
5. `B5 去重与噪音过滤最小规则`
6. `B6 检索问句专项`
7. `B7 回答复用沉淀专项`
8. `B8 学习记录回写闭环`

`Stage C` 默认任务卡：

1. `C1 状态字段模型冻结`
2. `C2 状态更新规则`
3. `C3 状态快照可见化`
4. `C4 续推问题专项样本`
5. `C5 下一步建议模板与规则`
6. `C6 建议质量验收`

最小 `Stage D` 默认任务卡：

1. `D1 低风险动作白名单`
2. `D2 执行结果摘要模板`
3. `D3 执行后验证模板`
4. `D4 执行结果回写状态/沉淀`
5. `D5 失败收口与交接`

### 12.5 固定下发模板

后续如果需要直接给实现型 AI 下发，可以优先使用下面模板：

1. `本轮目标：只推进知识沉淀主线，不扩能力域。`
2. `本轮范围：仅限本文第 12 节对应的 Day X 或任务卡 X。`
3. `本轮交付：真实样本 + 留证文件 + 对应文档回写 + 构建/核对结果。`
4. `本轮边界：不重写 planner，不扩接口，不改共享合同，不做页面大改。`
5. `本轮完成标准：必须回答“做了什么 / 没做什么 / 怎么验证 / 证据在哪”。`

### 12.6 使用规则

1. 如果是单人推进，默认顺序为：`B1-B4 -> B6-B8 -> C1-C5 -> D1-D4`。
2. 如果是双线并行，最多只允许一条 `Stage B` 线和一条 `Stage C/D` 线并行。
3. 每张任务卡都必须回答 4 个问题：`是否有真实样本 / 是否进入主链路 / 是否有留痕 / 是否减少重复劳动`。
4. 任一任务卡如果不能回答上面 4 个问题，默认不继续开下一张卡。

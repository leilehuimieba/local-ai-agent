## 1. 任务定位

本阶段只做一件事：

> 把现有前端从“结构能用”收口到“产品可用”，并稳定为当前首版产品前端。

本任务书不是纯视觉稿。
本任务书要求实现型 AI 直接按产品化标准完成页面、交互、状态和主链路回归。

---

## 2. 当前阶段目标

本阶段必须完成以下目标：

1. 首页稳定成为任务工作台，而不是聊天页变体。
2. 日志页稳定成为复盘与排障页，而不是日志列表。
3. 设置页稳定成为运行时控制面板，而不是表单集合。
4. 三页形成统一视觉语言、统一信息密度、统一组件节奏。
5. 不破坏聊天、确认、事件、日志、模型切换、模式切换、工作区切换主链路。

---

## 3. 本阶段硬约束

1. 正式产品主线仍是首页、日志页、设置页。
2. 独立对话页允许保留，但只能承接主聊天线程，不能扩成新产品域。
3. 不新增 skills 页、记忆页、多智能体页、IDE 式对象树导航。
4. 不新增顶层 API，不为了视觉重写主数据流。
5. 不允许把已接入的 repo context、风险确认、事件流、日志联动改回占位态。
6. 不允许把首页重新做成卡片拼盘或大杂烩控制台。

---

## 4. 设计与实现要求

### 4.1 设计语言

1. 整体气质为安静、专业、长期可用的本地智能体工作台。
2. 以雾白、矿物灰、石板灰为基底，强调色克制使用。
3. 靠留白、边框、分区和层级建立秩序，不靠大面积阴影和强渐变。
4. 输入区、确认区、任务区都要更像工作界面，不像社交聊天界面。

### 4.2 技术要求

1. 继续使用当前 React + Vite 前端结构，不引入重型 UI 框架重做页面。
2. 全局样式统一收口到 token 和基础层，不允许组件内散落大量硬编码颜色。
3. 页面结构优先复用现有模块，必要时只做受控拆分，不做大规模推翻重写。
4. 所有占位模块必须明确标识“后续接入”或“即将支持”。

### 4.3 产品要求

1. 首页第一眼要能看懂当前任务、当前状态、下一步和是否需要确认。
2. 日志页第一眼要能看懂筛查、时间线和详情联动关系。
3. 设置页第一眼要能看懂运行时、模型、模式、工作区、风险授权的管理边界。
4. 空状态、加载态、错误态必须简洁直接，不能依赖内部知识才能理解。

---

## 5. 推荐实现顺序

1. 收口全局视觉 token 和布局壳层。
2. 收口首页顶栏、主工作区、右侧 Task Hub、底部 Investigation Layer。
3. 收口日志页的摘要区、筛查工具条、时间线和详情联动区。
4. 收口设置页六大模块区块与状态展示。
5. 做主链路回归，确认聊天、确认、日志、模型、模式、工作区都未被改坏。

## 7. 文件级任务清单

### 7.1 App 与全局样式

1. 统一三页壳层、导航、主容器间距和页面切换结构。
2. 建立统一 token、surface、border、text、status 色值。
3. 清理明显偏旧稿或拼装感过强的样式。

### 7.2 首页工作台

1. 顶栏展示产品框架感、模型、模式、工作区、运行状态。
2. 主区突出当前任务、任务线程、确认卡、输入区。
3. 右栏固定收口为 Task Hub，不再漂成多个无主次卡片。
4. 底部只保留当前任务事件调查，不再重复全局日志。

### 7.3 日志页

1. 页面头部补齐标题、摘要、筛查入口。
2. 接通关键词、级别、重点等真实筛查逻辑。
3. 左侧时间线与右侧详情区必须真实联动。
4. 详情区优先展示 context_snapshot、verification_snapshot、error 等正式字段。

### 7.4 设置页

1. 固定六大模块区块。
2. 已真实接入模块要展示成正式功能，不允许和占位模块混写。
3. 工作区、授权、风险、模式、模型切换必须集中管理。
4. Memory Policy 与 Diagnostics 可占位，但要明确未接入状态。

### 7.5 主链路回归

1. 对话提交、消息展示、确认流必须继续可用。
2. 事件流和日志页分层必须继续成立。
3. 模型切换、模式切换、工作区切换必须继续可用。
4. repo context 展示不能因 UI 收口而丢字段。

---

## 8. 验收标准

做到以下程度，才算本阶段完成：

1. 首页明显像任务工作台。
2. 右侧栏明显像任务中枢。
3. 底部面板明显像调查层。
4. 日志页明显像复盘排障页。
5. 设置页明显像运行时控制面板。
6. 页面视觉语言统一，信息层级稳定。
7. 当前真实功能没有被前端改版破坏。

以下情况直接判不通过：

1. 做成普通后台模板。
2. 做成概念图式页面，真实功能退化。
3. 首页重新膨胀成聊天页或卡片拼盘。
4. 日志页仍是列表壳子，筛查和详情联动不真实。
5. 设置页仍是普通表单堆叠。

---

## 9. 构建与验证要求

实现完成后至少提交以下验证：

1. `npm run build`
2. `go build ./...`
3. `cargo build`

手工验证至少覆盖：

1. 首页工作台进入与任务查看
2. 日志页筛查与详情联动
3. 设置页工作区与风险设置
4. observe / standard / full_access 模式边界展示
5. Runtime 不可达或无任务时的空态与错误态

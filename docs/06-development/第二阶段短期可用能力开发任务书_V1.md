## 1. 任务定位

第二阶段短期目标只解决一件事：

> 把当前本地智能体收口到“接入在线大模型 API 后，已经足够可用、可执行、可沉淀、可持续使用”的状态。

这里强调的是：

1. `短期可用`
2. `长期方向不丢`
3. `不过度发散`

本任务书不是：

1. 多智能体开发任务书
2. 语音能力开发任务书
3. 浏览器全局观察开发任务书
4. 桌面自动化平台开发任务书
5. 大规模前端重做任务书

---

## 2. 当前阶段目标

本阶段必须把以下能力正式收口：

1. 用户接入在线模型后，可以稳定对话。
2. 智能体可以在当前工作区内真实读取文件。
3. 智能体可以在当前工作区内真实写入文件。
4. 智能体可以执行明确、受控的本地命令或操作。
5. 智能体具备清晰可见的主链路与思考收口。
6. 智能体具备本地缓存，降低重复思考与重复消耗。
7. 记忆、知识、偏好、失败教训继续作为长期资产沉淀。

一句话：

> 本阶段要把系统推进到“已经可以开始替代用户频繁去 VS Code 里找 AI 帮忙”的程度。

---

## 3. 当前阶段硬约束

1. 不推翻当前 Go + Rust + React 主架构。
2. 不新增多智能体正式主链路。
3. 不新增语音输入输出正式主链路。
4. 不新增浏览器全局观察正式主链路。
5. 不新增桌面 GUI 自动化正式主链路。
6. 不重做首页、日志页、设置页一级结构。
7. 不把“短期可用”扩写成重型平台化重构。
8. 不为了未来能力预演，引入当前无法稳定验收的重型链路。

所有改动都必须能明确归类到：

1. `主链路可用性`
2. `执行能力收口`
3. `本地缓存与成本优化`
4. `记忆与知识沉淀`
5. `专业能力接口预留`
6. `本地小模型兜底预研`

## 4. 本阶段实现原则

### 4.1 先可用，再扩展

1. 先把核心能力做稳。
2. 先把真实使用路径做顺。
3. 先把失败态和恢复态做清楚。
4. 后续能力只做预留，不抢占当前主资源。

### 4.2 先主链路，再体验增强

1. 先保证对话、读写、执行、收口成立。
2. 再增强缓存、解释性、沉淀质量。
3. 前端体验优化只能服务主链路，不单独发散。

### 4.3 先低成本增强，再重能力设想

1. 先做本地缓存与复用。
2. 先做结构化记忆和知识复用。
3. 先做 skill 接口预留。
4. 暂不直接实现高成本双智能体与全局观察。

### 4.4 先留证，再判断完成

1. 所有关键能力都必须能真实跑通。
2. 所有关键能力都必须有构建或样本留证。
3. 不允许只靠“感觉能用了”作为完成依据。

## 6. 推荐实现顺序

第二阶段按以下顺序推进：

1. 收口在线模型对话主链路。
2. 收口文件读写与命令执行主链路。
3. 收口任务分析、验证、失败恢复与结果解释。
4. 接入本地缓存，减少重复思考与重复回答。
5. 继续补厚记忆与知识沉淀质量。
6. 预留 skill / 本地小模型后续接口。

## 8. 文件级任务清单

### 8.1 工作包 A：在线模型对话主链路

目标：

1. 在线模型接入后，用户可以稳定发起对话。
2. `project_answer / context_answer / explain` 等默认路径更稳定。
3. 主回答失败时存在受控恢复，而不是直接中断。

要求：

1. 模型输出必须继续禁止工具调用伪指令、XML、标签式动作。
2. 对话回答必须与当前项目上下文、会话上下文、记忆与知识命中结果挂钩。
3. 事件与日志必须能看到主要回答路径和恢复路径。

完成标准：

1. 用户正常提问时，能得到稳定中文回答。
2. 项目说明类问题优先落到 `project_answer`。
3. 会话续答类问题优先落到 `context_answer`。
4. 回答异常时，存在单次受控恢复。

### 8.2 工作包 B：文件读写与执行能力

目标：

1. 智能体能在当前工作区内真实读文件。
2. 智能体能在当前工作区内真实写文件。
3. 智能体能执行明确、受控的本地命令。

要求：

1. 路径必须继续受工作区边界约束。
2. 高风险动作必须继续经过风险确认。
3. 写入与执行结果必须在事件流中可追踪。

完成标准：

1. 文件读取结果真实可用。
2. 文件写入结果真实可见。
3. 命令执行结果真实返回。
4. 风险确认链路不回退。

### 8.3 工作包 C：思维链路与结果解释

目标：

1. 让用户能理解系统为什么这样回答、为什么这样执行。
2. 让前端详情区不再只依赖黑盒 metadata。

要求：

1. 事件正式字段应逐步带出：
   1. 上下文摘要
   2. 记忆摘要
   3. 知识摘要
   4. 思考摘要
   5. 验证摘要
2. 失败时应能给出明确错误分类和下一步线索。

完成标准：

1. 对话主链路具备基本可解释性。
2. 日志详情区可直接展示关键上下文字段。
3. 用户能区分命中缓存、正常回答、恢复路径、失败路径。

### 8.4 工作包 D：本地缓存与成本优化

目标：

1. 降低重复思考。
2. 降低重复检索。
3. 降低重复解释。

要求：

1. 先做轻量本地缓存，不引入重型缓存系统。
2. 缓存必须可留痕。
3. 缓存命中与未命中原因必须可解释。
4. 缓存不能突破现有工作区和会话边界。

完成标准：

1. 至少一条主回答路径支持本地缓存。
2. 事件中能看到缓存状态。
3. 缓存命中后确实减少重复生成。

### 8.5 工作包 E：记忆与知识沉淀继续增强

目标：

1. 用户偏好继续沉淀。
2. 失败教训继续沉淀。
3. 知识召回继续保持 `README / 06-development 优先、07-test 降权`。
4. 低质量历史数据继续治理。

要求：

1. 记忆写入应继续受质量约束。
2. 知识召回应继续可解释。
3. 不允许 runtime 自己生成的大量低质量内容反向污染长期层。

完成标准：

1. 偏好沉淀、失败教训沉淀继续增长。
2. 召回质量不回退。
3. 历史脏数据治理继续可留证。

### 8.6 工作包 F：专业能力与本地兜底预留

目标：

1. 给后续 CTF / 网络安全 skill 留清晰接口位置。
2. 给本地小模型兜底留接口位置。

要求：

1. 只做接口和边界预留，不做重型实现。
2. 不引入当前无法稳定验收的新执行域。
3. 预留位置必须能与现有工具、记忆、知识、设置体系自然接上。

完成标准：

1. 架构上明确有落点。
2. 不破坏当前主链路。
3. 后续扩展不会重新推翻现有结构。

## 9. 当前阶段明确不做

以下内容本阶段不允许抢占主资源：

1. 双智能体正式实现
2. 全局语言智能体正式实现
3. 语音输入输出主链路
4. 浏览器悬浮窗正式实现
5. 浏览器选中文本联动正式实现
6. 桌面 GUI 理解正式实现
7. 大规模多专业 skill 同时铺开
8. 前端整站重新设计

## 10. 阶段完成标准

满足以下条件，才算第二阶段短期可用目标达成：

1. 接入在线模型后，系统可以稳定进行自然语言对话。
2. 系统可以在当前工作区内真实读写文件。
3. 系统可以执行明确、受控的本地命令。
4. 用户可以通过日志与事件看懂主要执行链路。
5. 至少一条主回答路径已经具备本地缓存与留痕。
6. 记忆与知识沉淀继续增强，且没有明显回退。
7. 主链路构建与联调验证通过。

以下情况直接判不通过：

1. 在线模型接入后仍大量落入无效回答。
2. 读写或执行能力不稳定。
3. 缓存只存在代码里，用户侧完全不可感知。
4. 记忆与知识召回质量明显回退。
5. 为了未来设想破坏了当前首版可用性。

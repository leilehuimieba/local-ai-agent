## 1. 一句话定位

本项目第二阶段的正式定位是：

> `一个面向长期学习、长期成长、长期积累、可持续专精的本地个人智能体平台。`

它不是只做一次性问答，也不是只做一个聊天壳层。

它的核心目标是：

1. 接入在线大模型后，成为一个真正可执行、可沉淀、可持续使用的本地智能体。
2. 长期服务用户学习、开发、知识积累、专业成长。
3. 为后续 skill、专业能力、双智能体和本地小模型兜底预留清晰接口。

---

## 2. 第二阶段总原则

第二阶段必须同时满足：

1. 先把“能用且舒服”做出来，再做重型想象能力。
2. 优先增强当前单智能体主链路，不推翻已冻结首版架构。
3. 优先做对用户长期价值最大的能力，不做为了“看起来先进”的空扩展。
4. 优先做低风险、可复用、可验收、可逐版迭代的落地点。

当前短期最重要的判断固定为：

1. 先把接入 API 后的对话、读写、执行、思维链路、本地缓存做稳。
2. 先把记忆、知识、偏好、失败教训做成长期资产。
3. 先让系统达到“可以逐步代替用户在 VS Code 里频繁找 AI”的程度。

## 3. 四栏清单

| 主题 | 已确认 | 后置 | 暂不做 | 风险点 |
|---|---|---|---|---|
| 产品总定位 | 本地长期个人成长型智能体平台。强调长期使用、本地沉淀、持续进化。 | 后续可延展为统一本地智能体平台。 | 不再定义成单纯聊天工具或临时 demo。 | 如果定位漂移回“普通聊天页”，后续架构会继续发散。 |
| 核心价值 | 本地知识库、个性化规划、专业能力接入、长期记忆与陪伴感。 | 后续再强化人格层与长期关系层。 | 不做纯情绪陪聊产品。 | 价值点过多会导致短期开发分散。 |
| 短期主目标 | 接入在线模型后，可对话、可读文件、可写文件、可执行操作、可持续推进任务。 | 后续再做更深自治与复杂工作流。 | 不把短期目标扩成全自动多智能体平台。 | 如果短期目标定义不清，会持续陷入“什么都想做”。 |
| 思维链路 | 系统必须具备真实的任务分析、执行路径、结果验证与可见链路。 | 后续再增强多层 reasoning、可解释评分、链路压缩策略。 | 不做研究型复杂评分系统。 | 链路过重会增加 token 和复杂度。 |
| 本地缓存 | 必须有本地缓存，避免重复思考、重复检索、重复解释。 | 后续再扩展到知识摘要缓存、工具结果缓存、项目上下文缓存治理。 | 不做重型分布式缓存或复杂缓存中台。 | 缓存若没有命中解释与失效策略，会制造错误答案复用。 |
| 本地知识库 | 明确作为核心优势。学习资料、网页总结、长期知识都要沉淀在本地。 | 后续再做更细知识分层、知识治理、知识质量评分。 | 不做只会简单向量检索的空壳知识库。 | 知识库如果持续写入低质量内容，会拖累召回质量。 |
| 记忆系统 | 必须保留并继续增强用户偏好、失败教训、项目规则、工作流模式等长期记忆。 | 后续再做更强版本化、可解释化、人格层记忆。 | 不做无边界的全量会话原文堆积。 | 记忆脏数据会污染主链路判断。 |
| 个性化规划 | 明确是产品优势之一。可用于学习规划、考研规划、阶段目标调整。 | 后续再接提醒、节奏纠偏、习惯跟踪。 | 暂不做完整日历系统或重型习惯打卡产品。 | 如果没有稳定记忆与上下文，规划容易退化为模板话术。 |
| 学习场景 | 明确服务用户学习电脑软件、浏览器内容、资料查阅、知识总结。 | 后续再做浏览器联动、选中文本发送、悬浮对话窗。 | 当前不重做成浏览器插件主产品。 | 浏览器观察与内容采集会带来注入和隐私风险。 |
| 专业化方向 | 明确预留“可专精”能力口子，首个明确方向是网络安全 / CTF。 | 后续逐步接入脚本、工具、题型分析、技能推荐、解题复盘。 | 不在当前阶段一次铺满大量专业方向。 | skill 接口不稳会导致后续能力接入混乱。 |
| skill 体系 | 必须预留 skill / script / tool 接入口，让系统能向专业方向成长。 | 后续再做 skill marketplace、分类管理、权限边界。 | 当前不做复杂插件市场和生态系统。 | skill 执行若缺权限边界，会带来安全风险。 |
| 本地小模型兜底 | 已确认值得做，目标是防止系统在无 API 时瘫痪。 | 后续再设计双模型分层：强思考模型 + 本地执行兜底模型。 | 当前不要求本地小模型具备强推理和长思考能力。 | 本地模型能力不足时，容易误判其能承担的任务范围。 |
| 本地小模型能力范围 | 至少适合做日志查看、配置切换、简单规则执行、知识写入、调用已有 skill。 | 后续再评估是否承担部分学习整理或轻量总结。 | 不让本地小模型承担复杂浏览器理解或深度规划。 | 若让本地模型承担过重任务，会导致体验明显变差。 |
| 双智能体设想 | 已记录为长期重要方向。包括执行智能体与全局语言智能体。 | 后续再做全局观察、语音理解、任务分发、结果汇总。 | 当前不进入正式主链路开发。 | token 消耗、全局监听、安全边界、调度复杂度都很高。 |
| 全局语言智能体 | 已确认这个想法有价值，尤其适合降低打字负担和统一全局理解。 | 后续再论证是否接语音、浏览器、桌面观察。 | 当前不直接实现。 | 一旦仓促上马会显著拉高成本和系统复杂度。 |
| 前端方向 | 当前前端以“够用、清楚、可继续优化”为目标，不再回头大改产品结构。 | 后续再专项优化前端视觉、层级、对话体验。 | 当前不为了视觉重构阻断智能体主能力。 | 前端若继续反复改，会拖慢主链路能力建设。 |
| 工作台与对话关系 | 已确认工作台保留，但内容要克制；对话页应独立承载主要聊天体验。 | 后续再优化工作台与日志、调查、对话之间的节奏。 | 不再把所有内容塞回首页主线程。 | 页面角色不清会继续导致体验拥挤。 |
| 日志入口 | 已确认独立日志页是唯一稳定日志入口。 | 后续再增强筛选、详情联动、快照展示。 | 不恢复首页和日志页双重主入口冲突。 | 如果日志入口分裂，用户会持续困惑。 |
| Token 优化 | 已确认是核心竞争点之一。应通过缓存、结构化记忆、按需召回、skill 替代长推理来优化。 | 后续再做更系统的上下文预算与成本策略。 | 不单靠“缩短 prompt”当成全部优化方案。 | 若优化只停留在表层，会很难形成和同类产品的差异。 |
| 与同类产品差异 | 明确不只强调“本地”，而是强调“本地沉淀 + 个性化规划 + 专业专精 + 记忆成长”。 | 后续再形成更明确的对外叙事和产品包装。 | 不把产品描述成模糊的“另一个 AI 壳”。 | 差异点不清会削弱后续产品竞争力。 |
| 安全边界 | 已确认要重点关注 prompt injection、联网攻击面、权限边界、隐私边界。 | 后续做专项安全策略和授权治理。 | 当前不开放无边界全局观察与任意外部执行。 | 这类风险是真正的产品级阻塞项，不能轻视。 |
| 统一平台方向 | 已确认长期要成为统一管理本地 AI、知识、记忆、skill、工具的平台。 | 后续再接更多专业智能体或外部 AI 平台协调。 | 当前不做“大一统控制中心”式重平台化重构。 | 平台化做太早会压垮当前可用性建设。 |
| 智能体自我完善能力 | 已确认这是重要方向，希望后续它能借助现有能力继续完善自己。 | 后续再逐步让它生成、修改、调用更多内部能力。 | 当前不直接放开高自治自修改。 | 如果主链路与验证链不稳，自我完善容易失控。 |

## 4. 第二阶段正式开发重点

第二阶段只优先做下面这些：

1. `在线模型接入后的主链路可用性`
   目标是稳住对话、读写、执行、验证、收口。

2. `本地缓存与上下文复用`
   目标是减少重复思考，降低 token 消耗，提高续答稳定性。

3. `记忆与知识沉淀继续补厚`
   目标是让偏好、经验、失败教训、知识资产继续变成长期价值。

4. `skill / 专业能力接入口预留`
   目标是为 CTF / 网络安全等专精方向预留清晰落点。

5. `本地小模型兜底方案预研`
   目标是先确定能力边界和接口，不急着把重逻辑压给本地模型。

---

## 5. 当前明确不进入第二阶段主开发的内容

以下内容保留设想，但不应在当前阶段抢占主资源：

1. 双智能体正式实现
2. 语音输入输出主链路
3. 浏览器全局观察主链路
4. 桌面 GUI 全局理解与自动操作主链路
5. 大规模插件市场或技能生态系统
6. 为了视觉效果继续推翻前端主结构

---

## 6. 当前结论

第二阶段不是“继续发散想法”的阶段，而是：

1. 把产品重新明确成长期个人成长型本地智能体平台。
2. 把短期主目标收敛到“接入 API 后足够可用、可执行、可沉淀、可长期用”。
3. 把未来重要方向按后置与风险项先记录清楚，避免开发顺序失控。

后续若继续推进第二阶段，应优先基于本文继续补：

1. 第二阶段开发任务书
2. 第二阶段验收文档
3. 对应的代码实现与逐项留证

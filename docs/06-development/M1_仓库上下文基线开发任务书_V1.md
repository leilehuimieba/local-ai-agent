## 1. 任务书定位

本任务书只服务于 M1。

M1 的唯一目标是：

> 给当前 Rust runtime 建立一套轻量、稳定、可降级、可展示的仓库上下文基线。

这里的“仓库上下文基线”不是全文索引系统，不是语义检索系统，也不是完整项目分析器。

它只回答一个问题：

> 每次用户发起任务时，runtime 能否以低成本知道“我当前在哪个工作区、这个仓库大概处于什么状态、有哪些高价值项目说明文件”。

## 2. M1 的战略意义

### 2.1 为什么 M1 必须先做

当前 runtime 还主要依赖：

1. `user_input`
2. `session summary`
3. 少量 `context_hints`

这会导致 planner 的问题是：

1. 对当前仓库状态缺少感知
2. 对 Git 分支和工作树变化无感知
3. 对项目说明文件无系统性加载
4. 很容易退化成“只对用户字面文本做反应”

M1 的作用是先把这个底座补上。

### 2.2 为什么 M1 要轻量实现

本项目后端主路线之所以坚持 Go + Rust，是因为我们明确追求：

1. 更低内存占用
2. 更高并发能力
3. 更快响应速度
4. 更适合长期本地运行

因此 M1 的实现必须体现这种路线。

M1 明确不采用：

1. Python 运行时过渡层
2. 重型全仓索引器
3. 常驻大对象缓存
4. 每轮全量递归扫描

M1 应采用：

1. Rust 原生实现
2. 快照式采集
3. 只读取最小高价值信息
4. 失败可降级

## 3. M1 范围边界

### 3.1 M1 必须完成的内容

M1 必须至少完成以下能力：

1. 识别当前工作区根路径
2. 判断当前工作区是否位于 Git 仓库中
3. 获取当前分支
4. 获取默认分支或主分支候选
5. 判断工作树是否存在未提交修改
6. 获取最近若干条提交摘要
7. 查找项目说明文件
8. 对说明文件做轻量摘要
9. 将以上信息装配为结构化 `RepoContextSnapshot`
10. 将最小 repo context 摘要回流到事件 metadata
11. 让 planner 可以消费 repo context 摘要
12. 让前端右侧上下文栏具备显示最小 repo context 的能力

### 3.2 M1 明确不做的内容

M1 不做：

1. 语义级项目分析
2. LSP 集成
3. 全仓代码依赖图
4. AST 分析
5. 文件 embedding
6. 向量数据库
7. 多工作区联合检索
8. 持续后台监控索引
9. 多代理上下文协同

### 3.3 M1 与 M2 的边界

M1 只负责：

1. 采集仓库上下文
2. 形成结构化快照

M2 才负责：

1. Context Envelope
2. Prompt 分层
3. 统一上下文装配

实现对话不得在 M1 中提前做 M2 的完整逻辑，只能为 M2 预留稳定接口。

## 4. 技术路线要求

### 4.1 正式技术路线

M1 后端正式实现要求如下：

1. Rust 负责 repo context 采集与装配
2. Go 只负责 request bridge 和事件回流
3. Frontend 只负责展示和消费 metadata

### 4.2 关于技术选型的进一步说明

你刚刚补充的意思已经在这里固化为正式约束：

1. 后端核心、长驻、主链路必须大部分由 Go 和 Rust 承担
2. 原因是 Go 和 Rust 在本项目目标下更适合低内存、快响应、高并发
3. 如果未来讨论引入其他语言，只能讨论非热点、非长驻、非核心辅助工具
4. 在没有专门批准前，实现对话一律不得引入 Python、Node 或其他运行时作为 M1 正式依赖

### 4.3 如果要讨论“更合适技术”，允许讨论的边界

可以讨论，但必须满足以下前提：

1. 不能破坏 Go + Rust 主线
2. 不能增加常驻内存负担
3. 不能引入新的核心运行时依赖
4. 不能让部署复杂度明显上升

在当前 M1 阶段，我的明确建议仍然是：

1. 不引入新的技术分支
2. 不做任何语言层面的摇摆
3. 先把 Rust 仓库上下文能力做稳

## 6. 数据结构规范

M1 必须优先产出结构化对象，而不是把 repo context 当成长字符串。

### 6.1 `RepoContextSnapshot`

建议字段如下：

1. `workspace_root: String`
2. `repo_root: Option<String>`
3. `git_available: bool`
4. `git_snapshot: Option<GitSnapshot>`
5. `doc_summaries: Vec<WorkspaceDocSummary>`
6. `warnings: Vec<String>`
7. `collected_at: String`

设计要求：

1. `repo_root` 允许为空
2. `git_snapshot` 允许为空
3. `warnings` 用于存放降级信息
4. `collected_at` 便于日志与调试

### 6.2 `GitSnapshot`

建议字段如下：

1. `current_branch: Option<String>`
2. `default_branch: Option<String>`
3. `is_dirty: bool`
4. `recent_commits: Vec<GitCommitSummary>`

### 6.3 `GitCommitSummary`

建议字段如下：

1. `commit_id: String`
2. `short_message: String`
3. `author: Option<String>`
4. `timestamp: Option<String>`

注意：

1. 不要求保留完整 commit message
2. 不要求保留 diff
3. 不要求保留过多历史

### 6.4 `WorkspaceDocSummary`

建议字段如下：

1. `path: String`
2. `kind: String`
3. `exists: bool`
4. `summary: String`
5. `truncated: bool`

`kind` 建议允许以下取值：

1. `readme`
2. `project_instruction`
3. `agent_instruction`
4. `workspace_note`
5. `other`

### 6.5 `RepoContextLoadResult`

如实现需要，可定义中间对象：

1. `snapshot: RepoContextSnapshot`
2. `degraded: bool`
3. `error_count: u32`

## 8. Git 信息采集规则

### 8.1 必须获取的 Git 信息

M1 至少获取：

1. 当前分支名
2. 默认分支候选
3. 当前工作树是否 dirty
4. 最近 3 到 5 条提交摘要

### 8.2 推荐采集方法

建议通过 `std::process::Command` 调用 Git。

推荐思路：

1. 先判断 `git` 是否可用
2. 再判断当前路径是否位于仓库中
3. 再获取 branch 与 status
4. 最后获取最近提交

### 8.3 降级要求

若以下场景发生，M1 必须降级而不是失败：

1. 机器未安装 Git
2. 当前目录不是 Git 仓库
3. 某条 Git 命令执行失败

降级后至少应满足：

1. `git_available` 正确
2. `warnings` 有说明
3. 其他可采集信息继续生效

## 9. 说明文件摘要规则

### 9.1 摘要目标

摘要的目的不是全文保留，而是让 runtime 知道这些文件大概在说什么。

### 9.2 推荐实现方法

建议复用现有：

1. `crates/runtime-core/src/text.rs`

建议方法：

1. 读取前若干 KB 或前若干行
2. 做统一截断
3. 形成短摘要

### 9.3 摘要长度要求

建议每个说明文件摘要控制在：

1. 80 到 240 字符之间

推荐上限：

1. 不超过 300 字符

### 9.4 不允许的做法

不允许：

1. 把 README 原文直接放进事件
2. 把多份说明文件完整塞回 session summary
3. 在 M1 中做复杂 NLP 摘要系统

## 10. 运行时接入规范

### 10.1 在主循环中的接入位置

M1 的 repo context 采集建议接在：

1. `RunRequest` 进入 runtime 后
2. `Analyze` 阶段之前或 `Analyze` 阶段内部最前面

不建议接在：

1. Execute 阶段之后
2. Finish 阶段才补采集

### 10.2 `lib.rs` 接入要求

`lib.rs` 中应体现明确顺序：

1. 读取 session context
2. 采集 repo context
3. 形成分析输入
4. 进入 planner

要求：

1. 顺序清晰
2. 异常可降级
3. 不将 repo context 逻辑直接扩散到多个地方

### 10.3 `planner.rs` 接入要求

`planner.rs` 在 M1 阶段不要求完全重构，但至少要满足：

1. 可以读取 repo context 摘要
2. 可以在分析说明中体现“基于当前仓库状态”
3. 不再完全只基于输入前缀判断

注意：

1. M1 不要求 planner 变成大模型驱动
2. M1 只要求 planner 开始拥有仓库环境意识

## 12. 性能与资源约束

M1 是第一批正式运行时增强，必须体现性能路线。

### 12.1 内存约束

实现对话必须尽量做到：

1. 不长期缓存大块 README 原文
2. 不把多份文档全文保存在内存结构里
3. 不在事件和 session 中重复保存同一批信息

### 12.2 延迟约束

M1 的采集不应显著拖慢单次 run。

建议目标：

1. 普通本地仓库下，repo context 采集应尽量控制在低百毫秒级

如果实际无法做到，至少应：

1. 有清晰解释
2. 有超时或截断策略

### 12.3 并发约束

M1 不要求复杂并发，但要求：

1. 不引入新的常驻后台任务
2. 不让一次 repo context 采集阻塞整个系统过久
3. 失败和慢查询能安全降级

## 13. 实施步骤建议

实现对话建议严格按以下步骤推进。

### Step 1：定义结构与模块边界

完成内容：

1. 新建 `repo_context.rs`
2. 定义核心结构体
3. 定义模块对外函数

输出标准：

1. 编译通过
2. 不接业务逻辑也能独立存在

### Step 2：实现 Git 可用性与仓库识别

完成内容：

1. 判断 Git 是否存在
2. 判断当前路径是否在仓库中
3. 获取 repo root

输出标准：

1. 非 Git 目录能降级
2. 错误不会直接中断主链路

### Step 3：实现 Git 状态采集

完成内容：

1. 获取当前分支
2. 获取默认分支候选
3. 获取 dirty 状态
4. 获取最近提交摘要

输出标准：

1. 结构稳定
2. 字段清晰

### Step 4：实现说明文件发现与摘要

完成内容：

1. 固定白名单查找说明文件
2. 对命中文件做轻量摘要

输出标准：

1. 不做深度全仓扫描
2. 结果简洁可显示

### Step 5：接入 runtime 主链路

完成内容：

1. 在 `lib.rs` 中接入 repo context
2. 在 `planner.rs` 中开始消费摘要
3. 事件 metadata 补充 repo context 信息

输出标准：

1. 事件流可见
2. planner 已开始具备仓库环境意识

### Step 6：接入前端最小展示

完成内容：

1. 合同同步
2. 右侧栏展示最小 repo context

输出标准：

1. 展示不过量
2. 不破坏现有 V2 结构

## 15. 验收标准

### 15.1 M1 通过条件

以下条件全部满足，M1 才算通过：

1. Rust runtime 已存在正式 `repo_context` 模块
2. 主链路 run 前能生成 `RepoContextSnapshot`
3. planner 能消费最小 repo context 摘要
4. 事件 metadata 能回流最小 repo context 信息
5. 前端右侧栏能显示最小 repo context
6. Git 不可用和非仓库目录场景可降级
7. 不引入 Python 或其他额外运行时依赖
8. 不破坏当前 V2 前端形态

### 15.2 M1 不通过条件

出现以下任一情况，M1 不通过：

1. repo context 逻辑主要散落在 `lib.rs` 中，没有形成独立模块
2. Git 异常会让 run 整体失败
3. README 等说明文件全文被塞进事件或 session
4. 为实现 M1 引入 Python 或新的正式后端技术栈
5. 前端为展示 repo context 被迫改变当前 V2 主结构

4. 顺手把说明文件做成全文知识库
5. 顺手增加新的页面与导航
6. 继续引入 Python 作为任何正式后端组成部分

## 18. 我对 M1 的当前建议

如果你问我 M1 有没有更优秀、更合适的建议，我当前给出的判断是：

1. 对 M1 来说，没有必要引入新的语言或新的大技术栈
2. 直接用 Rust 把 repo context 做成快照模块，是目前最符合你目标的做法
3. Go 只负责桥接和控制面，不要把 repo context 采集放到 Go 里主导
4. 前端只做最小展示，不做结构变化

也就是说：

> M1 最好的方案不是“更复杂”，而是“更纯净地用 Rust 做轻量仓库上下文快照，再用 Go 回流、前端展示”。

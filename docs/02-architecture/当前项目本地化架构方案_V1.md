## 1. 文档目的

本文只解决 5 个问题：

1. 旧项目 `local-agent-control-hub` 的哪些架构经验值得保留。
2. ChatGPT、Claude Desktop / Claude Code、Cursor、AnythingLLM / Open WebUI / LM Studio 的哪些能力组织方式值得学习。
3. 这些经验进入当前项目后，应该怎样做本地化处理，而不是生搬硬套。
4. 当前项目应该固定成什么样的正式架构形态，才能兼顾主链路、模块化和长期扩展。
5. 哪些接口和边界现在就要定下来，避免后期反复返工。

本文不是竞品分析集合，也不是具体编码任务书。

本文的作用是：

> 把“应该学什么、不能学什么、最终要落成什么形态”一次性定清，作为当前项目后续几轮架构优化的总依据。

## 2. 当前正式判断

基于现有项目状态、旧项目经验和竞品对比，当前正式判断固定为：

1. 当前项目与旧项目 `local-agent-control-hub` 高度同源，但不是旧项目的继续扩建，而是一次 `主链路化 + 本地化 + 去历史包袱` 的重构。
2. 当前项目已经具备 `产品导向模块化`，但还没有达到 `高程度可插拔模块化`。
3. 当前项目最值得避免的风险，不是“功能太少”，而是未来新增能力时再次回到 `中心文件持续膨胀、多层重复定义、多处同步改动` 的老路。
4. 当前项目应学习成熟竞品的 `能力组织模式` 与 `接口分层方式`，但不能照搬它们的产品入口、云侧依赖和重型扩展体系。

一句话结论：

> 当前项目应被固定为一个 `本地优先、单主线智能体、控制台化交互、能力分层注册、连接器可扩展、记忆与知识可沉淀` 的产品架构，而不是继续朝“全能控制中台”或“重型插件平台”发展。

## 3. 当前项目的正式产品形态

当前项目的正式产品形态固定为：

1. `本地优先`
2. `单主线智能体`
3. `工作台型产品界面`
4. `Go 控制面 + Rust 运行时 + React 前端`
5. `记忆、知识、工作区、连接器共同服务主聊天链路`

这里的“本地优先”具体含义固定为：

1. 用户的工作区、偏好、日志、知识、记忆默认留在本地。
2. 高价值沉淀默认优先进入本地数据库、本地知识源或本地文件。
3. 外部模型、外部连接和未来扩展服务都属于可接入能力，不是产品唯一依赖中心。

这里的“单主线智能体”具体含义固定为：

1. 所有正式能力都服务同一条主任务链路。
2. 不重新回到旧式多 Agent 编排中台。
3. 即使未来存在后台作业、连接器同步或研究型任务，也应作为主链路的扩展能力，而不是平行主系统。

## 4. 向旧项目学习什么

参考项目：

1. `D:\newwork\local-agent-control-hub`

旧项目最值得保留的不是具体代码，而是以下 6 类思想资产：

1. `控制面独立存在`
2. `会话 / 事件 / 日志是正式系统概念`
3. `工具与能力需要注册层`
4. `能力实现与主运行时要有边界`
5. `风险 / 审批 / 策略不能只是附属功能`
6. `前端应该是工作台，而不是单纯调用后端的壳`

旧项目最不应该直接带回来的部分固定为：

1. 过宽的中台面
2. 历史累积的平行功能面
3. 研究型微策略堆叠
4. 复杂多 Agent 残留
5. 重型但未完全收口的扩展面

因此，对旧项目的本地化处理原则固定为：

1. `继承边界`
2. `重写实现`
3. `缩小产品面`
4. `保留主价值`
5. `去除历史膨胀路径`

## 5. 向竞品学习什么

### 5.1 ChatGPT

值得学习：

1. `项目容器` 视角
2. `连接器` 作为正式产品概念
3. `异步研究任务` 与主对话解耦
4. `记忆边界` 和工作上下文边界

本地化处理后应变成：

1. 工作区、会话、知识沉淀共同形成 `本地项目容器`
2. 外部数据源通过 `连接槽位` 接入，而不是直接散落在设置页或执行层
3. 未来深度研究、长时任务进入 `jobs` 骨架，而不是侵入主聊天运行时

不学习：

1. 云侧黑盒状态
2. 过多隐藏式上下文注入
3. 强依赖远程连接器生态

### 5.2 Claude Desktop / Claude Code

值得学习：

1. `MCP / 外部能力接入` 思想
2. `分层设置`
3. `权限与确认边界`
4. `hooks / subagents / modes` 的扩展意识

本地化处理后应变成：

1. 当前项目先保留 `连接器槽位 + 能力注册 + 模式矩阵`
2. hooks、subagents 先只预留合同，不进入当前主产品复杂度
3. MCP 只作为可接入能力来源之一，不强迫所有能力全走 MCP

不学习：

1. 命令行优先的产品入口
2. 为编码单场景过度优化的交互形态
3. 直接照搬其配置结构

### 5.3 Cursor

值得学习：

1. `代码库上下文` 作为正式能力层
2. `规则系统` 与模式矩阵
3. `后台任务 / 后台 agent` 的产品方向
4. `工作流围绕当前项目组织`

本地化处理后应变成：

1. 当前项目中的工作区、知识、记忆一起构成 `项目上下文层`
2. 模式与策略进入运行时和网关的正式合同
3. 后台任务先通过 `jobs` 接口预留，而不是现在就做完整后台 agent 系统

不学习：

1. 强绑定 IDE 的入口方式
2. 大量依赖编辑器宿主能力的耦合设计

### 5.4 AnythingLLM / Open WebUI / LM Studio

值得学习：

1. `本地优先` 的产品心智
2. `模型服务抽象`
3. `连接器 / 知识源抽象`
4. `轻扩展与重扩展分层`

本地化处理后应变成：

1. 模型、知识源、连接器全部通过合同化结构暴露
2. 首轮只做轻扩展接口，不直接进入插件市场或动态脚本插件
3. 工作区、本地 SQLite、思源等本地能力优先成为一等公民

不学习：

1. 先做一个很重的插件广场
2. 让产品主链路被“接什么插件”主导

## 6. 本地化后的正式架构目标

当前项目的正式架构目标固定为：

1. 新增一个能力时，尽量只新增一个能力模块和少量注册信息。
2. 新增一个连接器时，尽量只在连接器层落地，而不是同时侵入运行时、网关、前端多个中心文件。
3. 主聊天链路、连接器管理、记忆沉淀、日志诊断共享统一元信息模型。
4. 工作区、本地知识源、思源、未来外部连接都能进入同一套接入框架。
5. 未来存在后台作业、深度研究、自动同步时，可以复用现有会话、能力、连接器合同，而不必推翻重来。

不追求的目标固定为：

1. 当前不做插件市场
2. 当前不做重型多智能体编排
3. 当前不让所有能力都动态脚本化
4. 当前不为了“无限未来”先做大量抽象层

## 7. 正式架构分层

当前项目的正式架构固定为 7 层。

### 7.1 Shell 与产品交互层

对应：

1. `frontend/`

职责：

1. 承接用户输入
2. 展示聊天、日志、设置、工作台
3. 消费会话、事件、能力摘要、连接器摘要
4. 承接确认和模式切换

不负责：

1. 智能体裁决
2. 风险策略判断
3. 直接执行业务能力

### 7.2 Gateway 控制面层

对应：

1. `gateway/`

职责：

1. 提供统一 HTTP API
2. 管理设置、会话、事件、日志、确认、配置
3. 桥接 Rust 运行时
4. 向前端暴露能力与连接器摘要
5. 为未来 jobs、同步任务、诊断面提供统一入口

不负责：

1. 智能推理主循环
2. 复杂 planner 策略
3. 大量工具实现细节

### 7.3 Runtime 主链路层

对应：

1. `crates/runtime-core`
2. `crates/runtime-host`

职责：

1. 分析任务
2. 规划能力
3. 分发执行
4. 处理观察结果
5. 验证结果
6. 决定 finish / confirm / recover

不负责：

1. Web API 托管
2. 页面消费组织
3. 前端配置交互

### 7.4 能力注册层

建议落点：

1. `crates/runtime-core/src/capabilities/`

职责：

1. 提供统一能力元信息
2. 管理能力 ID、域、风险、输入输出、验证策略、连接槽位
3. 供运行时、网关、前端共享

这是当前项目最关键的收口层。

### 7.5 能力执行层

建议落点：

1. `crates/runtime-core/src/executors/`

职责：

1. 按域承接具体执行逻辑
2. 减少中心执行文件膨胀
3. 为未来同域扩展提供稳定落点

建议域固定为：

1. `workspace`
2. `memory`
3. `knowledge`
4. `project`
5. `session`
6. `siyuan`

### 7.6 连接与接入层

建议落点：

1. `crates/runtime-core/src/connectors/`

职责：

1. 抽象外部连接、本地知识源、本地笔记、本地文件入口
2. 给每种连接分配稳定槽位
3. 为能力提供边界，而不是让连接逻辑散落在执行链路里

### 7.7 存储与沉淀层

当前已涉及或未来应统一收口的对象：

1. 设置
2. 事件日志
3. 运行日志
4. 长期记忆
5. 知识条目
6. 工作记忆
7. 后续 jobs 元数据

职责：

1. 本地优先持久化
2. 为诊断、沉淀、复盘、长期复用提供基础

## 8. 当前项目一次性定死的 10 个架构决策

从现在开始，以下 10 个决策作为正式固定口径：

1. `Go + Rust + React` 是长期主架构，不再把 Python 拉回正式主链路。
2. 产品主形态是 `单主线智能体 + 工作台控制台`，不是多 Agent 中台。
3. Gateway 是唯一正式控制面入口，不在前端或运行时之外再造平行服务入口。
4. 能力必须收敛到统一元信息模型，不允许运行时、网关、前端长期各写一套。
5. 连接器必须有统一槽位模型，不允许继续散落在设置页或工具代码里临时硬编码。
6. 工作区、记忆、知识、连接器都服务同一条主任务链路。
7. 后台作业、深度研究、同步任务走 `jobs` 骨架，不直接侵入聊天主运行时。
8. 当前允许 `Rust 原生能力 + 本地能力 + 外部接入能力` 共存，不强制一切先全 MCP 化。
9. 前端应逐步按领域消费能力和连接器合同，而不是继续扩大 `App.tsx` 的总装逻辑。
10. 当前优先做 `统一注册与接口预留`，再做 `执行器拆分`，最后做更高级扩展。

## 9. 必须统一的核心接口

### 9.1 CapabilitySpec

当前必须固定统一能力描述结构。

至少包含：

1. `capability_id`
2. `display_name`
3. `domain`
4. `risk_level`
5. `input_schema`
6. `output_kind`
7. `side_effect_level`
8. `supports_modes`
9. `verification_policy`
10. `connector_slot`

用途：

1. 供运行时选择能力
2. 供执行器做分发
3. 供网关对外暴露
4. 供前端展示和日志复盘

### 9.2 ConnectorSlotSpec

当前必须固定统一连接槽位结构。

至少包含：

1. `slot_id`
2. `display_name`
3. `priority`
4. `status`
5. `scope`
6. `current_capabilities`
7. `boundary`
8. `next_step`

用途：

1. 固定连接器优先级
2. 固定外部接入挂位
3. 让设置页、工作台、未来连接管理页读取同一事实源

### 9.3 CapabilityExecutor

当前必须预留统一执行接口。

建议能力执行侧统一具备：

1. `describe`
2. `execute`
3. `verify`
4. `recover`

要求：

1. 主调度只依赖接口
2. 每个域只承接自己的执行逻辑

### 9.4 Jobs Contract

当前可以先做骨架，但必须预留。

建议至少预留：

1. `job_id`
2. `job_kind`
3. `trigger`
4. `status`
5. `related_session_id`
6. `capability_scope`
7. `result_summary`

用途：

1. 为后续深度研究
2. 为后续连接器同步
3. 为后续周期任务
4. 为后续后台作业

### 9.5 Gateway API

当前必须固定 3 类接口地位：

1. `/api/v1/settings`
2. `/api/v1/capabilities`
3. `/api/v1/connectors`

后续预留：

1. `/api/v1/jobs`

含义：

1. `settings` 负责首屏摘要和现有控制项
2. `capabilities` 负责能力清单
3. `connectors` 负责连接槽位与后续连接管理
4. `jobs` 负责异步任务和后台作业

## 10. 本地化后的能力来源模型

当前项目不应只有一种能力来源。

正式能力来源模型固定为：

1. `Runtime Native`
2. `Connector-backed`
3. `External Protocol-backed`

### 10.1 Runtime Native

代表：

1. 工作区读写
2. 本地记忆读写
3. 本地知识检索
4. 会话辅助能力

特点：

1. 最稳定
2. 最适合高频主链路
3. 最适合当前阶段优先建设

### 10.2 Connector-backed

代表：

1. 本地文件库
2. 思源
3. 本地笔记
4. 浏览器抓取入口
5. 个人管理类入口

特点：

1. 有稳定槽位
2. 可由本地优先连接器承接
3. 更适合做边界管理

### 10.3 External Protocol-backed

代表：

1. 未来 MCP
2. 未来远程能力
3. 未来标准化扩展协议

特点：

1. 不是当前必须主路径
2. 当前只需预留边界，不需先做重型平台

## 11. 前端的本地化消费方式

当前项目前端必须避免继续扩大“总装式消费”。

正式方向固定为：

1. `settings` 消费系统控制项和最小摘要
2. `capabilities` 消费能力清单
3. `connectors` 消费连接槽位
4. `logs` 逐步消费能力元信息，而不是硬编码工具名
5. `workspace` 消费项目上下文和核心工作摘要

前端后续推荐新增领域：

1. `frontend/src/capabilities/`
2. `frontend/src/connectors/`
3. `frontend/src/jobs/`

当前不强制引入重型全局状态方案。

原则固定为：

1. 先按领域拆 API 和消费层
2. 确实出现跨页面稳定共享状态，再考虑状态收束

---

## 12. 当前项目的本地项目容器定义

借鉴 ChatGPT Projects、Cursor codebase 和本地知识工具的经验后，当前项目应把“项目容器”固定为以下组合：

1. `workspace`
2. `session`
3. `memory`
4. `knowledge`
5. `connector scope`
6. `mode / policy`

这意味着：

1. 当前项目的工作单位不是单条聊天记录。
2. 当前项目的工作单位也不是单纯一个文件夹。
3. 当前项目真正的工作单位，是一个带有上下文、记忆、知识、连接边界和策略状态的 `本地项目容器`。

这个定义非常关键。

后续不论是能力扩展、连接器扩展、深度任务还是复盘诊断，都应围绕这个容器组织，而不是每个模块单独发明一套作用域模型。

## 13. 实施顺序

当前项目的实施顺序固定为 4 步。

### 13.1 第一阶段：统一能力与连接器元信息

必做项：

1. 正式落地 `CapabilitySpec`
2. 正式落地 `ConnectorSlotSpec`
3. 网关新增 `/api/v1/capabilities`
4. 网关新增 `/api/v1/connectors`
5. 前端新增 `capabilities` 与 `connectors` 基础消费层

### 13.2 第二阶段：拆执行器

必做项：

1. 按域拆 `execution.rs`
2. 主执行入口只保留分发
3. 新增能力优先进入域执行器，而不是中心文件

### 13.3 第三阶段：拆验证与恢复

必做项：

1. 把验证策略围绕能力收口
2. 把恢复建议围绕能力收口
3. 日志和事件能直接展示 `verification_policy`

### 13.4 第四阶段：补 jobs 骨架与前端领域消费

必做项：

1. 预留 jobs 合同
2. 为深度任务、连接同步、后台作业提供统一入口
3. 前端逐步降低 `App.tsx` 总装负担

---

## 14. 当前阶段不要做的事

从现在开始，以下方向都不应优先进入当前主仓库复杂度：

1. 直接做插件市场
2. 直接做动态脚本插件执行框架
3. 直接做多智能体总控平台

## 16. 本方案的正式结论

当前项目的正式本地化架构方案固定为：

1. 继承旧项目的控制面、会话、能力注册、风险边界思想。
2. 学习竞品的项目容器、连接器、能力组织、后台任务和本地优先理念。
3. 通过 Go + Rust + React 的长期主架构，把这些经验本地化为单主线智能体产品。
4. 当前先做统一元信息与接口预留，再做执行器拆分和 jobs 骨架，而不是直接进入重型插件系统。

一句话执行要求：

> 后续凡是会影响能力扩展、连接器接入、记忆沉淀、前端消费方式和后台任务方向的设计，默认都应先对齐本文，再进入实现。

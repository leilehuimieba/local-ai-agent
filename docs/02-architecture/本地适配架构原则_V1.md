## 1. 文档目的

本文只解决一件事：

> 把“复刻 Claude Code 思路，但必须做本地适配”的原则正式固化，避免后续实现跑偏。

后续 AI 在做运行时、记忆、工具协议、模型适配、事件合同、网关桥接时，必须先遵守本文，再进入具体任务书。

---

## 2. 当前正式口径

当前项目的目标不是：

1. 逐文件照搬 Claude Code
2. 把 Python 旧项目整体翻译成 Rust
3. 为了复刻而牺牲本地性能和可维护性

当前项目的目标是：

1. 复刻 Claude Code 的运行时骨架
2. 复刻成熟代码智能体的能力组织方式
3. 吸收旧项目已验证过的边界与经验
4. 在本地场景下用更适合的工程栈重新落地

一句话：

> 我们复刻的是架构模式，不是原始技术栈；进入本项目主链路的能力，必须先做本地适配。

---

## 3. 技术栈硬约束

### 3.1 后端主栈

后端必须以 `Rust + Go` 为主。

职责原则：

1. `Rust` 负责高频核心运行时
2. `Go` 负责服务编排、会话控制、事件桥接、对外接口
3. `React` 负责前端界面与事件消费

### 3.2 Python 使用边界

`Python` 不进入长期主后端链路。

允许存在的场景：

1. 一次性迁移脚本
2. 临时分析脚本
3. 辅助数据处理工具

禁止进入的场景：

1. Agent 主循环
2. 正式网关服务
3. 长期运行的后端核心模块
4. 记忆主存储链路
5. 正式工具调度主链路

---

## 4. 性能与工程目标

后续实现必须同时服务这 5 个目标：

1. `低常驻内存`
2. `高并发可控`
3. `高响应速度`
4. `本地优先`
5. `稳定可扩展`

如果某个实现方案更像参考项目，但会明显增加内存、阻塞、耦合、调试成本，则不采用。

---

## 5. Claude Code 复刻边界

### 5.1 必须复刻的部分

后续实现要优先吸收这些能力：

1. `Agent Loop / Query Engine`
2. `Tool Protocol / Tool Registry`
3. `Context Envelope / Prompt 分层`
4. `Permission / Approval Gate`
5. `Memory Digest / Structured Memory`
6. `Artifact / Compaction`
7. `Verify / Finish`
8. `Model Adapter`

### 5.2 只复刻思路，不照搬实现

以下内容只允许学习思路：

1. 原仓库的模块组织方式
2. 工具协议设计
3. 上下文压缩方法
4. 记忆摘要与回填方式
5. 会话与事件回流方式

以下内容不允许直接照搬：

1. 原项目的语言栈
2. 与本地项目边界不符的服务组织
3. 不适合本地部署的重型依赖
4. 与当前 Go / Rust 分层冲突的实现方式

## 6. 旧项目经验吸收边界

参考项目：`D:\newwork\local-agent-control-hub`

### 6.1 必须吸收的经验

1. `planner / execution` 分离
2. `session / event / memory` 正式落点
3. `tool registry` 独立存在
4. `policy / security broker` 边界意识
5. `worker` 与主运行时解耦

### 6.2 不再延续的部分

1. Python 超大主循环
2. 研究型微策略堆叠
3. 过重的 heuristics 体系
4. 超前进入复杂多智能体编排

---

## 7. Rust 与 Go 的正式分工

### 7.1 Rust 负责

以下能力默认由 `Rust` 承担：

1. Agent 主循环
2. Planner 串接
3. Tool Registry
4. Tool 执行协议
5. 上下文装配
6. 压缩与 Artifact 外置
7. 记忆读写路由
8. 验证链与完成判定
9. 模型适配核心

### 7.2 Go 负责

以下能力默认由 `Go` 承担：

1. API 网关
2. 会话管理
3. SSE / WebSocket 事件流
4. 确认请求存储与恢复
5. 配置服务
6. 运行状态编排
7. Rust 合同桥接

### 7.3 前端负责

前端只负责：

1. 对话展示
2. 运行状态展示
3. 事件消费
4. 确认交互
5. 结果与日志呈现

前端不承担运行时裁决逻辑。

---

## 8. 本地智能体必须补齐的本地化能力

本项目和 Claude Code 最大的差异，是我们要做“本地智能体”。

因此必须额外补齐：

1. `短期工作记忆`
2. `长期记忆`
3. `本地知识沉淀`
4. `本地知识库存储`

要求：

1. 记忆不做全量注入
2. 知识按需召回
3. 有价值结论要能沉淀到本地数据库或知识库
4. 记忆服务的是后续任务复用，不只是当前聊天

---

## 9. 复用策略

后续实现允许复用优秀项目，但必须遵守下面顺序：

1. 先判断是否值得复用思路
2. 再判断是否适合本地场景
3. 再判断是否符合 Go / Rust 分层
4. 最后才决定直接改写还是重新实现

推荐参考来源：

1. Claude Code 相关源码还原仓库
2. `claw-code`
3. 旧项目 `local-agent-control-hub`

复用原则：

1. 能直接借鉴模块边界，就不重复从零摸索
2. 能复刻协议设计，就不重复发明接口
3. 不为“原创”牺牲稳定性和工期

---

## 10. 当前阶段禁止跑偏的事项

后续 AI 不要把时间优先花在这些方向：

1. 重做大面积前端概念设计
2. 提前进入复杂多智能体系统
3. 提前进入浏览器自动化主链路
4. 提前堆大量记忆评分算法
5. 把运行时重新做成单一超大文件
6. 把本应放在 Rust 的高频核心链路写回脚本语言

---

## 11. 实现决策优先级

如果后续遇到实现分歧，统一按下面优先级裁决：

1. 是否符合本地性能目标
2. 是否符合 Go / Rust 主分工
3. 是否符合单智能体主干收口方向
4. 是否利于后续记忆与知识库扩展
5. 是否便于事件合同和前端消费
6. 是否接近参考项目原做法

最后一条优先级最低。

---

## 12. 给后续代码 AI 的执行要求

开始实现前，必须先确认以下 6 条：

1. 当前实现是否仍以 `Rust + Go` 为主
2. 当前实现是否在复刻“架构模式”而不是“原语言实现”
3. 当前实现是否保留本地记忆与知识沉淀扩展位
4. 当前实现是否服务低内存、高响应目标
5. 当前实现是否没有把 Python 拉回主链路
6. 当前实现是否没有超前进入高级枝叶能力

如果这 6 条里有任意一条不满足，应先调整方案，再开始写代码。

---

## 13. 当前正式结论

本项目的正式路线已经确定为：

1. 以 `Go + Rust + React` 作为长期主架构
2. 以 `Rust runtime` 作为正式智能体主链路
3. 复刻 Claude Code 的运行时模式与能力组织
4. 吸收旧项目的边界经验和模块意识
5. 强化本地记忆、长期记忆、知识沉淀与本地知识库

一句话执行要求：

> 后续任何实现，都必须先做本地适配，再进入代码开发。

## 1. 文档目的

本文件用于回答以下问题：

1. 新项目第一版的系统分层应该是什么
2. 各层分别负责什么，不负责什么
3. 为什么采用 Go + Rust + React 的结构
4. 任务请求如何从前端流到执行层，再回流到前端
5. 记忆、Skills、风险控制应该嵌在系统的哪一层
6. 怎样在保证 MVP 简洁的同时，为后续扩展留出空间

本文件不是接口定义文档，也不是 crate 设计文档。
它的作用是固定“系统长什么样”。

---

## 2. 总体架构目标

新项目第一版的总体架构目标是：

> 以 React Web 控制台承载用户交互，以 Go 承载控制面，以 Rust 承载运行时和执行层，通过多层路由组织记忆、知识、Skill 和工具能力，构建一个轻量、可成长、风险可控、可观测的本地单智能体系统。

该架构同时满足以下目标：

1. 本地优先
2. 单智能体优先
3. 最小闭环优先
4. 模块化优先
5. token 效率优先
6. 风险可控
7. 后续可扩展

---

## 3. 架构总览

第一版建议采用以下总体分层：

1. 入口层
2. 交互层
3. 控制面层
4. 运行时核心层
5. 多层路由层
6. 能力实现层
7. 存储与沉淀层

可概括为：

> 桌面启动器负责把用户带入系统，React 负责交互，Go 负责控制面，Rust 负责理解任务和推进执行，路由层负责控制记忆与能力调用范围，能力层负责实际执行，存储层负责长期沉淀和日志。

## 4. 分层结构定义

### 4.1 入口层

#### 角色

入口层负责把用户稳定带入正式产品路径。

#### 第一版组成

1. 桌面启动器
2. 本地 Web 控制台入口 URL

#### 负责什么

1. 提供稳定的人类启动入口
2. 隔离旧项目与新项目的运行路径
3. 启动并引导用户进入主界面

#### 不负责什么

1. 不负责业务逻辑
2. 不负责任务执行
3. 不负责记忆和权限判断

---

### 4.2 交互层

#### 角色

交互层负责用户输入、过程可视化和最终结果展示。

#### 第一版组成

1. 聊天页
2. 日志页
3. 设置页
4. 模式切换控件
5. 模型切换控件
6. 风险确认交互

#### 负责什么

1. 接收用户任务
2. 展示执行进度
3. 展示工具调用与结果摘要
4. 展示风险确认
5. 展示模型状态和基本设置

#### 不负责什么

1. 不直接调用底层系统工具
2. 不承载运行时状态机
3. 不直接实现记忆检索
4. 不直接判断风险策略

#### 技术方向

第一版前端采用：

1. TypeScript
2. React
3. Vite

选择理由：

1. 工程成熟
2. 响应速度可接受
3. 与旧项目已有经验连续
4. 适合快速重写和模块化拆分

---

### 4.3 控制面层

#### 角色

控制面层是整个系统的统一组织入口，负责承接前端请求并对接 Rust 运行时。

#### 技术方向

由 Go 承担。

#### 负责什么

1. 对外提供统一 API
2. 管理会话
3. 回流事件
4. 管理配置与模型信息
5. 组织日志回流
6. 组织风险确认交互
7. 管理模式状态
8. 作为前端与运行时之间的唯一正式桥梁

#### 不负责什么

1. 不直接承担大部分智能体推理逻辑
2. 不直接承载复杂执行状态机
3. 不直接实现底层文件和命令逻辑

#### 设计原则

1. 可复用原项目中成熟的 Go 设计
2. 允许重构，不机械继承
3. 保持控制面边界清晰

---

### 4.4 运行时核心层

#### 角色

运行时核心层是新系统的智能执行中枢，负责理解任务、推进任务、决定下一步动作以及完成交付。

#### 技术方向

由 Rust 承担。

#### 负责什么

1. 分析任务
2. 规划下一步
3. 调用能力
4. 观察结果
5. 决定是否继续
6. 决定是否需要确认
7. 决定是否写入记忆
8. 生成阶段反馈和最终结果

#### 不负责什么

1. 不直接承担前端渲染
2. 不直接承担 Web 路由
3. 不直接管理前端会话界面状态

#### 第一版运行时主状态

建议第一版运行时保持最小状态机：

1. Analyze
2. Plan
3. Execute
4. Observe
5. Verify
6. Finish

旁路状态：

1. PausedForConfirmation
2. Failed

如需恢复能力，可后续扩展 `Recovering`，但第一版不应过度复杂化。

---

### 4.5 多层路由层

#### 角色

多层路由层负责控制“读什么、调什么、什么时候调”，它是系统省 token、保持清晰边界和支持成长性的关键。

#### 设计原则

分层可以多层，不限定为两层。

第一版建议保留以下逻辑层：

1. Core Context Layer
2. Task Routing Layer
3. Capability Group Layer
4. Retrieval Layer
5. Execution Dispatch Layer
6. Reflection & Memory Layer

#### 各层职责

##### Core Context Layer

负责提供最小必载信息：

1. 系统规范
2. 安全约束
3. 用户稳定偏好
4. 当前工作区/当前项目高层摘要

##### Task Routing Layer

负责判断当前任务属于什么领域，例如：

1. 开发
2. 文件整理
3. 知识问答
4. 日常助理
5. 系统维护

##### Capability Group Layer

负责决定当前应该开放哪些能力组，而不是一上来暴露所有 Skills 和工具。

例如：

1. 开发能力组
2. 文件管理能力组
3. 知识检索能力组
4. 日常助理能力组

##### Retrieval Layer

负责在已选定的能力组范围内检索：

1. 相关记忆
2. 相关知识块
3. 相关 Skill 包
4. 相关历史经验

##### Execution Dispatch Layer

负责把高层动作分发到具体能力实现：

1. Rust 原生能力
2. Skill 包
3. MCP 适配能力

##### Reflection & Memory Layer

负责：

1. 压缩上下文
2. 沉淀长期记忆
3. 记录经验教训
4. 归集每日日记 / 日报素材

#### 价值

该层的价值在于：

1. 不让模型全量面对所有能力
2. 降低 token 消耗
3. 提高能力匹配精度
4. 为后续 Skills 增长提供结构支撑

---

### 4.6 能力实现层

#### 角色

能力实现层负责真正执行动作。

#### 第一版正式纳入的能力类型

1. 命令执行能力
2. 文件读写能力
3. 工作区能力
4. 知识检索能力
5. 记忆读写能力

#### 能力来源

架构上支持三类来源：

1. Rust 原生能力
2. Skill 包
3. MCP 适配能力

#### 第一版建议

第一版不要求所有能力都走 MCP。
应优先使用更轻、更高效、更适合 MVP 的实现方式。

指导原则：

1. 先实现主链路
2. 再考虑标准化接入
3. 不为了“看起来高级”而引入不必要的协议成本

#### 对模型暴露原则

模型不应直接面对过细的底层能力。

应暴露粗粒度能力，如：

1. workspace_read
2. workspace_write
3. workspace_search
4. run_command
5. retrieve_knowledge
6. save_memory
7. load_relevant_memory

---

### 4.7 存储与沉淀层

#### 角色

存储与沉淀层负责长期信息保留与可观测性基础设施。

#### 第一版应覆盖的对象

1. 会话状态
2. 日志事件
3. 长期记忆
4. 知识数据
5. 用户偏好
6. 每日记录素材

#### 设计原则

1. 长期记忆必须分块
2. 日志和记忆分离存放
3. 工作记忆和长期记忆分层
4. 不把所有历史内容直接塞入当前上下文

---

## 5. 数据流与控制流

### 5.1 主任务流

第一版主任务流如下：

1. 用户从桌面入口启动系统
2. 用户进入聊天页并发起任务
3. 前端将任务发送给 Go 控制面
4. Go 控制面创建或更新会话，并调用 Rust 运行时
5. Rust 运行时进入分析、计划、执行流程
6. Rust 通过多层路由决定读取哪些记忆、调用哪些能力
7. 能力执行结果回到 Rust 运行时
8. Rust 生成阶段事件、结果摘要、确认请求或最终结果
9. Go 将事件与结果回流给前端
10. 前端持续展示日志、进度、确认与最终结果

### 5.2 高风险动作流

高风险动作流程如下：

1. Rust 运行时准备执行高风险动作
2. 风险判断模块判定该动作需要确认
3. Rust 生成确认请求
4. Go 控制面将确认请求发回前端
5. 前端展示确认信息
6. 用户确认或取消
7. Go 控制面将用户决策发回 Rust 运行时
8. Rust 再决定继续执行或中止

### 5.3 记忆调用流

记忆调用流程如下：

1. 任务进入运行时
2. Core Context Layer 注入必载最小摘要
3. Task Routing Layer 判断当前任务领域
4. Retrieval Layer 在相关模块范围内召回记忆
5. 返回摘要化后的相关内容
6. 运行时继续执行
7. 任务结束后，Reflection & Memory Layer 决定是否写入长期记忆或日报素材

---

## 6. 风险控制在架构中的位置

风险控制不是附属模块，而是系统核心横切能力。

### 6.1 风险控制应嵌入的位置

1. 控制面层
   - 管理模式
   - 组织确认交互

2. 运行时核心层
   - 判断何时触发风险评估
   - 决定是否暂停执行

3. 能力实现层
   - 执行动作前接受最终风控约束

4. 存储与沉淀层
   - 记录日志
   - 记录高风险操作痕迹

### 6.2 风险判断原则

架构上保留内部细粒度分级：

1. 低风险
2. 中风险
3. 高风险
4. 不可逆风险

但前端展示可根据体验简化为：

1. 可直接执行
2. 需要确认

或在必要时显示具体风险等级。

---

## 7. 记忆系统在架构中的位置

记忆系统不应被设计成“一个单独大仓库”，而应嵌入在运行时和路由层之间。

### 7.1 第一版记忆结构

建议分为：

1. 必载层
2. 按需检索层
3. 会话工作层
4. 沉淀层

### 7.2 第一版记忆目标

1. 保持会话连续
2. 控制上下文膨胀
3. 沉淀长期价值
4. 支持项目知识、偏好、教训和日报方向

### 7.3 不应做的设计

1. 每次进入工作区就全量加载所有记忆
2. 把所有长期信息拼成一个超大上下文块
3. 让模型直接在全量记忆里盲选

---

## 8. Skills 与成长性在架构中的位置

### 8.1 Skills 的角色

Skills 是后续“系统成长性”的重要载体，但第一版不应把全部复杂性压在 Skills 上。

### 8.2 架构定位

Skills 更适合作为：

1. 任务级能力包
2. 领域级能力扩展点
3. 可替换、可增加、可裁剪的能力单元

### 8.3 组织原则

Skills 必须按模块组织，而不是全部平铺。

例如：

1. 开发模块
2. 文件管理模块
3. 知识模块
4. 日常模块
5. 后续浏览器模块
6. 后续桌面模块

### 8.4 架构价值

这种设计使系统具备：

1. 成长性
2. 自定义扩展能力
3. 更低 token 消耗
4. 更高的模块隔离度

---

## 9. 为什么这个架构适合 Rust

### 9.1 Rust 的优势匹配点

Rust 特别适合承担以下部分：

1. 运行时状态机
2. 多层路由与调度
3. 工具调用分发
4. 风险约束
5. 低占用、本地长期运行
6. 较强的模块边界控制

### 9.2 不建议的做法

不建议把旧 Python 的复杂性逐文件原样翻译到 Rust。

建议做法是：

1. 保留正式主链路职责
2. 压缩不必要的复杂策略
3. 用 Rust 重建清晰的执行模型

### 9.3 架构要求

为适配 Rust，整体架构必须：

1. 状态清晰
2. 模块边界清晰
3. 协议优先
4. 不依赖动态脚本式隐性行为

---

## 10. 为什么这个架构适合后续扩展

### 10.1 面向未来的扩展点

该架构已经为以下方向预留了接口：

1. 语音输入输出
2. 浏览器自动化
3. 桌面自动化
4. 更丰富的 Skill 包
5. 更复杂的记忆系统
6. 更复杂的模型策略
7. 后续多智能体

### 10.2 扩展原则

扩展必须遵守：

1. 不破坏首版主链路
2. 不把所有能力直接暴露给模型
3. 不显著增加 token 浪费
4. 优先保留模块边界

---

## 11. 第一版不采用的架构方向

为避免系统过早复杂化，第一版不采用以下方向：

1. 多智能体优先架构
2. 浏览器和桌面自动化优先架构
3. 重型工作流编排台优先架构
4. 所有能力全 MCP 化
5. 所有能力平铺暴露给模型
6. 把长期记忆视为一次性全量上下文

---

## 12. 推荐目录层级方向

本文件不冻结最终目录，但建议新项目后续沿以下结构发展：

1. `frontend/`
2. `gateway/`
3. `crates/`
4. `shared/`
5. `config/`
6. `docs/`
7. `scripts/`

其中：

1. `frontend/` 负责交互层
2. `gateway/` 负责控制面层
3. `crates/` 负责 Rust 执行层与能力层
4. `shared/` 负责合同与共享结构
5. `config/` 负责配置与策略模板
6. `docs/` 负责文档驱动开发
7. `scripts/` 负责本项目必要启动和维护脚本

## 14. 使用规则

1. 后续模块设计不得违背本文件定义的总体分层。
2. 若后续技术实现需要新增层级，必须说明新增层的职责与价值。
3. 若某能力无法清晰归入本架构中的某一层，则应先回到架构层修正，而不是直接实现。

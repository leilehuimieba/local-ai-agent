## 1. 文档目的

本文只解决 4 个问题：

1. 当前前端到底应该管理哪些状态。
2. 这些状态应如何分层，而不是继续集中在单个入口文件里。
3. 页面、领域、组件、API 消费层之间的边界应该如何划分。
4. 后续如何在不推翻当前主链路的前提下，把前端逐步演进到更稳定的模块结构。

本文不是视觉文档，也不是实现任务书。

本文的作用是：

> 把前端的状态模型、数据分层、领域边界和模块落点一次性定清，作为后续前端重构和功能扩展的架构依据。

---

## 2. 当前正式判断

当前前端的正式判断固定为：

1. 当前前端最主要的问题不是组件不够多，而是状态与职责边界不清。
2. 当前 `App.tsx` 事实性承担了壳层、页面、状态、事件流、设置联动和 props 总装多重职责。
3. 当前项目已经到达必须显式建立前端领域边界的阶段，否则后续会继续回到中心文件膨胀和多处同步改动。

一句话结论：

> 当前前端应该先做状态与模块分层，再继续做页面与体验收口。

---

## 3. 前端设计原则

后续前端架构必须同时遵守以下原则：

1. `任务状态优先于页面状态`
2. `运行时事件优先于展示文案`
3. `领域边界优先于组件边界`
4. `全局层优先于局部临时提示`
5. `先统一状态模型，再拆组件和目录`

这些原则的含义是：

1. 页面切换不能再定义任务状态。
2. UI 不能依赖中文文案字符串判断业务状态。
3. 先定义 `runtime / history / settings / resources` 等领域，再决定组件怎么拆。
4. 风险确认、错误、连接提示应由全局层统一承接。

---

## 4. 前端状态总览

当前前端后续正式状态应固定分成 5 类：

1. `Shell State`
2. `Runtime State`
3. `Connection State`
4. `History State`
5. `Settings & Resource State`

### 4.1 Shell State

定义：

1. 当前导航视图
2. 当前布局开关
3. 全局层开关

它只回答：

1. 当前显示哪个主信息域
2. 哪些辅助面板处于展开状态

它不应回答：

1. 当前任务是否在运行
2. 当前连接是否正常
3. 当前结果是否失败

### 4.2 Runtime State

定义：

1. 当前 `session`
2. 当前 `run`
3. 当前消息流
4. 当前事件流
5. 当前确认请求
6. 当前任务标题
7. 当前任务生命周期状态

它只服务当前任务现场。

### 4.3 Connection State

定义：

1. 当前 SSE 连接状态
2. 最近成功接收事件时间
3. 是否正在重连
4. 是否允许手动恢复

它是独立于 Runtime State 的一类系统状态。

### 4.4 History State

定义：

1. 稳定日志列表
2. 历史筛查条件
3. 当前聚焦历史项
4. 历史加载状态

它只服务历史复盘，不应被当前会话事件节奏直接驱动。

### 4.5 Settings & Resource State

定义：

1. 设置快照
2. 工作区与授权
3. 风险策略
4. 记忆资源摘要
5. 后续能力与连接器摘要

它服务于系统配置与资源上下文，不应混入当前任务生命周期。

## 5. 任务状态模型

### 5.1 正式任务状态

前端后续正式任务状态建议固定为：

1. `idle`
2. `submitting`
3. `streaming`
4. `awaiting_confirmation`
5. `resuming`
6. `completed`
7. `failed`
8. `archived`

### 5.2 各状态定义

`idle`

1. 当前无活跃任务
2. 可发起新任务

`submitting`

1. 前端已提交请求
2. 仍在等待 run 被正式接纳

`streaming`

1. 当前任务已开始
2. 正在持续接收事件或等待下一个事件

`awaiting_confirmation`

1. 当前任务被风险门阻塞
2. 需要用户明确决策

`resuming`

1. 用户完成确认或恢复动作
2. 前端在等待主链路继续推进

`completed`

1. 当前任务已经完成
2. 已有最终答复或正式完成结论

`failed`

1. 当前任务主链路失败
2. 需用户理解原因并决定是否恢复

`archived`

1. 当前任务已退出高优先级现场状态
2. 进入历史或静态查看阶段

### 5.3 迁移原则

状态迁移必须遵守：

1. `idle -> submitting -> streaming`
2. `streaming -> awaiting_confirmation`
3. `awaiting_confirmation -> resuming -> streaming`
4. `streaming -> completed`
5. `streaming -> failed`
6. `completed / failed -> archived`

不允许：

1. 直接由页面切换触发任务状态变化
2. 通过中文展示文案反向判断状态

---

## 6. 连接状态模型

### 6.1 正式连接状态

前端连接状态建议固定为：

1. `connecting`
2. `connected`
3. `reconnecting`
4. `disconnected`
5. `closed`

### 6.2 各状态定义

`connecting`

1. 正在建立事件流连接

`connected`

1. 当前流连接可用
2. 可以持续接收事件

`reconnecting`

1. 连接中断后正在自动恢复

`disconnected`

1. 当前连接不可用
2. 需要用户知晓

`closed`

1. 当前连接生命周期已经结束
2. 可能因为任务结束，也可能因为主动关闭

### 6.3 与任务状态的关系

连接状态和任务状态必须并存，而不能合并。

例如：

1. 任务可处于 `streaming`，连接处于 `reconnecting`
2. 任务可处于 `awaiting_confirmation`，连接仍为 `connected`
3. 任务可处于 `completed`，连接随后转为 `closed`

## 7. 前端领域分层

结合当前项目架构方向，前端正式建议分成 4 个一级领域和 1 个壳层。

### 7.1 `shell`

职责：

1. 应用壳层
2. 页面导航
3. 布局容器
4. 全局层挂载

代表内容：

1. 顶部导航
2. 主内容容器
3. 全局 overlay 容器

### 7.2 `runtime`

职责：

1. 当前任务现场
2. 会话事件流
3. 确认流
4. 运行状态
5. 连接状态

代表内容：

1. 聊天主区
2. 当前任务头部
3. 风险确认
4. 任务中枢
5. 调查层

### 7.3 `history`

职责：

1. 稳定日志
2. 复盘筛查
3. 详情联动
4. 历史任务回放

### 7.4 `settings`

职责：

1. 模型
2. 模式
3. 工作区
4. 授权
5. 风险策略
6. 诊断工具

### 7.5 `resources`

职责：

1. 记忆摘要
2. 仓库上下文摘要
3. 后续连接器与能力摘要

当前阶段不一定需要独立页面，但应在架构层保留独立领域意识。

## 8. 模块边界设计

### 8.1 壳层模块

建议职责：

1. 不直接维护复杂业务状态
2. 只消费各领域暴露出的稳定选择器和动作

典型问题：

1. 当前 `App.tsx` 不应继续承担事件流与设置联动的全部细节

### 8.2 页面模块

页面模块只负责：

1. 组织当前领域 UI
2. 调用领域动作
3. 装配展示模块

页面模块不负责：

1. 定义业务状态机
2. 定义跨领域同步逻辑

### 8.3 组件模块

组件应尽量分为两类：

1. `展示组件`
2. `领域容器组件`

展示组件：

1. 不接触异步与复杂状态迁移
2. 尽量接收稳定、低耦合 props

领域容器组件：

1. 只承接本领域数据装配
2. 不跨多个领域做大范围总装

### 8.4 API 模块

API 层必须按领域分开，而不是继续集中混用。

建议领域：

1. `runtime/api.ts`
2. `history/api.ts`
3. `settings/api.ts`
4. `resources/api.ts`

原则：

1. 一个 API 文件只服务一个领域
2. 不在页面中直接写零散 `fetch`

## 10. 当前代码到目标结构的映射

### 10.1 当前 `App.tsx`

当前问题：

1. 负责页面切换
2. 负责聊天状态
3. 负责事件流订阅
4. 负责设置联动
5. 负责 props 总装

目标：

1. 退化为壳层装配
2. 仅连接各领域模块

### 10.2 当前 `useSessionEventStream`

当前问题：

1. 只管理最小事件消费
2. 没有正式连接状态输出

目标：

1. 归入 `runtime` 领域
2. 成为连接状态的一部分

### 10.3 当前 `useLogs`

当前问题：

1. 通过当前事件数驱动刷新
2. 历史语义与当前会话耦合

目标：

1. 归入 `history` 领域
2. 独立于当前事件节奏

### 10.4 当前 `useSettings`

当前问题：

1. 配置和资源数据混在一起
2. 存在互相回写和派生污染

目标：

1. `settings` 和 `resources` 边界分离
2. 配置只管理配置，资源只管理资源

## 11. 状态共享原则

当前前端后续共享状态应遵守：

1. 跨页面稳定共享状态才进入正式共享层
2. 局部组件临时状态继续放在组件内部
3. 领域内部状态优先留在领域内，不上升为全局单例

结合当前目录规则，后续新增跨组件或跨页面状态管理时，统一使用 `zustand`。

但必须强调：

1. `zustand` 只是承载工具
2. 真正关键的是领域边界和状态语义先定义清楚

---

## 12. 数据流原则

前端正式数据流必须满足：

1. 用户动作进入领域动作层
2. 领域动作调用对应 API
3. API 结果更新领域状态
4. 领域状态驱动 UI

不再推荐：

1. 页面中直接混写异步请求、状态迁移和展示逻辑
2. 通过 props 链条把大量 setter 继续向下传

---

## 13. 全局层设计要求

全局层应只承接三类内容：

1. `Confirmation`
2. `Connection`
3. `Critical Error`

原因：

1. 这些内容天然跨页面
2. 这些内容优先级高
3. 这些内容与任务状态强绑定

全局层不应承接：

1. 普通信息卡片
2. 页面级次要提示
3. 任意业务说明块

---

## 14. 后续演进顺序

必须严格按以下顺序推进：

1. 先建立正式状态模型
2. 再拆 `runtime / history / settings / resources` 边界
3. 再把全局层抬出来
4. 再把 `App.tsx` 退化为壳层
5. 最后再做更细目录和组件收口

如果顺序反过来，会出现：

1. 目录看起来更漂亮
2. 但真正的状态混乱没有解决

---

## 15. 当前阶段不要做的事

1. 不要为了分层一次性重命名和迁移大量文件
2. 不要先做完整路由重构再补状态模型
3. 不要先上大量新目录再想清楚职责边界
4. 不要让 `history`、`settings`、`runtime` 继续互相直接读写内部状态
5. 不要让全局层承接所有提示和说明

---

## 16. 一句话结论

当前前端的状态与模块分层正式方向固定为：

1. 壳层负责结构
2. Runtime 负责当前任务
3. History 负责历史复盘
4. Settings 负责系统配置
5. Resources 负责资源上下文
6. 全局层负责高优先级信号

一句话执行要求：

> 后续凡是会影响前端状态模型、目录拆分、App 壳层职责和跨页面共享状态的实现，默认都应先对齐本文，再进入代码开发。

## 2. 目标定义

### 2.1 总目标

在当前 Rust 本地智能体项目上完成产品级改造，实现以下目标：

1. 能力达到 Hermes 同级：CLI 交互、运行时循环、工具系统、记忆沉淀、可观测与恢复。
2. 在 Windows 原生体验上达到更优：安装、运行稳定性、诊断与权限可控性优于 Hermes 当前推荐路径。
3. 建立可持续工程化能力：可测、可发布、可回滚、可审计。

### 2.2 结果标准

1. 功能等价标准：核心任务链路覆盖率不低于 Hermes 常用链路的 90%。
2. 稳定性标准：长任务运行崩溃率 < 2%，中断恢复成功率 >= 95%。
3. 质量标准：关键能力具备自动化回归，发布前必须通过 Gate 验收。
4. 平台标准：Windows 全新环境 10 分钟内可安装并完成首次任务。

## 3. 现状基线

### 3.1 现有基础（可复用）

1. 已有 Rust workspace 结构：`runtime-core` 与 `runtime-host`。
2. 已具备运行时基础模块：`planner`、`execution`、`memory`、`tool_registry`、`verify` 等。
3. 已具备基本主机入口，可作为 API Host 与 Runtime 总线承载。

### 3.2 主要缺口（必须补齐）

1. Agent Loop 稳定状态机和跨阶段恢复机制尚未产品化固化。
2. 工具协议、权限策略、审计链路尚未形成统一治理闭环。
3. 记忆与知识沉淀需要提升到可检索、可解释、可回放级别。
4. CLI/TUI 体验与网关多入口能力尚未达到产品级一致性。
5. Windows 原生适配与安装分发链路尚未收口。

## 4. 架构改造原则

1. 单一主链路原则：所有执行入口必须收敛到统一 Runtime 状态机。
2. 协议优先原则：模块间以明确 contract 交互，不做隐式耦合。
3. 先稳后快原则：优先交付可运行、可恢复、可验证，再扩功能面。
4. Windows 一等支持原则：默认开发和验收路径必须含 Windows 原生场景。
5. 渐进替换原则：尽量复用现有 crate，避免一次性推翻重写。

## 5. 分阶段路线

### 阶段 A：基线收口与架构冻结（1 周）

目标：冻结改造边界与验收口径。

交付物：

1. 能力对标矩阵（Hermes vs 当前项目）。
2. 当前模块映射图与缺口清单。
3. 风险登记册与回滚策略表。
4. 阶段 Gate 定义文档。

Gate-A：

1. 全体核心能力均有“归属模块+验收指标+失败回退”。
2. 不存在未定义的关键架构分歧。

### 阶段 B：Runtime 内核升级（2 周）

目标：完成可持续运行的 Agent Loop 主干。

交付物：

1. 状态机固化：`analyze -> plan -> act -> verify -> finish`。
2. 中断、取消、超时、重试策略统一。
3. 运行快照与恢复机制（checkpoint + resume）。
4. 运行事件合同升级。

Gate-B：

1. 50 轮混合任务无致命崩溃。
2. 中断恢复成功率 >= 95%。

### 阶段 C：工具与权限治理（2 周）

目标：建立“能用且可控”的工具系统。

交付物：

1. Tool Contract（参数、错误、超时、重试）统一。
2. 命令执行后端抽象（PowerShell/CMD/WSL/bash）。
3. 权限策略引擎（风险分级+确认链）。
4. 审计日志可追溯。

Gate-C：

1. 高风险动作拦截准确率 >= 99%。
2. 工具调用可定位率 >= 95%。

### 阶段 D：记忆与技能体系（2 周）

目标：形成跨会话连续性和可演进能力。

交付物：

1. 三层记忆模型（短期/长期/知识）。
2. 检索路由（规则召回 + 语义召回预留）。
3. 技能加载、版本管理、执行隔离。
4. 记忆写回与治理策略。

Gate-D：

1. 连续 7 天跨会话任务可复现。
2. 核心技能执行链路稳定。

### 阶段 E：交互与网关统一（2 周）

目标：CLI 与网关能力一致。

交付物：

1. CLI/TUI 改造（历史、补全、中断、状态面板）。
2. 网关统一会话协议（先完成 1-2 个入口）。
3. 跨入口会话上下文一致性。

Gate-E：

1. CLI/网关会话切换一致性 >= 95%。
2. 核心场景操作不丢状态。

### 阶段 F：Windows 产品化与发布（2 周）

目标：完成 Windows 原生可交付。

交付物：

1. 安装与升级链路（至少 1 条稳定分发方式）。
2. `doctor` 环境诊断和一键自检。
3. 发布清单、迁移说明、回滚预案。

Gate-F：

1. 新机器 10 分钟内完成安装并执行首任务。
2. 发布候选通过回归与故障注入测试。

### 阶段 G：发布后稳定运营与持续治理（2 周）

目标：把“可发布”升级为“可持续稳定运营”。

交付物：

1. 证据保鲜策略与自动复跑入口（Gate 证据时效治理）。
2. 发布后巡检与告警审计流程（含 warning 跟踪闭环）。
3. 最小回归基线与失败分流规则。
4. 运行手册与值守职责规范。

Gate-G：

1. 关键验收脚本具备自动复跑入口，证据时效满足阈值约束。
2. warning 告警项具备责任人、追踪编号、到期时间三要素。
3. 最小回归基线执行通过率 >= 95%。
4. 无未处理 P0/P1 阻塞项。

## 6. 组织与职责

### 6.1 模块归属

1. `runtime-core`：状态机、工具协议、记忆、验证、事件。
2. `runtime-host`：API Host、会话调度、网关对接入口。
3. `gateway/`：多入口协议接入与会话桥接。
4. `frontend/`：状态可视化、确认交互、审计查看。
5. `docs/11-hermes-rebuild/`：计划、门禁、执行记录。

### 6.2 决策机制

1. 任何结构性改动必须在阶段计划中写明变更理由与回退方式。
2. 未通过当前 Gate，不得推进到下一阶段开发。
3. 紧急修复可插单，但必须补齐对 Gate 的影响评估。

## 7. 质量与验收体系

### 7.1 测试层级

1. 单元测试：核心状态机、协议解析、策略判断。
2. 集成测试：工具调用链、记忆写回链、恢复链。
3. 端到端测试：真实任务脚本与跨入口会话。
4. 可靠性测试：长任务、失败注入、网络抖动、中断恢复。

### 7.2 发布门禁

1. 功能门禁：阶段交付项全部通过。
2. 稳定门禁：关键指标达到阈值。
3. 安全门禁：高风险动作拦截与审计可用。
4. 回滚门禁：可在 30 分钟内回退到上一个稳定版本。

## 8. 风险与缓解

1. 范围蔓延风险：每阶段固定“必须项/可延期项”，未过 Gate 禁止扩项。
2. 混写复杂度风险：Rust 为主，Go 仅在明确收益时引入 sidecar。
3. 兼容风险：保留关键配置兼容层与迁移脚本。
4. 进度风险：每周设风险复盘，触发阈值后自动降 scope。

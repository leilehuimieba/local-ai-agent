# 产品化封装上线向导使用模拟

更新时间：2026-04-26

## 用户画像

用户准备把本地智能体当成日常项目推进工具使用，不想理解脚本体系，只想知道“现在能不能上线 / 如何打包 / 如何诊断 / 能不能作为发布候选”。

## 场景 1：进入上线向导

用户从左侧 Rail 点击“上线”。页面展示“上线向导”，不要求用户记脚本名，而是按业务目标排列：

1. 上线前检查
2. 安装包构建
3. Doctor 诊断
4. 发布候选验证
5. 发布建议

用户第一眼能理解：这是发布前必走路径。

## 场景 2：点击运行上线前检查

用户点击“运行上线前检查”。页面立即显示“正在运行脚本，请稍候”，后端只会执行白名单脚本：

- `scripts/run-full-regression.ps1 -OutFile tmp/release-wizard/prelaunch.json`

执行完成后，用户看到“通过 / 失败”、耗时和产物路径，不需要打开终端记命令。

## 场景 3：点击运行安装包构建

用户点击“运行安装包构建”。系统把 gateway、runtime、frontend dist、配置和启动说明组装到安装目录：

- `tmp/release-wizard-install`

用户看到结果状态后，可以继续 Doctor 诊断；失败时优先检查构建日志、目标目录权限和 frontend dist。

## 场景 4：点击运行 Doctor 诊断

用户点击“运行Doctor 诊断”。系统执行：

- `scripts/doctor.ps1 -RepoRoot <repo> -OutFile tmp/release-wizard/doctor.json`

用户看到通过后，知道依赖、配置、端口、前端产物、服务健康和日志写入已按发布口径检查。

## 场景 5：点击运行发布候选验证

用户点击“运行发布候选验证”。系统执行：

- `scripts/run-stage-f-rc-acceptance.ps1 -Rounds 1 -RequirePass`

通过后，用户获得“可以作为发布候选”的明确信号；未通过时不进入发布，先修复阻塞项并重跑。

## 用户体验结论

1. 用户不再需要记脚本名，也不需要切到终端执行脚本。
2. 每一步都有清晰目标、通过标准、失败处理和一键执行按钮。
3. 后端只允许固定白名单 step，不暴露任意命令执行能力。
4. 产品化封装从“工程脚本集合”升级为“可执行上线前向导”。

## 后续建议

2. 增加“导出发布报告”按钮。
3. 增加“回退当前安装版本”入口。

## 方案

- 核心做法：
  1. 基于 `G-01~G-03` 已有脚本与证据，补齐值守主流程（日常、发布窗口、故障分流、交接清单）。
  2. 定义三类角色（值守人/变更负责人/评审人）与责任边界，避免职责重叠导致空窗。
  3. 引入“失败分流 -> 升级路径 -> 回写位置”固定模板，保证值守动作可回放。
  4. 产出一次值守演练记录作为样本，绑定具体时间与证据文件。
- 状态流转或调用链变化：
  1. 文档流程从“脚本执行”升级为“执行-判定-升级-交接”闭环。
  2. `G-04` 完成后，阶段 G 主线只剩 `G-G1` 签收任务。

## 风险与回退

- 主要风险：
  1. 值守流程描述过宽会导致执行歧义。
  2. 演练样本缺少失败分支，可能无法覆盖升级动作。
- 回退方式：
  1. 若流程不清晰，按 `G-01/G-02/G-03` 脚本入口回退到最小可执行路径。
  2. 若演练证据不足，补 1 次针对性演练并更新 artifacts，不扩实现范围。

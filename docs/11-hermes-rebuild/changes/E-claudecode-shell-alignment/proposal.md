# 变更提案

## 背景

- 本次变更要解决的问题：
  1. 用户要求基于 `ClaudeCodeRev` 源码，对当前项目的 `run_command/shell_command` 能力做机制级对照，而不是经验性建议。
  2. 当前仓库虽然具备命令执行、风险确认、artifact 落盘与基础压缩能力，但在“大输出回灌、权限决策链、压缩边界”上与 Claude Code 的成熟实现存在结构性差距。
  3. 若不先固化对照结论与任务拆分，后续实现容易出现范围漂移或重复返工。
- 对应阶段目标：
  1. 阶段 E / Gate-E 下的交互一致性与运行时治理收口。
  2. 为阶段 C（工具与权限治理）遗留项提供可执行补齐入口。

## 目标

- 本次要完成什么：
  1. 固化 `ClaudeCodeRev` vs 当前实现的源码证据与差距矩阵。
  2. 输出“可执行最小 task 拆分”，覆盖工具合同、输出外置、权限链路、压缩边界四条主线。
  3. 明确每条主线的验收口径与非目标，作为后续实现依据。

## 非目标

- 本次明确不做什么：
  1. 不在本次 change 内直接改 Rust/Go/前端业务代码。
  2. 不做 Gate-E 完成声明。
  3. 不调整 `current-state.md` 当前活跃 change（保持 `E-sensitive-pattern-expansion` 不变）。

## 验收口径

- 通过标准：
  1. change 目录内形成完整五件套（`proposal/design/tasks/status/verify`）。
  2. 形成一份可追溯到源码行号的差距矩阵文档。
  3. `tasks.md` 给出可直接执行的最小任务分解，且每项有完成判据。

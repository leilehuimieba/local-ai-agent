# 变更提案

## 背景

- 本次变更要解决的问题：
  1. 当前仓库已形成 `runtime-core / gateway / frontend / docs` 的一级模块边界，但多个热点文件仍持续膨胀，模块内聚度高于可维护阈值。
  2. 当前 GitHub 主仓已具备连续提交与阶段推进记录，但仓库对外表达、文档口径一致性与热点文件治理仍存在可见缺口。
  3. 若继续在现有热点文件上叠加 H-02 / H-03 / H-05 后续改动，后续 review、回滚和验收成本会继续上升。
- 对应阶段目标：
  1. 阶段 H（产品差异化与透明执行）下的工程化稳定性补强：在不改写 Gate-H 当前结论的前提下，为后续结构性收敛准备独立工作区。

## 目标

- 本次要完成什么：
  1. 固定“仓库模块化优化”的问题边界、优先级与执行顺序。
  2. 将文档口径一致性、gateway 热点拆分、runtime-core 热点拆分、frontend 热点减重收敛为一组可执行任务。
  3. 为后续是否正式启动模块化收敛主线提供独立的 change 工作区与验收口径。

## 非目标

- 本次明确不做什么：
  1. 不在本 change 内直接切换当前 active change。
  2. 不一次性重构 `runtime-core / gateway / frontend` 全部热点文件。
  3. 不把模块化草案直接表述为 Gate-H 已通过或阶段 H 已完成。

## 验收口径

- 通过标准：
  1. 至少明确一轮模块化收敛的优先级、边界、文件落点与验证方式。
  2. 能明确回答“为什么当前要做模块化收敛、先拆哪里、哪些先不动”。
  3. 草案状态下即可支持后续新建正式任务，而不与当前 `current-state.md` 冲突。

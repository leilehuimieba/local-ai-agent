# 变更提案

## 背景

- 本次变更要解决的问题：
  1. 当前项目虽然已完成阶段 D 的三层记忆与关键词检索主路径，但“生命周期采集 -> 结构化沉淀 -> 分层检索注入”的工程闭环仍不完整。
  2. 现有跨会话能力在“可解释检索、按需注入、异步恢复、隐私治理”四个方面仍有提升空间，难以支撑长周期任务稳定复用。
  3. 用户明确要求基于 `claude-mem` 思路，形成本项目配套的成熟路线、核心功能实现计划、验收标准与 AI 可执行任务分解。
- 对应阶段目标：
  1. 不改变当前 `F-install-upgrade-20260414` 主推进前提下，先建立一套可直接落地的记忆增强实施蓝图，作为后续切主推进时的执行输入。

## 目标

- 本次要完成什么：
  1. 在本项目约束下冻结“Memory Progressive Disclosure 专项计划”（不引入 Bun/Node 常驻服务）。
  2. 明确 `M0 -> M5` 全链路里程碑、Gate、回退策略、证据路径。
  3. 产出可直接下发给 AI 执行的任务拆分（任务 ID、DoD、验证命令、证据要求）。

## 非目标

- 本次明确不做什么：
  1. 不切换当前主推进项，不改变 `current-state.md` 的阶段与活跃 change。
  2. 不在本刀内直接实施 runtime/gateway 代码改造。
  3. 不引入外部向量数据库作为首轮主依赖。

## 验收口径

- 通过标准：
  1. 本 change 五件套完整，且可单独指导后续实现推进。
  2. `design.md` 包含核心能力全链路计划（采集、存储、检索、注入、治理、回退）。
  3. `tasks.md` 包含可执行任务 ID、完成判据、证据落点，满足 AI 逐项执行要求。
  4. `verify.md` 提供可量化的指标门禁与验收矩阵，不以口头结论替代证据。

## 方案

### 1. 先收口，再实现

- 当前窗口仅做 Step1/Step2（需求收口 + 方案定稿），不直接进代码实现。
- 保持“任务页唯一主工作区、日志页跨会话复盘、设置页运行控制”的页面边界不变。
- 所有优化任务必须可映射到现有四页与三个任务页辅助区，禁止新增一级入口。

### 2. 下一轮前端优化优先级

1. 任务页状态闭环可视化（运行中/待确认/失败/完成）。
2. 调查层阅读闭环（自动跟随与手动查看切换可感知、可回到最新）。
3. 首页恢复态信息密度优化（只保留“继续任务”所需信息）。
4. 记录页复盘效率优化（风险、错误、验证信息分组更清晰）。
5. 设置页文案与状态一致性收口（运行控制优先，弱化噪音）。

### 3. 实施分段（交给实现窗口）

- Wave 1（低风险，无合同改动）：
  - 任务页状态表达统一
  - 调查层交互与空/错态统一
  - 首页恢复区密度收口
- Wave 2（中风险，依赖合同稳定）：
  - 日志页结构化筛选增强
  - 设置页诊断与风险状态联动增强

### 4. 阶段约束

- 本 change 当前是并行设计窗口，不视为阶段 E 已进入实现。
- 在阶段 C 的合同收口任务未完成前，本 change 不推进大规模前端实现。

### 5. Wave 1 文件级交接清单

1. `frontend/src/workspace/WorkbenchOverview.tsx`
   - 收口首页恢复态内容密度，只保留继续任务所需信息。
2. `frontend/src/workspace/TopBar.tsx`
   - 统一任务状态短文案与视觉优先级，减少噪音标签。
3. `frontend/src/workspace/BottomLogsDrawer.tsx`
   - 强化“自动跟随/手动查看/回到最新”三态提示与入口。
4. `frontend/src/workspace/BottomPanel.tsx`
   - 统一调查层空态、等待态、失败态表达。
5. `frontend/src/chat/ChatPanel.tsx`
   - 统一主线程状态表达顺序，保证提交后立即可见反馈。
6. `frontend/src/chat/chatResultModel.ts`
   - 收口 `summary -> action -> detail -> risk -> next` 各段落状态映射。
7. `frontend/src/workspace/ContextSidebar.tsx`
   - 收口任务中枢分组顺序，强化“当前任务/风险/下一步”可扫读性。

### 6. Wave 1 验收清单（三态判定）

1. 提交任务后，主线程在首个事件前有明确等待态。
2. 调查层可明确区分自动跟随与手动查看，且“回到最新事件”入口可用。
3. 失败且无事件时，主线程与调查层均有恢复建议，不回落普通空态。
4. 首页恢复态不承接主线程推进，只承接继续入口与阻塞提示。
5. 桌面宽屏和手机窄屏无明显溢出、挤压、滚动错乱。
6. `npm run build` 通过。

判定规则：

- `通过`：以上 6 项全部满足。
- `有条件通过`：最多 1 项轻微问题，不影响主链路闭环。
- `不通过`：出现主链路断裂、状态错位或构建失败。

### 7. 实现启动条件

1. 当前 change 保持设计态，只有在实现窗口接手时才切为实现态。
2. 实现窗口每轮只改 Wave 1 中一个主题，禁止跨主题扩散。
3. 每轮结束必须补最小回归结果到本 change `verify.md`。
4. 如发现与阶段 C 合同收口冲突，立即暂停并回到本 change 更新方案。

## 风险与回退

- 主要风险：前端优化窗口范围扩散成“重构前端”。
- 主要风险：体验改动与现有任务页边界冲突。
- 回退方式：每轮只改一个体验主题，始终按 `frontend-acceptance` 最小回归清单回归；若出现边界冲突，回退到上轮页面结构并保留样式层改动。

# Proposal: 竞品用户迁移体验优化

## 背景

项目已经具备本地执行、记录、验证与工程治理能力，但产品表达仍偏工程控制台。竞品忠实用户迁移时，首要判断不是底层能力是否完整，而是：是否能马上理解价值、是否敢交付真实任务、失败时是否可控、长期使用是否有连续性。

## 目标

把现有能力包装成更适合竞品用户迁移的产品体验：

1. 第一次打开就知道为什么使用本项目。
2. 能力说明从“功能列表”升级为“任务范式 + 安全边界”。
3. 首页强化示例任务、继续上次任务、上线前检查入口。
4. 工作记录从技术日志感转向用户可读的工作历史。
5. 失败与验证信息以“原因 / 影响 / 建议 / 证据”方式表达。

## 非目标

1. 不重构 runtime 主流程。
2. 不新增复杂后端接口。
3. 不改 provider 配置与模型调用协议。
4. 不引入新的发布系统。

## 验收标准

1. 用户问“你能做什么”时，返回产品化使用引导。
2. 首页示例围绕竞品用户迁移场景：检查项目、跑测试、改功能、上线前检查。
3. 记录页默认展示“工作历史”语义，不暴露复杂筛选与统计。
4. 前端测试与 TypeScript 编译通过。
5. 全量回归通过或明确记录非本次变更阻塞。

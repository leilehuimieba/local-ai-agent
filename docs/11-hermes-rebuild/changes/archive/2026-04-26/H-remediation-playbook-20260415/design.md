## 方案

1. 核心做法：
   - 先冻结 H-02 的问题范围、样本分流与指引模板，不直接进入大规模实现。
   - H-02 最小主链路固定为：`diagnose -> classify -> auto_fix_or_manual_takeover -> verify -> replay`。
   - 自动修复只覆盖低风险、可验证、可回退的本地问题；超出边界一律降级为人工接管指引。
   - 人工接管指引统一采用固定模板，要求步骤可复制、预期结果明确、失败后有下一步分流。
   - H-02 当前先在 gateway / doctor 基线侧证明闭环可回放，再决定是否下沉到 runtime 主链。
2. 状态流转或调用链变化：
   - `diagnose`：采集报错、命令上下文、最近动作、目标环境信息。
   - `classify`：把问题归入固定 playbook 类型，并输出风险等级。
   - `auto_fix_or_manual_takeover`：
     - 命中低风险白名单：生成自动修复计划并执行前后校验。
     - 未命中白名单：生成手动接管步骤，不自动代执行。
   - `verify`：校验问题是否消失，补成功、失败、未完成三类结论。
   - `replay`：把聚合结果、接管指引、回放结果落到证据目录。

## H02-01 冻结结论

### 1. 固定问题分类

H-02 仅覆盖以下五类高频 Windows 本地问题：

1. 依赖缺失
   - 典型表现：命令不存在、运行时/包管理器缺失、基础依赖未安装。
2. PATH / 环境变量错误
   - 典型表现：已安装但不可达、版本命中错误、会话环境与系统环境不一致。
3. 权限问题
   - 典型表现：权限不足、目录不可写、需要管理员操作。
4. 端口占用 / 进程不可达
   - 典型表现：服务启动失败、健康检查失败、端口被占用或目标进程未就绪。
5. 配置错误
   - 典型表现：配置文件缺字段、路径错误、格式不合法、引用不存在资源。

H-02 当前不覆盖：

1. 驱动级/内核级修复。
2. 数据损坏恢复。
3. 跨机器、跨云、跨容器复杂联调。
4. 无法稳定复现的问题猎奇排查。

### 2. 自动修复白名单边界

允许进入自动修复的动作必须同时满足以下条件：

1. 低风险：不会删除用户数据，不修改高敏系统策略，不写入不可逆配置。
2. 可验证：执行后可以用明确命令或状态检查确认是否成功。
3. 可回退：失败时至少能提示恢复方式、原值或手动兜底动作。
4. 单步可解释：能向用户清楚说明“为什么执行、会改什么、预期看到什么”。

当前允许进入白名单的方向：

1. 当前会话级 PATH 补全或命令定位校验。
2. 非破坏性的目录可写性修复建议与安全替代目录切换。
3. 明确缺失依赖的安装前检查与安装指令生成。
4. 配置文件的非破坏性校验、缺省值补全建议、路径存在性修正建议。

当前禁止自动执行的动作：

1. 注册表批量修改。
2. 系统级环境变量持久写入。
3. 防火墙、服务、启动项、计划任务等高风险系统配置修改。
4. 删除用户文件、覆盖未知配置、杀死不明关键进程。
5. 需要管理员权限但无法明确影响面的操作。

### 3. 人工接管触发条件

出现以下任一情况时，必须转人工接管，不继续自动执行：

1. 动作超出自动修复白名单。
2. 需要管理员权限或用户确认高风险影响面。
3. 当前问题缺足够上下文，无法安全判定根因。
4. 自动修复已尝试但校验未通过。
5. 同一问题链路存在多种可能修复路径，自动选择风险过高。

### 4. 人工接管模板冻结

每份手动接管指引必须包含以下字段：

1. `issue_summary`：问题摘要。
2. `why_this_happens`：原因解释。
3. `risk_level`：风险等级与是否需要管理员权限。
4. `prerequisites`：前置条件与需要用户准备的内容。
5. `steps[]`：
   - `step_no`
   - `action`
   - `expected_result`
   - `if_failed`
   - `next_step`
6. `success_criteria`：完成判据。
7. `rollback_hint`：失败或中断时如何回退。
8. `evidence_ref`：对应日志、截图或回放编号。

### 5. 证据目录冻结

H-02 证据统一冻结为以下结构：

1. `tmp/stage-h-remediation/latest.json`
   - 聚合记录问题类型、处理路径、是否自动修复、最终结论。
2. `tmp/stage-h-remediation/manual-guides/*.md`
   - 按问题样本输出手动接管指引。
3. `tmp/stage-h-remediation/replay-results.json`
   - 回放结果、成功率、失败原因、未覆盖样本。

## H02-02 冻结结论

### 1. 样本矩阵冻结原则

H02-02 统一按以下原则冻结样本：

1. 优先复用 `doctor` / diagnostics 已能识别的问题，不先引入新的诊断源。
2. 先覆盖会真实阻断本地启动、构建、日志写入和基础配置的高频样本。
3. 每个样本都必须能映射到：
   - `scenario`
   - `route`
   - `auto_fix_allowed`
   - `source_field`
   - 对应 `manual-guides/*.md`
4. 当前只把已落证据的样本纳入 H02-02，未落证据的泛化问题保留到后续扩项。

### 2. 优先级定义

1. `P0`：直接阻断本地主链路启动或当前任务继续。
2. `P1`：不一定阻断，但会显著增加人工排查与修复成本。
3. `P2`：当前只做观察，不进入本轮最小实现。

### 4. 自动修复 / 人工接管分流表

#### 4.1 自动修复候选

当前仅冻结 1 个首批自动修复候选：

1. `H02-S04 / logs_not_writable`

入选原因：

1. 问题边界清晰，能直接映射到日志目录不可写。
2. 可优先尝试低风险目录切换、权限校验或提示用户选择安全替代目录。
3. 修复后可通过写入检查即时验证结果。

#### 4.2 人工接管样本

当前冻结为人工接管样本：

1. `H02-S01 / runtime_unreachable`
2. `H02-S02 / gateway_unreachable`
3. `H02-S03 / frontend_dist_missing`
4. `H02-S05 / config_missing_or_invalid`

进入人工接管的理由：

1. 需要用户判断是否重启进程、重建前端产物或修复配置内容。
2. 自动修复影响面未完全可控，当前不进入真实执行。
3. 已能生成逐步指引，适合作为先验证“接得住”的闭环样本。

### 5. 证据映射冻结

H02-02 当前固定使用以下证据：

1. 样本聚合：`tmp/stage-h-remediation/latest.json`
2. 手动接管模板：
   - `tmp/stage-h-remediation/manual-guides/runtime-unreachable.md`

3. 回放口径：`tmp/stage-h-remediation/replay-results.json`

## H02-10 签收级缺口设计与验收矩阵

### 1. 剩余缺口分层

H-02 当前剩余 warning 统一按以下两层处理：

1. `L1 / 已有最小闭环但不足签收`
   - 已存在低风险自动修复、自动降级人工接管、回放与人工评分证据。
   - 目前可证明“能接住一部分高频问题”，但不能证明“已覆盖签收所需高风险边界与权限边界”。
2. `L2 / 签收级缺口`
   - 高风险配置写入边界尚未冻结为可验收矩阵。
   - 更广权限类真实验收范围尚未冻结为最小必测集。
   - 在这两项未冻结前，不再把新增低风险样本视为 H-02 的有效收口手段。

### 2. 高风险配置写入边界

以下场景统一定义为“高风险配置写入”，当前只允许纳入设计与验收矩阵，不进入自动实现：

1. 持久化修改系统级环境变量或机器级 PATH。
2. 写入注册表、服务配置、启动项、计划任务、防火墙规则。
3. 覆盖已有用户配置值，且原值并非明确缺失默认项。
4. 修改端口、工作目录、外部服务地址、凭据引用等会影响现有运行拓扑的字段。
5. 需要跨文件联动写入，或写入后无法用单一健康检查直接确认影响面的配置。

当前仅允许保留在低风险范围内的配置动作：

1. 缺失字段补安全默认值，且字段语义清晰、影响面局部、可立即复核。
2. 写入前后差异可记录，失败后可回退到原文件。
3. 不涉及管理员权限、不涉及系统级持久配置。

### 3. 权限类验收范围

H-02 再提审前，权限类至少要覆盖以下验收分层，但当前只冻结验收矩阵，不要求立即实现：

1. `P-A / 当前用户目录可切换`
   - 原目录不可写，但可安全切换到当前用户可写目录。
2. `P-B / 当前用户目录不可写且需人工改权限`
   - 指引需明确最小权限修改步骤、预期结果与失败分流。
3. `P-C / 需要管理员执行的系统目录或系统配置`
   - 必须人工接管，不得自动代执行。
4. `P-D / 权限问题与配置写入耦合`
   - 例如配置文件存在但不可写、日志目录与配置目录同时受限。
   - 必须先停止自动修复扩展，改为人工验收样本。

其中，签收级最小覆盖要求不是“样本数量更多”，而是上述四层都已有明确归类、人工接管边界与不可自动化说明。

### 4. 必须人工接管的场景

出现以下任一情况，必须人工接管：

1. 需要管理员权限才能完成。
2. 写入目标属于系统级或跨应用共享配置。
3. 无法明确给出写入前后差异与回退步骤。
4. 同一修复动作可能影响现有端口、工作区、服务发现或外部依赖。
5. 涉及凭据、密钥、令牌、账户权限提升。
6. 权限不足与根因不清同时出现，无法安全区分是路径问题、ACL 问题还是进程占用。

### 5. 后续才允许考虑自动化的场景

以下场景仅在 H-02 被重新评审并单独扩 scope 后，才允许考虑自动化：

1. 已经由人工验收反复证明稳定、可回退、影响面局部的配置写入。
2. 可形成固定差异模板且每次写入前都能做 dry-run 对比的配置修复。
3. 权限问题仅限当前用户工作区，且无需管理员、无需修改系统 ACL 策略。
4. 自动执行失败后可无损回退，并能稳定降级回人工接管。

### 6. 高风险配置写入验收矩阵

| 编号 | 场景类型 | 示例 | 当前归类 | 当前处理要求 | 再提审前要求 |
|---|---|---|---|---|---|
| C-A | 缺失字段补安全默认值 | 缺 `runtime_port`、缺 `default_workspace.workspace_id` | 低风险已落最小闭环 | 允许保留现有最小自动写入证据 | 只作为低风险基线，不代表高风险已覆盖 |
| C-B | 覆盖已有业务配置值 | 已存在端口、工作目录、服务地址但值异常 | 高风险配置写入 | 一律人工接管 | 需先冻结写前差异、影响面说明、回退步骤，未满足前不得自动化 |
| C-C | 系统级持久配置写入 | 机器级 PATH、系统环境变量 | 高风险配置写入 | 一律人工接管 | 仅允许作为验收排除项，不进入 H-02 自动修复 |
| C-D | 系统策略/服务联动写入 | 注册表、服务、启动项、防火墙、计划任务 | 高风险配置写入 | 一律人工接管 | 仅允许写入“必须人工”口径，不得转自动化 |
| C-E | 跨文件联动配置写入 | 同时改 app 配置、脚本、服务发现地址 | 高风险配置写入 | 一律人工接管 | 需明确联动图与回退链，H-02 当前不承接实现 |
| C-F | 凭据/敏感引用写入 | token、密钥、账户、网络凭据 | 禁止自动化 | 一律人工接管 | 作为永久排除项，不纳入 H-02 自动修复 |

签收级口径：

1. H-02 当前仅允许 `C-A` 作为已落低风险基线。
2. `C-B` ~ `C-F` 必须先冻结人工接管边界与验收排除口径，才可视为“签收级缺口已定义清楚”。
3. H-02 当前不追求证明 `C-B` ~ `C-F` 可自动修复，只要求证明这些场景不会被误判进入自动执行。

### 7. 权限类最小必测验收矩阵

| 编号 | 权限层 | 场景 | 当前归类 | 当前处理要求 | 再提审前要求 |
|---|---|---|---|---|---|
| P-A | 当前用户可切换 | 原目录不可写，但可改到用户目录 | 可保留为未来低风险候选 | 当前优先人工接管或安全切换建议 | 需明确切换条件、验证方式、失败分流 |
| P-B | 当前用户需手工改权限 | 当前用户有权限调整工作区 ACL | 人工接管 | 不自动代执行 | 需冻结最小修改步骤、预期结果、回退提示 |
| P-C | 需要管理员权限 | 系统目录、Program Files、系统服务目录 | 必须人工接管 | 严禁自动代执行 | 需显式写成“签收级必测人工场景” |
| P-D | 权限与配置写入耦合 | 配置文件不可写且涉及配置修复 | 必须人工接管 | 停止自动修复扩展 | 需先判定权限，再判定配置，不得合并成自动动作 |

签收级口径：

1. H-02 再提审前，至少要证明 `P-A` ~ `P-D` 四层边界已冻结。
2. 这里的“覆盖”是覆盖验收分层与人工边界，不是要求四层都已有自动化实现。
3. 只要涉及 `P-C` 或 `P-D`，当前默认停止自动修复扩展。

### 8. 后续路径裁决

#### 8.1 高风险配置写入后续路径

1. 继续禁止自动化：
   - `C-C / 系统级持久配置写入`
   - `C-D / 系统策略或服务联动写入`
   - `C-F / 凭据与敏感引用写入`
2. 后续可考虑进入更高门槛验证，但当前阶段 H 不进入实现：
   - `C-B / 覆盖已有业务配置值`
   - `C-E / 跨文件联动配置写入`
   - 前提是先有写前差异、影响面、回退链、人工复核口径四项同时冻结。
3. 即使后续推进也必须保持人工接管：
   - `C-C`、`C-D`、`C-F`
   - 当前阶段 H 内把它们视为“永久人工接管或排除自动化”场景。
4. 当前阶段 H 的处理结论：
   - `C-A` 维持现有低风险基线，不扩项。
   - `C-B`、`C-E` 只保留为“未来可进入更高门槛验证”的候选，不在本轮实现、不进入当前验证窗口。
   - `C-C`、`C-D`、`C-F` 在阶段 H 内不建议推进到实现验证。
5. 停止边界：
   - 一旦需要系统级持久写入、管理员权限、敏感凭据处理或无法提供回退链，立即停止，不再作为 H-02 后续推进方向。

#### 8.2 权限类场景后续路径

1. 继续观察、不执行：
   - `P-C / 需要管理员权限`
   - `P-D / 权限与配置写入耦合`
   - 当前阶段 H 内只保留人工接管说明与风险分流，不进入真实执行。
2. 未来可考虑纳入验证样本：
   - `P-A / 当前用户目录可切换`
   - `P-B / 当前用户需手工改权限`
   - 前提是仍限定在当前用户工作区、无需管理员、且失败后可稳定降级回人工接管。
3. 应永久维持人工接管或只保留指引：
   - `P-C`
   - `P-D`
   - 以及任何需要修改系统 ACL 策略、系统目录授权或与配置写入耦合不清的场景。
4. 当前阶段 H 的处理结论：
   - 不建议把任何权限类高风险场景推进到真实验证。
   - 若后续确需补样，只允许在 `P-A`、`P-B` 中选择不触及管理员权限的最小人工接管验证样本。

### 10. 受限验证窗口选择

#### 10.1 本轮唯一允许的受限验证场景

本轮唯一允许进入验证计划的最小场景固定为：

#### 10.2 为什么该场景符合受限验证窗口

#### 10.4 受限验证计划

### 12. 停止条件与回退条件

#### 9.1 停止条件

出现以下任一情况，H-02 必须停止继续扩样或扩实现，只保留文档收口：

1. 新样本开始进入系统级持久配置写入。

3. 新样本无法给出清晰回退边界。

#### 9.2 回退条件

1. 配置写入方案无法证明仅修改安全默认缺失项。
2. 权限类样本一旦落地就需要系统级授权或高敏策略调整。
3. 新增验证依赖人工主观判断，无法沉淀为稳定验收矩阵。
4. 文档口径与现有证据冲突，无法继续保守表达当前状态。

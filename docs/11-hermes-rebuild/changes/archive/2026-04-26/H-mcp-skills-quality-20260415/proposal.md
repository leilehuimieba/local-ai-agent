# H-mcp-skills-quality-20260415（proposal）

更新时间：2026-04-16
状态：草案

## 背景

1. 阶段 H 路线要求补齐 H-03“`MCP + Skills` 执行质量体系”，当前仓库尚无对应正式 change 工作区。
2. Gate-H 要求 H-01 ~ H-05 全部具备可复现证据；H-02 已进入真实修复样本阶段，但 H-03 仍缺 proposal/design/tasks/status/verify 与证据口径。
3. 当前仓库已具备可复用基础：
   - `crates/runtime-core/src/skill_catalog.rs`：技能清单加载、版本 pin、隔离校验。
   - `crates/runtime-core/src/tool_registry.rs`：工具可见性、能力声明、规划入口。
   - `crates/runtime-core/src/run_visibility.rs`：失败分流、等待原因、证据引用等可观测字段。
4. 当前代码里尚未发现独立的“H-03 聚合证据目录”，因此需要先建立最小执行入口，再决定是否进入实现与评测补齐。
5. 2026-04-16 对外部项目 `rtk-ai/rtk` 的静态调研表明，其“薄适配层 + 单一裁决核心、增强层失败不阻断主链路、摘要与原始证据双轨、收益量化闭环”四个做法，对 H-03 的执行质量体系有直接借鉴价值，但其 WSL-first 路径不适合作为本项目默认口径。

## 目标（一句话）

为 H-03 建立“技能可加载、工具可路由、失败可定位、回退可审计”的最小执行与验收口径，让 `MCP + Skills` 从理念变成可测的质量体系。

## 范围

### In-Scope

1. 冻结 H-03 的质量目标、模块边界、证据结构与任务拆分。
2. 明确 Skills 装载、版本治理、隔离校验的验收口径。
3. 明确工具路由、失败分流、fallback 证据的验收口径。
4. 为后续最小评测包与回放脚本预留 `tmp/stage-h-mcp-skills/`。
5. 把外部参考项目中可迁移的治理原则收敛为 H-03 的显式设计约束，而不是停留在聊天结论。

### Out-of-Scope

1. 不在本轮直接扩出新的 MCP 适配层实现。
2. 不与 H-02 修复链、H-04 学习模式、H-05 记忆路由混合推进。
3. 不在缺乏证据的前提下宣称 H-03 已通过。
4. 不把外部项目的 Unix / WSL 默认路径照搬为本项目 Windows-first 的执行口径。

## 验收标准

1. H-03 change 工作区齐备，可作为后续正式推进入口。
2. H-03 的模块范围、最小评测包、fallback 证据和 Gate-H 映射已明确写入文档。
3. 已明确当前可复用实现基础与尚缺证据，不把历史口号误写成“已完成”。
4. 已把外部参考项目可迁移的治理原则映射到 H-03 的设计、任务与验证口径，且明确哪些点只可借鉴、不可照搬。

## 风险与回退

1. 风险：把 H-03 写成泛化“智能体质量”而失去仓库落点。
2. 风险：把 Skills 路由与 MCP 接入混成一层，后续难以定位失败原因。
3. 风险：把外部项目的 token 节省目标、Unix shell 集成方式或 WSL-first 路径误当成本项目默认目标。
3. 回退：若 H-03 设计评审未通过，保留 change 草案，不切主推进，也不进入业务实现。

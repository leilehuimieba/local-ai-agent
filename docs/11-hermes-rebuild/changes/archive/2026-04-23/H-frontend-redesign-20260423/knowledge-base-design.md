## 一、现状分析

### 1.1 现有能力

| 模块 | 当前状态 | 问题 |
|------|----------|------|
| **Memory（记忆）** | Settings 页内展示，含偏好/教训/知识沉淀/运行过程 | 混入运行时自动沉淀内容，用户无法主动管理 |
| **思源外挂知识库** | 后端已接入，前端仅显示 `source_type = siyuan` 标记 | 没有独立页面，用户看不到思源里的内容 |
| **API** | 只有 `fetchMemories` / `deleteMemory` | 无添加/编辑/分类能力 |

### 1.2 关键区分

| | Memory（记忆） | Knowledge Base（知识库） |
|---|---|---|
| **来源** | 运行时自动沉淀 | 用户主动添加/编辑 |
| **内容** | 偏好、教训、运行时摘要 | 专业资料、文档、笔记、经验总结 |
| **管理** | 浏览、筛选、删除 | 添加、编辑、分类、标签、搜索 |
| **用途** | 系统自用（跨会话续接） | 用户+系统共用（回答问题时引用） |

**结论**：Memory 和 Knowledge Base 是两层东西，不能混在同一个页面里。

## 二、知识库页面设计

### 2.1 页面定位

- 独立工作区视图，和 首页/任务/记录/设置 并列
- 左侧导航新增"知识库"图标
- 作为用户主动管理专业资料的入口

### 2.2 布局（参考 Dribbble + Claude Code）

```
┌─────────────────────────────────────────────────────────────┐
│  [●] 本地智能体                                       [≡]   │
├─────┬──────────────────────────────────────────┬────────────┤
│     │  [知识库 A ▼]        [+ 添加资料]        │            │
│  🏠  │                                          │   详情     │
│  💬  │  [全部] [分类1] [分类2] [标签 ▼] [搜索]  │   面板     │
│  📋  │                                          │   (点击    │
│  📚  │  ┌─────────┐ ┌─────────┐ ┌─────────┐   │   展开)    │
│  ⚙️  │  │ 资料 1  │ │ 资料 2  │ │ 资料 3  │   │            │
│     │  │ 摘要... │ │ 摘要... │ │ 摘要... │   │            │
│     │  │ #标签   │ │ #标签   │ │ #标签   │   │            │
│     │  └─────────┘ └─────────┘ └─────────┘   │            │
│     │                                          │            │
└─────┴──────────────────────────────────────────┴────────────┘
```

### 2.3 页面结构

#### 顶部操作栏
- **知识库切换器**：下拉选择当前知识库（默认知识库 / 工作区知识库 / 自定义知识库）
- **添加资料按钮**：打开添加面板
- **搜索框**：在当前知识库内搜索

#### 筛选栏
- 分类筛选（全部 / 文档 / 笔记 / 代码 / 链接）
- 标签云（可点击筛选）
- 排序（最近更新 / 最近添加 / 最常引用）

#### 内容区
- **空状态**：大图标 + "还没有资料，添加第一篇知识" + 添加按钮
- **卡片网格**：每个卡片展示
  - 标题（1 行）
  - 摘要（2 行，灰色）
  - 标签 pills（1 行）
  - 底部：更新时间 + 被引用次数

#### 右侧面板（点击卡片展开）
- 标题（可编辑）
- 标签（可添加/删除）
- 正文（Markdown，可编辑）
- 来源信息（URL / 文件路径 / 手动输入）
- 操作按钮：[保存] [删除]

#### 添加资料面板（模态框或侧滑面板）
- 标题输入
- 分类选择
- 标签输入（逗号分隔）
- 正文输入（支持 Markdown）
- 来源（可选）
- [确认添加] [取消]

### 2.4 与任务流的结合

**场景 1：提问时手动引用**
- 任务页输入框上方增加"引用知识库"选择器
- 默认"自动"，可下拉选择特定知识库
- 选定后，该次提问优先检索该知识库内容

**场景 2：系统自动调用**
- 系统在 analyze 阶段自动判断是否需要检索知识库
- 如果用户问题涉及专业领域，自动检索相关知识库
- 在事件流中显示"已检索知识库：XXX，命中 Y 条"

**场景 3：知识库内直接提问**
- 在知识库页面内，有一个"基于这些资料提问"的输入框
- 类似 RAG 对话，只基于当前知识库内容回答

## 三、信息架构

### 3.1 数据模型（前端先行，可 mock）

```typescript
type KnowledgeBase = {
  id: string;
  name: string;
  description: string;
  itemCount: number;
  createdAt: string;
  updatedAt: string;
};

type KnowledgeItem = {
  id: string;
  kbId: string;
  title: string;
  summary: string;
  content: string;       // Markdown
  category: string;      // 文档 / 笔记 / 代码 / 链接
  tags: string[];
  source?: string;       // URL 或文件路径
  citationCount: number; // 被引用次数
  createdAt: string;
  updatedAt: string;
};
```

### 3.2 需要后端支持的 API

| API | 方法 | 用途 |
|-----|------|------|
| `/api/v1/knowledge-bases` | GET | 知识库列表 |
| `/api/v1/knowledge-bases/:id/items` | GET | 知识库内条目列表 |
| `/api/v1/knowledge-bases/:id/items` | POST | 添加条目 |
| `/api/v1/knowledge-bases/:id/items/:itemId` | PUT | 编辑条目 |
| `/api/v1/knowledge-bases/:id/items/:itemId` | DELETE | 删除条目 |
| `/api/v1/knowledge-bases/:id/search` | POST | 知识库内搜索 |
| `/api/v1/chat`（扩展） | POST | 支持 `knowledgeBaseId` 参数 |

**前端先行策略**：
- 先实现 UI，用 localStorage 做 mock 数据层
- 等后端 API 完成后，替换 API 调用即可
- 不阻塞前端重新设计的整体进度

## 四、实现节奏建议

### 方案 A：知识库作为 Wave 3 的一部分（推荐）

1. **Wave 2**：完成首页 + 任务页 + 记录页 + 设置页简化
2. **Wave 3**：新增知识库页面
   - 3.1：知识库列表 + 条目卡片网格
   - 3.2：添加/编辑/删除条目
   - 3.3：任务页输入框加入"引用知识库"选择器
   - 3.4：与后端 API 对接

### 方案 B：知识库独立 change

- 前端重设计先按原计划推进（Wave 1-4）
- 知识库页面作为独立 change 并行开发
- 避免一次改太多，用户不好验收

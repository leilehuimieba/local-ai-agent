# Wave 3 保守收口口径

## 目的

本文件用于在主控已经确认页面截图与最小 walkthrough 已补齐、但仍不准备把并行 change 直接写成“全部完成”的情况下，为 Wave 3 提供一套可稳定落地的保守收口口径。

这份口径的核心作用是：

1. 避免把 Wave 3 误写成“已全部完成”
2. 明确 Wave 3 当前已经达到的代码侧状态
3. 允许本并行 change 在证据待补阶段稳定挂住，而不是悬空

## 当前建议口径

当前建议统一使用：

- **“Wave 3 代码侧已基本收口，页面证据已显著补齐；本并行 change 可结束（并行），但不写成全部完成。”**

或更完整一点：

- **“Wave 3 当前已完成代码侧最小收口，页面截图与最小 walkthrough 已补齐；主控判断可结束（并行），但不切主推进、不写成全项目前端已全部验收。”**

不建议使用：

1. “Wave 3 已全部完成”
2. “Wave 3 已正式关闭”
3. “附属工作区改造已全部验收”

## 这套口径适用的前提

只有在以下条件同时成立时，才建议使用本口径：

1. `Logs / History / Settings / Resources` 的核心代码改造已完成
2. Wave 3 的收口评估文档已存在
3. 多视图一致性检查清单已存在
4. 截图补证入口已存在
5. 当前主要缺口已经从“实现”转为“验证”

结合当前 change 工作区，以上条件已经满足。

## 当前可以稳定表述为“已完成”的部分

### 1. 代码侧最小收口已完成

当前可以稳定写为：

1. `LogsPanel.tsx` 已进入 workbench logs / review 语言
2. `history/components/*` 已进入统一 review workspace 语言
3. `settings/*` 已进入 settings workspace 语言
4. `resources/*` 已在 settings 内按资源工作区语义收口
5. `HistoryDetailRail.tsx` 已完成最小外层语义收口

### 2. 文档侧收口入口已完成

当前可以稳定写为：

1. `wave3-最小入口.md` 已有
2. `wave3-文件级实施计划.md` 已有
3. `wave3-收口评估.md` 已有
4. `wave3-多视图一致性检查清单.md` 已有
5. `wave3-截图补证入口.md` 已有

### 3. 构建侧已具备连续复验记录

当前可以稳定写为：

1. Wave 3 各轮最小收口后均已执行构建复验
2. 当前不存在已知的构建级阻塞

## 当前不能稳定表述为“已完成”的部分

### 1. 最终验收口径

当前不能写为：

1. “Wave 3 已全部完成”
2. “Wave 3 已全部验收”
3. “附属工作区收口已完全闭合”

## 在 `status.md` 中的推荐写法

如果需要在 `status.md` 中表达当前状态，建议使用类似表述：

### 状态

- **主控判断本并行 change 已具备收口条件，建议标记为可结束（并行），不切主推进。**

### 进行中

- **当前已无新增实现动作，后续仅保留按需引用和最小说明。**

### 下一步

- **如后续需要，只补最小引用说明或归档口径，不继续扩大代码 diff。**

## 在 `verify.md` 中的推荐写法

建议保留下面这类结论：

1. **Wave 3 在代码侧已基本具备阶段性收口条件**
2. **多视图一致性在代码与结构层面基本通过**
3. **当前页面证据已显著补齐，本并行 change 可结束（并行）**

并继续明确：

- **当前不建议直接写成“Wave 3 已全部完成”或“前端工作台重构已全部验收”。**

## 对后续推进的约束

在使用本口径期间，建议保持以下约束：

1. 不继续扩大代码 diff
2. 不再新开额外实现波次
3. 不切主推进
4. 不修改 `docs/11-hermes-rebuild/current-state.md`
5. 如需对外说明，优先引用 `wave3-收口交接说明.md`

## 何时从“保守收口”切换到“正式关闭”

主控当前已完成裁决：

1. 截图已补齐
2. walkthrough 已补齐
3. `verify.md` 已回填
4. 当前可使用：

- **“Wave 3 代码侧已基本收口，页面证据已显著补齐；本并行 change 可结束（并行），但不写成全部完成。”**

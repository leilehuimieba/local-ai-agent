# Design: 竞品用户迁移体验优化

## 设计原则

1. 小白优先：默认隐藏内部术语。
2. 任务范式优先：用“你可以让我做什么”替代“我有哪些模块”。
3. 安全感优先：强调改动前确认、验证证据和影响范围。
4. 连续性优先：突出“继续上次任务”。

## 改动点

### Runtime explain

`explain` 使用本地产品化文案，覆盖：

- 为什么用本项目
- 可以直接输入的任务范式
- 相比竞品的优势
- 安全边界

### 首页 / 工作台

优化示例 prompt 与引导文案：

- 检查当前项目状态
- 跑测试并定位失败
- 修改功能并保留验证证据
- 准备上线前检查

### 记录页

把“任务记录”进一步产品化为“工作历史”：

- 默认只显示时间、任务标题、状态、一句话结果
- 展开显示工具调用、验证结果、耗时

## 风险与回退

1. 文案变更可能影响测试断言：同步更新测试。
2. `explain` 静态文案可能与能力注册表不完全同步：后续可升级为 registry 驱动。
3. 回退方式：恢复 `explain` 旧文案与首页示例旧值。

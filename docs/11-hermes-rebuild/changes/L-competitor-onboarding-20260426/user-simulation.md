# 竞品用户真实使用场景模拟

更新时间：2026-04-26

## 用户画像

用户长期使用 ChatGPT / Claude / Cursor 类竞品，已经习惯自然语言输入、快速反馈和少量配置。他愿意尝试本项目的前提不是“能力更多”，而是本项目能更懂本地项目、能安全执行、能留下验证证据。

## 场景 1：第一次打开

### 用户心理

我不想读文档，也不想理解 runtime、gateway、change、Gate。我只想知道这个工具是否比竞品更适合推进我的本地项目。

### 当前体验

首页直接给出承诺：

- “把本地项目放心交给我推进”
- 强调不需要学习内部概念
- 强调理解项目状态、影响范围、执行修改与验证证据

快速入口：

- 检查项目状态
- 上线前检查
- 安全修改功能
- 继续上次任务

### 留存判断

用户能在 10 秒内理解差异化：这不是普通聊天工具，而是本地项目推进助手。

## 场景 2：询问“你能做什么”

### 用户心理

我希望它告诉我能完成哪些真实任务，而不是列技术模块。

### 当前体验

`explain` 返回任务范式：

- 检查项目状态
- 跑测试并定位失败
- 修改功能并说明影响范围
- 准备上线前检查
- 继续上次任务

并明确差异化：围绕本地工作区、项目文档、执行记录和验证证据持续推进。

### 留存判断

相比竞品，用户获得更强的“本地项目连续推进”感知。

## 场景 3：用户准备交付真实任务

### 用户输入

“帮我修改一个功能；先说明会影响哪些文件，再做最小改动并验证。”

### 当前体验预期

首页示例已把用户引导到安全修改范式：先影响范围，再最小改动，再验证。

### 留存判断

这个范式降低了竞品用户对自动改文件的恐惧。

## 场景 4：用户查看历史

### 用户心理

我想知道它上次到底做了什么，不想看 trace_id 和 event_type。

### 当前体验

记录页已改为“工作历史 / 工作时间线”：默认看任务、状态、一句话结果，展开后才看工具调用、验证结果和耗时。

### 留存判断

用户能把它当作工作历史，而不是日志控制台。

## 场景 5：上线前检查

### 用户心理

我要的是“能不能上线”的答案，不是让我记脚本名。

### 当前体验

首页提供“上线前检查”入口文案，prompt 已引导系统输出：失败原因、影响范围、不影响什么和建议修复。

### 留存判断

用户开始把本项目用于交付前质量把关，而不只是聊天问答。

## 回答三个核心问题

### 1. 他为什么要使用我们的项目？

因为竞品擅长回答问题，但本项目更适合推进本地项目：能读取当前项目状态，能执行命令和验证，能沉淀工作历史，并能围绕 change/status/verify 形成连续工作链路。

### 2. 如果他是竞品忠实使用者，我们怎么留住这个用户？

必须持续强化三件事：

1. 比竞品更懂他的本地项目上下文。
2. 比竞品更安全：改动前说明影响范围，关键动作可确认。
3. 比竞品更可验证：每次执行后留下测试、证据和工作历史。

### 3. 我们怎么给他更好的体验？

把内部工程治理翻译成用户语言：

- `change/status/verify` → 当前任务、进展、验证结果
- `logs/events` → 工作历史
- `provider/runtime/gateway` → 服务状态
- `run_failed` → 失败原因、影响范围、不影响项、建议修复

## 下一轮建议

1. 新增“影响范围卡片”：改动后自动列出修改文件、风险和回退方式。
2. 新增“上线前检查”独立页面或向导，不让用户面对脚本名。
3. Provider 健康状态独立展示，避免模型问题被误认为产品不可用。
4. `explain` 后续可从 capability registry 自动生成，避免静态文案漂移。

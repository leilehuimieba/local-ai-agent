## 1. 先看结论

如果你只想快速知道“现在看什么、做什么”，按下面三条执行：

1. 先看 `docs/README.md`，只认其中“当前唯一执行入口”。
2. 再看 `docs/11-hermes-rebuild/current-state.md`，确认当前阶段、当前 Gate、当前活跃 change。
3. 然后看 `docs/11-hermes-rebuild/changes/INDEX.md`，只用于定位变更目录和观察项。
4. 只有在主入口文档无法回答问题时，才回看 `docs/archive/` 或旧 `V1` 资料。

## 2. 当前状态来源（单一事实源）

1. 当前阶段、当前 Gate、当前活跃 change 统一以 `docs/11-hermes-rebuild/current-state.md` 为准。

4. 本指引不单独维护“当前阶段 / 当前 Gate / 当前活跃 change”字段；发现此类字段时应改为引用 `current-state.md`。

## 4. 一页式阅读顺序（避免看花）

1. `README`：确认当前唯一入口。
2. `current-state`：确认当前阶段、当前 Gate、当前活跃 change。
3. 总路线：确认阶段目标与 Gate。
4. 阶段计划：确认本阶段必做与失败回退。
5. `changes/INDEX`：定位活跃 change 和观察项目录。
6. 活跃 change 的 `tasks + status + verify`：确认今天具体动作。
7. 需要细节时，再跳到领域文档或历史证据。

## 5. 如何判断一份文档是否“过时”

满足任一条，可判定为“非当前执行输入”：

1. 位于 `docs/archive/`。
2. 文档自身声明“历史归档、只读参考、不再作为当前执行入口”。
3. 与 `docs/README.md` 的“当前唯一执行入口”不一致。
4. 与 `docs/11-hermes-rebuild/current-state.md` 冲突。
5. 与 `11-hermes-rebuild` 的总路线或阶段计划冲突。
6. 无法映射到当前阶段和当前 Gate。

## 6. 当前需要做什么（不是泛泛建议）

基于当前文档状态，下一步应先做“状态对齐”，再做实现：

1. 先在 `current-state.md` 确认主推进状态，再同步 `changes/INDEX.md` 导航。
2. 为下一条推进任务创建 change（至少补齐 `proposal/design/tasks/status/verify`）。
3. 任务实施时只按该 change 的 `tasks.md` 勾选推进，验证证据写入 `verify.md`。
4. 未满足当前 Gate 前，不切到下一阶段文档和实现。

## 7. 常见误区

1. 误区：看到 `V1` 就认为不能用。
   纠正：`V1` 文档有很多仍可参考，但默认不是第一执行入口。
2. 误区：`docs/07-test/evidence` 太多就全跳过。
   纠正：做 Gate 判定时应精确看对应样本，不需要全看。
3. 误区：任何文档写了阶段/Gate就可以直接作为执行口径。
   纠正：只以 `current-state.md` 作为状态裁决源，其他文档仅作上下文说明。

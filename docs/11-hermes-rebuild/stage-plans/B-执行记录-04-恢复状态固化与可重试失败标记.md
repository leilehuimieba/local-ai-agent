# B 阶段执行记录 04：恢复状态固化与可重试失败标记

更新时间：2026-04-10
阶段：B（进行中）

## 1. 本次目标

1. 把 checkpoint 从“只有 id 和 resumable”升级为“带恢复状态语义”。
2. 区分确认恢复与失败重试两类恢复原因。
3. 为后续阶段级恢复提供稳定 schema。

## 2. 本次改动范围

1. Rust：`crates/runtime-core/src/checkpoint.rs`
2. Rust：`crates/runtime-core/src/sqlite_store.rs`

## 3. 关键改动

### 3.1 checkpoint 状态字段升级

1. `RunCheckpoint.resume_reason`
2. `RunCheckpoint.resume_stage`

当前固定语义：

1. `confirmation_required -> PausedForConfirmation`
2. `retryable_failure -> Execute`
3. `none -> final_stage`

### 3.2 resumable 判定升级

1. 不再只按 `awaiting_confirmation` 判定。
2. 若 `RunResult.error.retryable=true`，checkpoint 也会被标记为可恢复。

### 3.3 SQLite schema 升级

1. `runtime_checkpoints` 表新增：
2. `resume_reason`
3. `resume_stage`
4. 兼容性通过 `alter table` 迁移补齐。

## 4. 测试补充

新增定向测试覆盖：

1. checkpoint 正常写入与事件插入。
2. confirmation 恢复路径。
3. retryable failure checkpoint 被标记为可恢复。

## 5. 验证结果

1. `cargo test -p runtime-core checkpoint -- --nocapture`：通过。
2. `cargo build`：通过。
3. `go build ./...`：通过。

## 6. 当前结论

1. B-04 已完成“恢复状态固化”。
2. checkpoint 现在具备最小恢复语义，不再只是布尔标记。
3. 后续做真正阶段级恢复时，可以直接基于 `resume_reason + resume_stage` 推进。

## 7. 下一步（B-05）

1. 为 retryable failure 增加 gateway 侧恢复入口。
2. 明确失败重试与确认恢复的幂等边界。
3. 决定是否引入真正的阶段级继续执行，而不是入口重入。

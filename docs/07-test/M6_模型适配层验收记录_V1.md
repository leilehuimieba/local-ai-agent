# 本地智能体 - M6 模型适配层验收记录 V1

更新时间：2026-04-02

状态：已完成实现，待最终人工复核。

## 1. 本轮目标

按 M6 要求，建立统一模型接入点，使 runtime 不再直接散落 provider 调用细节。

## 2. 实现范围

1. 新增 `ProviderConfig`
2. 新增 `ModelRequest / ModelResponse / ModelError`
3. 新增 `ModelAdapter` trait
4. 新增 `OpenAiCompatibleAdapter`
5. 新增 `complete_with_model()` 统一模型调用入口
6. `llm.rs` 改为只做统一入口封装

## 3. 修改文件清单

1. `crates/runtime-core/src/model_adapter.rs`
2. `crates/runtime-core/src/model_client.rs`
3. `crates/runtime-core/src/llm.rs`
4. `crates/runtime-core/src/contracts.rs`
5. `gateway/internal/contracts/contracts.go`
6. `frontend/src/shared/contracts.ts`

## 4. 新增结构与字段

### 4.1 新增结构

1. `ProviderConfig`
2. `ModelRequest`
3. `ModelResponse`
4. `ModelError`
5. `ModelAdapter`

### 4.2 当前 Provider 形态

当前正式验证 provider：

1. `provider_id = scnet`
2. `base_url = https://api.scnet.cn/api/llm/v1`
3. `model_id = MiniMax-M2.5`

## 5. 运行链路变化

调整前：

1. `llm.rs` 直接负责 body 文件写入、curl 请求和响应解析

调整后：

1. `llm.rs` 只保留统一入口
2. `model_client.rs` 负责请求构建
3. `model_adapter.rs` 负责 provider 调用与解析
4. 后续替换 provider 时不需要重写上层主循环

## 6. 手工测试步骤

1. 启动 `runtime-host`
2. 启动 `gateway`
3. 在设置中确认模型为 `scnet / MiniMax-M2.5`
4. 发送中文 `chat/run`
5. 检查返回结果、事件流和日志
6. 模型配置错误时，确认错误可以回流

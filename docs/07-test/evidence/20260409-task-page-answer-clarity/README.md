# 任务页回答分层与右栏收口证据

## 本轮目标

本轮只补齐任务页体验收口的页面级证据，证明当前页面已经能够区分正式回答、恢复结果、系统说明，并证明右侧栏主视野已经从调试面板收紧为任务辅助信息。

## 留证方式

本轮未修改前端逻辑。页面证据来自本地前端真实页面 `http://127.0.0.1:4173/` 的受控复现：

1. 启动 `frontend` 本地开发环境。
2. 在真实任务页中通过浏览器控制台动态导入 `frontend/src/runtime/state.ts`。
3. 用 `useRuntimeStore.setState(...)` 注入受控运行态，让页面分别渲染三种结果模式。

该方式只用于页面留证，不会写回仓库代码，也不新增接口或共享合同字段。

## 证据文件与映射

### 1. 正式回答证据

文件：`task-page-formal-answer-and-sidebar.png`

页面可见内容：

- 主消息区顶部出现 `正式回答 / 正式`
- 结果摘要块标题为 `正式回答`
- 页面下方仍保留简化后的 `当前进度`

该图同时也是右栏收口证据，因为右侧主结构只保留：

- `当前任务`
- `下一步与最近动作`
- `状态沉淀与续接依据`
- `风险与记忆`

图中未出现 repo 主区块，也没有把仓库路径、文档路径等开发者视角内容放在右栏主视野。

### 2. 恢复结果证据

文件：`task-page-recovery-result.png`

页面可见内容：

- 主消息区顶部出现 `恢复结果 / 恢复`
- 内容以恢复路径说明为主，不再伪装成正式回答
- 页面仍保留当前进度条带，但过程展示已收紧到少量关键动作

该图用于证明“恢复路径结果”已经和正式回答做了明确区分。

### 3. 系统说明证据

文件：`task-page-system-status.png`

页面可见内容：

- 主消息区顶部出现 `系统说明 / 说明`
- 摘要块标题为 `当前状态`
- 说明文案明确写出：当前没有新的正式答复输出，页面只展示本轮系统状态和下一步线索

该图用于证明“系统说明”不会再默认混入正式答复，而是被单独标识为状态型输出。

## 为什么这些证据足以支持本轮放行

本轮放行重点不是继续开发，而是确认上一轮体验收口已经在真实页面中成立。三张截图共同覆盖了：

1. 正式回答
2. 恢复结果
3. 系统说明
4. 右栏收口后不再出现 repo 主区块

因此它们可以直接回答验收关注点：

- 用户是否能看出当前看到的是正式回答还是系统性说明
- 恢复路径是否仍在冒充最终答案
- 任务页右栏是否还以开发者调试信息为主

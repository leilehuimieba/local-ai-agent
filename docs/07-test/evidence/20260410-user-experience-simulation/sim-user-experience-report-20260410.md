# 真实用户模拟与同类体验对比（2026-04-10）

## 1. 模拟范围

本次按两类用户做了 6 条真实问句模拟：

1. 新手用户（3 条，四六级备考场景）
2. 程序员用户（3 条，项目推进/读文档/验收判断场景）

原始样本：

1. `sim-user-experience-before-20260410.json`
2. `sim-user-experience-after2-20260410.json`
3. `sim-user-experience-compare-summary-20260410.json`

## 2. 实测结果（修补前后）

修补前（before）：

1. 新手 3 条四六级问句全部落到 `recovery`，且回复偏“通用技术兜底”，可执行性弱。
2. 程序员“是否可开始验收”问句落到 `project_answer`，更像项目介绍，不够决策化。

修补后（after2）：

1. 新手 3 条四六级问句全部转为 `answer`，且命中稳定模板（第一步、30 分钟计划、背词复习法）。
2. 程序员“是否可开始验收”问句转为 `session_context` 的稳定模板，给出“结论+依据+下一步”。
3. 其余工程向读文件问句保持 `answer`，未出现回退。

## 3. 已修补问题

本轮做的是最小修补，不扩接口、不改共享合同、不改主循环：

1. 在 `context` 增加 3 个四六级高频模板：
   `cet_first_step_template`、`cet_daily_plan_template`、`cet_vocab_review_template`
2. 在 `planner/context` 增加“验收就绪问句”最小路由与模板：
   让“是否可以开始验收”直接得到可执行结论。

代码位置：

1. `crates/runtime-core/src/executors/context.rs`
2. `crates/runtime-core/src/planner.rs`

## 4. 同类型项目体验对比（OpenClaw 类）

### 4.1 OpenClaw（架构与体验）

从 OpenClaw 官方架构文档看，系统是 `Gateway + Agent Host + Runtime` 分层，强调多会话、异步执行、跨设备连接和 CLI/移动端入口。[来源](https://docs.openclaw.ai/architecture)

对新手体感：

1. 好处：能力强、可扩展、执行型任务覆盖广。
2. 成本：概念多、组件多，初次使用理解门槛较高。

对程序员体感：

1. 好处：可控性高，适合复杂自动化任务编排。
2. 成本：调试面更大，失败定位链路更长。

### 4.2 OpenHands（开发者代理）

OpenHands 的定位是软件开发代理，官方本地部署文档涉及 Docker、LLM 配置、运行参数等步骤。[来源](https://docs.openhands.dev/openhands/usage/local-setup)

对新手体感：

1. 不适合“零技术用户”直接上手，操作负担偏高。

对程序员体感：

1. 更接近工程研发流，适合代码任务，但对“学习陪练”并非主场景。

### 4.3 Continue（IDE 内协作）

Continue 的优势是把能力嵌入 IDE，启动路径短、对程序员友好。[来源](https://docs.continue.dev/cli/quickstart)

对新手体感：

1. 依赖 IDE 语境，学习型场景（如四六级）不够自然。

对程序员体感：

1. 上手快，适合“边写边问”。
2. 但跨任务流程编排能力通常不如完整 Agent Runtime。

### 4.4 对我们项目的启发

1. 新手场景不能把质量押注在实时模型稳定性上，必须有最低可用模板与兜底。
2. 程序员场景需要“可执行结论”，不能只给概述。
3. 能力越强越要强调收口边界和验证证据，否则会出现“看起来很强、落地不稳”。

## 5. 新手与程序员体感结论

新手（四六级）：

1. 修补前：容易得到“你先补信息”的模板化兜底，挫败感强。
2. 修补后：能直接拿到可执行动作，第一步更清晰。

程序员：

1. 修补前：部分问句会变成“项目介绍”，对决策帮助有限。
2. 修补后：关键验收判断问句已可直接返回结论与动作。

## 6. 四六级场景当前可提供的帮助

当前系统可以稳定提供的最小价值：

1. 30 天起步第一步（当天可执行三步）
2. 每天 30 分钟固定学习节奏（听力/词汇/阅读翻译）
3. 词汇遗忘问题的可执行复习法（1-3-7-14）

这三项都已通过真实 run 样本验证为 `completed + answer + verified`。

## 7. 仍需观察的点

1. 非模板覆盖的新问法，仍可能受 provider 波动影响。
2. 学习类模板目前是“最低可用版”，后续可在不扩功能前提下迭代为“按周追踪+错题回看”。
3. Agent 型系统的工具权限与安全边界仍需持续验证。相关研究显示，具备工具调用能力的代理系统在复杂环境下对安全策略与隔离机制要求更高。[来源](https://arxiv.org/abs/2604.04759)

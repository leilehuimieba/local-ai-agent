## 1. 文档目的

本文用于回答本轮前端重构是否已经真正完成验收闭环，重点确认：

1. 前端是否已经从页面驱动推进到任务状态驱动，并完成 `runState / connectionState` 的正式接管。
2. `App.tsx`、工作台、日志页、设置页是否已经按领域边界完成一轮真实收口。
3. 风险确认、连接提示、严重错误是否已经提升为全局层，不再依赖局部页面偶然可见。
4. 主链路在重构过程中是否保持可构建、可运行、可手工回归。
5. 是否允许在当前基础上继续进入下一阶段体验深化，而不是继续回到结构补课。

## 6. 状态模型验收

### 6.1 任务状态模型

实际填写：

1. 状态定义：`已正式落地 idle / submitting / streaming / awaiting_confirmation / resuming / completed / failed / archived，并由 runtime store 统一承载。`
2. 迁移规则：`提交任务后进入 submitting；收到 run_started 或流事件后进入 streaming；收到 confirmation_required 进入 awaiting_confirmation；批准确认后进入 resuming；收到 run_finished 进入 completed；收到 run_failed 进入 failed。`
3. 风险点或偏差：`archived 已完成类型与 store 挂位，但本轮正式验收中未覆盖归档动作的真实前台入口。`

### 6.2 连接状态模型

实际填写：

1. 状态定义：`已正式区分 connecting / connected / reconnecting / disconnected / closed。`
2. 展示位置：`应用顶部状态带、对话页状态区、全局连接提示层均已消费正式 connectionState。`
3. 风险点或偏差：`断连恢复目前以用户点击“重新连接”为主；恢复成功后，严重错误提示不会自动撤下，仍需手动关闭。`

### 6.3 事件到状态映射

实际填写：

1. 映射规则：`frontend/src/events/useSessionEventStream.ts` 只负责 EventSource 生命周期、连接态和事件转发；运行事件统一交给 runtime store 做状态迁移；App.tsx 不再散落写迁移分支。`
2. 已清理的旧逻辑：`移除了页面侧靠中文状态文案反推业务状态的旧写法，也修掉了 zustand 选择器未浅比较造成的死循环问题。`
3. 残留问题：`个别展示仍保留中文标签，但只承担展示职责，不再反向驱动状态逻辑。`

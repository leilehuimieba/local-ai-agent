# 本地智能体 - M5 验证链与完成判定验收记录 V1

更新时间：2026-04-02

状态：已完成实现，待最终人工复核。

## 1. 本轮目标

按 M5 要求，把 `Verify` 从阶段名升级为实际验证能力，并让 `Finish` 基于显式完成判定收口。

## 2. 实现范围

1. 新增 `VerificationNeed`
2. 新增 `VerificationOutcome`
3. 新增 `VerificationReport`
4. 新增 `CompletionDecision`
5. 主循环中接入 `verify_tool_execution()`
6. `RunResult.status` 改为基于 `CompletionDecision`

## 3. 修改文件清单

1. `crates/runtime-core/src/verify.rs`
2. `crates/runtime-core/src/completion.rs`
3. `crates/runtime-core/src/query_engine.rs`
4. `crates/runtime-core/src/lib.rs`
5. `crates/runtime-core/src/contracts.rs`

## 4. 新增结构与字段

### 4.1 新增结构

1. `VerificationNeed`
2. `VerificationOutcome`
3. `VerificationReport`
4. `CompletionDecision`
5. `VerificationSnapshot`

### 4.2 回流字段

当前会通过事件 metadata 回流：

1. `verification_code`
2. `verification_passed`
3. `verification_summary`
4. `completion_status`
5. `completion_reason`

## 5. 运行链路变化

调整前：

1. `Verify` 阶段主要是文案型事件
2. `Finish` 更接近“动作成功就结束”

调整后：

1. `execute_stage` 后会生成 `VerificationReport`
2. `Finish` 前会生成 `CompletionDecision`
3. `verification_completed` 事件详情来自真实验证结果
4. `RunResult.status` 使用显式完成判定

## 6. 手工测试步骤

1. 发起正常只读或项目问答任务
2. 检查 `verification_completed` 事件
3. 确认 `metadata` 中有验证字段
4. 发起失败任务
5. 确认失败时仍有验证结果与完成判定

## 7. 构建验证结果

1. `cargo build -p runtime-host`：通过
2. `go build ./...`：通过
3. `npm run build`：通过

## 8. 未完成项

2. 还没有更复杂的子目标级别验证器

## 9. 偏离本规范的地方

1. 未单独建立多层 completion 策略模块
2. 仍复用当前主循环结构，没有把 Verify 再拆成更细阶段

## 1. 文档目的

本文件用于回答以下问题：

1. Rust 运行时在新系统中到底负责什么
2. Rust 运行时不应该负责什么
3. 第一版运行时的最小状态机应该是什么
4. 运行时应该如何组织记忆、能力、风险和事件
5. 如何避免把原 Python runtime 的复杂性原样搬到 Rust
6. 后续 crate 拆分和实现顺序该如何围绕运行时展开

本文件不是最终接口定义文档，也不是最终 crate 目录结构文档。
它是运行时层面的设计约束文档。

---

## 2. 运行时定位

Rust 运行时是新系统中的“执行中枢”和“任务推进中枢”，不是前端，也不是控制面。

它的核心角色是：

1. 理解当前任务
2. 决定下一步动作
3. 组织能力调用
4. 观察动作结果
5. 判断是否继续、暂停、确认、失败或完成
6. 压缩上下文并沉淀关键记忆

一句话定义：

> Go 负责系统对外协调，Rust 负责系统内部推进任务。

---

## 3. 设计目标

第一版 Rust 运行时应同时满足以下目标：

1. 轻量
2. 低占用
3. 低 token 浪费
4. 可模块化拆分
5. 可长期维护
6. 支持主链路可观测
7. 支持风险控制
8. 支持未来成长

### 3.1 运行时层面的具体目标

1. 让单智能体稳定完成文件、命令、工作区和知识相关任务
2. 让执行过程持续可见
3. 让高风险动作可被拦截和确认
4. 让记忆系统按需工作，而不是全量灌入
5. 让模型看到更少、更粗粒度、更高价值的能力接口

### 3.2 运行时层面的体验目标

1. 响应尽量快
2. 不因过度包装而拖慢实际使用
3. 不把用户常见任务搞成过长流程
4. 在危险动作前给出足够清楚的说明

---

## 4. 非目标

为了保证第一版可实现，Rust 运行时明确不承担以下目标：

1. 不做多智能体编排
2. 不做浏览器自动化核心
3. 不做桌面 GUI 自动化核心
4. 不做复杂调度系统
5. 不做自动 fallback 和模型测速
6. 不做重型研究框架式 candidate scoring 系统
7. 不做大量彼此强耦合的实验性 bias / suppression / entropy 机制

---

## 5. 核心设计原则

### 5.1 最小状态机优先

第一版运行时必须保持最小可解释状态机，不允许演化成一个巨型隐式流程。

### 5.2 Contract First

运行时与外界交互必须建立在稳定合同上，而不是隐式对象结构上。

### 5.3 Module Boundary First

运行时内部模块边界必须先于实现存在。

### 5.4 Token Efficiency First

运行时必须主动减少：

1. 模型可见工具数量
2. 冗长原始输出回灌
3. 全量记忆加载
4. 不必要的重复推理

### 5.5 安全和确认不是附属能力

风险判断和确认机制是运行时主路径的一部分。

## 6. 第一版运行时最小状态机

第一版建议的主状态机如下：

1. `Analyze`
2. `Plan`
3. `Execute`
4. `Observe`
5. `Verify`
6. `Finish`

旁路状态：

1. `PausedForConfirmation`
2. `Failed`

可选后续扩展状态：

1. `Recovering`

### 6.1 状态说明

#### Analyze

负责：

1. 解析用户任务
2. 识别任务类型
3. 判断当前任务是否可直接回答或需要行动

#### Plan

负责：

1. 形成当前最小执行计划
2. 决定下一步动作
3. 决定是否需要读记忆、读知识、调能力

#### Execute

负责：

1. 调用能力
2. 执行动作
3. 收集原始结果

#### Observe

负责：

1. 解析执行结果
2. 提取结果摘要
3. 更新当前任务状态

#### Verify

负责：

1. 判断当前动作是否达成子目标
2. 判断是否需要继续下一步
3. 判断是否需要用户确认
4. 判断是否已可结束

#### Finish

负责：

1. 生成最终答复
2. 生成完成摘要
3. 决定是否写入长期记忆或日报素材
4. 输出最终事件

#### PausedForConfirmation

负责：

1. 暂停高风险动作
2. 产出确认请求
3. 等待控制面回传用户决策

#### Failed

负责：

1. 统一错误收口
2. 输出失败原因
3. 输出失败摘要
4. 为复盘和日志保留必要信息

---

## 7. 状态转移规则

建议的主转移如下：

1. `Analyze -> Plan`
2. `Plan -> Execute`
3. `Execute -> Observe`
4. `Observe -> Verify`
5. `Verify -> Plan`
6. `Verify -> Finish`
7. `Verify -> PausedForConfirmation`
8. 任意主状态 -> `Failed`

确认完成后的转移：

1. `PausedForConfirmation -> Execute`
2. `PausedForConfirmation -> Failed`
3. `PausedForConfirmation -> Finish`

说明：

1. 第一版不建议设计过多中间状态。
2. 第一版不建议允许状态机出现大量环形隐式跳转。

---

## 8. 运行时职责边界

### 8.1 运行时负责什么

1. 任务分析
2. 当前步计划
3. 能力选择
4. 结果解析
5. 风险中断
6. 记忆调度
7. 阶段事件生成
8. 最终结果生成

### 8.2 运行时不负责什么

1. 不负责前端页面状态管理
2. 不负责 HTTP 路由
3. 不负责桌面启动器逻辑
4. 不负责前端设置页具体呈现
5. 不负责浏览器扩展
6. 不负责复杂运维面板

### 8.3 与 Go 控制面的边界

Go 负责：

1. API
2. 会话
3. 事件分发
4. 设置
5. 模式外显
6. 风险确认交互桥接

Rust 负责：

1. 执行决策
2. 能力调用
3. 结果判断
4. 记忆调用
5. 任务推进

---

## 9. 运行时内部模块划分建议

第一版建议按职责拆成以下运行时模块，而不是按旧 Python 文件名迁移。

### 9.1 `runtime-core`

负责：

1. 状态机
2. run 生命周期
3. step 组织
4. 统一上下文对象
5. 主循环调度

### 9.2 `runtime-planner`

负责：

1. 任务分析
2. 子目标提取
3. 当前步计划
4. 下一步能力选择建议

第一版只保留最少必要计划逻辑，不引入复杂研究型打分器。

### 9.3 `runtime-executor`

负责：

1. 执行动作
2. 调用具体能力
3. 统一封装执行结果
4. 控制超时与取消

### 9.4 `runtime-router`

负责：

1. 模块路由
2. 能力组路由
3. 记忆召回范围控制
4. Skill / MCP / native 能力选择

### 9.5 `runtime-memory`

负责：

1. 工作记忆
2. 长期记忆
3. 记忆检索
4. 记忆写入
5. 上下文压缩

### 9.6 `runtime-verify`

负责：

1. 结果验证
2. 继续/完成判断
3. 是否需要确认判断

### 9.7 `runtime-risk`

负责：

1. 风险判断
2. 危险操作识别
3. 确认请求构建
4. 替代方案建议

### 9.8 `runtime-events`

负责：

1. 生成阶段事件
2. 生成日志摘要
3. 统一事件 envelope

### 9.9 `runtime-output`

负责：

1. 最终答复生成
2. 最终摘要生成
3. 结果压缩
4. 结束态输出

## 10. 建议的核心对象模型

第一版不需要过度复杂的数据结构，但至少建议保留以下核心对象概念。

### 10.1 `RunContext`

表示一次任务运行的稳定上下文，例如：

1. 会话标识
2. 当前工作区
3. 当前模式
4. 当前模型
5. 用户输入目标
6. 必载上下文摘要

### 10.2 `RunState`

表示一次任务运行的动态状态，例如：

1. 当前状态机阶段
2. 当前步编号
3. 计划摘要
4. 最近观察摘要
5. 是否等待确认
6. 是否完成
7. 是否失败

### 10.3 `StepRecord`

表示单步动作的记录，例如：

1. 第几步
2. 执行动作是什么
3. 调用了什么能力
4. 结果摘要
5. 是否成功

### 10.4 `ActionRequest`

表示运行时准备执行的单次动作。

### 10.5 `ActionResult`

表示能力层返回的动作结果。

### 10.6 `VerificationResult`

表示验证层对当前动作的判断结果，例如：

1. 继续
2. 结束
3. 需要确认
4. 失败

### 10.7 `MemoryPacket`

表示从记忆系统召回的相关内容包。

### 10.8 `FinalReport`

表示运行结束后返回给控制面和前端的最终结果对象。

## 11. 记忆系统在运行时中的嵌入方式

### 11.1 必载层

每次任务开始时只注入最小必要信息：

1. 系统规范
2. 安全约束
3. 用户稳定偏好
4. 当前工作区或项目的高层摘要

### 11.2 按需检索层

在 `Analyze` 或 `Plan` 阶段，根据任务相关性召回：

1. 项目知识
2. 历史经验
3. 失败教训
4. 用户偏好块

### 11.3 工作记忆层

在运行过程中维护：

1. 当前目标
2. 当前计划
3. 最近关键动作
4. 当前未完成事项

### 11.4 沉淀层

在 `Finish` 阶段决定是否写入：

1. 长期记忆
2. 每日记录素材
3. 经验教训

### 11.5 运行时约束

运行时必须避免：

1. 每次都全量加载所有长期记忆
2. 让模型直接面对全部记忆块
3. 让冗长历史记录占满上下文

## 12. Skills 与能力调度在运行时中的位置

### 12.1 能力来源类型

第一版架构允许三类能力来源：

1. Rust 原生能力
2. Skill 包
3. MCP 适配能力

### 12.2 运行时策略

运行时不应让模型直接面对海量底层动作。

应先由 `runtime-router` 做：

1. 任务领域判断
2. 能力组判断
3. 能力来源选择

### 12.3 技术策略

优先级建议如下：

1. 原生能力优先
2. Skill 包其次
3. MCP 作为标准接入能力

说明：

1. 这里的优先级不是绝对规则，而是第一版为了效率和轻量化的经验性建议。
2. 若某能力通过 MCP 更稳定，也可直接采用 MCP。

### 12.4 token 约束

运行时应把 Skill 和 MCP 的复杂性封装在本地，不应把所有底层细节回灌给模型。

---

## 13. 风险控制在运行时中的嵌入方式

### 13.1 风险判断时机

风险判断应至少发生在：

1. 执行动作前
2. 高影响范围动作前
3. 删除类动作前
4. 明显危险命令前

### 13.2 风险判断维度

运行时内部应支持：

1. 动作类型
2. 目标路径敏感度
3. 影响范围
4. 可逆性
5. 是否属于计划内动作
6. 是否存在替代方案

### 13.3 风险结果

风险结果可以内部区分：

1. 低风险
2. 中风险
3. 高风险
4. 不可逆风险

对前端可转成：

1. 直接执行
2. 需要确认

或按需要展示完整等级。

### 13.4 确认请求对象应包含

1. 动作是什么
2. 为什么执行
3. 影响范围
4. 目标路径
5. 是否可回退
6. 潜在危害
7. 风险等级
8. 可选替代方案

---

## 14. 事件与可观测性模型

第一版运行时必须产生可供前端展示和复盘的阶段事件。

### 14.1 事件目标

1. 让用户知道系统在做什么
2. 让日志页能复盘任务推进过程
3. 让问题排查有可用线索

### 14.2 建议的基础事件类型

1. `run_started`
2. `analysis_ready`
3. `plan_ready`
4. `action_requested`
5. `action_completed`
6. `confirmation_required`
7. `verification_completed`
8. `memory_written`
9. `run_failed`
10. `run_finished`

### 14.3 事件内容原则

每个事件不应无节制地携带原始长文本。

应优先输出：

1. 阶段名称
2. 摘要
3. 关键元信息
4. 与用户理解相关的少量上下文

## 15. token 效率设计要求

Rust 运行时必须主动承担 token 优化责任，而不是把 token 浪费完全交给模型端承担。

### 15.1 必须做的约束

1. 工具暴露粗粒度化
2. 工具返回结果先本地摘要
3. 记忆按需加载
4. 长上下文分层压缩
5. 能力先分模块路由再检索

### 15.2 不建议的做法

1. 暴露几十个细碎底层动作给模型
2. 把命令原始输出全文塞回上下文
3. 每一步都重新加载大段历史
4. 对所有 Skills 做全量扫描

---

## 16. 错误处理与恢复原则

第一版不做复杂恢复系统，但必须具备基本错误收口能力。

### 16.1 必须具备

1. 统一错误对象
2. 统一失败摘要
3. 统一失败事件
4. 对用户可理解的错误提示

### 16.2 第一版可不做

1. 复杂自动修复策略
2. 复杂失败回放
3. 大量恢复偏置和打分器

### 16.3 后续可扩展

在第二版或后续版本，可引入更明确的恢复状态：

1. `Recovering`
2. `RetryWithAdjustment`
3. `FallbackCapability`

但第一版不要过早设计重型恢复系统。

---

## 17. 明确不继承的旧复杂性

为了让 Rust 运行时可维护，以下旧复杂性默认不应直接迁入第一版：

1. 过度细粒度的候选评分体系
2. 大量彼此耦合的 suppression / bias / entropy 模块
3. 多套接近但重复的 completion proof / dossier / route / contract 包装体系
4. 研究型的元诊断堆叠
5. 过长且职责混杂的主循环结构

第一版应保留的是“职责”，而不是“旧实现的全部形态”。

## 21. 使用规则

1. 后续 Rust 运行时实现不得突破本文件中的职责边界。
2. 若新增运行时模块，必须说明该模块是否属于第一版必需。
3. 若后续有人尝试把旧 Python 的研究型复杂性直接搬进来，应优先回到本文件判断是否违背第一版目标。

## 2. 项目定位

新项目定位为：

> 一个本地优先、桌面启动、以 Web 控制台为主界面的单智能体控制平台。

长期方向不是单纯的聊天助手，而是逐步演进为具备以下能力的本地智能体系统：

1. 能理解和推进用户任务
2. 能操作文件、工作区、命令和知识资源
3. 能沉淀长期记忆、日报和经验教训
4. 能在未来逐步扩展到浏览器、桌面应用、语音和更多本地能力
5. 最终朝“本地版 Jarvis”方向演进

---

## 3. 产品原则

### 3.1 总体原则

1. 新项目不追求与原项目逐文件、逐模块、逐行为等价。
2. 新项目只吸收原项目中对正式产品主链路有价值的部分。
3. 新项目第一版以最小可用闭环为优先目标。
4. 新项目从第一天开始按模块化、文档驱动、阶段开发的方式推进。
5. Go 保留控制面角色，Rust 负责替代 Python 运行时与执行层。
6. 第一版聚焦个人长期使用，同时按“他人可用”的产品标准设计。

### 3.2 体验原则

1. 轻量级：代码尽量少、运行占用尽量低、存储占用尽量低。
2. 无感化：实际工作时尽量融入电脑环境，而不是频繁打断用户。
3. 体验优：核心流程流畅、反馈明确、风险可控、操作直觉。
4. 不浪费 token：新增能力必须考虑 token 成本和实际收益。
5. 成本效益优先：优先建设高收益、高频、低阻力能力。

### 3.3 工程原则

1. 架构允许多层分层，不限定为两层。
2. 文件夹层级必须模块化。
3. 功能实现必须模块化。
4. 避免冗余代码。
5. 提高代码复用性。
6. 不允许把新项目重新长成另一个“大 loop.py”。

## 5. 第一版 MVP 范围

### 5.1 第一版必须具备的核心能力

第一版的智能体必须至少具备：

1. 能聊天
2. 能调用工具
3. 能返回结果
4. 最好能执行系统中的常见操作

### 5.2 第一版必须交付的能力域

1. 命令执行
2. 文件读写
3. 工作区操作
4. 知识检索
5. 记忆系统

### 5.3 第一版界面范围

第一版前端只保留以下页面：

1. 首页工作台
2. 日志页
3. 设置页

第一版不做：

1. 复杂运维面板
2. 多面板业务工作台
3. 多智能体面板
4. 浏览器扩展界面

### 5.4 第一版后置能力

以下能力明确后置：

1. 浏览器自动化
2. 桌面自动化
3. 任务调度
4. 语音输入输出
5. 多智能体协作
6. 复杂 GUI 控制
7. 全系统所有应用控制
8. 自动测速和自动 fallback

### 5.5 第一版接受的明确限制

第一版只承接以下结果与任务类型：

1. 文本型工具结果
2. 文件类任务
3. 命令类任务
4. 工作区类任务
5. 知识检索类任务

## 6. 运行模式

第一版保留三种模式：

1. 观察模式
2. 标准模式
3. 全权限模式

### 6.1 观察模式

特点：

1. 以分析、检查、建议为主
2. 不主动执行修改类动作
3. 适合先看方案、先做只读分析

### 6.2 标准模式

特点：

1. 作为默认模式
2. 常见读写行为允许执行
3. 高危行为仍受确认机制控制

### 6.3 全权限模式

特点：

1. 面向高频使用场景
2. 尽量减少普通操作阻断
3. 但对高危和不可逆动作仍然必须人工确认

不保留“严格模式”。

---

## 7. 工作区与权限边界

### 7.1 工作区模型

第一版不是单一固定工作区，而是：

1. 支持多个工作区
2. 支持工作区切换
3. 允许跨工作区访问
4. 默认以当前工作区为操作中心

### 7.2 系统范围能力方向

长期来看，系统需要具备更大范围的本地文件管理能力，甚至逐步扩展到更广的系统控制能力。

但第一版不把“全盘控制能力”作为验收目标。
第一版只要求架构上为后续扩展保留可能性，不堵死未来路径。

### 7.3 新目录访问交互

首次接触新目录时，系统应提供提示。

建议交互：

1. 弹出提示框或显式交互提醒
2. 允许用户选择本次继续或取消
3. 提供“默认不提醒 / 记住选择”之类的选项
4. 后续可在设置中管理目录授权策略

## 8. 风险控制与确认机制

### 8.1 风险展示与内部判断

系统内部应保留风险分级能力。
对外可以展示风险等级，当前冻结为：

1. 低风险
2. 中风险
3. 高风险
4. 不可逆风险

说明：

1. 工程上保留细粒度风险判断是必要的。
2. 界面上不必过度复杂，但要让用户能理解当前动作的风险性质。

### 8.2 默认执行原则

1. 常见、低风险、日常型操作可直接执行。
2. 文件修改、增加内容等常规开发动作默认不需要每次确认。
3. 高危和不可逆动作必须人工确认。

### 8.3 文件操作风险原则

#### 读文件

1. 默认风险较低
2. 首次接触新目录时可提示

#### 写文件

1. 默认允许
2. 必须记录日志
3. 若目标路径敏感或范围异常，可升级风险判断

#### 删除文件

1. 属于敏感动作
2. 必须结合重要性、影响范围和可逆性判断
3. 高风险删除必须人工确认

### 8.4 风险判断维度

风险判断采用综合判断，不依赖单一指标。

建议至少考虑：

1. 动作类型
2. 可逆性
3. 影响范围
4. 目标路径敏感度
5. 命令危险特征
6. 是否属于当前计划
7. 是否会影响系统整体稳定性

### 8.5 替代方案原则

如果存在更安全的替代方案，系统应优先采用替代方案。

例如：

1. 可移动到回收站时，优先不做永久删除
2. 可只读检查时，优先不直接覆盖修改
3. 可局部处理时，优先不做大范围处理

### 8.6 高危确认时必须展示的信息

在执行高危动作前，至少展示：

1. 在做什么
2. 影响范围
3. 目标路径
4. 为什么要这么做
5. 是否可回退
6. 潜在危害
7. 当前风险等级

## 9. 记忆系统

### 9.1 第一版记忆目标

第一版必须支持：

1. 会话内上下文连续
2. 重要信息长期沉淀
3. 重要事项写入长期记忆或每日记忆
4. 上下文压缩
5. 失败教训沉淀
6. 用户偏好沉淀

### 9.2 记忆调用原则

不是每次进入工作区都加载所有记忆。

记忆必须按需加载：

1. 必载记忆只加载最小必要集合
2. 其他记忆按任务相关性召回
3. 长期记忆应分块，而不是做成单体大记忆池

### 9.3 建议的记忆分层

#### 必载层

包含：

1. 系统执行规范
2. 安全约束
3. 用户稳定偏好
4. 当前工作区/当前项目高层摘要

#### 模块路由层

根据任务类型决定应该检索哪些模块的记忆与能力。

#### 按需检索层

只召回当前任务需要的：

1. 项目知识
2. 历史经验
3. 失败教训
4. 个人偏好
5. 相关长期记忆块

#### 会话工作层

包含当前任务的：

1. 目标
2. 最近关键动作
3. 当前发现
4. 未完成子任务

#### 沉淀层

包含长期保留的：

1. 项目知识
2. 个人偏好
3. 操作经验
4. 失败教训
5. 每日日报

### 9.4 上下文压缩原则

必须保留：

1. 项目整体理解
2. 稳定关键事实
3. 当前目标与约束
4. 已确认的重要结论

可以压缩：

1. 中间执行步骤
2. 已完成动作的详细流水
3. 临时性细节
4. 低价值冗长输出

可以忽略或归档：

1. 无复用价值的琐碎过程
2. 任务完成后不再重要的低价值细节

### 9.5 反思与教训

系统应支持记录失败教训和反思信息，以帮助后续避免重复犯错。

建议后续将长期记忆至少分为：

1. `project_knowledge`
2. `preference`
3. `daily_note`
4. `lesson_learned`

## 10. Skills、能力与可成长性

### 10.1 总体方向

系统应具备“可成长性”。
后续可以通过增加数据、增加能力块、增加 Skills 来持续进化。

### 10.2 Skills 的组织原则

Skills 不应全量平铺读取，而应按模块或领域分层组织。

例如：

1. 开发模块
2. 文件管理模块
3. 日常助理模块
4. 知识检索模块
5. 系统操作模块
6. 后续可扩展的媒体/浏览器/桌面模块

### 10.3 Skills 调用原则

1. 先做模块级筛选
2. 再做模块内检索
3. 不在每次任务中全量读取全部 Skills

### 10.4 能力来源

系统后续可支持多种能力来源共存：

1. Rust 原生能力
2. Skill 包
3. MCP 适配能力

能力调度必须服务于两个目标：

1. 提高任务完成效率
2. 降低 token 消耗

## 11. 模型与 Provider

### 11.1 第一版要求

1. 单智能体
2. 支持手动切换不同模型
3. 最好支持不同厂商模型切换
4. 模型不可用时要明确提示

不可用提示包括但不限于：

1. 超时
2. 鉴权失败
3. 配额不足
4. 余额不足
5. 接口异常

### 11.2 第一版不做的内容

第一版后置：

1. 自动 fallback
2. 模型测速
3. 模型自动排序
4. 复杂模型路由策略

### 11.3 前台交互原则

模型切换尽量在 UI 中完成，而不是通过代码手动修改。

## 12. 日志与可观测性

### 12.1 第一版日志面板定位

日志面板服务于：

1. 复盘
2. 测试
3. 遇到问题时排查

### 12.2 第一版至少应能看到

1. 当前正在做什么
2. 调用了什么工具
3. 工具返回了什么结果摘要
4. 是否触发确认
5. 是否报错
6. 当前使用的模型
7. 任务如何结束

### 12.3 进度反馈要求

前端必须能看到任务执行过程，而不是只看到最终结果。

至少要支持：

1. 工具调用流
2. 阶段状态变化
3. 结果摘要回流

## 13. 前端方向

前端第一版允许重写，不要求复刻原项目界面。

前端原则：

1. 轻量
2. 响应快
3. 性能好
4. 适合长期维护
5. 适合模块化开发

当前建议方向：

1. 继续采用 TypeScript
2. 前端框架优先考虑成熟生态与工程效率
3. 首版正式界面固定为首页工作台、日志页、设置页三部分
4. 具体视觉语言与界面实现以 V2 前端方案文档为准

具体前端技术选型在后续《前端技术方案》文档中确定。

## 14. Rust 化目标

Rust 替代 Python 的目标不是机械翻译旧 runtime，而是重建一套更清晰的执行层。

Rust 化核心目标：

1. 更低运行占用
2. 更快执行速度
3. 更强模块边界
4. 更适合长期维护
5. 更适合多层能力路由
6. 不牺牲实际使用体验

Rust 设计必须兼顾：

1. 调用速度
2. 实际使用体验
3. 架构可成长性
4. token 成本
5. 风险控制

## 15. 端口与环境隔离

新项目必须与原项目隔离运行。

要求：

1. 新项目端口与原项目不冲突
2. 新项目配置与原项目不共享运行状态
3. 新项目日志、缓存、数据库、构建产物独立

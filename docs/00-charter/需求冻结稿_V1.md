# 本地智能体 - 需求冻结稿 V1（正式版）

更新时间：2026-04-07  
状态：正式版（已按愿景优先规则完成冻结）

## 0. 整合原则（冲突判定）

本稿基于以下历史文档整合形成：

1. 《个人愿景与系统定位草案_V1》（历史草案）
2. 《需求冻结稿_V1》（历史版本）

冲突处理规则：

1. 若出现目标、范围、能力优先级冲突，以《个人愿景与系统定位草案_V1》为准。
2. 《需求冻结稿_V1》中无冲突的工程约束、阶段边界、验收口径继续保留。
3. 本稿生效后，可作为后续架构设计、开发计划、里程碑拆解的统一需求输入。

---

## 1. 一句话定位

做一个**本地电脑可执行型智能体**：不止会对话，更能在本机完成真实任务，具备命令执行、文件操作、本地程序调用与可控安装能力，并保证可控、可审计、可回溯、可扩展。

---

## 2. 目标用户与核心价值

### 2.1 目标用户

1. 第一优先为个人单机长期使用（你自己）。
2. 设计标准按“他人可用”要求推进，不做一次性脚本工程。

### 2.2 核心价值

1. 缩短“意图 -> 执行结果”路径。
2. 降低重复环境搭建与重复操作成本。
3. 将高频流程沉淀为可复用技能，形成长期复利。

---

## 3. 对标对象与学习立场

1. 对标 Codex / OpenClaw 等电脑操作型智能体的“可执行能力”。
2. 重点学习 Claude Code 的运行时主干能力（上下文、工具协议、权限、记忆、验证），不追求 UI 复刻。
3. 允许复用开源实现，但坚持“主干优先、可解释、可验证”。

---

## 4. V1 产品边界（整合后冻结）

### 4.1 正式入口

V1 正式主入口：

1. 桌面启动器
2. 本地 Web 控制台

### 4.2 V1 核心闭环能力（以愿景稿优先）

V1 必须具备：

1. 聊天与任务理解
2. 工具调用与结果返回
3. 命令执行（PowerShell/CLI）
4. 文件读写与工作区操作
5. 启动本地程序
6. 知识检索与记忆沉淀
7. **可控安装与验证能力**（可通过命令行安装时，允许执行并验证版本/路径）

说明：第 7 项由愿景稿明确提出，优先级高于冻结稿中的保守表述，纳入 V1 主闭环，但必须受风险控制机制约束。

### 4.3 V1 前端范围

V1 前端固定三页：

1. 首页工作台
2. 日志页
3. 设置页

前端正式视觉与界面口径以后续 V2 文档为准。

### 4.4 V1 明确后置

以下能力后置，不纳入 V1 验收：

1. 浏览器扩展作为正式主入口
2. 语音输入输出
3. 多智能体协作界面
4. 复杂运维多面板
5. 自动 fallback / 自动测速 / 模型自动排序
6. 全系统所有应用统一控制验证

说明：愿景稿中的浏览器插件“老师模式”归为可选增强，作为后续阶段能力。

---

## 5. 两条主线场景（产品演进主轴）

### 5.1 主线 A：CTF / 渗透 / AWDP 技能化

目标：把高频流程技能化，减少重复操作，提高解题效率。

能力方向：

1. 环境准备（依赖、服务、端口检查、必要配置）
2. 资产与信息收集（结构化结果输出）
3. 常见漏洞复现流程模板化
4. 赛后知识沉淀与 SOP 复用

### 5.2 主线 B：学习教练闭环（以英语四级为例）

目标：从一次性规划升级为“规划-执行-反馈-调整”的持续教练循环。

能力方向：

1. 初步学习计划生成
2. 学习过程记录与错题沉淀
3. 基于完成率与测验结果动态调参
4. 对话式讲解与掌握度验证

---

## 6. 运行模式与权限模型

V1 保留三种模式：

1. 观察模式（以分析建议为主，不主动执行修改）
2. 标准模式（默认，常见读写允许执行）
3. 全权限模式（高频场景下减少阻断，但高危动作仍需确认）

不保留“严格模式”。

---

## 7. 工作区与访问边界

1. 支持多工作区与工作区切换。
2. 默认以当前工作区为操作中心。
3. 首次接触新目录应显式提示，并允许记住授权策略。
4. V1 不以“全盘控制能力”作为验收目标，但架构上保留扩展可能。

---

## 8. 风险控制与确认机制（强制项）

### 8.1 风险分级

对外分级保留：

1. 低风险
2. 中风险
3. 高风险
4. 不可逆风险

### 8.2 默认执行原则

1. 常见低风险操作可直接执行。
2. 常规文件写入默认可执行，但必须记录日志。
3. 高危或不可逆动作必须人工确认。

### 8.3 高危确认最小展示信息

1. 正在执行什么
2. 影响范围
3. 目标路径/目标对象
4. 执行动机
5. 是否可回退
6. 潜在危害
7. 当前风险等级

### 8.4 安全替代优先

若存在更安全替代路径，优先替代方案（如回收站替代永久删除、只读检查替代直接覆盖）。

---

## 9. 记忆系统与知识库

### 9.1 V1 记忆目标

1. 会话连续
2. 长期记忆沉淀
3. 上下文压缩
4. 失败教训沉淀
5. 用户偏好沉淀

### 9.2 记忆调用原则

1. 按需加载，避免全量加载。
2. 必载层只保留最小必要集合。
3. 长期记忆按块组织，不做单体巨型记忆池。

### 9.3 知识库定位

1. 沉淀学习内容、解题过程、踩坑和 SOP。
2. 作为后续检索、规划、复用、自动化执行依据。

---

## 10. Skills / MCP / 能力可扩展性

1. 允许将本地 CLI、脚本、二进制封装为 Skills 或 MCP 工具。
2. Skills 按模块分层组织，避免全量平铺读取。
3. 能力来源支持 Rust 原生能力、Skill 包、MCP 适配共存。
4. 调度目标：提升任务完成效率并降低 token 成本。

---

## 11. 模型与 Provider

V1 冻结要求：

1. 单智能体
2. 支持手动模型切换
3. 最好支持多厂商模型切换
4. 模型不可用时必须明确报错（超时/鉴权失败/配额不足/余额不足/接口异常）

---

## 12. 日志与可观测性

日志面板用于复盘、测试与排障，至少可见：

1. 当前执行阶段
2. 工具调用链
3. 结果摘要
4. 是否触发确认
5. 错误信息
6. 当前模型
7. 任务结束状态

前端必须展示过程反馈，不能只展示最终答案。

---

## 13. 工程与技术约束

1. 新项目不与旧项目逐文件等价迁移，只吸收主链路价值。
2. 架构与实现坚持模块化，避免重新演化为单体大循环脚本。
3. Go 保留控制面角色，Rust 负责替代 Python 运行时与执行层。
4. Rust 化目标是重建清晰执行层，而非机械翻译旧 runtime。

---

## 14. 环境隔离要求

新项目必须与旧项目隔离运行：

1. 端口不冲突
2. 配置与运行状态不共享
3. 日志、缓存、数据库、构建产物独立

---

## 15. 当前不做（V1 之外）

V1 不做：

1. 浏览器扩展作为正式入口
2. 语音输入输出
3. 多智能体
4. 浏览器自动化大范围能力
5. 桌面自动化大范围能力
6. 任务调度
7. 复杂运维面板
8. 自动 fallback / 自动测速

## 1. 文档目的

本文件用于回答以下问题：

1. 第一版到底必须做什么
2. 第一版哪些能力属于“最好做”
3. 第一版哪些内容明确后置
4. 每个功能做到什么程度才算“完成”
5. 后续开发顺序该如何围绕 MVP 组织

本文件是第一版功能范围的直接清单，不讨论具体实现技术细节。

---

## 2. MVP 总目标

第一版 MVP 的目标是：

> 做出一个可真实使用的、本地优先的、桌面启动的单智能体控制台，使用户能够通过聊天方式发起任务，并让系统完成文件、命令、工作区、知识、记忆、本地程序调用和可控安装验证相关的核心操作，同时在前端看到执行进度、结果和必要确认。

第一版重点不是做大而全，而是确保以下主链路成立：

1. 启动系统
2. 进入聊天界面
3. 发起任务
4. 智能体理解任务并执行
5. 前端持续看到进度
6. 最终返回结果
7. 高风险动作会被拦住并要求确认

---

## 3. MVP 成功标准

如果第一版要被判定为“达到了 MVP”，至少需要满足：

1. 用户可以从桌面入口进入系统
2. 用户可以在聊天页发起任务
3. 系统可以完成基本命令、文件、工作区、知识、本地程序调用与可控安装验证任务
4. 用户能看到执行过程，而不是只有最终答复
5. 系统具备基本记忆能力和上下文连续能力
6. 系统具备基本风险控制能力
7. 用户可以在界面中切换模型
8. 整体使用体验清楚、稳定、可重复

---

## 4. MVP 必做功能

以下功能为第一版必须交付项。

### 4.1 桌面启动与进入主界面

#### 功能描述

用户通过桌面启动器进入本地 Web 控制台。

#### 最小完成标准

1. 存在明确的桌面启动入口
2. 启动后可进入新项目的本地 Web 控制台
3. 新项目端口与旧项目不冲突
4. 启动路径稳定，用户无需手动拼命令进入主流程

---

### 4.2 聊天页

#### 功能描述

提供一个聊天主界面，用户通过自然语言发起任务。

#### 最小完成标准

1. 用户可以输入任务或问题
2. 智能体可以返回文本结果
3. 智能体可以基于当前会话连续推进任务
4. 聊天页是第一版主工作界面

---

### 4.3 单智能体任务执行

#### 功能描述

系统以单智能体方式理解任务、推进任务并返回结果。

#### 最小完成标准

1. 第一版仅支持单智能体
2. 智能体可以执行多步任务
3. 智能体可以在任务过程中调用工具
4. 智能体可以根据执行结果继续下一步

---

### 4.4 命令执行与本地程序调用（含可控安装验证）

#### 功能描述

系统可以执行受控命令、启动本地程序，并在可控边界内完成命令行安装与安装结果验证。

#### 最小完成标准

1. 支持在受控环境中执行命令
2. 命令输出可以回流到运行时
3. 命令结果可被摘要展示给前端
4. 高风险命令具备风险控制
5. 支持通过系统指令启动本地程序
6. 在命令行安装可行时，支持执行安装并输出执行结果
7. 安装后支持校验版本号或可执行路径

---

### 4.5 文件读写

#### 功能描述

系统可以读取文件、修改文件、创建文件，并将结果纳入任务推进。

#### 最小完成标准

1. 支持读文件
2. 支持改文件
3. 支持新建文件
4. 常见写入不必每次人工确认
5. 写入操作必须记录日志

---

### 4.6 工作区操作

#### 功能描述

系统支持多个工作区，并围绕当前工作区执行主要任务。

#### 最小完成标准

1. 支持多个工作区
2. 支持切换工作区
3. 允许跨工作区访问
4. 仍以当前工作区为默认操作中心
5. 首次接触新目录时可提示用户

---

### 4.7 知识检索

#### 功能描述

系统可以从本地知识来源中检索与当前任务相关的信息。

#### 最小完成标准

1. 能够存取基础知识数据
2. 能根据任务进行相关检索
3. 检索结果能够服务于任务推进
4. 检索结果应能压缩为适合上下文使用的摘要

---

### 4.8 记忆系统

#### 功能描述

系统具备会话连续、长期沉淀和按需调用记忆的能力。

#### 最小完成标准

1. 支持会话内上下文连续
2. 支持长期记忆基础分块
3. 不是每次进入工作区都全量加载所有记忆
4. 支持按需检索相关记忆
5. 支持写入重要事项
6. 支持记录用户偏好
7. 支持记录基础经验或教训

---

### 4.9 上下文压缩

#### 功能描述

系统能够在上下文较长时保留关键内容并压缩低价值细节。

#### 最小完成标准

1. 能保留项目整体理解
2. 能保留当前目标和关键约束
3. 能压缩中间流水和低价值冗余输出
4. 不因上下文过长而持续堆积无用信息

---

### 4.10 日志页

#### 功能描述

系统提供一个用于复盘、测试和排障的日志页。

#### 最小完成标准

1. 用户能看到当前任务在做什么
2. 用户能看到调用了什么工具
3. 用户能看到工具返回结果摘要
4. 用户能看到错误和确认事件
5. 用户能大致看懂任务是如何结束的

---

### 4.11 任务进度可视化

#### 功能描述

前端应持续显示任务推进过程，而不是黑盒执行。

#### 最小完成标准

1. 能看到开始执行
2. 能看到阶段推进
3. 能看到工具调用
4. 能看到结果摘要回流
5. 能看到任务完成或失败

---

### 4.12 模型切换

#### 功能描述

用户可以在 UI 中手动切换模型。

#### 最小完成标准

1. 支持在界面中手动切换模型
2. 支持基础 provider / model 配置
3. 模型不可用时有明确提示
4. 不需要通过改代码完成模型切换

---

### 4.13 运行模式切换

#### 功能描述

用户可以在界面中切换三种模式。

#### 最小完成标准

1. 支持观察模式
2. 支持标准模式
3. 支持全权限模式
4. 默认模式为标准模式
5. 模式切换结果对行为产生明确影响

---

### 4.14 风险控制与人工确认

#### 功能描述

高风险和不可逆动作必须经过用户确认。

#### 最小完成标准

1. 常见低风险动作可直接执行
2. 写文件默认允许但记录日志
3. 删除类动作进入更高风险判断
4. 高危动作必须人工确认
5. 确认信息至少展示：
   - 在做什么
   - 影响范围
   - 目标路径
   - 为什么要做
   - 是否可回退
   - 潜在危害
   - 风险等级

---

### 4.15 新目录首次接触提示

#### 功能描述

智能体第一次接触新目录时，系统向用户提示。

#### 最小完成标准

1. 首次进入新目录时可提醒
2. 用户可选择继续或取消
3. 提供“默认不提醒 / 记住选择”之类的选项
4. 后续可被设置统一管理

---

### 4.16 设置页

#### 功能描述

第一版提供最基础但可用的设置页。

#### 最小完成标准

1. 可管理模型相关基础设置
2. 可查看或调整模式相关基础设置
3. 可容纳后续目录授权和风险设置的入口
4. 不要求做成复杂运维台

---

## 5. MVP 应做功能

以下功能建议在第一版尽量完成，但如果实现复杂度过高，可在不破坏主链路的前提下适度降级。

### 5.1 长期记忆基础分类

建议至少具备基础分类能力，例如：

1. 项目知识
2. 用户偏好
3. 每日记录
4. 经验教训

### 5.2 每日总结入口

建议为“日报 / 每日总结”保留基础入口或基础数据归集能力。

第一版不要求复杂格式，但建议预留：

1. 今日做了什么
2. 完成了什么
3. 关键学习内容
4. 基础积极反馈

### 5.3 替代方案优先

当存在更安全方案时，系统应尽量优先建议替代方案，而不是直接走高风险动作。

### 5.4 多厂商模型基础支持

若实现成本可控，建议第一版支持多个厂商模型的基础配置和手动切换。

以下功能不要求在第一版完成，但建议在架构上预留接口。

### 6.2 浏览器与网页理解

1. 浏览器自动化
2. 浏览器上下文理解
3. 网页学习内容辅助

### 6.3 桌面自动化

1. 本地 GUI 应用操作
2. 鼠标键盘模拟
3. 窗口识别与控制

### 6.4 更复杂的能力成长机制

1. 更复杂的 Skill 包系统
2. 更复杂的能力块自定义增长
3. 更复杂的经验沉淀和反思系统

### 6.5 更复杂的模型编排

1. 自动 fallback

3. 模型自动排序
4. 更高级路由

## 7. MVP 明确不做的内容

以下内容第一版明确不做，不应进入首版验收。

1. 多智能体协作
2. 浏览器扩展
3. 浏览器自动化
4. 桌面自动化
5. 任务调度
6. 语音输入输出
7. 复杂多面板运维控制台
8. 复杂 GUI 结果展示
9. 自动 fallback
10. 自动测速
11. 全系统所有应用统一接入

---

## 8. MVP 交互范围定义

第一版交互范围明确限制为：

1. 自然语言输入
2. 文本结果输出
3. 文本型日志展示
4. 风险确认弹窗或显式确认交互
5. 模式切换
6. 模型切换

不要求第一版支持：

1. 复杂图形化任务流编辑
2. 复杂拖拽式工作台
3. 复杂富媒体结果展示

---

## 9. MVP 数据与状态范围

第一版至少应管理以下状态：

1. 当前会话状态
2. 当前模型状态
3. 当前模式状态
4. 当前工作区状态
5. 记忆读写状态
6. 风险确认状态
7. 日志与事件状态

第一版不要求：

1. 复杂调度状态
2. 多 agent 状态
3. 浏览器和桌面自动化状态

---

## 10. MVP 性能与体验目标

第一版虽然不是性能极限版本，但至少应遵守以下方向：

1. 尽量轻量
2. 尽量快速反馈
3. 不制造大量无意义 token 消耗
4. 不让用户频繁手动补救系统混乱行为
5. 重要反馈要直观
6. 风险提示要清晰

---

## 11. MVP 开发优先级建议

建议按以下顺序实施：

### P0 - 必须先做

1. 桌面启动入口
2. 聊天页
3. Go 控制面最小闭环
4. Rust 运行时最小闭环
5. 命令执行
6. 文件读写
7. 工作区切换与目录提示
8. 基础日志页
9. 任务进度回流
10. 模式切换
11. 风险确认

### P1 - 第一版建议完成

1. 知识检索
2. 长期记忆基础分块
3. 上下文压缩
4. 多模型基础切换
5. 每日总结入口

### P2 - 明确后续版本

1. 浏览器能力
2. 桌面能力
3. 语音能力
4. 调度能力
5. 多智能体

## 12. MVP 通过验收时的最小使用场景

第一版至少应能稳定支持以下典型场景：

### 场景 1：项目文件修改

1. 用户说明要修改某个项目文件
2. 智能体读取相关文件
3. 智能体修改文件
4. 用户在前端看到过程和结果
5. 日志页能复盘这次操作

### 场景 2：命令执行与排错

1. 用户要求执行命令
2. 智能体执行命令
3. 输出摘要返回
4. 如有错误，前端可见
5. 用户能继续追问和修正

### 场景 3：知识检索辅助

1. 用户询问项目或知识相关问题
2. 智能体检索相关内容
3. 智能体返回总结和结论
4. 重要信息可写入记忆

### 场景 4：高风险动作确认

1. 用户要求执行高危或删除类动作
2. 系统识别风险
3. 明确展示确认信息
4. 用户确认后再执行或取消

### 场景 5：切换模型继续工作

1. 用户在界面切换模型
2. 当前会话继续使用新模型
3. 如果模型不可用，有明确提示

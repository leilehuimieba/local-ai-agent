## 2. 产品边界总原则

### 2.1 第一版产品边界定义

第一版产品边界只包含那些直接服务于以下核心闭环的能力：

1. 用户通过桌面启动器进入系统
2. 用户在本地 Web 控制台与单智能体交互
3. 智能体理解任务并进行多步推进
4. 智能体调用文件、命令、工作区、知识与记忆相关能力
5. 系统将执行进度、结果、日志和必要确认回流给用户

凡是不直接服务于这条闭环的内容，默认不属于第一版正式产品内核。

### 2.2 边界控制原则

1. 第一版不追求“大而全”。
2. 第一版优先确保主链路可用、稳定、清晰。
3. 所有后续扩展能力必须先判断是否会干扰主链路。
4. 架构要为后续扩展留接口，但第一版不提前承诺未来全部能力。

## 3. 正式产品内核

第一版正式产品内核包含以下部分。

### 3.1 入口层

1. 桌面启动器
2. 本地 Web 控制台

### 3.2 交互层

1. 首页工作台
2. 日志页
3. 设置页
4. 模式切换
5. 模型切换

### 3.3 控制层

1. Go 控制面
2. 会话与事件流
3. 配置与模型信息管理
4. 日志与审计回流
5. 风险确认编排

### 3.4 执行层

1. Rust 运行时
2. 单智能体任务推进
3. 工具调用编排
4. 结果观察与阶段反馈
5. 记忆与知识调用

### 3.5 核心能力层

第一版正式能力限定为：

1. 命令执行
2. 文件读写
3. 工作区操作
4. 知识检索
5. 记忆读写与压缩

### 3.6 安全与治理层

1. 三种模式
   - 观察模式
   - 标准模式
   - 全权限模式
2. 新目录首次接触提示
3. 写入日志
4. 删除/高危动作确认
5. 风险判断与替代方案优先

---

## 4. 正式人类入口

### 4.1 第一版唯一正式人类入口

第一版唯一正式人类入口定义为：

1. 桌面启动器
2. 由桌面启动器打开的本地 Web 控制台

这意味着：

1. 用户不需要先理解内部服务结构
2. 用户不应依赖内部 CLI 作为主产品入口
3. 人类面对的正式界面是桌面启动后的 Web 控制台

### 4.2 正式人类入口的要求

正式人类入口应满足：

1. 明确可见
2. 启动路径稳定
3. 与旧项目不冲突
4. 能引导用户进入标准模式下的主界面
5. 能为后续模式切换、模型切换、日志查看提供统一载体

---

## 5. 正式系统入口

### 5.1 第一版唯一正式系统入口

第一版唯一正式系统入口定义为：

1. Go 控制面服务入口

这意味着：

1. 前端不直接绑定 Rust 内部结构
2. Rust 执行层不直接暴露为用户主入口
3. 所有前端交互、事件流、配置、日志、确认流应围绕 Go 控制面组织

### 5.2 系统入口的职责

正式系统入口应负责：

1. 承接前端请求
2. 向运行时发起任务执行请求
3. 回流事件与日志
4. 管理模式、模型、设置和会话
5. 组织风险确认流程

---

## 6. 第一版明确纳入范围的功能

第一版明确纳入正式产品范围的功能如下。

### 6.1 智能体交互

1. 单智能体对话
2. 任务推进
3. 结果返回

### 6.2 工具与执行

1. 命令执行
2. 文件读写
3. 工作区相关操作
4. 知识检索
5. 记忆调用

### 6.3 过程可见性

1. 前端可看到任务执行进度
2. 至少能看到调用了什么工具
3. 至少能看到结果摘要
4. 能看到错误、确认和任务结束状态

### 6.4 模型能力

1. 手动切换模型
2. 支持多厂商模型配置的基本能力
3. 模型不可用时有明确提示

### 6.5 记忆与日报方向

第一版正式纳入：

1. 会话内上下文连续
2. 长期记忆基础分块
3. 重要信息写入
4. 基础上下文压缩
5. 为每日日报/每日总结预留路径

说明：

1. “每日日报”方向已纳入正式产品方向。
2. 但第一版不要求复杂的日报模板系统，只要求预留和基础能力。

## 8. 不属于第一版正式产品边界的内容

### 8.1 旧项目中默认不纳入新产品边界的内容

以下内容即使存在于原项目，也默认不属于新项目第一版正式边界：

1. 比赛相关流程
2. 临时脚本型工作流
3. 实验性能力区
4. 导入保留区内容
5. 隔离区内容
6. 历史归档内容
7. 与第一版主闭环无直接关系的附属面板和工具

### 8.2 不能被误描述为第一版能力的内容

以下内容即使未来会做，也不能在第一版中被表述为“已支持”：

1. 浏览器深度自动化
2. GUI 桌面控制
3. 全盘统一管理
4. 多智能体合作系统
5. 语音 Jarvis 式交互
6. 全系统应用接入

可以表述为：

1. 已预留扩展方向
2. 已为后续能力增长保留结构

## 9. 扩展边界原则

### 9.1 第一版允许的扩展方式

第一版允许在不破坏主链路的前提下，为未来扩展保留以下能力接口：

1. 能力模块注册机制
2. Skill 包机制
3. MCP 适配层
4. 记忆分层扩展点
5. 领域模块路由器
6. 后续语音接口保留点

### 9.2 第一版不允许的扩展方式

第一版不允许为了“预留未来”而做以下事情：

1. 先引入大量未使用模块
2. 先做复杂但当前不用的 UI
3. 先接入高复杂度自动化能力
4. 先把能力暴露得过细，导致 token 浪费
5. 为未来假设写出大而空的基础设施

### 9.3 扩展判断原则

后续每增加一个能力，都应判断：

1. 是否影响主链路清晰度
2. 是否显著增加 token 消耗
3. 是否能带来真实收益
4. 是否值得第一时间建设
5. 是否适合下沉为 Skill 包、本地能力或 MCP 适配层

## 10. 技术边界原则

### 10.1 Go 的边界

Go 继续承担控制面角色。

原则：

1. 可以复用成熟代码
2. 不为了保留而保留
3. 允许按需要重构
4. 不重复造轮子

### 10.2 Rust 的边界

Rust 负责替代原 Python 运行时与执行层。

原则：

1. 不做 Python 逐文件机械翻译
2. 以更清晰的运行时结构重建
3. 强调性能、低占用、模块化、长期维护性

### 10.3 前端的边界

前端第一版采用：

1. TypeScript
2. React
3. Vite

原则：

1. 允许重写
2. 不要求复刻旧前端
3. 只围绕第一版主界面服务
4. 前端正式界面口径以后续 V2 文档为准

## 11. 新旧项目边界

### 11.1 原项目角色

原项目 `D:\newwork\local-agent-control-hub` 的角色是：

1. 参考源仓库
2. 需求来源
3. 架构经验来源
4. 可复用代码来源

不是：

1. 新项目直接继续开发的主仓库
2. 新项目必须保持兼容的实现对象

### 11.2 新项目角色

新项目 `D:\newwork\本地智能体` 的角色是：

1. 正式重构仓库
2. 未来文档、设计、实现、验收的主工作区
3. 新架构和新主链路的唯一正式载体

### 11.3 环境隔离

新旧项目必须隔离：

1. 端口隔离
2. 配置隔离
3. 日志隔离
4. 状态隔离
5. 构建产物隔离

## 12. 第一版验收时必须符合的边界判断

如果某个能力或页面要被宣称为“第一版已完成”，它必须同时满足：

1. 属于本文件定义的正式产品边界
2. 服务于第一版核心闭环
3. 不依赖后置能力才能成立
4. 已纳入正式交互路径
5. 已具备基本可观测性和风险控制

如果不满足以上条件，则应归类为：

1. 预留能力
2. 后续方向
3. 实验能力
4. 非首版范围

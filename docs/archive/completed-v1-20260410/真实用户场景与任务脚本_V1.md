# 真实用户场景与任务脚本 V1

## 1. 文档目的与适用范围

本文用于将《知识沉淀型个人助手产品定义_V1》中已经确定的产品主线，进一步落到真实用户场景与任务脚本层，作为后续场景优先级、状态维护、知识沉淀、下一步建议、验收设计与开发拆解的上游输入。

本文不重复定义产品定位、核心价值或总体边界，而是重点回答以下问题：

1. 当前产品在真实使用中优先承接哪些高频场景
2. 用户在这些场景中的典型起点、输入过程与中途变化是什么
3. 助手在这些场景下应如何接住输入、维护状态、沉淀知识并给出建议
4. 一轮场景结束后，系统应保留哪些可回看的结果、状态与后续入口
5. 第一版在这些场景中不处理什么，避免主线发散

本文适用于以下类型的真实使用场景建模：

1. 工程开发场景
2. 通用学习场景
3. 浏览学习场景
4. CTF / 安全学习场景

本文不展开以下内容：

1. 具体工程实现方案
2. Skill、MCP 或浏览器接入的底层技术细节
3. 多智能体协作设计
4. 高权限自动化与复杂控制链路
5. 验收指标与测试细则的最终定义

本文的作用，是把产品定义进一步转化为可执行、可比较、可验证的真实任务脚本，为后续场景优先级判断、能力设计与验收文档编写提供统一基础。

## 2. 场景脚本统一模板

为保证不同场景可比较、可拆解、可验收，本文后续所有场景均按统一模板编写。

每个场景脚本至少应包含以下部分：

1. `场景名称`
   用一句话说明该场景的核心任务，例如“工程开发中的文档共创与方向收口”或“浏览网页时的学习理解与知识沉淀”。

2. `场景目标`
   说明用户在该场景中真正想完成什么，以及该场景对产品的价值是什么。

3. `用户起点`
   说明用户进入该场景时通常处于什么状态，包括已知信息、已有材料、当前困惑、当前目标或已有进度。

4. `典型输入`
   说明用户在该场景中会持续提供哪些输入，例如自然语言描述、网页内容、笔记、文件结果、阶段总结或人工补充信息。

5. `中途变化`
   说明用户在场景推进过程中可能出现的补充材料、目标调整、问题变化、阻塞出现或理解深化等情况。

6. `助手应做什么`
   说明助手在该场景下应如何接住输入、维护状态、沉淀知识、给出建议，并在必要时执行少量辅助动作。

7. `用户典型追问`
   说明用户在该场景中最可能提出的追问，例如“我现在做到哪了”“还差什么”“下一步该做什么”“前几天看过什么”。

8. `场景结束后应留下什么`
   说明一轮结束后系统应保留哪些知识、状态、阶段结论、方法经验、待办事项或后续入口。

9. `第一版不做什么`
   说明该场景在第一版中明确不展开的能力，避免将场景脚本写成无限扩张的需求池。

10. `对后续文档的影响`
   说明该场景会影响哪些上游设计，例如知识资产模型、状态维护模型、下一步建议标准、场景优先级或验收用例。

上述模板的重点不是描述理想流程，而是尽可能贴近真实使用过程，使后续产品设计、开发拆解与验收判断都能基于统一脚本展开。

## 3. 工程开发场景

### 3.1 场景名称

工程开发中的问题推进、知识沉淀与阶段收口

### 3.2 场景目标

用户在工程开发过程中，不只是希望助手回答单个问题或完成单次修改，而是希望助手能够持续接住开发过程中的背景、材料、判断、结果与阻塞，帮助自己推进任务、沉淀经验，并在阶段变化时给出更贴近实际情况的下一步建议。

该场景对产品的价值在于：

1. 它是当前最贴近真实高频使用的场景之一
2. 它能够直接验证知识沉淀、状态维护、下一步建议与有限执行这条主线是否成立
3. 它能够为后续其他场景提供较强的任务推进与阶段收口样本

### 3.3 用户起点

用户进入该场景时，通常已经处于某个开发任务中，例如：

1. 正在定义或调整产品方向
2. 正在修改文档、代码或配置
3. 正在排查问题、理解现有实现或收口某项能力
4. 对下一步该做什么已有模糊判断，但尚未完全理清

此时用户往往不会一次给出完整上下文，而是先提供当前目标、局部背景、已有材料或当前困惑，再在后续过程中逐步补充。

### 3.4 典型输入

该场景下的典型输入包括：

1. 当前任务描述
2. 项目文档与历史记录
3. 代码片段、配置内容或命令结果
4. 阶段判断、个人想法与临时结论
5. 新出现的问题、阻塞与补充材料
6. 一轮修改后的结果与复盘信息

### 3.5 中途变化

在工程开发过程中，用户通常会出现以下变化：

1. 对问题的理解逐步收紧，而不是一开始就清楚
2. 中途补充新的文档、代码、日志、需求或限制条件
3. 原本以为要做代码改动，后来发现应先改产品口径或任务范围
4. 原本聚焦某个局部问题，后来发现需要先解决更上游的定义问题
5. 某一轮工作结束后，需要重新判断“现在做到哪了、还差什么、下一步做什么”

### 3.6 助手应做什么

在该场景下，助手应能够：

1. 接住用户当前任务、已有上下文与阶段目标
2. 持续记录开发过程中的关键判断、变更原因、阻塞与阶段结论
3. 帮助用户从零散问题中提炼出当前真正要解决的核心问题
4. 在中途上下文变化时更新对当前任务状态的理解
5. 基于已有沉淀与当前进展，回答“做到哪了、还差什么、下一步做什么”
6. 在明确授权下，完成少量与当前任务直接相关的辅助动作
7. 在一轮结束时保留可回看的交接信息、阶段结论与后续入口

### 3.7 用户典型追问

该场景中，用户典型会追问：

1. 我们现在真正要解决的问题是什么
2. 这一轮已经收口到什么程度了
3. 还缺哪些信息或步骤
4. 下一步最应该先做什么
5. 之前为什么做这个判断
6. 上一次改动后留下了哪些未完成项

### 3.8 场景结束后应留下什么

一轮工程开发场景结束后，系统应保留：

1. 当前任务的阶段结论
2. 本轮关键判断与判断依据
3. 已完成事项与未完成事项
4. 当前阻塞与待确认问题
5. 可继续接上的后续入口
6. 对未来类似问题可复用的经验与知识沉淀

### 3.9 第一版不做什么

该场景在第一版中明确不做以下内容：

1. 全自动接管整个项目开发流程
2. 高权限、无边界的自动化改动与批量执行
3. 为了工程场景而把产品重新定义为纯研发副驾
4. 复杂多智能体协作开发链路
5. 与知识沉淀主线无关的工具广度扩张

### 3.10 对后续文档的影响

该场景将直接影响以下后续文档：

1. 场景优先级与范围定义
2. 知识资产模型
3. 状态维护模型
4. 下一步建议能力定义
5. 有限执行与权限边界
6. 验收标准与评估用例

## 4. 通用学习场景

### 4.1 场景名称

通用学习过程中的理解辅助、知识沉淀与阶段推进

### 4.2 场景目标

用户在学习一个新主题时，希望助手不仅帮助解释当前内容，还能够持续接住学习过程中的材料、理解结果、阶段困惑与阶段总结，将其沉淀为长期可复用的知识与学习状态，并据此给出下一步学习建议。

该场景对产品的价值在于：

1. 它直接体现“知识沉淀型个人助手”的核心定位
2. 它能够验证产品是否真正具备持续维护学习状态的能力
3. 它能够为浏览学习场景和其他具体学习场景提供上游通用模板

### 4.3 用户起点

用户进入该场景时，通常处于某个学习主题的过程中，例如：

1. 正在学习一个新概念、新领域或新工具
2. 已经接触过一些资料，但理解还不稳定
3. 知道自己有困惑，但不清楚问题卡在哪一层
4. 想把一段学习过程整理清楚，并明确下一步该继续学什么

用户在这个阶段通常提供的信息并不完整，往往是边学边补充、边理解边修正。

### 4.4 典型输入

该场景下的典型输入包括：

1. 学习目标与当前主题
2. 用户当前正在阅读、观看或整理的内容
3. 用户自己的理解、疑问、摘录与总结
4. 线下学习后的人工补充输入，例如纸质阅读总结
5. 阶段性结论、复盘内容与自我判断
6. 对某些知识点“已理解 / 未理解 / 半理解”的主观反馈

### 4.5 中途变化

在学习过程中，用户通常会出现以下变化：

1. 从完全陌生逐步进入初步理解
2. 原本以为不懂的是某个知识点，后来发现卡在更基础的前置概念
3. 中途补充新资料、笔记、网页内容或自己的新理解
4. 某些内容短时看懂，但过一段时间又记不清
5. 一轮学习结束后，希望回看“我到底学了什么、哪里还不稳、接下来该学什么”

### 4.6 助手应做什么

在该场景下，助手应能够：

1. 接住用户当前学习主题、已有背景与当前困惑
2. 理解并整理用户提供的学习材料、摘录、总结与问题
3. 将学习过程中的知识点、阶段结论、疑问与理解变化沉淀下来
4. 维护用户当前学习状态，包括已掌握内容、薄弱点、阶段位置与待补内容
5. 回答“我已经学到哪里了、哪些还不稳、下一步该学什么”
6. 在合适的时候提供总结、解释、回顾建议或适度的巩固支持
7. 在一轮学习结束后，保留可继续接上的学习入口与可回看的学习记录

### 4.7 用户典型追问

该场景中，用户典型会追问：

1. 我现在到底理解到哪一步了
2. 哪些内容我其实还没真正掌握
3. 我今天学过什么
4. 我前几天看过的内容现在还记得多少
5. 接下来最应该补哪一块
6. 能不能帮我把这一段内容重新讲清楚
7. 能不能帮我做一个简短回顾

### 4.8 场景结束后应留下什么

一轮通用学习场景结束后，系统应保留：

1. 当前学习主题与阶段位置
2. 本轮学习涉及的关键知识点
3. 已理解内容、待巩固内容与未理解内容
4. 用户的阶段性总结与疑问
5. 下一步学习建议与后续入口
6. 可回看的学习轨迹与知识沉淀记录

### 4.9 第一版不做什么

该场景在第一版中明确不做以下内容：

1. 将产品扩展成完整教育平台或课程系统
2. 复杂、重型的学习评分与精细化能力画像体系
3. 高强度自动教学编排或多角色教学协作
4. 与主线无关的大量学习功能堆叠
5. 为了学习场景而引入高权限、重自动化的控制能力

### 4.10 对后续文档的影响

该场景将直接影响以下后续文档：

1. 场景优先级与范围定义
2. 知识资产模型
3. 状态维护模型
4. 下一步建议能力定义
5. 浏览学习场景定义
6. 验收标准与评估用例

## 5. 浏览学习场景

### 5.1 场景名称

浏览网页过程中的内容理解、学习辅助与轨迹沉淀

### 5.2 场景目标

用户在浏览网页、在线文档、知识社区或学习材料时，希望助手能够理解当前页面内容，帮助解释关键信息、总结当前要点、记录学习过程中的理解与问题，并将这一过程沉淀为后续可回看的学习轨迹与知识记录。

该场景对产品的价值在于：

1. 它是学习场景中最具连续输入特征的典型场景之一
2. 它能够验证产品是否可以在动态内容环境中持续接住用户输入
3. 它能够为后续网页内容接入、学习巩固与回看能力提供直接样本

### 5.3 用户起点

用户进入该场景时，通常处于在线学习或资料查阅过程中，例如：

1. 正在阅读技术文章、教程、文档或论坛讨论
2. 正在查找某个问题的背景资料或解决思路
3. 已经浏览了一部分内容，但不确定哪些信息最重要
4. 对当前页面有局部理解困难，希望即时得到解释或总结

用户此时往往并不会完整描述背景，而是基于当前页面和即时疑问发起交互。

### 5.4 典型输入

该场景下的典型输入包括：

1. 当前网页或在线文档中的文本内容
2. 用户对当前内容的即时提问
3. 用户标记的重点、摘录或个人理解
4. 页面切换、主题变化或新的补充资料
5. 用户对“已理解 / 未理解 / 需要回顾”内容的反馈
6. 某次浏览结束后的阶段性总结

### 5.5 中途变化

在浏览学习过程中，用户通常会出现以下变化：

1. 从被动阅读转为主动提问和理解
2. 因当前页面引出新的前置概念或相关主题
3. 在多个页面之间切换，逐步形成某个主题的整体理解
4. 对同一类内容先理解一部分，后续又希望回看或巩固
5. 浏览结束后，想知道“今天我到底看了什么、哪些值得沉淀、下一步该继续看什么”

### 5.6 助手应做什么

在该场景下，助手应能够：

1. 理解当前页面或当前浏览内容与用户学习目标之间的关系
2. 帮助总结当前内容的重点、结构与关键概念
3. 对用户当前不理解的部分给出贴近上下文的解释
4. 记录用户在浏览过程中的重点、疑问、理解结果与阶段变化
5. 将值得保留的内容沉淀为知识记录、学习轨迹或后续回顾材料
6. 在合适的时候给出回顾建议、巩固建议或下一步浏览建议
7. 在一轮浏览结束后，形成可回看的学习摘要与后续入口

### 5.7 用户典型追问

该场景中，用户典型会追问：

1. 这一页最重要的内容是什么
2. 这里讲的概念到底是什么意思
3. 这一段和我当前在学的主题有什么关系
4. 我今天已经看过哪些内容
5. 哪些内容我后面应该再回顾一次
6. 下一步应该继续看哪一块
7. 能不能基于这次浏览帮我做个简短巩固

### 5.8 场景结束后应留下什么

一轮浏览学习场景结束后，系统应保留：

1. 当前浏览主题与相关页面内容摘要
2. 本轮浏览中的重点概念、关键问题与阶段理解
3. 用户标记的重要内容与待回顾内容
4. 与长期学习主题相关的知识沉淀记录
5. 下一步推荐浏览方向或学习建议
6. 可回看的浏览学习轨迹

### 5.9 第一版不做什么

该场景在第一版中明确不做以下内容：

1. 高复杂度浏览器自动化控制
2. 模拟用户完成复杂网页操作链路
3. 无边界的全局网页监控与高风险行为采集
4. 为规避网站策略而进行复杂自动化伪装机制设计
5. 将产品重新定义为浏览器自动化工具

### 5.10 对后续文档的影响

该场景将直接影响以下后续文档：

1. 场景优先级与范围定义
2. 输入渠道定义
3. 知识资产模型
4. 状态维护模型
5. 下一步建议能力定义
6. 有限执行与权限边界
7. 验收标准与评估用例

## 6. CTF / 安全学习场景

### 6.1 场景名称

CTF / 安全学习过程中的分析辅助、方法沉淀与下一步判断

### 6.2 场景目标

用户在进行 CTF 练习、安全学习或受控分析任务时，希望助手不仅帮助理解当前题目、目标或现象，还能够持续记录分析过程、沉淀方法与经验、识别当前卡点，并据此给出更贴近当前阶段的下一步建议。

该场景对产品的价值在于：

1. 它验证产品能否承接高专业度、强步骤性的学习与分析过程
2. 它能够体现 Skill 在领域场景中的增强价值，而不改变产品主定义
3. 它能够为后续方法沉淀、学习轨迹维护与场景化建议能力提供高价值样本

### 6.3 用户起点

用户进入该场景时，通常处于某个 CTF 题目、安全练习或受控分析任务中，例如：

1. 刚接触一个新题型或新目标，不知道从哪里开始
2. 已做过初步信息收集，但还没有形成清晰判断
3. 已识别出部分现象，但不确定漏洞类型、技术栈或下一步路径
4. 有若干工具和经验，但希望通过助手更快梳理问题与推进步骤
5. 希望在做题或分析的同时，把过程沉淀为以后可复用的方法经验

### 6.4 典型输入

该场景下的典型输入包括：

1. 题目描述、靶机说明或任务背景
2. 初步观察结果、页面信息、响应特征或环境线索
3. 用户执行后的阶段结果、命令结果、笔记与判断
4. 用户对漏洞类型、技术实现或分析路径的猜测
5. 中途新增的线索、异常现象或失败尝试
6. 一轮分析后的阶段总结与复盘信息

### 6.5 中途变化

在 CTF / 安全学习过程中，用户通常会出现以下变化：

1. 从不知道怎么开始，到逐步形成初步分析路径
2. 原本以为是某类问题，后来发现真正卡点在更基础的信息识别或前置知识
3. 中途加入新的线索、页面、响应、工具结果或环境信息
4. 某条分析路径被证明无效，需要回退并重新判断
5. 一轮任务结束后，希望回看“这题属于什么类型、我为什么卡住、以后类似题怎么更快入手”

### 6.6 助手应做什么

在该场景下，助手应能够：

1. 接住当前题目、目标或分析任务的背景与阶段状态
2. 帮助整理已有线索、观察结果、假设与阶段判断
3. 识别当前分析路径中的关键缺口、阻塞点与待确认问题
4. 结合场景 Skill 提供贴近上下文的分析建议与下一步方向判断
5. 记录分析过程中的方法、尝试、失败路径与有效结论
6. 将可复用的题型经验、方法套路与个人薄弱点沉淀下来
7. 在一轮结束后保留可回看的过程记录、阶段结论与后续学习入口

### 6.7 用户典型追问

该场景中，用户典型会追问：

1. 我现在最应该先判断什么
2. 当前更像哪一类题型或问题
3. 我已经拿到了哪些有效线索
4. 这一步为什么不通
5. 下一步最值得验证的方向是什么
6. 这次分析里我暴露出的薄弱点是什么
7. 类似题下次我应该先看什么

### 6.8 场景结束后应留下什么

一轮 CTF / 安全学习场景结束后，系统应保留：

1. 当前题目或任务的类型判断与阶段结论
2. 本轮分析中的关键线索、关键判断与判断依据
3. 已尝试路径、失败路径与有效路径
4. 与该场景相关的方法经验、题型知识与常见坑点
5. 用户在该类问题上的薄弱点与下一步学习建议
6. 可继续接上的后续分析入口与可回看的过程记录

### 6.9 第一版不做什么

该场景在第一版中明确不做以下内容：

1. 无边界的自动攻击、自动利用或高风险自动化链路
2. 脱离授权的目标操作与高权限控制行为
3. 将产品重新定义为渗透平台、漏洞利用工具或全自动安全执行系统
4. 复杂多智能体协作式安全分析流程
5. 为追求自动化能力而削弱知识沉淀、状态理解与学习支持主线

## 7. 场景间共同主线与差异总结

前述工程开发、通用学习、浏览学习、CTF / 安全学习四类场景，虽然具体内容不同，但都服务于同一条产品主线：

> 持续接住用户在不同场景中的输入，将其沉淀为长期可复用的知识与状态，并据此给出贴近个人真实情况的下一步建议，在必要时辅以少量受控执行。

### 7.1 四类场景的共同主线

这四类场景在结构上具有以下共同点：

1. `都有持续输入`
   用户不会一次给出完整信息，而是在过程中持续补充材料、问题、结果、总结与判断。

2. `都需要状态维护`
   系统都需要持续维护用户当前目标、阶段、进度、阻塞、近期结果与理解变化。

3. `都需要知识沉淀`
   场景中的输入、过程、结论、方法、错误与经验都应逐步沉淀为长期可复用的知识资产。

4. `都需要下一步建议`
   用户在每类场景中都会反复追问“现在做到哪了、还差什么、下一步做什么”。

5. `都需要阶段收口`
   一轮场景结束后，系统都应保留交接信息、阶段结论、待办事项与可继续接上的后续入口。

### 7.2 四类场景的主要差异

这四类场景的差异主要体现在以下方面：

1. `输入来源不同`
   工程开发场景以任务描述、文档、代码、结果为主；
   通用学习场景以学习材料、总结、理解反馈为主；
   浏览学习场景以网页内容、即时提问、浏览轨迹为主；
   CTF / 安全学习场景以题目、线索、分析结果、方法尝试为主。

2. `状态重点不同`
   工程开发更关注任务推进状态与阶段收口；
   通用学习更关注理解程度、薄弱点与学习阶段；
   浏览学习更关注当前页面上下文、即时理解与回顾需求；
   CTF / 安全学习更关注分析路径、关键判断、卡点与题型经验。

3. `知识沉淀重点不同`
   工程开发更偏决策依据、经验结论与任务上下文；
   通用学习更偏主题知识、阶段理解与学习轨迹；
   浏览学习更偏页面重点、问题记录与回顾材料；
   CTF / 安全学习更偏方法套路、题型经验、失败路径与薄弱点记录。

4. `建议形式不同`
   工程开发更偏任务推进建议；
   通用学习更偏学习顺序与理解巩固建议；
   浏览学习更偏当前内容解释、回顾与下一步浏览建议；
   CTF / 安全学习更偏分析方向、验证顺序与后续学习建议。

### 7.3 Skill 在不同场景中的位置

Skill、MCP 与其他场景增强能力，在四类场景中的作用应统一理解为：

1. 用于增强特定场景下的理解、分析、建议与辅助动作能力
2. 用于补充某一领域中的专业知识、步骤经验与常见模式
3. 不改变产品主定义，不构成多个并列产品
4. 不应替代知识沉淀、状态维护与下一步建议这条共性主线

## 8. 第一版场景优先级建议

结合前述四类场景的真实使用价值、主线验证强度与当前实现复杂度，第一版的场景优先级建议分为两层。

### 8.1 第一优先级场景

第一版建议优先深做以下两类场景：

1. `工程开发场景`
2. `通用学习场景`

优先原因如下：

1. 这两类场景最能共同验证“知识沉淀、状态理解、下一步建议、有限执行”这条主线是否真正成立
2. 工程开发场景是当前最接近真实高频使用的场景之一，能够较快形成稳定反馈闭环
3. 通用学习场景最能体现产品作为“知识沉淀型个人助手”的核心价值，避免产品重新滑回纯研发副驾
4. 这两类场景共同覆盖“任务推进”和“长期成长”两个最关键的产品方向

### 8.2 第二优先级场景

第一版可同步保留并进行验证，但不作为最先深做重点的场景包括：

1. `浏览学习场景`
2. `CTF / 安全学习场景`

排序原因如下：

1. 浏览学习场景具有很高价值，但本质上属于通用学习场景在网页内容环境下的强化版本，适合作为后续增强方向逐步补强
2. CTF / 安全学习场景具有较强差异化价值，但对领域 Skill、边界约束与建议质量的要求更高，适合作为重点验证场景而不是第一版唯一核心场景
3. 若在第一版同时深做上述两类场景，容易拉高整体复杂度并分散主线收口能力

### 8.3 当前优先级判断原则

当前场景优先级的判断依据包括：

1. 是否能够直接验证产品主线是否成立
2. 是否具有持续、高频、真实的使用需求
3. 是否能够为后续知识资产模型、状态维护模型与建议能力提供稳定样本
4. 是否会在当前阶段显著拉高系统复杂度或削弱主线收口

因此，第一版的核心目标不是同时把所有场景做深，而是先在最能代表产品主线的场景中稳定成立，再逐步扩展到更强差异化的场景。

## 9. 文档结论与后续衔接

通过前述四类真实场景脚本，可以确认当前产品虽然服务于不同场景，但其主线结构是一致的：都需要持续接住用户输入、维护当前状态、沉淀长期知识，并据此给出下一步建议，在必要时辅以少量受控执行。

基于这一判断，当前产品不应被理解为多个并列助手或多个并列产品，而应被理解为同一个“知识沉淀型个人助手”在不同场景中的应用与增强。工程开发、通用学习、浏览学习、CTF / 安全学习等场景的差异，主要体现在输入来源、状态重点、知识沉淀重点与建议形式上，而不改变产品主定义本身。

结合真实使用价值、主线验证强度与当前复杂度控制，第一版建议优先围绕以下两类场景做深：

1. `工程开发场景`
2. `通用学习场景`

浏览学习场景与 CTF / 安全学习场景可作为同步验证与后续增强方向保留，但当前不宜与第一优先级场景并行深做，以避免分散主线收口能力。

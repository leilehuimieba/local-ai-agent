## 1. 文档目的

本文的目标不是描述技术实现细节，而是明确第二阶段到底要把这个产品做成什么、为什么这样做、哪些必须先做、哪些先不要做。

本文重点解决以下问题：

1. 当前产品的真实定位是什么。
2. 相比同类产品，当前产品的优势应该如何表达。
3. 第二阶段短期内必须达成的用户价值是什么。
4. 长期方向有哪些，哪些现在先后置。
5. 开发时哪些边界不能被突破。

---

## 2. 第二阶段一句话定义

第二阶段的产品定义是：

> `一个面向长期学习、长期成长、长期积累、可持续专精的本地个人智能体平台。`

它的核心不是一次性聊天，而是：

1. 帮助用户学习。
2. 帮助用户规划。
3. 帮助用户沉淀知识和记忆。
4. 帮助用户接入专业能力并持续成长。
5. 在在线模型可用时提供强思考，在在线模型不可用时保持基本可执行能力。

---

## 3. 当前产品定位

### 3.1 产品本质

当前产品不应被定义成普通 AI 聊天工具，也不应被定义成一个只会执行命令的开发辅助壳。

更准确的定义是：

1. `本地长期个人成长型智能体`
2. `本地知识与长期记忆的统一入口`
3. `用户学习、实践、复盘、沉淀经验的长期平台`
4. `未来可向专业方向持续专精的本地 AI 底座`

### 3.2 用户想要的核心感受

用户希望这个产品不是“每次都重新认识我”的 AI。

用户真正想要的是：

1. 它能长期记住我。
2. 它能跟着我一起学习。
3. 它能根据我的目标和习惯做判断。
4. 它能逐步掌握我所在专业的工具和 skill。
5. 它在我本地电脑上长期可用，而不是一次性临时助手。

## 4. 产品优势定义

第二阶段必须把产品优势说清楚，否则开发容易继续做成“什么都做一点，但没有记忆点”的状态。

### 4.1 优势一：本地知识库不是附件，而是核心资产

本产品的本地知识库不是一个普通附加功能，而是产品核心。

优势体现在：

1. 用户在学习软件、浏览器页面、资料内容时，可以随时让智能体帮忙查资料、解释信息、总结重点。
2. 这些结果不会像普通聊天那样用完即丢，而是会进入本地知识库。
3. 知识库会随着用户学习过程持续成长，成为长期资产。
4. 本地知识库能与对话、规划、记忆、后续 skill 调用形成联动。

这与用户提到的同类产品差异在于：

1. 不只是“也有本地 RAG”。
2. 而是“把学习和沉淀变成长期闭环”。
3. 本地知识真正服务后续判断，而不是只做检索装饰。

### 4.2 优势二：个性化规划能力

当前产品不能只回答问题，还要能帮助用户做真正贴近个人情况的规划。

用户明确期望的规划价值包括：

1. 学习规划
2. 考研规划
3. 目标变更后的重新排程
4. 结合用户认知程度、分数段、习惯与执行情况做动态调整
5. 在用户学习不足时给出合适提醒

这类规划不应是网上模板拼接，而应是：

1. 结合用户背景
2. 结合用户历史记忆
3. 结合用户真实目标
4. 结合用户已经执行到哪一步

### 4.3 优势三：专业能力可接入、可专精

用户明确希望这个智能体不是全能泛助手，而是能向某个专业方向越长越强。

当前最明确的专业方向是：

1. `网络安全`
2. `CTF`
3. `渗透测试相关能力`

这意味着产品必须预留：

1. skill 接入口
2. 工具调用接口
3. 专业脚本执行接口
4. 知识点与工具推荐接口
5. 专业复盘和专业沉淀接口

用户对这个方向的理解是：

1. 这个智能体像“龙虾”一样，能向某个方向专精。
2. 接入某些 skill 后，就能在对应行业或专业里越来越强。
3. 未来这会直接服务学习、实习、求职和专业实践。

### 4.4 优势四：记忆与“灵魂”

用户不想要一个只会执行命令的冷工具。

用户明确希望它具备：

1. 长期记忆
2. 用户偏好
3. 失败教训
4. 用户日志
5. 更长期的人格层或语言层沉淀

用户已经提出要预留的正式载体包括：

1. `soul.md`
2. `user.md`
3. 用户日志
4. 项目规则和习惯性偏好

这里强调的不是表面拟人，而是：

1. 连续性
2. 温度感
3. 长期关系
4. 语言风格稳定
5. 能陪用户一起成长

### 4.5 优势五：Token 和成本优化

用户明确在意成本问题，尤其是在长期使用场景下。

因此本产品的优势不能只写成功能多，还必须写成：

1. 尽量少重复思考
2. 尽量少重复检索
3. 尽量少重复解释
4. 尽量把高成本 Token 用在真正需要深思考的地方

核心优化方向包括：

1. 本地缓存
2. 结构化记忆
3. 本地知识复用
4. 工具和 skill 替代部分长推理
5. 本地小模型承担低成本执行型任务

## 5. 第二阶段核心用户场景

### 5.1 学习辅助场景

用户在电脑软件、网页、文档、资料中学习时，希望能够：

1. 快速提问
2. 快速查资料
3. 快速获得概念解释
4. 快速获得页面重点总结
5. 把学习结果沉淀到本地知识库

### 5.2 长期规划场景

用户希望智能体能帮助自己做长期规划，而不是只回答即时问题。

重点场景包括：

1. 考研
2. 阶段学习计划
3. 技能成长路径
4. 目标变化后的重新规划

### 5.3 专业实战场景

用户希望未来在 CTF / 网络安全等方向中，智能体能帮助自己：

1. 调用工具
2. 执行脚本
3. 分析代码
4. 推荐适合当前问题的 skill
5. 整理题目知识与解题经验

### 5.4 长期陪伴场景

用户明确希望它未来不只是执行者，也可以是聊天对象或长期陪伴对象。

这要求产品未来支持：

1. 更强语言层
2. 更稳定长期记忆
3. 更自然表达方式
4. 更温和的交互体验

---

## 6. 第二阶段短期必须实现的目标

第二阶段短期不是把所有梦想一次做完，而是先做到“能用且舒服”。

当前短期必须实现的目标固定为：

1. 接入在线大模型 API 后，用户可以直接对话。
2. 智能体可以读取文件。
3. 智能体可以写入文件。
4. 智能体可以执行明确、受控的本地操作。
5. 智能体具备基本思维链路和结果解释能力。
6. 智能体具备本地缓存，减少重复思考和 Token 消耗。

这部分是第二阶段短期验收的核心，不得偏移。

---

## 7. 本地小模型兜底需求

### 7.1 为什么需要本地小模型

用户并不要求本地小模型非常聪明。

用户需要本地小模型的根本原因是：

1. 防止系统在没有在线 API 时完全瘫痪。
2. 让系统在离线或低成本状态下仍有最基本可执行性。
3. 让它能承担简单调度、简单执行、简单配置、简单知识整理。

### 7.2 本地小模型的定位

本地小模型应被定义为：

1. `低成本兜底执行模型`
2. `不是主思考模型`
3. `不是复杂规划模型`
4. `不是长上下文推理模型`

### 7.3 本地小模型可承担的能力

用户明确希望本地小模型至少能做：

1. 查看最新日志并做简单理解
2. 更换 API 或切换配置
3. 调用已有脚本或已有 skill
4. 按规则把信息写入知识库或记忆库
5. 执行明确、低歧义、短链路的本地操作

### 7.4 本地小模型当前不应承担的能力

1. 长链路复杂规划
2. 深度策略性思考
3. 长上下文复杂分析
4. 浏览器 GUI 理解
5. 高成本多工具自治流程

## 8. 双智能体与全局语言智能体设想

### 8.1 用户提出的核心设想

用户希望未来有两层智能体：

1. `执行智能体`
   位于对话域，负责当前任务推进。
2. `全局语言智能体`
   位于更高层，负责全局理解、语音沟通、任务分发和结果汇总。

### 8.2 全局语言智能体的预期作用

用户明确期待它未来可以：

1. 看到用户与执行智能体的全部对话
2. 在条件允许时观察浏览器行为
3. 接收直接指令
4. 代替用户总结和表达意图
5. 向执行智能体分发任务
6. 用文字或语音对任务结果做回报

### 8.3 这类能力的价值

用户认为它的核心价值在于：

1. 减少打字负担
2. 提高自然表达容错率
3. 形成“全局理解 + 局部执行”的协作结构
4. 增强陪伴感和温度感

### 8.4 当前判断

这条线值得记录并预留接口，但当前不进入短期主开发范围。

主要原因是：

1. Token 成本高
2. 架构复杂度高
3. 权限与隐私风险高
4. 当前阶段更重要的是先把单智能体主链路做稳

---

## 9. Token 与本地缓存需求

用户明确提出，省 Token 的重要方法之一是本地缓存。

当前应明确的缓存需求包括：

1. 缓存常用数据和资料
2. 缓存稳定知识摘要
3. 缓存稳定对话结果
4. 避免每轮都重新做重推理

缓存的价值不只是省钱，还包括：

1. 提高回答一致性
2. 提高响应速度
3. 降低重复检索
4. 提高系统长期稳定性

---

## 10. 统一平台诉求

用户希望最终通过一个本地平台统一操控电脑上的多个 AI 与能力，而不是在不同平台之间重复操作。

这意味着长期方向应包括：

1. 统一模型入口
2. 统一 skill 管理
3. 统一知识与记忆管理
4. 统一工具调用入口
5. 统一桌面侧使用体验

但当前阶段不应直接重做成大平台。

当前应该先确保：

1. 单智能体主链路稳
2. 接口可扩展
3. 后续接其他平台时不需要推翻当前结构

---

## 11. 浏览器学习辅助设想

用户希望未来在浏览器学习场景中具备更顺手的辅助方式，包括：

1. 桌面置顶悬浮对话窗
2. 选中网页内容后可直接发送给本地智能体
3. 智能体能解释选中内容
4. 智能体能辅助知识总结
5. 智能体能在一定程度上理解当前软件页面或 GUI 信息

当前判断：

1. 这个方向有价值
2. 但它明显属于后续能力
3. 需要在算力、权限、注入风险、隐私边界上单独收口

## 12. 当前边界

### 12.1 当前必须优先做的

1. 对话主链路
2. 文件读写
3. 命令执行
4. 思维链路
5. 本地缓存
6. 记忆与知识沉淀

### 12.2 当前只做预留的

1. skill 接入口
2. 本地小模型 adapter 接入口
3. 双智能体接口位置
4. 浏览器联动接口位置

### 12.3 当前不做的

1. 双智能体正式实现
2. 语音正式实现
3. 浏览器全局观察正式实现
4. 桌面 GUI 正式理解与操作
5. 重型平台化重构
6. 前端整站大翻修

---

## 13. 安全与风险需求

用户明确提到后续必须考虑：

1. Prompt Injection
2. 联网时可能面临的攻击面
3. 浏览器内容输入本地系统后的风险
4. 全局观察能力的权限和隐私边界

因此本需求文档明确要求：

1. 后续联网能力必须考虑输入清洗和权限边界。
2. 未来全局观察能力必须建立正式授权模式。
3. 不允许为了赶进度，默认开放高风险全局能力。

## 15. 第二阶段完成定义

当且仅当满足以下条件时，才能认为第二阶段短期目标成立：

1. 接入在线模型后，系统已经具备可用对话主链路。
2. 系统已经具备真实读写与执行能力。
3. 系统已经具备最小可见思维链路。
4. 系统已经具备本地缓存与复用。
5. 用户已经可以开始用它替代一部分在 VS Code 里频繁提问 AI 的行为。
6. 后续 skill、本地兜底模型、双智能体等方向已经有清晰文档口径，但没有破坏当前短期目标。

---

## 16. 当前结论

第二阶段的正确方向不是继续把产品说成“普通本地 AI”，而是：

1. `一个本地长期成长型智能体`
2. `一个学习、规划、专业化、记忆化、可进化的平台`
3. `一个短期先做到可用，长期再逐步发展为更强平台的产品`

当前开发必须优先保证：

1. 不跑偏到空泛平台化
2. 不跑偏到重型未来能力
3. 不跑偏到只有界面、没有核心价值

后续任务书、实现和验收，都应以本文为第二阶段总需求口径。

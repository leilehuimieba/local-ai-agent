## 3. 总体实现顺序

这 7 个能力不建议并行无序推进，推荐按“先主干、再增强、最后适配”的顺序落地。

补充原则：

1. 这里的实现顺序同时服务于低内存、低延迟和高并发目标
2. 所有新增模块都应优先落到 Rust runtime 或 Go 控制面现有结构里
3. 不允许为了赶进度引入 Python 过渡实现
4. 不允许因为参考仓库中存在某种实现方式，就直接沿用同类技术手段

### 3.1 第一阶段：上下文与工具主干

优先实现：

1. 实时仓库上下文加载
2. Prompt 分层与上下文装配
3. Tool Registry / Tool Contract

原因：

1. 这是 runtime 从“最小 demo”走向“工程化主循环”的基础
2. 如果不先做这三项，后续记忆、压缩、验证都没有稳定挂点
3. 这三项完成后，planner 和 execution 的边界会明显更清楚

### 3.2 第二阶段：记忆与压缩能力

优先实现：

4. 结构化记忆 schema 与按需召回
5. 大输出外置与上下文压缩

原因：

1. 这两项直接解决长任务可持续性问题
2. 没有结构化记忆和压缩，主循环很快会因为输出和上下文膨胀失控
3. 这两项应建立在第一阶段的上下文装配和工具合同之上

### 3.3 第三阶段：质量与适配能力

优先实现：

6. 更正式的验证链与完成判定
7. 统一模型适配层

原因：

1. 验证链要依赖前面的工具、记忆、压缩产物
2. 模型适配层适合在 runtime 数据结构趋稳后再接入
3. 如果过早做多模型和复杂验证，容易反过来把当前骨架重新搞乱

### 3.4 不建议提前投入的方向

在以上 7 项完成前，不建议优先做：

1. 多智能体编排
2. 浏览器自动化主链路
3. 桌面自动化主链路
4. 复杂自治工作流
5. 大量研究型评分器和偏置模块

## 4. 能力 1：实时仓库上下文加载

### 4.1 目标

让 runtime 在每次任务开始前，不只拿到 `user_input`，还拿到工作区和仓库的基础运行态上下文。

最低目标：

1. 当前工作区根目录
2. Git 是否可用
3. 当前分支
4. 主分支或默认分支
5. 工作树是否脏
6. 最近提交摘要
7. 工作区中存在的说明文件摘要
8. 与当前任务直接相关的上下文提示

### 4.2 推荐模块落点

Rust runtime 新增：

1. `crates/runtime-core/src/repo_context.rs`

Rust runtime 调整：

1. `crates/runtime-core/src/lib.rs`
2. `crates/runtime-core/src/contracts.rs`
3. `crates/runtime-core/src/planner.rs`
4. `crates/runtime-core/src/session.rs`
5. `crates/runtime-core/src/paths.rs`

Go 控制面可能补充：

1. `gateway/internal/contracts/contracts.go`
2. `gateway/internal/api/chat.go`

Frontend 只做展示增强，不做主逻辑：

1. `frontend/src/workspace/ContextSidebar.tsx`
2. `frontend/src/shared/contracts.ts`

### 4.3 文件级任务清单

#### A. Rust 新增文件

1. 新建 `crates/runtime-core/src/repo_context.rs`
2. 定义 `RepoContextSnapshot`
3. 定义 `GitSnapshot`
4. 定义 `WorkspaceDocSummary`
5. 定义 `load_repo_context(request: &RunRequest) -> RepoContextSnapshot`

#### B. Rust 修改文件

1. 在 `crates/runtime-core/src/lib.rs` 中接入 `repo_context`
2. 在 `crates/runtime-core/src/contracts.rs` 中给 `RunRequest` 或事件 metadata 预留 repo context 引用字段
3. 在 `crates/runtime-core/src/planner.rs` 中让 planner 能读取 repo context 摘要
4. 在 `crates/runtime-core/src/paths.rs` 中补齐说明文件候选路径解析

#### C. Go 修改文件

1. 在 `gateway/internal/api/chat.go` 中补充必要的 `context_hints`
2. 在 `gateway/internal/contracts/contracts.go` 中同步合同字段

#### D. Frontend 修改文件

1. 在 `frontend/src/shared/contracts.ts` 中补充 repo context 相关 metadata 类型
2. 在 `frontend/src/workspace/ContextSidebar.tsx` 中显示当前分支、工作树状态、说明文件命中结果

### 4.4 完成标准

完成后至少应满足：

1. 每次 run 都能拿到一个 repo context 快照
2. planner 不再只基于用户文本裸奔
3. 前端能看到最小 repo context 摘要
4. 失败时不会阻断整个 run，而是以降级模式继续

4. 手工跑一次聊天主链路
5. 检查事件流是否仍然完整
6. 检查确认流是否仍然可用

如果某批次引入了新合同字段，还应额外检查：

1. `gateway/internal/contracts/contracts.go`
2. `crates/runtime-core/src/contracts.rs`
3. `frontend/src/shared/contracts.ts`

三者是否仍然一致。

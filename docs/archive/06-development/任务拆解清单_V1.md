## 1. 文档目的

本文件用于把“下一步执行计划”进一步拆成实现型 AI 可直接照着推进的任务清单。

本文件目标：

1. 尽量降低实现偏航概率
2. 避免 AI 自己擅自扩展首版范围
3. 让实现工作优先围绕收口、拆分、对齐、补缺进行

本文件不是需求文档，也不是设计文档，而是执行导向的任务表。

---

## 2. 总执行原则

后续实现型 AI 必须遵守：

1. 先审查，再修改
2. 先收口，再扩展
3. 先拆重，再补缺
4. 先主链路，再体验优化
5. 不做后置能力
6. 每完成一阶段都给出阶段总结

禁止事项：

1. 不得先做浏览器自动化
2. 不得先做桌面自动化
3. 不得先做语音
4. 不得先做多智能体
5. 不得先做自动 fallback
6. 不得先把更多逻辑继续堆到大文件中

## 4. 阶段 A - 差异审查任务

### A1. 文档对照审查

任务：

1. 审查当前代码是否符合产品边界文档
2. 审查当前代码是否符合 MVP 文档
3. 审查当前 API 和事件对象是否符合合同文档
4. 审查当前 Rust runtime 是否符合运行时设计草案

输出：

1. 一份 gap review

gap review 至少包含：

1. 已符合项
2. 不符合项
3. 高优先级修复项
4. 中优先级修复项
5. 暂不处理项

完成标准：

1. 后续实现者对“当前项目到底差什么”有明确认识

禁止事项：

1. 不要一边审查一边乱改大量代码

### A2. 大文件风险审查

任务：

1. 评估 [App.tsx](D:/newwork/本地智能体/frontend/src/App.tsx) 当前承担了哪些职责
2. 评估 [lib.rs](D:/newwork/本地智能体/crates/runtime-core/src/lib.rs) 当前承担了哪些职责
3. 标出哪些职责最先应该拆出去

输出：

1. 前端拆分建议清单
2. Rust 拆分建议清单

完成标准：

1. 明确优先拆分项

---

## 5. 阶段 B - 前端收口任务

### B1. 建立前端模块边界

任务：

1. 把聊天提交逻辑从顶层页面中拆出
2. 把事件订阅逻辑从顶层页面中拆出
3. 把模型切换逻辑拆出
4. 把工作区切换逻辑拆出
5. 把确认流逻辑拆出
6. 把全局日志读取逻辑拆出

建议形成的模块方向：

1. `chat`
2. `events`
3. `settings`
4. `confirmations`
5. `logs`

完成标准：

1. 顶层 `App.tsx` 只保留页面组合与少量壳层状态
2. 前端逻辑不再高度集中

### B2. 页面结构收口

任务：

1. 确保前端只保留聊天、日志、设置三条主线
2. 不额外增加复杂面板
3. 保证确认卡片、模式切换、模型切换、工作区切换都合理放置

完成标准：

1. 页面结构与边界文档一致

### B3. 前端错误展示统一

任务：

1. 统一提交错误提示
2. 统一模型不可用提示
3. 统一运行时不可达提示
4. 统一确认提交失败提示

完成标准：

1. 用户可理解失败原因
2. 不同错误不会以完全不同方式杂乱显示

---

## 6. 阶段 C - Go 控制面收口任务

### C1. 核心 API 对齐

任务：

1. 对照 API 合同审查 `chat/run`
2. 对照 API 合同审查 `chat/confirm`
3. 对照 API 合同审查 `events/stream`
4. 对照 API 合同审查 `settings`
5. 对照 API 合同审查 `logs`

完成标准：

1. 核心接口字段与语义稳定
2. 前端无需针对“临时字段”写特殊兼容

### C2. 控制面职责清理

任务：

1. 确保 Go 只承担控制面职责
2. 不让 Go 层继续吞掉运行时逻辑
3. 保持 session / event / settings / confirmation / runtime bridge 的边界

完成标准：

1. Go 层仍是控制面，不变成第二个运行时

### C3. 设置与工作区状态收口

任务：

1. 确认模式切换状态持久化是否稳定
2. 确认模型切换状态持久化是否稳定
3. 确认工作区切换状态持久化是否稳定
4. 确认目录授权记忆机制是否有合理落点

完成标准：

1. 用户切换设置后，行为与状态一致

---

## 7. 阶段 D - Rust 运行时收口任务

### D1. runtime-core 拆分

任务：

1. 从 `lib.rs` 中拆出状态机相关逻辑
2. 拆出风险判断相关逻辑
3. 拆出文件与命令执行逻辑
4. 拆出记忆读写逻辑
5. 拆出知识检索逻辑
6. 拆出事件构造逻辑
7. 拆出输出总结逻辑

完成标准：

1. `lib.rs` 不再承载所有职责
2. 代码结构更接近运行时设计草案

### D2. 状态机校正

任务：

1. 检查当前实现是否符合 `Analyze -> Plan -> Execute -> Observe -> Verify -> Finish`
2. 检查确认流是否确实进入 `PausedForConfirmation`
3. 检查失败流是否统一进入 `Failed`

完成标准：

1. 状态流可解释
2. 前端看到的事件与状态机阶段一致

### D3. 风险与模式规则校正

任务：

1. 检查 `observe` 模式是否真的不修改
2. 检查 `standard` 模式是否允许常见写入
3. 检查 `full_access` 模式是否仍保留高危确认
4. 检查删除类动作与新目录提示逻辑是否可靠

完成标准：

1. 模式切换不是表面文章
2. 高危动作不会绕过确认

### D4. 记忆与上下文收口

任务：

1. 检查是否有全量记忆加载风险
2. 检查会话压缩摘要是否真实有效
3. 检查长期记忆写入是否已分块
4. 检查 `project_knowledge / preference / daily_note / lesson_learned` 这些类别是否已预留或合理承接

完成标准：

1. 记忆体系不只是名义存在
2. 不会因为历史堆积把上下文直接撑爆

### D5. 知识检索收口

任务：

1. 检查知识检索目标路径和范围是否合理
2. 检查结果摘要是否足够可读
3. 检查知识检索是否真正能参与任务推进

完成标准：

1. 知识检索不是孤立功能
2. 检索结果可被智能体和前端消费

## 8. 阶段 E - 合同与对象校正任务

### E1. 统一关键对象

任务：

1. 对齐 `RunRequest`
2. 对齐 `RunResult`
3. 对齐 `RunEvent`
4. 对齐 `ConfirmationRequest`
5. 对齐 `ConfirmationDecision`
6. 对齐模型、工作区、日志、错误对象

完成标准：

1. 字段命名稳定
2. 字段语义稳定
3. 三端理解一致

### E2. 统一事件类型

任务：

1. 确认 `run_started`
2. 确认 `analysis_ready`
3. 确认 `plan_ready`
4. 确认 `action_requested`
5. 确认 `action_completed`
6. 确认 `confirmation_required`
7. 确认 `verification_completed`
8. 确认 `memory_written`
9. 确认 `run_failed`
10. 确认 `run_finished`

完成标准：

1. 前端日志和事件流都能稳定识别这些类型

### E3. 错误对象统一

任务：

1. 统一运行时不可达错误
2. 统一模型不可用错误
3. 统一工具执行失败错误
4. 统一确认流相关错误

完成标准：

1. 不同错误类型有清晰来源与说明

2. 让日志页能看出一次任务经历了什么
3. 让错误和确认事件更明显

完成标准：

1. 日志页能服务复盘、测试和排障

### F3. 每日总结基础预留

任务：

1. 确保每日总结/日报素材有合理落点
2. 不要求复杂格式，但要有基础归集路径

完成标准：

1. 后续加日报功能时不需要推翻当前结构

## 11. 下一步实现的总体目标

后续实现型 AI 的下一阶段总体目标不是“做出更多功能”，而是：

1. 让当前首版原型模块化
2. 让当前首版原型更稳定
3. 让当前首版原型更贴近验收标准
4. 让后续继续开发不会再堆回大文件

---

## 12. 下一步项目要实现的具体目标

以下目标必须在下一阶段尽量完成：

1. 前端模块化拆分完成第一轮
2. Rust runtime-core 模块化拆分完成第一轮
3. Go / Rust / 前端关键合同对齐
4. 高危确认流稳定
5. 新目录首次接触提示稳定
6. 工作区切换稳定
7. 模型切换稳定
8. 会话连续与压缩具备首版可用水平
9. 日志页具备复盘能力

---

## 13. 下一阶段验收标准

下一阶段通过，不要求“首版最终完成”，但必须满足：

1. 前端和 Rust 不再继续把核心逻辑堆到单一大文件
2. 首版主链路继续可用，且更稳定
3. API 与事件字段基本稳定
4. 高危动作不会绕过确认
5. 至少 5 个核心场景可大体稳定跑通

---

## 14. 暂停事项

在下一阶段任务完成前，暂停以下开发方向：

1. 浏览器扩展
2. 浏览器自动化
3. 桌面自动化
4. 语音能力
5. 多智能体
6. 自动 fallback
7. 自动测速
8. 复杂 Skill 平台
9. 花哨视觉改版

## 15. 如果后续还需要你本人确认的事项

当前文档已经足够支持实现工作，不强依赖你继续补需求。

如果后续实现型 AI 需要你再确认，最可能的事项是：

1. 日志页最终更偏技术还是更偏用户视图
2. 模型配置页是否要更像 Cherry Studio
3. 每日日报是否纳入首版
4. 长期记忆分类是否进一步细化

这些都不阻断当前执行计划。

---

## 16. 使用规则

1. 后续实现型 AI 应先完成本清单中的阶段 A，再继续后面的工作。
2. 若某一阶段完成后出现新的结构性问题，应先修结构，再继续下一阶段。
3. 后续如果新增任务拆解版本，应明确写出与本版本的差异。

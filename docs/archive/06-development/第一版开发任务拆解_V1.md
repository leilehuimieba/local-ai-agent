## 1. 文档目的与作用

本文用于把已经收口的产品定义、场景范围、知识资产模型、状态维护模型、下一步建议标准与验收标准，进一步转化为可直接指导 AI 开发的第一版任务拆解文档。

本文重点不是重复讲产品是什么，也不是展开底层实现细节，而是回答以下执行问题：

1. 第一版应该按什么顺序推进
2. 每个阶段到底要完成什么
3. 每个阶段做到什么程度才算完成
4. 每个阶段明确不能顺手扩张什么
5. 各阶段如何与验收标准对齐

本文作为第一版开发推进文档，直接服务于以下目标：

1. 降低 AI 开发过程中的跑偏概率
2. 把产品文档转化为顺序明确的开发主线
3. 让实现型 AI 能按阶段推进，而不是同时铺开所有能力
4. 让每个阶段都能被检查、被回归、被验收

本文不展开以下内容：

1. 具体技术选型细节
2. 具体数据库结构
3. 具体提示词写法
4. 多智能体协作设计
5. 浏览器高复杂度自动化方案
6. 自动 fallback 机制设计
7. 重型记忆评分与复杂画像机制
8. 大范围架构重写方案

本文的作用，是把“第一版要做什么”进一步收口为“先做什么、做到什么算完成、哪些不能做”的执行顺序文档。

## 2. 使用方式

后续实现型 AI 在使用本文时，应按以下方式执行：

1. 先阅读本文列出的上游文档，不得只看本任务拆解文档就擅自改写产品方向
2. 严格按阶段顺序推进，不得在前置阶段未完成时跳到后置阶段大规模开发
3. 每完成一个阶段，都要检查该阶段的完成标准与禁止扩张项
4. 若某个新增需求不在本文阶段范围内，应先判断是否属于第一版边界，再决定是否延后
5. 若某个阶段完成后发现上游文档存在冲突，应优先回到上游文档口径，而不是自行改写产品定义

本文适合指导的是：

1. 第一版能力主线实现
2. 场景主线落地顺序安排
3. 回归与验收对齐

本文不适合被用于：

1. 借机重新定义产品
2. 擅自扩大第一版范围
3. 以“顺手优化”为名引入与主线无关的大型改动

## 3. 开发总原则

第一版开发必须围绕“知识沉淀、状态理解、下一步建议、有限执行”这条主线展开，不以功能数量、工具接入数量或自动化炫技作为主要推进目标。

后续实现型 AI 必须遵守以下原则：

1. `先主线，后增强`
   先让知识沉淀、状态理解、下一步建议与有限执行在第一优先级场景中成立，再考虑差异化增强。

2. `先第一优先级场景，后第二优先级场景`
   先完成工程开发场景与通用学习场景，再处理浏览学习场景与 CTF / 安全学习场景的基础验证。

3. `先可持续使用，后功能堆叠`
   先保证多轮使用、状态续接、沉淀复用与建议有效，再考虑添加更多功能点。

4. `先完成闭环，后扩大边界`
   先完成“输入进入 -> 形成沉淀 -> 状态更新 -> 给出建议 -> 有限执行桥接 -> 留下续接入口”的闭环，再考虑更大范围的增强。

5. `先满足验收，后追求复杂`
   只要当前还未达到验收标准，就不应因为新点子有吸引力而偏离主线。

6. `先守边界，后谈扩展`
   Skill、MCP 与场景增强能力只能作为增强层使用，不能把产品重新写成技能平台、能力平台或多 Agent 平台。

## 4. 当前固定边界

后续实现型 AI 必须将以下内容视为当前固定边界，不得在第一版开发过程中擅自突破。

### 4.1 当前必须围绕的核心主线

1. `知识沉淀`
2. `状态理解`
3. `下一步建议`
4. `有限执行`

### 4.2 当前必须优先服务的场景

1. `工程开发场景`
2. `通用学习场景`

### 4.3 当前可做基础验证但不得抢主线资源的场景

1. `浏览学习场景`
2. `CTF / 安全学习场景`

### 4.4 当前明确不扩的内容

1. `多智能体编排`
2. `浏览器高复杂度自动化`
3. `自动 fallback`
4. `重型记忆评分`
5. `重型能力画像`
6. `高权限自动化控制`
7. `为了当前版本成立而进行的大范围架构重写`

## 5. 开发顺序总览

第一版开发顺序建议正式收口如下：

1. `阶段 0：开发入口冻结与对齐`
2. `阶段 1：工程开发场景主线落地`
3. `阶段 2：通用学习场景主线落地`
4. `阶段 3：知识资产与状态维护联动落地`
5. `阶段 4：下一步建议能力落地`
6. `阶段 5：有限执行桥接落地`
7. `阶段 6：第二优先级场景基础验证`
8. `阶段 7：回归、验收与收口`

上述顺序的核心逻辑如下：

1. 先统一第一版到底做什么，避免 AI 一边开发一边改产品
2. 先让最能验证主线的工程开发场景成立
3. 再让最能体现产品本体的通用学习场景成立
4. 然后补齐知识资产与状态维护的联动能力
5. 在此基础上收口下一步建议质量
6. 再让有限执行与建议形成桥接
7. 最后用第二优先级场景验证主线可迁移性
8. 以验收标准作为最终放行依据，而不是以功能多少作为判断标准

## 6. 阶段 0：开发入口冻结与对齐

### 6.1 阶段目标

在真正进入实现之前，先统一第一版开发口径，确保后续实现型 AI 使用的是同一套产品边界、场景范围、建模口径与验收标准。

### 6.2 必须完成项

1. 明确当前产品唯一主线是“知识沉淀型个人助手”
2. 明确第一优先级场景为工程开发场景与通用学习场景
3. 明确第二优先级场景只做基础验证，不做主线抢跑
4. 明确知识资产模型、状态维护模型与下一步建议标准为后续能力实现依据
5. 明确验收标准文档是阶段完成判断依据，而不是开发完成后才去补看的文档
6. 明确 Skill、MCP 属于增强层，不属于产品本体

### 6.3 完成标准

1. 后续实现型 AI 不再重新讨论产品定位
2. 后续任务不再把产品写成多 Agent 平台或能力平台
3. 后续所有开发任务都能明确映射到主线能力或优先场景

### 6.4 禁止扩张项

1. 不在此阶段新增额外场景主线
2. 不在此阶段重开产品定义讨论
3. 不在此阶段擅自补充复杂自动化能力

### 6.5 对应验收关系

本阶段不单独构成产品验收项，但决定后续所有阶段是否会跑偏；若本阶段口径不稳，后续开发结果均不应直接放行。

---

## 7. 阶段 1：工程开发场景主线落地

### 7.1 阶段目标

先在工程开发场景中跑通第一版主线，验证系统是否能围绕真实任务推进形成连续使用价值，而不是停留在一次性问答。

### 7.2 必须完成项

1. 能接住用户当前开发任务、背景、目标与中途补充信息
2. 能围绕同一任务持续维护当前主题、当前目标、当前阶段、当前进度、当前阻塞与最近结果
3. 能把开发过程中的阶段结论、问题与解法记录、方法经验与可继续接上的上下文记录沉淀下来
4. 能在多轮推进中稳定回答“现在做到哪了、还差什么、下一步做什么”
5. 一轮任务结束后，能保留阶段结论、未完成事项与后续入口，支持下一轮续接

### 7.3 完成标准

1. 用户在同一开发任务中连续多轮使用时，不需要每轮都重新解释完整背景
2. 系统能说清当前任务推进到了哪里，而不是只重复用户刚刚输入的内容
3. 阶段结论与问题记录能被保留，并在后续轮次中继续参与理解与建议
4. 续接任务时，系统能基于已有沉淀继续推进，而不是像新对话一样从零开始

### 7.4 禁止扩张项

1. 不把工程开发场景扩成“全能研发副驾平台”
2. 不为了工程场景体验而先做复杂浏览器自动化
3. 不为了工程场景能力而引入多智能体分工
4. 不因为工程场景高频使用，就把通用学习场景降为附属功能

### 7.5 对应验收关系

本阶段直接对应以下验收要求：

1. 工程开发场景验收用例 A：多轮任务推进
2. 工程开发场景验收用例 B：阶段收口与续接
3. 核心验收维度中的知识沉淀验收
4. 核心验收维度中的状态理解验收
5. 核心验收维度中的连续使用验收

---

## 8. 阶段 2：通用学习场景主线落地

### 8.1 阶段目标

在工程开发场景之外，再让通用学习场景成立，验证产品本体不是“只适合任务推进”，而是能真正承接长期学习过程。

### 8.2 必须完成项

1. 能接住用户当前学习主题、学习目标、学习材料、学习困惑与阶段反馈
2. 能持续维护学习场景下的当前主题、当前目标、当前阶段、当前掌握情况、当前阻塞与最近结果
3. 能沉淀主题知识、阶段理解、方法经验、问题记录与可继续接上的上下文记录
4. 能稳定回答“学到哪了、哪些还不稳、下一步该学什么”
5. 能支持一轮学习后的回顾、暂停与后续续接

### 8.3 完成标准

1. 用户围绕同一学习主题连续使用时，系统能区分已理解、待巩固与未理解内容
2. 学习过程中的阶段理解与关键结论可以被沉淀并回看
3. 用户在后续轮次追问时，系统能基于已有沉淀给出续接建议
4. 通用学习场景中的建议质量达到高质量建议标准，而不是只有泛泛鼓励

### 8.4 禁止扩张项

1. 不把通用学习场景扩成复杂课程平台
2. 不为了做学习体验而提前引入重型能力画像与复杂评分
3. 不把浏览学习场景中的所有能力都提前并入通用学习主线
4. 不把 CTF / 安全学习场景直接作为学习主线替代品

### 8.5 对应验收关系

本阶段直接对应以下验收要求：

1. 通用学习场景验收用例 C：连续学习主题跟踪
2. 通用学习场景验收用例 D：回顾与再进入
3. 核心验收维度中的知识沉淀验收
4. 核心验收维度中的状态理解验收
5. 核心验收维度中的下一步建议验收
6. 核心验收维度中的连续使用验收

---

## 9. 阶段 3：知识资产与状态维护联动落地

### 9.1 阶段目标

在两个第一优先级场景都初步成立之后，进一步把知识沉淀与状态理解联动起来，避免知识资产成为静态存档，也避免状态维护变成脱离沉淀的短期缓存。

### 9.2 必须完成项

1. 围绕第一优先级知识资产类型建立稳定沉淀路径
2. 围绕第一优先级状态类型建立稳定维护路径
3. 明确哪些内容进入知识资产，哪些内容进入状态维护，避免混用
4. 让状态更新能够引用已有知识资产，而不是完全脱离历史沉淀
5. 让知识资产在后续状态理解与任务续接中真正被复用

### 9.3 本阶段优先处理的知识资产

1. `主题知识`
2. `阶段结论`
3. `方法经验`
4. `问题与解法记录`
5. `可继续接上的上下文记录`

### 9.4 本阶段优先处理的状态类型

1. `当前场景`
2. `当前主题`
3. `当前目标`
4. `当前阶段`
5. `当前进度`
6. `当前阻塞`
7. `最近结果`
8. `当前掌握情况`

### 9.5 完成标准

1. 高价值输入不会只停留在原始记录层，而能形成可回看、可复用的资产
2. 当前状态不会只反映当前一句话，而能吸收阶段结论与最近结果
3. 用户追问“之前结论是什么、现在处于什么阶段”时，系统能给出一致回答
4. 资产与状态之间的边界清晰，不出现“所有内容都既是状态又是资产”的混乱情况

### 9.6 禁止扩张项

1. 不在本阶段引入复杂知识图谱
2. 不在本阶段引入重型关系网络
3. 不在本阶段引入重型能力画像
4. 不在本阶段引入高噪声全量行为记录

### 9.7 对应验收关系

本阶段直接对应以下验收要求：

1. 核心验收维度中的知识沉淀验收
2. 核心验收维度中的状态理解验收
3. 核心验收维度中的连续使用验收
4. 第一版通过总体标准中的“知识沉淀可回看可复用”
5. 第一版通过总体标准中的“状态理解可持续成立”

---

## 10. 阶段 4：下一步建议能力落地

### 10.1 阶段目标

在知识资产与状态维护形成基本联动后，收口下一步建议能力，让系统不仅能描述现状，还能给出真正推动用户继续前进的建议。

### 10.2 必须完成项

1. 基于当前状态给出进度判断型建议
2. 基于当前阻塞与当前掌握情况给出缺口识别型建议
3. 基于当前目标、最近结果与已有知识资产给出下一步行动型建议
4. 在一轮结束时给出收口与续接型建议
5. 建议必须能够解释为什么先做这一步，而不是只给结论

### 10.3 本阶段建议的核心依据

1. 当前状态
2. 当前轮次新输入
3. 已有知识资产

### 10.4 完成标准

1. 工程开发场景中的建议稳定达到高质量建议标准
2. 通用学习场景中的建议稳定达到高质量建议标准
3. 建议能明确回应当前阶段和当前阻塞，而不是模板化输出
4. 用户看完建议后，能明确知道自己下一步最应该先做什么
5. 建议能体现连续性，说明其与已有沉淀和最近结果之间的关系

### 10.5 禁止扩张项

1. 不在本阶段引入复杂长周期规划系统
2. 不在本阶段引入重型个性化推荐系统
3. 不在本阶段引入自动决策型建议能力
4. 不在本阶段引入多智能体协同建议能力

### 10.6 对应验收关系

本阶段直接对应以下验收要求：

1. 核心验收维度中的下一步建议验收
2. 第一版通过总体标准中的“下一步建议具备推进价值”
3. 工程开发场景与通用学习场景中的关键建议质量要求

---

## 11. 阶段 5：有限执行桥接落地

### 11.1 阶段目标

在建议能力基本成立之后，再让有限执行承担桥接作用，使系统能够在明确授权下把建议推进到少量实际动作，但不滑向高权限自动化系统。

### 11.2 必须完成项

1. 明确有限执行只服务当前目标，不单独成为产品主能力
2. 建立“建议 -> 执行 -> 结果回写”的基本桥接链路
3. 明确哪些动作属于第一版允许的有限执行，哪些属于越界能力
4. 让执行结果能够被回写进知识资产与状态维护链路
5. 保证执行动作有明确授权边界与结果留痕

### 11.3 第一版有限执行应收的范围

1. 与当前目标直接相关的少量辅助动作
2. 与当前任务或学习过程直接相关的整理、记录与结果回写
3. 能明显帮助用户把建议转成行动的低风险桥接动作

### 11.4 完成标准

1. 用户能感受到建议不是停留在口头层面，而是可以被顺畅推进
2. 执行结果不会游离在主线之外，而能回写到状态与知识沉淀中
3. 有限执行没有越界扩张为高风险自动化控制
4. 有限执行在第一优先级场景中能形成基本闭环价值

### 11.5 禁止扩张项

1. 不在本阶段引入高权限自动化控制
2. 不在本阶段引入全局行为监控
3. 不在本阶段引入浏览器高复杂度自动化
4. 不在本阶段引入脱离授权的自动执行链路

### 11.6 对应验收关系

本阶段直接对应以下验收要求：

1. 核心验收维度中的有限执行验收
2. 第一版通过总体标准中的“有限执行能形成桥接”
3. 放行判断中的边界守住要求

---

## 12. 阶段 6：第二优先级场景基础验证

### 12.1 阶段目标

在第一优先级场景主线成立后，再用浏览学习场景与 CTF / 安全学习场景验证主线能力是否具备可迁移性，但不改变第一版重心。

### 12.2 浏览学习场景必须完成项

1. 能接住当前页面内容、即时提问与阶段总结
2. 能对当前页面内容做理解支持与基础总结
3. 能把浏览过程中的重点、疑问与待回顾内容沉淀下来
4. 能给出下一步浏览或回顾建议

### 12.3 CTF / 安全学习场景必须完成项

1. 能接住题目背景、线索、阶段结果与当前判断
2. 能整理分析路径、已尝试步骤、当前卡点与下一步方向
3. 能沉淀方法经验、失败路径、有效路径与后续入口
4. 能在场景增强能力帮助下给出方向性建议

### 12.4 完成标准

1. 浏览学习场景中的建议至少稳定达到基本合格建议标准
2. CTF / 安全学习场景中的建议至少稳定达到基本合格建议标准
3. 两类场景都能体现主线能力的迁移性，而不是另起一套产品主线
4. 第二优先级场景没有反向拉高第一版复杂度

### 12.5 禁止扩张项

1. 不把浏览学习场景扩成复杂浏览器自动化产品
2. 不把 CTF / 安全学习场景扩成自动攻击或自动利用平台
3. 不为了第二优先级场景表现而引入多智能体编排
4. 不允许第二优先级场景抢占第一优先级场景的收口资源

### 12.6 对应验收关系

本阶段直接对应以下验收要求：

1. 浏览学习场景验证用例
2. CTF / 安全学习场景验证用例
3. 第一版通过总体标准中的主线可迁移性要求

## 13. 阶段 7：回归、验收与收口

### 13.1 阶段目标

在主线能力与场景验证都完成后，按验收文档逐项回归，判断第一版是否达到“可以放行”或“只能限制通过”的状态。

### 13.2 必须完成项

1. 按工程开发场景与通用学习场景的关键用例进行完整回归
2. 按浏览学习场景与 CTF / 安全学习场景进行基础验证回归
3. 对照知识沉淀、状态理解、下一步建议、有限执行与连续使用五个验收维度逐项检查
4. 保留原始输入样本、沉淀结果、状态变化、建议输出与执行留痕
5. 根据验收文档判断是“通过”“限制通过”还是“不通过”

### 13.3 完成标准

1. 第一优先级场景关键用例稳定通过
2. 第二优先级场景完成基础验证
3. 主线四项能力都能提供可说明、可回看的证据
4. 能明确指出当前剩余问题是否影响第一版成立

### 13.4 禁止扩张项

1. 不在验收阶段通过新增大功能来掩盖主线问题
2. 不因为局部亮点就忽略整体主线未成立
3. 不在证据不足的情况下凭主观感觉宣布通过

## 14. 各阶段依赖关系

为避免开发顺序被打乱，当前各阶段依赖关系明确如下：

1. 阶段 0 完成前，不应启动大规模实现
2. 阶段 1 与阶段 2 是第一版主线成立的基础，优先级高于后续所有增强
3. 阶段 3 依赖阶段 1 与阶段 2 中已有的真实输入、沉淀与状态样本
4. 阶段 4 依赖阶段 3 已能提供稳定的状态与知识资产依据
5. 阶段 5 依赖阶段 4 已能给出足够可执行的建议
6. 阶段 6 必须在阶段 1 至阶段 5 已基本成立后再开展
7. 阶段 7 必须在前六个阶段已有可回归样本后再进行

若依赖关系被打乱，最常见的问题会是：

1. 建议能力缺乏真实状态依据
2. 知识沉淀变成孤立存档
3. 第二优先级场景把主线资源带偏
4. 验收阶段只能凭感觉判断

---

## 15. 开发过程中的判断规则

后续实现型 AI 在推进过程中，遇到新增需求或临时想法时，应按以下顺序判断是否进入当前版本：

1. 该内容是否直接强化知识沉淀、状态理解、下一步建议或有限执行
2. 该内容是否优先服务工程开发场景或通用学习场景
3. 没有它时，第一版是否仍然可以成立
4. 加入它后，是否会显著提高复杂度或资源消耗
5. 它更适合作为当前版本内容，还是更适合作为后续增强项

若判断结果是：

1. 直接强化主线且优先服务第一优先级场景
   则可纳入当前阶段开发

2. 有价值但不影响第一版成立
   则应进入后续增强列表，而不是插队进入主线

3. 主要价值来自平台化、自动化或复杂扩张
   则当前不应纳入第一版

---

## 16. 当前明确暂停的方向

在第一版完成前，以下方向明确暂停，不得抢占主线开发资源：

1. 多智能体协作与编排
2. 浏览器高复杂度自动化
3. 自动 fallback
4. 重型记忆评分与复杂画像
5. 高权限桌面控制
6. 与主线无关的平台化能力拼装
7. 以未来公开版设想为名的超前重构

这些方向不是永久否定，而是明确不属于当前第一版开发任务拆解范围。

---

## 17. 第一版通过前必须满足的收口条件

在宣称第一版已经成立前，后续实现型 AI 至少应确认以下条件同时满足：

1. 工程开发场景主线成立
2. 通用学习场景主线成立
3. 知识资产可回看、可复用
4. 状态维护能持续更新并支持续接
5. 下一步建议在第一优先级场景中达到高质量建议标准
6. 有限执行能形成桥接，但未越界扩张
7. 浏览学习场景与 CTF / 安全学习场景完成基础验证
8. 验收证据链完整，能够支撑放行判断

若上述任一条件未满足，则不应宣称第一版已经真正完成。

## 18. 文档结论与执行要求

基于当前已经确定的产品定义、场景优先级、知识资产模型、状态维护模型、下一步建议标准与验收标准，第一版开发任务拆解当前正式收口如下：

1. 第一版开发必须优先围绕工程开发场景与通用学习场景展开
2. 第一版开发必须按“场景主线成立 -> 资产与状态联动 -> 建议能力收口 -> 有限执行桥接 -> 第二优先级场景验证 -> 验收放行”这一顺序推进
3. 第一版开发不得重新扩展到多智能体、复杂浏览器自动化、自动 fallback、重型记忆评分或平台化重构
4. 第一版开发的完成判断，必须以验收标准与评估用例为准，而不是以功能数量为准

后续实现型 AI 若直接根据本文推进开发，应默认遵守以下执行要求：

1. 不跳阶段
2. 不擅自改产品定位
3. 不擅自扩版本范围
4. 不在第一优先级场景未收口前抢做第二优先级场景增强
5. 不在验收标准未满足前宣称第一版已成立

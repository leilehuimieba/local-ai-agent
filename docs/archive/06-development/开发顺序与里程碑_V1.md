## 1. 文档目的

本文件用于规定：

1. 第一版实现工作按什么顺序推进
2. 哪些工作必须先完成，哪些才能开始
3. 每个阶段的目标、产出物和完成标志是什么
4. 哪些事情明确不应该在前期提前做

本文件的作用是防止实现阶段“想到哪做到哪”，而是确保开发遵循统一节奏。

---

## 2. 开发总原则

### 2.1 先骨架，后细节

先建立主链路骨架，再补细节优化。

### 2.2 先主链路，后扩展

先保证桌面启动、聊天、Go 控制面、Rust 运行时、日志回流、风险确认这一条主链路跑通，再考虑更复杂能力。

### 2.3 先合同，后并行

只有在合同和边界稳定后，前端、Go、Rust 才适合并行实现。

### 2.4 先最小可用，后优化体验

第一版先跑通最小闭环，不在前期过度追求花哨 UI 和复杂能力。

### 2.5 先减法，后增量

实现过程中优先做减法，不把旧项目复杂性原样搬进新项目。

---

## 3. 阶段总览

第一版建议按以下阶段推进：

1. Phase 0 - 文档冻结与项目脚手架
2. Phase 1 - 最小系统骨架打通
3. Phase 2 - 最小聊天与事件闭环
4. Phase 3 - 核心执行能力接入
5. Phase 4 - 风险控制与确认流
6. Phase 5 - 记忆与知识闭环
7. Phase 6 - 打磨、验收与首版冻结

---

## 4. Phase 0 - 文档冻结与项目脚手架

### 4.1 目标

在进入实现前，冻结首版边界并建立新仓库最小目录骨架。

### 4.2 主要任务

1. 冻结需求文档
2. 冻结产品边界文档
3. 冻结 MVP 文档
4. 冻结总体架构文档
5. 冻结 Rust 运行时设计文档
6. 冻结 API 与事件合同文档
7. 冻结迁移边界文档
8. 建立新项目目录基础结构

### 4.3 输出物

1. 文档体系
2. 空白但正确的新仓库结构

### 4.4 完成标志

1. 后续实现者不再需要回头补最上层边界文档
2. 新项目目录结构已经建立
3. 原项目和新项目边界明确

---

## 5. Phase 1 - 最小系统骨架打通

### 5.1 目标

让系统从“目录存在”变成“基本能启动的空壳系统”。

### 5.2 主要任务

1. 建立 `frontend/` 最小 React + Vite 骨架
2. 建立 `gateway/` 最小 Go 控制面骨架
3. 建立 `crates/` 最小 Rust workspace 骨架
4. 打通桌面启动器到 Web 控制台的最小链路
5. 配置新项目独立端口
6. 建立最小设置和环境配置读取能力

### 5.3 输出物

1. 桌面启动器可打开新项目 Web 控制台
2. 前端、Go、Rust 三端都有最小工程骨架
3. 新旧项目互不冲突

### 5.4 完成标志

1. 能启动新项目
2. 能看到新项目基础页面
3. Go 控制面与 Rust 服务或运行时入口可互通

### 5.5 此阶段不要做

1. 不做复杂页面
2. 不做记忆系统
3. 不做知识检索
4. 不做浏览器/桌面能力

---

## 6. Phase 2 - 最小聊天与事件闭环

### 6.1 目标

打通“聊天输入 -> Go -> Rust -> 事件流 -> 前端展示 -> 最终答复”这条最小闭环。

### 6.2 主要任务

1. 实现聊天页基础交互
2. 实现 `chat/run` API
3. 实现最小 `RunRequest` / `RunResult`
4. 实现事件流订阅
5. 实现最小运行时状态机
6. 支持返回简单文本结果

### 6.3 输出物

1. 用户可在聊天页输入任务
2. 系统能返回答复
3. 前端能看到运行开始、执行中、结束事件

### 6.4 完成标志

1. 聊天主链路打通
2. 事件流基础可见
3. 日志页能显示基础运行记录

### 6.5 此阶段不要做

1. 不做复杂动作执行
2. 不做复杂记忆写入
3. 不做自动恢复

---

## 7. Phase 3 - 核心执行能力接入

### 7.1 目标

让运行时具备第一版必须的核心能力。

### 7.2 主要任务

1. 接入命令执行能力
2. 接入文件读写能力
3. 接入工作区切换与工作区上下文
4. 设计并接入粗粒度能力调用接口
5. 把执行结果摘要化后回流前端

### 7.3 输出物

1. 基础命令执行闭环
2. 基础文件操作闭环
3. 基础工作区切换闭环

### 7.4 完成标志

1. 系统能完成典型文件修改任务
2. 系统能完成典型命令执行任务
3. 前端能看到工具调用和结果摘要

### 7.5 此阶段不要做

1. 不做浏览器自动化
2. 不做桌面自动化
3. 不做全量 Skill 系统

---

## 8. Phase 4 - 风险控制与确认流

### 8.1 目标

让系统从“能执行”升级为“能安全执行”。

### 8.2 主要任务

1. 实现三种模式
   - 观察模式
   - 标准模式
   - 全权限模式
2. 实现风险判断基础逻辑
3. 实现确认请求对象
4. 实现确认弹窗或确认交互
5. 实现新目录首次接触提示
6. 实现删除和高危动作拦截

### 8.3 输出物

1. 风险确认流闭环
2. 新目录提示闭环
3. 模式切换影响行为的基础逻辑

### 8.4 完成标志

1. 常见低风险动作可直接执行
2. 删除和高风险动作需要确认
3. 前端能展示风险等级与关键说明

### 8.5 此阶段不要做

1. 不做复杂政策编辑台
2. 不做复杂风险策略 DSL

---

## 9. Phase 5 - 记忆与知识闭环

### 9.1 目标

让系统不只是一次性执行器，而是具备上下文连续和基础沉淀能力。

### 9.2 主要任务

1. 实现会话连续上下文
2. 实现长期记忆基础分块
3. 实现按需记忆召回
4. 实现基础知识检索
5. 实现上下文压缩
6. 实现基础记忆写入摘要事件
7. 为每日总结预留基础数据归集

### 9.3 输出物

1. 记忆读写闭环
2. 知识检索闭环
3. 上下文压缩基础逻辑

### 9.4 完成标志

1. 系统能在同一会话中持续推进任务
2. 重要信息能写入长期记忆
3. 不会每次都全量加载所有记忆
4. 检索结果能支持任务推进

### 9.5 此阶段不要做

1. 不做复杂自动反思引擎
2. 不做复杂经验归因系统

---

## 10. Phase 6 - 打磨、验收与首版冻结

### 10.1 目标

把已经跑通的主链路整理成可稳定验收的第一版。

### 10.2 主要任务

1. 打磨聊天页、日志页、设置页基础体验
2. 清理不必要的接口和临时逻辑
3. 对照验收标准逐条验证
4. 修复关键缺陷
5. 冻结第一版交付边界

### 10.3 输出物

1. 首版可用版本
2. 首版验收记录
3. 首版待办清单

### 10.4 完成标志

1. 通过验收标准
2. 核心场景稳定
3. 明确哪些能力属于第二版

---

## 11. 建议的实现顺序细化

建议实现顺序进一步细化为：

1. 项目骨架
2. 桌面启动链路
3. 前端聊天页骨架
4. Go 控制面最小 API
5. Rust 运行时最小状态机
6. 事件流
7. 日志页
8. 命令执行
9. 文件读写
10. 工作区切换
11. 模式切换
12. 风险确认
13. 新目录提示
14. 记忆基础分块
15. 知识检索
16. 上下文压缩
17. 设置页完善
18. 首版验收

---

## 12. 不建议并行推进的内容

以下内容不建议过早并行：

1. 浏览器自动化
2. 桌面自动化
3. 语音能力
4. 复杂 Skill 平台
5. 自动 fallback
6. 多智能体

原因：

1. 容易拖慢首版主链路
2. 容易让架构发散
3. 容易增加无效 token 和无效开发成本

## 13. 里程碑定义

### M1 - 空壳系统启动

标志：

1. 桌面启动器可进入新前端
2. 新项目端口独立
3. Go、Rust、前端骨架可启动

### M2 - 聊天与事件流打通

标志：

1. 前端可提交任务
2. 事件流可见
3. 可返回简单结果

### M3 - 核心执行闭环完成

标志：

1. 命令执行可用
2. 文件读写可用
3. 工作区操作可用
4. 日志可复盘

### M4 - 风险与模式闭环完成

标志：

1. 模式切换可用
2. 风险确认可用
3. 新目录提示可用

### M5 - 记忆与知识闭环完成

标志：

1. 会话连续可用
2. 长期记忆基础可用
3. 知识检索可用
4. 上下文压缩可用

### M6 - 首版验收完成

标志：

1. 通过验收标准
2. 首版边界冻结
3. 第二版工作清单形成

## 14. 每阶段输出文档建议

实现过程中建议同步维护：

1. 阶段完成记录
2. 待决问题清单
3. 已知限制清单
4. 下一阶段进入条件

---

## 15. 阶段进入条件

### 15.1 进入下一阶段前必须满足

1. 上一阶段核心目标已经达成
2. 上一阶段输出物已整理
3. 不存在阻断主链路的关键缺陷
4. 不存在会推翻前序设计的重大分歧

### 15.2 不满足时怎么办

1. 优先补齐上一阶段
2. 如需改文档，先回到相应规划文档修订
3. 不建议带着结构性问题进入下一阶段

---

## 16. 第一版不应追求的里程碑

以下内容不能作为首版里程碑：

1. 技术栈看起来很高级
2. UI 看起来很炫
3. 能力很多但主链路不稳定
4. 浏览器/桌面功能先做出来
5. 技能很多但调度混乱

第一版真正的里程碑应始终围绕：

1. 能启动
2. 能聊天
3. 能执行
4. 能看过程
5. 能控风险
6. 能记住关键内容

---

## 17. 与实现型 AI 协作时的建议

如果后续把实现交给新的 AI，对方必须遵守：

1. 先完成 M1，再做 M2
2. 先完成 M2，再做 M3
3. 不得擅自把后置功能提前纳入主线
4. 不得用实验性复杂度替代首版可用性
5. 每个阶段都要产出阶段结果说明

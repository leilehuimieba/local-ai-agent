## 0. 降级说明

本文属于旧的产品深化方向材料，当前保留为历史参考。

现在继续推进时，应优先围绕“知识沉淀型个人助手”主线，而不是继续按“忠实用户转化导向”组织任务。

---

## 1. 文档目的

本文只解决一件事：

> 把当前已经“可用、可执行、可沉淀”的本地智能体，继续收口到“值得长期使用、值得迁移主工作流、值得持续沉淀个人资产”的程度。

本文不是新的愿景文档，也不是自由 brainstorming 文档。
本文是后续实现型 AI 的正式执行入口，用于回答：

1. 为什么用户已经会用，但还不一定会长期留下来。
2. 现在最该补哪几类能力，才能提高忠实用户转化概率。
3. 每个工作包应该先做什么，再做什么，做到什么程度才算通过。
4. 后续实现型 AI 应该如何一步一步推进，避免重新发散。

---

## 2. 当前阶段判断

当前系统已经完成：

1. 第二阶段短期目标正式通过。
2. 在线模型对话、文件读写、命令执行、风险确认、缓存、记忆和知识沉淀主链路成立。
3. 项目状态类回答已经可以稳定输出“已完成能力 / 当前阶段 / 待收口项”并与真实样本留证绑定。

当前系统还没有完成：

1. 让普通用户强烈感受到“它记得我、懂我现在做到哪、能继续推进我昨天的事”。
2. 让知识沉淀从“已有能力”升级到“像长期资产一样可管理、可信任、可持续复用”。
3. 让日志、事件、记忆、知识、项目推进在前端形成真正自然的日常工作台体验。

一句话判断：

> 当前项目已经通过“可用性验收”，但还处于“忠实用户体验尚未完全产品化”的阶段。

---

## 3. 用户视角下的关键问题

本阶段不再只从工程视角看“功能是否成立”，而要从忠实用户视角看“为什么用户会留下来”。

核心问题固定为：

1. `连续性是否足够强`
   用户第二天回来时，系统是否真的知道他昨天在做什么、做到哪里、接下来应该继续什么。

2. `本地知识是否足够像长期资产`
   用户整理过、学习过、执行过的内容，是否能稳定变成后续判断依据，而不是散在日志里。

3. `执行是否足够可信`
   用户是否敢把更多真实任务交给系统，而不是只把它当成聊天壳。

4. `体验是否足够顺`
   用户打开系统后，是否能一眼看懂当前任务、当前阶段、下一步和当前沉淀。

5. `是否形成日常闭环`
   用户是否会自然形成“打开系统 -> 查看当前状态 -> 推进任务 -> 形成沉淀 -> 第二天继续”的稳定使用习惯。

---

## 4. 本阶段总目标

本阶段必须把以下六个目标正式收口：

1. `把连续性做成用户可见能力`
2. `把记忆做成可查看、可理解、可管理的正式产品能力`
3. `把知识沉淀做成长期资产，而不是文本堆积`
4. `把执行能力做成可信闭环`
5. `把项目推进体验做成日常工作台`
6. `把真实留证和验收标准继续绑定到主链路`

一句话：

> 本阶段要让系统从“已经能用”推进到“用户愿意长期把项目、知识和工作习惯放进来”。

---

## 5. 本阶段硬约束

1. 不推翻当前 Go + Rust + React 主架构。
2. 不为了做忠实用户体验而重写整套前端或整套运行时。
3. 不新增双智能体正式主链路。
4. 不新增语音主链路、浏览器全局观察主链路、桌面自动化主链路。
5. 不把任务扩写成重型平台化大重构。
6. 所有新增能力都必须能给出真实样本留证。
7. 所有新增结论都必须回填验收文档。
8. 任何实现如果破坏当前第二阶段已通过主链路，该轮实现直接视为失败。

---

## 6. 实现原则

### 6.1 先做忠诚度闭环，不扩花哨能力

1. 先让用户感到“它记得我、能继续做、值得托付”。
2. 再考虑更大的连接器、更多入口和更重的自动化。

### 6.2 先做可见性，再做复杂智能

1. 先让用户看见系统记住了什么、为什么这么做、当前进度是什么。
2. 再做更复杂的推理和自动规划。

### 6.3 先把已成立能力产品化，再扩新能力

1. 先把项目问答、记忆、知识、执行、留证串成统一体验。
2. 不要把主资源重新分散到大量新功能。

### 6.4 先留证，再宣称能力成立

1. 所有新增工作包都必须有真实样本。
2. 所有新增工作包都必须有构建结果。
3. 所有新增工作包都必须回填验收文档。

## 9. 工作包 A：连续性与记忆产品化

### 9.1 目标

1. 用户能看见系统记住了什么。
2. 用户能理解为什么记住。
3. 用户能继续推进上一次的项目或任务，不需要重新解释。

### 9.2 必做项

1. 明确区分长期记忆、项目规则、阶段上下文、近期会话摘要。
2. 补齐“记忆来源、理由、更新时间、作用范围”的前端展示。
3. 为记忆可见性增加最小管理能力，至少支持查看与删除。
4. 增强项目续推问题的状态理解，不只回答“是什么”，还要回答“进展到哪”。

### 9.3 文件落点

1. `crates/runtime-core/src/memory.rs`
2. `crates/runtime-core/src/memory_router.rs`
3. `crates/runtime-core/src/memory_recall.rs`
4. `crates/runtime-core/src/contracts.rs`
5. `gateway/internal/contracts/contracts.go`
6. `gateway/internal/session/bus.go`
7. `frontend/src/settings/`
8. `frontend/src/logs/`

### 9.4 逐步实施步骤

1. 先盘点现有记忆类型，明确哪些属于长期记忆、哪些属于阶段上下文、哪些只属于会话压缩摘要。
2. 为正式记忆结构补足 `source / reason / updated_at / scope` 的稳定展示字段，优先复用已有合同，不随意扩散新字段。
3. 在前端增加最小可用的记忆查看入口，只展示真实存在的记录，不做空概念占位。
4. 只在已有数据结构支持的前提下补“删除”能力，不新增重型管理后台。
5. 为“继续推进”“上次做到哪”“还差什么”这类输入增加更稳定的状态续推逻辑。
6. 留证时必须至少包含：记忆查看样本、记忆删除样本、项目续推样本。

### 9.5 完成标准

1. 用户可以明确看到当前已激活的关键记忆。
2. 记忆内容带有来源与理由。
3. 会话续推类问题能稳定理解当前项目状态。
4. 记忆查看与删除都存在真实样本留证。

### 9.6 不通过情形

1. 仍然只能从日志碎片里间接猜系统记住了什么。
2. 记忆展示只有内容，没有来源、理由或作用范围。
3. 项目续推仍频繁退化成泛泛回复。

## 10. 工作包 B：知识沉淀与治理增强

### 10.1 目标

1. 本地知识库更像长期资产，而不是堆积文本。
2. 高价值内容能进入知识层，低价值内容会被拦截。
3. 知识召回时能说明来源和可信度。

### 10.2 必做项

1. 明确区分文档知识、运行时沉淀知识、用户确认知识。
2. 为知识条目补来源类型、可信度口径和更新时间。
3. 增强低价值知识拦截规则。
4. 增强任务完成后的高价值知识自动沉淀。

### 10.3 文件落点

1. `crates/runtime-core/src/knowledge.rs`
2. `crates/runtime-core/src/knowledge_store.rs`
3. `crates/runtime-core/src/query_engine.rs`
4. `crates/runtime-core/src/execution.rs`
5. `frontend/src/logs/`
6. `frontend/src/events/`

### 10.4 逐步实施步骤

1. 先盘点现有知识记录的 `knowledge_type / source_type / verified / updated_at` 使用情况，统一口径。
2. 先补知识召回的“命中原因 + 来源路径”展示，再补可信度口径。
3. 再做高价值知识自动沉淀，只允许对已验证、可复用、跨任务有价值的结论入库。
4. 对重复、低价值、时效性差的内容补治理规则，并把“跳过原因”记录到事件或日志中。
5. 留证时必须至少包含：知识召回样本、高价值知识自动沉淀样本、低价值知识拦截样本。

### 10.5 完成标准

1. 知识条目能区分来源与可信度。
2. 知识召回不再只是片段摘要，而是能说明命中依据。
3. 低价值知识持续被拦截，且有留痕。

### 10.6 不通过情形

1. 仍然分不清哪条知识来自文档、哪条来自运行时沉淀。
2. 知识召回只能看到一段文本，看不到命中原因。
3. 低价值知识继续持续污染知识层。

## 11. 工作包 C：可信执行与验证收口

### 11.1 目标

1. 用户能放心让系统执行工作区任务。
2. 用户能理解执行为何成功、为何失败。
3. 用户能在失败后继续推进，而不是流程断掉。

### 11.2 必做项

1. 增强执行前意图、执行后验证、失败后恢复的展示一致性。
2. 为关键任务增加“下一步建议”或“继续推进建议”。
3. 强化风险确认与恢复路径的解释性。

### 11.3 文件落点

1. `crates/runtime-core/src/execution.rs`
2. `crates/runtime-core/src/query_engine.rs`
3. `crates/runtime-core/src/contracts.rs`
4. `gateway/internal/contracts/contracts.go`
5. `frontend/src/events/`
6. `frontend/src/logs/`
7. `frontend/src/chat/`

### 11.4 逐步实施步骤

1. 先逐条核对命令执行、文件写入、删除、项目问答这些主路径的 `reasoning_summary / verification_summary / cache_reason` 是否口径统一。
2. 再对常见失败态补“继续建议”，优先覆盖真实高频失败，不做空泛提示。
3. 再校对高危动作确认卡和日志详情，确保用户在聊天页和日志页看到的是同一套解释。
4. 最后为执行成功后的高价值结论增加自动留证与沉淀。
5. 留证时必须至少包含：执行成功样本、执行失败样本、风险确认样本、恢复路径样本。

### 11.5 完成标准

1. 用户能从日志和事件里清楚看见“为何这样执行”。
2. 用户能在失败后直接知道下一步怎么做。
3. 高危动作、恢复动作、缓存动作都能明确区分。

### 11.6 不通过情形

1. 执行成功了但用户仍不知道系统为什么这么做。
2. 失败后只能看到错误，不能继续推进。
3. 风险确认、恢复留痕和普通执行混在一起无法理解。

## 12. 工作包 D：项目推进与状态理解增强

### 12.1 目标

1. 项目状态类回答更像项目经理，而不是文档摘要器。
2. 系统能稳定说明“做到什么程度、为什么这样判断、下一步做什么”。
3. 项目状态类回答要直接和真实留证绑定。

### 12.2 必做项

1. 保持“已完成能力 / 当前阶段 / 待收口项”三段式输出。
2. 让状态类回答能展开到对应样本、验证路径和完成标准。
3. 让“继续下一步”能优先给出当前阶段最合理动作。

### 12.3 文件落点

1. `crates/runtime-core/src/execution.rs`
2. `crates/runtime-core/src/prompt.rs`
3. `crates/runtime-core/src/query_engine.rs`
4. `frontend/src/logs/`
5. `frontend/src/events/`
6. `tmp/loyal-user-acceptance/`

### 12.4 逐步实施步骤

1. 先稳住 `project_answer` 对状态类问题的命中文档和缓存稳定性，不允许退回到泛化项目说明。
2. 再把阶段二核心文档与验收文档中的事实结构化成可复用状态摘要。
3. 再为“继续下一步”“还差什么”“现在最该做什么”补独立问答样本。
4. 最后再让回答展开到对应样本、验证路径和完成标准。
5. 项目状态类样本必须单独留证，不允许混在普通项目说明样本里。

### 12.5 完成标准

1. 项目状态类回答稳定命中核心文档和真实样本。
2. 回答里能明确指出已完成能力、当前阶段和待收口问题。
3. 回答里能进一步指向样本或完成标准。

### 12.6 不通过情形

1. 回答退回到“这个项目在持续开发中”这类泛化表述。
2. 回答虽然提到能力，但无法说明为什么这样判断。
3. 回答仍然与真实留证脱节。

## 13. 工作包 E：日常使用闭环与前端工作台收口

### 13.1 目标

1. 用户打开系统就能看懂“现在该做什么”。
2. 对话、日志、记忆、知识、进度形成统一体验。
3. 产品看起来更像“长期工作台”，而不是“技术功能集合”。

### 13.2 必做项

1. 首页突出当前任务、当前阶段、下一步建议。
2. 日志页突出复盘与证据，而不是只做事件列表。
3. 设置页或工作台侧栏增加对记忆、知识、工作区状态的最小可见入口。
4. 为空态、失败态、无任务态补更直接的引导。

### 13.3 文件落点

1. `frontend/src/App.tsx`
2. `frontend/src/chat/`
3. `frontend/src/logs/`
4. `frontend/src/events/`
5. `frontend/src/settings/`
6. `frontend/src/workspace/`
7. `frontend/src/shared/contracts.ts`

### 13.4 逐步实施步骤

1. 先收口首页，突出“当前项目 / 当前任务 / 下一步 / 是否需要确认”。
2. 再调整日志页详情区，把验证、缓存、记忆、知识、失败原因放在更高优先级。
3. 再补“继续上次任务”“查看当前长期规则”“查看最近沉淀”这类自然入口。
4. 最后做空态、失败态、无任务态引导，并对已有主链路做回归。
5. 留证时必须至少包含：首页状态样本、日志复盘样本、继续上次任务样本。

### 13.5 完成标准

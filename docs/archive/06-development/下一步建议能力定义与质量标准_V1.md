# 下一步建议能力定义与质量标准 V1

## 1. 文档目的与作用

本文用于在《知识沉淀型个人助手产品定义_V1》《真实用户场景与任务脚本_V1》《第一版场景优先级与范围_V1》《知识资产模型_V1》与《状态维护模型_V1》的基础上，进一步明确第一版“下一步建议”能力到底应如何定义、建议应基于哪些依据生成、什么样的建议才算真正可用，以及哪些能力当前不作为建议系统重点建设对象。

本文关注的不是提示词实现细节，也不是复杂推荐系统设计，而是产品层面的建议能力定义与质量标准问题，重点回答以下问题：

1. 第一版中的“下一步建议”具体指什么
2. 建议生成应优先依据哪些状态与知识资产
3. 什么样的建议才算有价值、可执行、贴合当前阶段
4. 哪些建议属于泛泛建议、失焦建议或不合格建议
5. 第一版应优先达到什么质量标准，暂不深做哪些能力

本文作为“状态理解 -> 下一步建议”主线的核心定义文档，直接服务于以下后续工作：

1. 验收标准与评估用例设计
2. 有限执行与权限边界收口
3. 输入渠道与结果留痕设计
4. 第一版开发任务拆解

本文不展开以下内容：

1. 模型提示词实现细节
2. 复杂推荐算法、复杂排序机制或重型个性化系统
3. 多智能体协同建议流程
4. 高风险自动化决策链路
5. 具体工程层的检索、缓存与召回优化细节

本文的作用，是把“下一步建议”从抽象价值进一步收口为“第一版到底应该给出什么样的建议、建议依据是什么、什么样算达标”的明确标准基础。

---

## 2. 下一步建议能力的总体原则

第一版“下一步建议”能力应围绕“知识沉淀、状态理解、下一步建议、有限执行”这条产品主线展开，不追求一次性给出最全面、最复杂的规划，而优先给出最贴近当前阶段、最能帮助推进、最适合在高频场景中稳定成立的建议。

第一版建议能力应遵循以下原则：

1. `基于当前状态优先`
   建议必须首先建立在当前场景、当前主题、当前目标、当前阶段、当前进度、当前阻塞、最近结果与当前掌握情况之上，而不是脱离当前状态单独生成。

2. `基于知识资产补强`
   建议应优先参考已有知识资产中的主题知识、阶段结论、方法经验、问题与解法记录与可继续接上的上下文记录，用于增强建议的连续性与针对性。

3. `当前最值得做优先`
   建议不是列出所有可能动作，而是优先指出当前阶段最值得先做、最可能带来推进的下一步。

4. `可执行优先`
   建议应尽量具体、可理解、可落地，避免空泛方向、口号式提醒或无法直接转化为行动的抽象表述。

5. `与场景匹配优先`
   建议形式应随场景变化而变化。工程开发更偏任务推进建议，通用学习更偏理解巩固与学习顺序建议，浏览学习更偏当前内容理解与回顾建议，CTF / 安全学习更偏分析路径与验证顺序建议。

6. `轻量有效优先`
   第一版不追求复杂长链规划、复杂个性化画像驱动的推荐或自动化决策系统，而优先保证建议在高频场景中的实用性、稳定性与连续性。

基于上述原则，第一版不以“建议得更多”为目标，而以“建议得更贴合当前阶段、更能帮助推进、更能形成持续使用价值”为目标。

## 3. 第一优先级建议能力定义

结合第一版产品主线、场景优先级与状态维护模型，第一版建议能力应优先做到以下四类输出。

### 3.1 进度判断型建议

`进度判断型建议` 指围绕“现在做到哪了”给出的阶段判断与收口提示。

核心价值：

1. 帮助用户确认当前是否真的在推进
2. 帮助用户识别当前已经完成的部分与仍待推进的部分
3. 帮助用户避免因上下文分散而失去阶段感

适用重点：

1. 工程开发场景
2. 通用学习场景
3. 浏览学习与 CTF / 安全学习场景中的阶段回看

### 3.2 缺口识别型建议

`缺口识别型建议` 指围绕“还差什么”给出的阻塞判断、缺口识别与待补信息提示。

核心价值：

1. 帮助用户明确当前最影响推进的缺口是什么
2. 帮助系统避免给出脱离阻塞实际情况的建议
3. 帮助后续行动建议建立在更真实的问题识别之上

适用重点：

1. 工程开发中的阻塞识别
2. 通用学习中的理解缺口识别
3. CTF / 安全学习中的分析路径缺口识别

### 3.3 下一步行动型建议

`下一步行动型建议` 指围绕“下一步该做什么”给出的明确行动建议、优先级建议与顺序建议。

核心价值：

1. 直接帮助用户推进当前任务或学习过程
2. 将状态理解与知识沉淀转化为实际推进价值
3. 为有限执行能力提供可桥接的行动入口

适用重点：

1. 工程开发中的下一步任务推进
2. 通用学习中的下一步学习顺序
3. 浏览学习中的下一步回顾或继续阅读方向
4. CTF / 安全学习中的下一步分析或验证方向

### 3.4 收口与续接型建议

`收口与续接型建议` 指围绕“这一轮结束后该怎么接下去”给出的收口建议、交接建议与后续入口建议。

核心价值：

1. 帮助用户在一轮结束后保留可继续推进的入口
2. 帮助系统形成连续多轮使用价值
3. 帮助知识资产与状态维护在后续轮次中继续发挥作用

适用重点：

1. 工程开发场景中的阶段交接
2. 通用学习场景中的阶段回顾与下一轮继续入口
3. 浏览学习与 CTF / 安全学习场景中的后续回看与继续分析入口

### 3.5 第一优先级建议能力的共同特点

上述四类建议被列为第一优先级，主要因为它们共同满足以下条件：

1. 能直接回答“做到哪了、还差什么、下一步做什么”这组核心问题
2. 能直接服务工程开发与通用学习两个第一优先级场景
3. 能直接连接状态维护与知识资产
4. 适合第一版轻量但有效地稳定落地

## 4. 建议生成的核心依据

第一版建议生成不应依赖模糊印象或单轮输入猜测，而应优先建立在以下依据之上。

### 4.1 状态依据

建议生成应优先使用以下状态作为直接依据：

1. `当前场景`
2. `当前主题`
3. `当前目标`
4. `当前阶段`
5. `当前进度`
6. `当前阻塞`
7. `最近结果`
8. `当前掌握情况`

这些状态共同决定：

1. 当前建议应该偏解释、推进、收口还是回顾
2. 当前建议应该优先解决什么问题
3. 当前建议应该落在哪个粒度上

### 4.2 知识资产依据

建议生成应优先参考以下知识资产：

1. `主题知识`
2. `阶段结论`
3. `方法经验`
4. `问题与解法记录`
5. `可继续接上的上下文记录`

这些资产共同决定：

1. 当前建议是否能利用用户已有沉淀，而不是重新从零开始
2. 当前建议是否能参考用户过往类似经验
3. 当前建议是否能保持跨轮次的一致性与连续性

### 4.3 当前输入依据

除状态与资产外，建议还应参考当前轮次的即时输入，包括：

1. 本轮问题描述
2. 新补充材料
3. 当前结果变化
4. 用户显式表达的困惑、限制与目标修正

这些内容的作用是：

1. 让建议保持对当前变化敏感
2. 防止建议只依赖历史沉淀而忽略当前变化

### 4.4 建议依据的优先顺序

当状态、知识资产与当前输入同时存在时，建议依据的优先顺序应为：

1. 当前状态
2. 当前轮次新输入
3. 已有知识资产

原因在于：

1. 状态决定当前最核心的问题是什么
2. 当前输入决定本轮是否出现了新变化
3. 知识资产用于增强连续性、稳定性与复用价值

## 5. 建议质量标准

第一版建议质量的判断，不应建立在“看起来聪明”或“表达流畅”上，而应建立在是否真正帮助用户推进当前阶段上。

第一版建议质量标准建议分为以下四层。

### 5.1 不合格建议

以下建议应视为不合格：

1. 与当前场景、当前主题或当前目标明显不相关
2. 只给空泛方向，不给可理解的推进路径
3. 忽略当前阻塞、最近结果或当前掌握情况
4. 仅重复用户输入，而没有形成新的推进价值
5. 给出过度宏大、脱离第一版边界或高风险的行动建议

### 5.2 基本合格建议

以下建议可视为基本合格：

1. 能围绕当前场景与当前目标作答
2. 能指出当前最值得关注的一步
3. 能避免明显脱离当前状态
4. 用户看完后知道“下一步大概做什么”

### 5.3 高质量建议

以下建议可视为高质量：

1. 能准确反映当前阶段与当前阻塞
2. 能指出当前最值得优先处理的一步及其原因
3. 能结合已有知识资产或最近结果给出连续性建议
4. 用户看完后不仅知道做什么，还知道为什么先做这一步
5. 建议粒度适中，既不过于空泛，也不过于碎片化

### 5.4 第一版目标质量

第一版不要求建议系统在所有场景中都达到高度个性化或复杂规划能力，但至少应达到以下标准：

1. 在工程开发与通用学习场景中稳定达到“高质量建议”
2. 在浏览学习与 CTF / 安全学习场景中至少稳定达到“基本合格建议”
3. 对明显脱离当前状态的建议能够减少到可接受范围

## 6. 泛泛建议与高价值建议的区分

为避免建议能力退化为通用问答，当前需要明确区分“泛泛建议”和“高价值建议”。

### 6.1 泛泛建议的典型特征

以下特征通常意味着建议过于泛：

1. 不结合当前阶段，只给通用方向
2. 不结合当前阻塞，只给宽泛目标
3. 不结合最近结果，只重复常识
4. 不结合知识资产，只给模板化回答
5. 一条建议几乎适用于任何用户、任何场景、任何阶段

### 6.2 高价值建议的典型特征

以下特征通常意味着建议具备较高价值：

1. 紧贴当前场景与当前目标
2. 明确回应当前阻塞与阶段缺口
3. 结合最近结果说明为什么此时应先做这一步
4. 能引用已有知识资产或历史结论，体现连续性
5. 让用户明确知道下一步的优先级与原因

### 6.3 当前判断原则

当前判断建议是否有价值，可优先看以下三个问题：

1. 这条建议是否只适用于“现在这个人、这个阶段、这个场景”
2. 这条建议是否真正回应了“当前还差什么”
3. 用户看完后是否能立刻知道自己下一步最应该先做什么

如果三者都不能成立，则该建议大概率仍然是泛泛建议。

## 7. 第二优先级建议能力

除第一优先级建议能力外，以下能力也具有明确价值，但在第一版中更适合作为保留方向或轻量兼容对象，而不作为重点建设对象。

### 7.1 长周期规划型建议

`长周期规划型建议` 指对多阶段任务、多周学习路径或长期成长目标给出的较长周期建议。

保留原因：

1. 它有助于提升长期连续使用价值
2. 它能够增强学习与成长场景中的路线感

当前不作为第一优先级的原因：

1. 第一版更应先保证“当前下一步”建议稳定成立
2. 长周期规划容易脱离当前状态、走向复杂规划系统

### 7.2 个性化风格适配型建议

`个性化风格适配型建议` 指根据用户长期偏好、节奏或习惯调整表达方式与建议风格的能力。

保留原因：

1. 它有助于提升建议体验
2. 它能够增强产品的个体贴合度

当前不作为第一优先级的原因：

1. 第一版更应先保证建议内容本身有用
2. 过早强化风格适配容易滑向复杂偏好建模

### 7.3 薄弱点强化型建议

`薄弱点强化型建议` 指基于用户反复暴露出的薄弱点、误判点给出的针对性强化建议。

保留原因：

1. 它对通用学习与 CTF / 安全学习场景很有价值
2. 它能增强长期成长支持能力

当前不作为第一优先级的原因：

1. 第一版应先保证基础状态与基础建议稳定成立
2. 过早引入薄弱点强化容易依赖复杂识别与复杂画像机制

### 7.4 第二优先级建议能力的定位

上述建议能力在第一版中的定位是：

1. 可保留轻量增强空间
2. 可在局部场景中进行有限验证
3. 不作为第一版建议质量判断的核心对象
4. 不在当前阶段引入复杂规划、复杂推荐或复杂个性化机制

## 8. 不作为第一版重点建设的建议能力

以下建议能力虽然在长期演进中可能具备价值，但当前不作为第一版重点建设对象。

### 8.1 重型个性化推荐能力

`重型个性化推荐能力` 指依赖复杂长期画像、复杂偏好推断与多维行为建模的深度推荐系统。

当前不重点建设的原因：

1. 第一版已明确不优先引入重型画像与复杂推断机制
2. 这类能力对数据积累周期与系统复杂度要求较高
3. 过早建设容易偏离主线收口

### 8.2 自动决策型建议能力

`自动决策型建议能力` 指系统在缺乏充分显式确认的情况下，直接替用户做出高影响决策或自动延伸为执行链路的能力。

当前不重点建设的原因：

1. 第一版建议应以帮助用户判断为主，而不是替用户做高影响决策
2. 这类能力容易与高风险自动化边界冲突
3. 它会放大误判风险与边界风险

### 8.3 多智能体协同建议能力

`多智能体协同建议能力` 指多个角色或多个 agent 共同生成、博弈或融合建议的复杂建议系统。

当前不重点建设的原因：

1. 第一版已明确不围绕多智能体编排展开
2. 这类能力会显著增加建议链路复杂度
3. 当前更需要先让单主线建议能力稳定成立

### 8.4 高风险场景自动化建议链路

`高风险场景自动化建议链路` 指在高权限操作、安全执行、复杂网页控制等场景中，直接导向高风险自动化动作的建议链条。

当前不重点建设的原因：

1. 第一版应优先守住有限执行与高风险边界
2. 这类建议一旦与自动化桥接，风险会显著放大
3. 当前主线更需要低风险、高价值、可解释的建议能力

### 8.5 当前排除原则

上述建议能力当前不作为重点建设对象，核心原因在于：

1. 对第一版主线的直接支撑不足
2. 容易引入高风险、高复杂度或高歧义
3. 容易分散当前建议能力主线的收口资源

这些建议能力并非永久排除，而是在第一版阶段不作为主要建模与实现投入方向。

## 9. 第一版建议能力优先级建议

结合产品主线、场景优先级、知识资产模型与状态维护模型，第一版建议能力优先级建议分为三层。

### 9.1 第一层：重点建设的建议能力

第一版应优先重点建设以下建议能力：

1. `进度判断型建议`
2. `缺口识别型建议`
3. `下一步行动型建议`
4. `收口与续接型建议`

这类建议优先建设的原因在于：

1. 它们直接支撑“做到哪了、还差什么、下一步做什么”这组核心问题
2. 它们直接服务工程开发与通用学习两个第一优先级场景
3. 它们能直接连接状态维护与知识资产
4. 它们适合第一版轻量但有效地稳定落地

### 9.2 第二层：轻量保留的建议能力

第一版可轻量保留，但不作为重点建设对象的建议能力包括：

1. `长周期规划型建议`
2. `个性化风格适配型建议`
3. `薄弱点强化型建议`

这类建议能力当前的定位是：

1. 可以保留局部增强空间
2. 可以为后续长期体验提供方向储备
3. 当前不投入过多建模资源，不引入复杂规划与复杂推荐

### 9.3 第三层：当前不重点建设的建议能力

第一版当前不重点建设的建议能力包括：

1. `重型个性化推荐能力`
2. `自动决策型建议能力`
3. `多智能体协同建议能力`
4. `高风险场景自动化建议链路`

这类建议能力当前不重点建设的原因在于：

1. 与第一版主线的直接关系不够强
2. 容易显著增加复杂度与风险
3. 容易分散当前建议主线的收口能力

### 9.4 当前建议优先级的核心判断

第一版建议能力建设的核心目标，不是尽可能给出更多建议，而是优先给出：

1. 最贴近当前阶段的建议
2. 最能帮助用户推进的建议
3. 最能在高频场景中稳定成立的建议

因此，第一版建议能力优先追求“建议得更有用”，而不是“建议得更全面”。

## 10. 文档结论与后续衔接

基于《知识沉淀型个人助手产品定义_V1》《真实用户场景与任务脚本_V1》《第一版场景优先级与范围_V1》《知识资产模型_V1》与《状态维护模型_V1》的上游结论，第一版下一步建议能力当前正式收口如下：

1. 第一版优先重点建设 `进度判断型建议`、`缺口识别型建议`、`下一步行动型建议` 与 `收口与续接型建议`
2. 第一版建议必须优先建立在当前状态、当前输入与高价值知识资产之上，而不是脱离当前阶段单独生成
3. 第一版对 `长周期规划型建议`、`个性化风格适配型建议` 与 `薄弱点强化型建议` 采取轻量保留策略
4. 第一版当前不重点建设重型个性化推荐、自动决策型建议、多智能体协同建议与高风险自动化建议链路
5. 第一版建议质量的核心标准，是建议是否真正贴近当前阶段、回应当前阻塞、指出当前最值得先做的一步

基于上述结论，后续设计与开发应优先围绕以下方向展开：

1. 为第一优先级建议能力建立清晰、稳定、可解释的输出标准
2. 在状态维护模型与知识资产模型中明确建议依据的使用边界
3. 在有限执行设计中明确建议与执行之间的桥接边界
4. 在验收标准中重点验证建议是否真正形成推进价值，而不是停留在泛泛表达

本文将直接作为以下后续文档的上游输入：

1. 验收标准与评估用例
2. 有限执行与权限边界
3. 第一版开发任务拆解

本文的作用，是将“下一步建议”进一步收口为“第一版到底应如何给建议、建议质量如何判断、哪些先不重做”的明确能力标准，为后续实现与验收提供稳定基础。

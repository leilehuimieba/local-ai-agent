# 第一版场景优先级与范围 V1

## 1. 文档目的与作用

本文用于在《知识沉淀型个人助手产品定义_V1》与《真实用户场景与任务脚本_V1》的基础上，进一步明确第一版优先支持哪些场景、每类场景收在什么范围，以及哪些内容暂不进入第一版深做范围。

本文的重点不是重复描述场景本身，而是回答以下范围决策问题：

1. 第一版优先深做哪些场景
2. 第一版中每类场景应做到什么深度
3. 哪些场景可作为同步验证或后续增强方向保留
4. 哪些内容当前不纳入第一版重点范围
5. 后续能力设计、开发拆解与验收文档应围绕哪些场景展开

本文作为范围决策文档，直接服务于以下后续工作：

1. 第一版能力拆解与开发顺序安排
2. 知识资产模型与状态维护模型设计
3. 下一步建议能力定义与质量标准制定
4. Skill 使用边界与接入规则收口
5. 验收标准与评估用例设计

本文不展开以下内容：

1. 真实场景脚本的详细过程描述
2. 具体工程实现方案
3. 各场景下的底层技术接入细节
4. 多智能体协作设计
5. 超出第一版范围的能力扩张讨论

本文的作用，是在产品主线与真实场景之间进一步完成一次“版本收口”，把“应该支持哪些场景”转化为“第一版优先做哪些、做到什么程度”的明确决策。

---

## 2. 场景优先级结论

结合《真实用户场景与任务脚本_V1》中对四类核心场景的分析，第一版场景优先级结论正式收口如下：

### 2.1 第一优先级场景

第一版优先深做以下两类场景：

1. `工程开发场景`
2. `通用学习场景`

### 2.2 第二优先级场景

第一版可同步保留并进行验证，但不作为最先深做重点的场景包括：

1. `浏览学习场景`
2. `CTF / 安全学习场景`

### 2.3 优先级判断依据

上述排序主要基于以下判断：

1. `主线验证强度`
   工程开发场景与通用学习场景最能共同验证“知识沉淀、状态理解、下一步建议、有限执行”这条产品主线是否真正成立。

2. `真实使用频率`
   工程开发场景是当前最贴近真实高频使用的场景之一，能够较快形成反馈闭环；通用学习场景则覆盖用户长期成长中的持续需求。

3. `产品本体表达`
   通用学习场景最能体现产品作为“知识沉淀型个人助手”的核心价值，有助于避免产品重新滑回纯研发副驾。

4. `复杂度控制`
   浏览学习场景与 CTF / 安全学习场景具有较强价值，但在当前阶段同时深做会显著提高整体复杂度，并削弱第一版的主线收口能力。

第一版的目标不是平均推进所有场景，而是先让最能代表产品主线的场景稳定成立，再逐步扩展到更强差异化的场景。

---

## 3. 第一优先级场景范围

第一版优先围绕“工程开发场景”和“通用学习场景”做深，其目标不是覆盖这两类场景中的所有可能能力，而是先让产品主线在这两类高价值场景中稳定成立。

### 3.1 工程开发场景的第一版范围

第一版在工程开发场景中应重点做到：

1. 能接住用户当前任务、已有背景、阶段目标与中途补充信息
2. 能持续维护当前任务状态，包括目标、进度、阻塞、阶段结论与待确认问题
3. 能沉淀开发过程中的关键判断、经验结论、未完成事项与后续入口
4. 能稳定回答“现在做到哪了、还差什么、下一步做什么”
5. 能在明确授权下完成少量与当前任务直接相关的辅助动作

第一版在工程开发场景中达到以下程度即可视为成立：

1. 用户能够连续多轮推进同一任务，而不是每轮都重新提供完整上下文
2. 一轮任务结束后，系统能够留下可回看的阶段结论与交接信息
3. 用户能够明显感受到助手在任务推进与阶段收口上的连续支持价值

### 3.2 通用学习场景的第一版范围

第一版在通用学习场景中应重点做到：

1. 能接住用户当前学习主题、已有背景、材料输入与阶段困惑
2. 能持续维护当前学习状态，包括已掌握内容、薄弱点、阶段位置与待补内容
3. 能沉淀学习过程中的主题知识、阶段理解、疑问记录与学习轨迹
4. 能稳定回答“学到哪了、哪些还不稳、下一步该学什么”
5. 能在合适的时候提供总结、解释、回顾建议与适度巩固支持

第一版在通用学习场景中达到以下程度即可视为成立：

1. 用户能够连续记录并回看同一学习主题的阶段进展
2. 系统能够区分已理解、待巩固与未理解内容
3. 用户能够明显感受到助手在学习状态维护与下一步学习建议上的连续支持价值

### 3.3 第一优先级场景的共性收口

无论是工程开发场景还是通用学习场景，第一版都应共同满足以下要求：

1. 不依赖高权限自动化或复杂场景编排
2. 重点验证知识沉淀、状态理解、下一步建议与有限执行主线
3. 能形成连续多轮使用，而不是停留在单次问答
4. 能在一轮结束后保留后续可继续接上的上下文与结果

第一优先级场景的目标，是先让产品主线在最具代表性的高频场景中跑通，而不是一次性覆盖工程开发或学习场景中的全部能力边界。

---

## 4. 第二优先级场景范围

第一版对“浏览学习场景”与“CTF / 安全学习场景”的处理原则是：保留、验证、增强，但暂不作为最先深做重点。

### 4.1 浏览学习场景的第一版范围

第一版在浏览学习场景中可先做到：

1. 能接住当前页面内容、用户即时提问与阶段性浏览总结
2. 能帮助用户理解当前页面重点、关键概念与当前主题的关系
3. 能记录浏览过程中的重点内容、疑问与待回顾内容
4. 能为后续学习回顾与知识沉淀提供基础记录
5. 能在当前页面上下文下给出简要的回顾建议或下一步浏览建议

第一版在浏览学习场景中的重点是验证：

1. 网页内容环境下的输入接入是否能够稳定承接
2. 浏览过程中的知识沉淀与学习轨迹记录是否成立
3. 当前页面上下文中的解释与建议能力是否有实际价值

第一版在浏览学习场景中暂不深做：

1. 高复杂度浏览器自动化控制
2. 复杂页面操作链路自动执行
3. 无边界的行为监控与高风险采集
4. 为规避网站策略而进行复杂自动化机制设计

### 4.2 CTF / 安全学习场景的第一版范围

第一版在 CTF / 安全学习场景中可先做到：

1. 能接住题目背景、初步线索、阶段结果与用户当前判断
2. 能帮助整理分析路径、关键假设、已尝试步骤与当前卡点
3. 能结合场景 Skill 给出下一步分析方向建议
4. 能沉淀题型经验、方法记录、失败路径与薄弱点信息
5. 能在一轮结束后留下可回看的过程记录与后续学习入口

第一版在 CTF / 安全学习场景中的重点是验证：

1. 高专业度领域场景下的状态维护与建议能力是否成立
2. 场景 Skill 对分析理解与建议质量是否能形成有效增强
3. 方法沉淀、题型经验与薄弱点记录是否具备长期价值

第一版在 CTF / 安全学习场景中暂不深做：

1. 无边界自动攻击或自动利用链路
2. 脱离授权的目标操作与高权限控制行为
3. 复杂多智能体协作式安全分析流程
4. 为追求自动化效果而扩张为安全执行平台

### 4.3 第二优先级场景的定位

浏览学习场景与 CTF / 安全学习场景在第一版中的定位是：

1. 用于验证主线能力在差异化场景中的可迁移性
2. 用于观察场景增强能力是否真正带来价值
3. 用于为后续版本积累更细化的输入样本、知识沉淀样本与建议样本
4. 不用于在第一版阶段拉高整体复杂度或重写主线优先级

## 5. 当前明确不纳入第一版深做范围的内容

在当前第一版范围内，以下内容明确不纳入深做范围：

1. `多智能体编排与多角色协作流程`
   第一版不以多个智能体之间的分工、调度与协作为重点，而以单一主线助手在不同场景中的稳定成立为重点。

2. `浏览器高复杂度自动化控制`
   第一版不深做复杂网页操作链路、自动点击跳转、自动完成任务等能力，而优先验证浏览学习中的内容理解、知识沉淀与建议价值。

3. `高权限自动化与远程控制能力`
   第一版不深做全电脑高权限操作、手机远程控制、即时通讯深接入等能力，以避免过早引入高风险与高复杂度。

4. `重型记忆评分与复杂能力画像体系`
   第一版不深做复杂评分、精细化能力评估、重型学习画像与复杂推荐机制，而优先验证基本知识沉淀、状态维护与建议能力。

5. `平台化扩张与能力拼装型建设`
   第一版不以接入更多能力、拼装更多模块或扩展更多场景为目标，而以让核心主线在优先场景中跑通为目标。

6. `超出场景主线的功能堆叠`
   无论是工程开发、通用学习、浏览学习还是 CTF / 安全学习场景，第一版都不因单一场景需求而扩展出与知识沉淀主线无关的大量功能。

7. `高风险安全自动化执行链路`
   在 CTF / 安全学习场景中，第一版不深做自动攻击、自动利用、脱离授权的目标操作或高权限执行系统。

上述内容并非永久否定，而是当前第一版明确不作为重点投入方向。

第一版的资源应优先用于：

1. 让工程开发与通用学习场景稳定成立
2. 让浏览学习与 CTF / 安全学习场景完成基础验证
3. 让知识沉淀、状态理解、下一步建议与有限执行主线形成闭环

## 6. 范围判断原则

为避免第一版范围在推进过程中不断扩张，后续新增场景、能力或需求时，应统一按以下原则判断是否纳入第一版。

### 6.1 可优先纳入第一版的条件

新增内容只有在同时满足以下条件时，才应优先纳入第一版：

1. `能够直接强化产品主线`
   即能够明显增强知识沉淀、状态理解、下一步建议或有限执行中的至少一项核心能力。

2. `能够自然接入已有高优先级场景`
   即能够直接服务工程开发场景或通用学习场景，而不是另起一条独立主线。

3. `能够形成真实持续使用价值`
   即不仅在单次体验中看起来有用，而且能在连续使用中提升任务推进、学习理解或知识积累效果。

4. `不会显著拉高当前复杂度`
   即不会在第一版阶段引入高风险自动化、复杂编排、大规模平台化扩张或明显超出当前资源承载能力的设计。

### 6.2 应暂不纳入第一版的信号

当新增内容出现以下任一信号时，当前应暂不纳入第一版：

1. `主要价值来自能力炫技，而不是主线增强`
2. `需要为其新开一整套独立流程、独立系统或独立场景主线`
3. `对高权限自动化、复杂协作或大范围接入有明显依赖`
4. `短期内缺乏真实高频使用场景支撑`
5. `会显著分散工程开发与通用学习两类核心场景的收口能力`

### 6.3 出现争议时的判断顺序

当某项新增内容同时具备价值与复杂度，是否纳入第一版应按以下顺序判断：

1. 它是否直接强化第一版核心主线
2. 它是否优先服务第一优先级场景
3. 没有它时，第一版是否仍然可以成立
4. 加入它后，是否会显著推迟主线收口
5. 它更适合作为第一版内容，还是更适合作为第二阶段增强方向

如果判断后发现该内容虽然有价值，但不影响第一版主线是否成立，则应优先放入后续阶段，而不是进入当前版本范围。

### 6.4 当前版本判断的核心原则

第一版的判断核心不是“能不能多做一些”，而是“是否能够先让最关键的主线在最具代表性的场景中稳定成立”。

因此，第一版优先收口主线，而不是优先扩张边界。

## 7. 文档结论与后续衔接

基于《知识沉淀型个人助手产品定义_V1》与《真实用户场景与任务脚本_V1》的上游结论，第一版场景优先级与范围当前正式收口如下：

1. 第一版优先深做 `工程开发场景` 与 `通用学习场景`
2. 第一版保留 `浏览学习场景` 与 `CTF / 安全学习场景` 的基础验证与后续增强空间，但当前不作为最先深做重点
3. 第一版不围绕多智能体编排、高权限自动化、平台化扩张或高风险执行链路展开
4. 第一版的核心目标是让“知识沉淀、状态理解、下一步建议、有限执行”这条主线，先在最具代表性的高频场景中稳定成立

基于上述结论，后续能力设计与开发拆解应优先围绕以下方向展开：

1. 工程开发场景中的任务推进、阶段收口与经验沉淀
2. 通用学习场景中的学习状态维护、知识积累与下一步学习建议
3. 浏览学习与 CTF / 安全学习场景中的基础样本验证与增强层设计
4. 所有场景共享的知识沉淀、状态维护、建议生成与受控执行主线能力

本文将直接作为以下后续文档的范围依据：

1. 知识资产模型
2. 状态维护模型
3. 下一步建议能力定义与质量标准
4. Skill 使用边界与接入规则
5. 有限执行与权限边界
6. 第一版开发任务拆解
7. 验收标准与评估用例

本文的作用，是将“支持哪些场景”进一步收口为“第一版优先做哪些、做到什么程度、哪些先不深做”的明确范围判断，为后续实现与验收提供稳定边界。

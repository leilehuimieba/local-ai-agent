# 前端验收标准

## A. 视觉一致性

1. 工作台、对话页、右侧上下文栏、调查层必须使用同一套视觉 token。
2. 指标信息统一使用 `metric-chip` 风格，禁止同类信息出现多套卡片样式。
3. 区块标题统一使用 `section-head` 结构，标题、kicker、状态徽标层级一致。
4. 核心卡片统一使用 `surface-card` 语义，不允许出现明显脱节的背景、边框和阴影体系。

## B. 主工作流闭环

1. 用户从工作台进入对话页后，可以清楚识别当前任务、当前状态、当前工作区和当前运行。
2. 提交任务后，对话主线程必须立即出现反馈，不能只有按钮禁用。
3. 在首个事件返回前，消息区必须显示明确的等待态，说明任务正在进入运行流。
4. 如果提交失败且还没有生成事件，消息区必须显示恢复建议，不能回落成普通空态。
5. 任务完成或失败后，主线程必须有明确结果消息或失败提示。

## C. 调查层闭环

1. 调查层展开后，必须能明确区分“自动跟随最新事件”和“手动查看历史事件”。
2. 用户未手动干预时，新事件到达后调查焦点自动跟随最新事件。
3. 用户选中旧事件或离开最新事件位置后，自动跟随必须停止，不能抢走阅读焦点。
4. 手动查看模式下，界面必须提供“回到最新事件”的显式入口。
5. 事件卡必须同时具备默认、悬停、选中、最新事件四种可感知状态。
6. 在首个事件返回前，调查层必须显示等待态，而不是空白时间线。
7. 提交失败且没有事件时，调查层必须显示失败态和恢复建议。

## D. 信息可读性

1. 长文本、长路径、运行 ID 必须能安全换行或截断，不能破坏布局。
2. 不同信息层级必须通过字号、字重、颜色和留白区分，而不是只靠颜色。
3. 风险、运行、失败、完成等状态必须在视觉上可快速区分。

## E. 响应式与稳定性

1. 桌面主工作流在 1180px 以上宽度下不允许出现明显布局挤压。
2. 窄屏时布局可以折叠，但信息层级不能失真，核心操作必须仍可用。
3. 构建必须通过 `npm run build`。
4. 交互增强不能破坏既有任务提交、事件流更新、确认流和日志入口。

## 当前阶段目标

当前优先收口顺序：

1. 消息流和调查层的状态设计闭环。
2. 移动端和窄屏布局收口。
3. 可访问性和键盘交互。
4. 真实页面验收与细节修正。

## 第七阶段持续演进规则

### 页面职责边界

1. 首页只负责恢复上下文、显示最近任务、暴露继续入口，不再承接任务推进、跨会话调查或运行配置。

3. 记录页只负责跨会话调查与复盘，不承接当前会话主线程交互。
4. 设置页只负责运行环境、权限、资源和诊断控制，不承接任务结果或记录详情。

新增前端能力只能优先评估以下落点：

1. 任务页主线程
2. 任务页检查器

4. 记录页详情区
5. 设置页模块区

## 第 8.5 步状态归属修正留证

### 收口规则

1. 资源区数据只保留一个可信状态源，统一由 `useMemories -> useSettings -> SettingsPanel / SettingsSections` 向下提供。
2. `MemoryResourcesSection.tsx` 只允许保留筛选、展开项、局部提示这类 UI 层本地状态，不允许再维护独立的 `items` 真相。
3. 资源区组件不允许再直接调用 `fetchMemories`、`deleteMemory` 或其他资源 API；刷新与删除必须回到外层传入的方法。
4. 这样收口的原因是资源列表、记忆数量和设置页统计已经彼此关联，如果组件自行维护第二份真相，后续很容易出现列表、数量、其他区域显示不同步。
5. 设置页动作错误与任务运行错误必须拆分：模型切换、模式切换、工作区切换等设置动作失败，只允许走设置页自己的 `pendingAction / actionError / lastSuccess` 反馈体系。
6. `runtime submitError` 只保留任务提交失败和确认流提交失败两类运行时错误，不再承接设置动作失败。
7. 这样拆分的原因是设置动作失败不代表任务提交失败，如果继续混写到 runtime，会污染任务页主线程、调查层和检查器的状态语义。
8. 顶部栏只做当前模型、当前模式的摘要展示，不再承担模型和模式修改。
9. 顶部栏如需引导用户调整设置，只允许提供“前往设置页调整”这类轻量入口，不重新承接真实修改动作。
10. 这样收口的原因是设置页已经具备完整反馈闭环，顶部栏继续保留修改入口会形成两套控制体验，并绕开设置页反馈体系。

### 最小回归补齐

1. 检查资源区刷新后，列表内容与设置页中的记忆数量展示同步更新。
2. 检查删除记忆后，资源区列表与 `long_term_memory_count` 等外层统计保持同步，不出现一处已删、一处未变。
3. 检查设置动作失败时，错误只出现在设置页反馈区域，不污染任务页主线程、检查器或调查层的运行错误提示。
4. 检查顶部栏仍能看到当前模型和当前模式，但不再可直接修改模型或模式。
5. 检查设置页仍是唯一真实设置修改入口，顶部栏只保留摘要展示和前往设置页提示。
6. 在 `frontend/` 目录执行 `npm run build`，构建必须通过。

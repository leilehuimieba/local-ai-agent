## 1. 文档目的

本文不是新的视觉稿，也不是前端代码实现说明。

本文只解决 4 个问题：

1. 当前前端为什么已经“能用但不好用”。
2. 从用户视角看，当前前端最需要被修正的体验模型是什么。
3. 在不推翻当前项目边界的前提下，前端应该演进成什么形态。
4. 后续前端优化、状态拆分、工作台重构和架构收口应按什么顺序推进。

一句话目标：

> 把当前前端从“页面驱动的单页应用”收口为“任务运行驱动的本地智能体工作台”。

---

## 2. 当前项目背景与约束

基于当前仓库状态，前端已经形成以下现实基础：

1. 采用 `React + TypeScript + Vite` 单页前端。
2. 已有首页工作台、日志页、设置页三类正式产品面。
3. 已有会话提交、事件流回放、确认流、日志查看、模型切换、模式切换、工作区切换等主链路能力。
4. 运行时与网关已稳定返回 `context_snapshot`、`verification_snapshot`、`risk_level`、`memory_digest`、`knowledge_digest` 等正式字段。
5. 当前前端已经通过首版产品化验收，但仍明显停留在“结构可用、体验未定型”的阶段。

同时，当前项目也有明确约束：

1. 当前阶段不宜推翻三页主结构。
2. 不宜为了体验优化直接重写整套数据流和接口体系。
3. 前端必须继续围绕 Go 控制面与正式事件合同组织。
4. 后续架构演进强调渐进式拆分，不允许一次性大重构失控。

因此，本方案的核心原则不是“重做”，而是：

> 在保持现有主链路成立的前提下，重建前端的体验模型、状态模型和信息架构。

## 3. 当前前端的核心问题

### 3.1 不是页面问题，而是模型问题

当前前端最根本的问题不是“样式还不够成熟”，而是：

1. `页面模型` 先于 `任务模型`
2. `局部 UI 逻辑` 先于 `统一运行状态`
3. `组件拼接` 先于 `体验编排`

用户感受到的结果就是：

1. 能看到很多信息，但不知道什么最重要。
2. 能看到任务在动，但不总能判断它现在到底处于什么状态。
3. 页面切换、底部调查层、确认流、错误提示之间缺少稳定关系。
4. 有时像聊天应用，有时像控制台，有时又像日志工具，整体心智不统一。

### 3.2 从用户视角的 5 个不满点

当前前端最容易让用户产生以下不满：

1. `我不知道它现在到底在工作、等待我、还是已经卡住了。`
2. `我需要的信息没有固定位置，每次都要重新找。`
3. `当前任务现场和历史复盘资料混在一起，阅读负担大。`
4. `当风险确认或错误出现时，界面没有形成足够稳定的接管机制。`
5. `整个前端更像“若干页面拼起来”，而不是一个长期可用的本地工作系统。`

---

## 4. 当前正式判断

基于当前实现状态与第二阶段产品方向，前端的正式判断固定为：

1. 当前前端的正确目标不再是“继续把三页做得更像成熟网页”。
2. 当前前端的正确目标是“让三页共同服务一个统一的任务运行体验”。
3. 后续前端应从“页面优先”切换为“当前任务优先”。
4. 工作台、日志、设置三页仍然保留，但它们必须重新获得更清晰的职责边界。

一句话结论：

> 当前前端最大的问题，是用页面思维承载实时任务系统；当前前端后续必须改成由任务状态驱动 UI，而不是由页面切换驱动体验。

---

## 5. 目标用户体验

优化后的前端，用户打开后应立即获得以下感受：

1. 我现在在哪个任务里。
2. 系统现在正在做什么。
3. 它是正常执行、等待确认、断开重连还是已经失败。
4. 如果我需要介入，入口会主动出现在我面前，而不是让我去找。
5. 当前任务的实时现场，与历史日志、系统设置、上下文资源之间边界清楚。

这意味着前端必须优先回答用户最在意的 5 个问题：

1. `当前任务是什么`
2. `当前状态是什么`
3. `现在需不需要我做决定`
4. `结果依据是什么`
5. `失败或中断后怎么恢复`

---

## 6. 前端总体优化方向

### 6.1 总体方向

前端优化方向固定为：

> 从“三页工作台”升级为“以任务运行为核心、以全局状态为骨架、以调查与复盘为辅助层的本地智能体工作界面”。

### 6.2 本轮不改变的边界

本轮仍保持：

1. 正式页面以工作台、日志页、设置页为主。
2. 不新增多智能体、记忆独立页、插件中心等新产品域。
3. 不推翻现有聊天、日志、设置 API。
4. 不把前端变成重运维控制台。

### 6.3 本轮必须改变的东西

本轮必须改变：

1. 页面结构围绕任务重新组织。
2. 全局状态可视化机制要建立起来。
3. 当前任务现场与历史信息要明确分层。
4. 高优先级事件要从局部卡片提升为全局体验层。

---

## 7. 目标体验模型

### 7.1 核心模型

前端后续应统一采用以下体验模型：

1. `任务运行驱动 UI`
2. `状态机驱动体验`
3. `全局层承接确认、错误、连接状态`
4. `页面只作为信息域切分，不再负责定义任务状态`

### 7.2 任务优先于页面

用户心智应从：

1. 首页
2. 日志页
3. 设置页

转为：

1. 当前任务现场
2. 历史复盘入口
3. 系统配置入口

也就是说，页面仍存在，但页面不再是体验主语。

### 7.3 全局层优先于局部层

以下能力必须被视为全局体验层，而不是某个页面里的附属卡片：

1. 风险确认
2. 连接状态
3. 严重错误
4. 恢复入口

原因很简单：

1. 它们由任务状态触发，不由页面位置触发。
2. 它们会打断用户当前工作流。
3. 它们决定任务是否可以继续推进。

## 8. 工作台页面优化方案

### 8.1 工作台的新角色

工作台不是聊天页变体，而是：

> 当前任务的统一运行容器

工作台要承接四类信息：

1. 主任务线程
2. 当前任务状态
3. 当前任务辅助信息
4. 当前任务调查入口

### 8.2 目标布局

建议工作台形成稳定四区结构：

1. 顶部全局状态带
2. 中央主任务区
3. 右侧任务中枢
4. 底部调查层

其中：

1. 顶部负责让用户“先安心”
2. 中央负责让用户“继续推进任务”
3. 右侧负责让用户“理解当前任务”
4. 底部负责让用户“在需要时深入调查”

### 8.3 顶部全局状态带

顶部除了品牌、导航、模型和模式外，应正式承载：

1. 当前任务标题摘要
2. 当前运行状态
3. 当前连接状态
4. 当前工作区
5. 当前模型与模式
6. 当前是否存在待确认动作

顶部的作用不是信息堆叠，而是让用户在任何时候都能“一眼看明白现在是怎么回事”。

### 8.4 中央主任务区

中央区域应坚持一个原则：

> 所有内容都要服务“继续推进当前任务”

中央区建议固定为：

1. 当前任务头部
2. 主线程消息流
3. 当前风险确认入口
4. 输入与控制区

具体要求：

1. 消息区弱化普通聊天感，强化任务线程感。
2. 最终结论、阶段结果、失败信息、验证结论要有更稳定层级。
3. 输入区要更像“任务指令输入器”，而不是社交聊天输入框。

### 8.5 右侧任务中枢

右侧区域的正式定位固定为：

> 当前任务驾驶舱

建议固定收口为以下 6 块：

1. `当前任务`
2. `下一步建议`
3. `最近动作`
4. `仓库上下文`
5. `风险确认`
6. `记忆与知识痕迹`

右侧的核心价值不是“多显示一些数据”，而是帮用户迅速建立判断：

1. 当前任务推进得是否正常
2. 有没有阻塞
3. 最近系统到底做了什么
4. 当前结果依赖了什么上下文

### 8.6 底部调查层

底部调查层不再理解为“首页底板”，而要理解为：

> 当前任务的深度观察层

建议分为以下标签：

1. `事件流`
2. `工具动作`
3. `验证结果`
4. `系统细节`

当前阶段如果不想扩太多，也至少应把现有底部面板明确收口为：

1. 当前会话事件流
2. 当前调查焦点
3. 当前下一观察点

底部层的原则是：

1. 默认不打断主任务区
2. 需要时可快速展开
3. 展开后足够支撑排障和解释

## 9. 日志页优化方案

### 9.1 日志页的新角色

日志页的正式定位不应再是“稳定日志列表”，而应固定为：

> 历史任务复盘与排障页面

### 9.2 体验目标

用户进入日志页后，应立刻知道：

1. 这里不是当前任务现场
2. 这里是历史记录与复盘入口
3. 我可以从任务视角理解发生过什么，而不是只盯着一行行记录

### 9.3 页面结构建议

日志页建议固定为：

1. 页面头部
2. 摘要统计区
3. 筛查工具条
4. 历史时间线主体
5. 详情联动区

### 9.4 内容组织原则

日志页应从“按日志行组织”逐步转为“按任务复盘组织”。

短期内保留现有日志记录模型，但体验上应优先支持：

1. 快速定位某次失败
2. 快速定位某次风险确认
3. 快速理解某条记录的上下文、验证和后续动作

中期建议逐步演进为：

1. 先按任务聚合
2. 再按记录展开

## 10. 设置页优化方案

### 10.1 设置页的新角色

设置页必须从“混合控制面”收口为：

> 系统配置与运行策略入口

### 10.2 正式职责

设置页当前只应承接：

1. Runtime 环境
2. 模型与模式
3. 工作区与授权
4. 风险与权限策略
5. Memory Policy 摘要或占位
6. Diagnostics 与导出工具

### 10.3 应减弱的内容

设置页应减少以下倾向：

1. 把当前任务信息塞进设置页
2. 把资源数据和配置动作混在一起
3. 把运行现场解释写成大量说明卡片

原因是：

1. 配置页不应承担工作台角色
2. 数据资源和系统配置属于不同心智
3. 当前任务相关信息应回到工作台与日志页

---

## 11. 状态模型设计

### 11.1 运行状态必须正式化

当前前端后续必须显式建立任务运行状态模型。

建议运行状态固定为：

1. `idle`
2. `submitting`
3. `streaming`
4. `awaiting_confirmation`
5. `resuming`
6. `completed`
7. `failed`
8. `archived`

### 11.2 连接状态必须独立建模

建议连接状态固定为：

1. `connecting`
2. `connected`
3. `reconnecting`
4. `disconnected`
5. `closed`

### 11.3 为什么要拆成两套状态

因为对用户来说：

1. 任务在运行，但连接暂时重连中
2. 任务等待确认，但连接仍然正常
3. 任务已经完成，但流连接已关闭

这是三种完全不同的体验。

如果继续把它们揉成一个“状态文案”，前端永远会出现错位。

### 11.4 UI 绑定原则

后续 UI 绑定应遵守：

1. 业务状态驱动 UI
2. 展示文案不反向驱动逻辑
3. 任何状态判断都不依赖中文标签本身

---

## 12. 数据分层与领域边界

当前前端后续必须明确区分以下 4 类数据：

1. `当前会话实时流`
2. `历史稳定日志`
3. `系统配置与策略`
4. `资源与上下文数据`

### 12.1 当前会话实时流

包含：

1. 当前 session
2. 当前 run
3. 事件流
4. 确认流
5. 当前输出

它只服务当前任务现场。

### 12.2 历史稳定日志

包含：

1. 持久化日志
2. 复盘筛选
3. 错误与风险历史
4. 历史任务回放材料

它只服务历史查看和复盘。

### 12.3 系统配置与策略

包含：

1. 模型
2. 模式
3. 工作区选择
4. 风险策略
5. 授权策略

它只服务系统配置。

### 12.4 资源与上下文数据

包含：

1. 记忆资源
2. 知识沉淀
3. 仓库上下文摘要
4. 连接器与能力摘要

它应作为工作台辅助信息和后续资源面基础，不应长期和设置混写。

## 13. 前端分层建议

结合当前项目本地化架构方案与高模块化演进方案，前端后续建议形成以下四层：

1. `App Shell`
2. `Runtime Domain`
3. `History Domain`
4. `Settings / Resources Domain`

### 13.1 App Shell

负责：

1. 页面路由或视图壳
2. 全局层挂载
3. 主布局组织

不负责：

1. 任务运行判断
2. 历史日志组织
3. 配置资源聚合逻辑

### 13.2 Runtime Domain

负责：

1. 当前 session
2. 当前 run
3. 事件消费
4. 确认流
5. 连接状态

它是未来最优先应被收口的领域。

### 13.3 History Domain

负责：

1. 稳定日志
2. 过滤与检索
3. 详情联动
4. 历史复盘

### 13.4 Settings / Resources Domain

负责：

1. 设置读取与变更
2. 工作区与授权
3. 记忆与资源摘要
4. 后续能力与连接器摘要消费

---

## 14. 分阶段实施路线

### 阶段一：先收口状态模型

目标：

1. 正式建立任务状态模型
2. 正式建立连接状态模型
3. 把运行状态与页面状态拆开

这是后续所有体验优化的前提。

### 阶段二：建立全局层

目标：

1. 风险确认提升为全局层
2. 连接提示提升为全局层
3. 严重错误提升为全局层

做到这一步后，用户对系统状态的理解会立刻变稳。

### 阶段三：重构工作台

目标：

1. 工作台真正成为统一任务页
2. 右侧任务中枢稳定成型
3. 底部调查层与主任务区形成明确主辅关系

### 阶段四：重构日志页

目标：

1. 强化历史任务复盘定位
2. 降低“看技术日志”的门槛
3. 为后续任务级历史聚合预留结构

### 阶段五：收口设置页与资源边界

目标：

1. 设置页只承接系统配置
2. 资源与上下文类内容逐步从设置页中剥离
3. 为后续 `capabilities / connectors / resources` 的前端消费预留位置

---

## 15. 与最新架构文档的对齐要求

### 15.1 与本地化架构方案对齐

前端后续必须继续遵守：

1. 前端不承担运行时裁决逻辑
2. 前端围绕正式合同消费系统状态
3. 前端逐步降低总装式消费方式

### 15.2 与高模块化演进方案对齐

前端后续应逐步预留或吸收：

1. `capabilities` 领域消费
2. `connectors` 领域消费
3. `jobs` 的未来挂位

但当前阶段不强求一次性全部接入。

### 15.3 与第二阶段需求文档对齐

第二阶段强调：

1. 长期使用
2. 长期记忆
3. 长期成长
4. 成本与缓存优化

因此前端必须体现：

1. 当前任务不是一次性对话
2. 日志与记忆不是附属功能
3. 当前工作区、知识与记忆共同服务长期使用

---

## 16. 本轮验收标准

后续以前端优化为目标的工作，至少应满足以下标准：

1. 用户能稳定判断当前任务处于什么状态。
2. 用户能稳定判断当前连接是否正常。
3. 用户收到确认请求时，不再需要切换页面寻找上下文。
4. 用户能区分“当前任务现场”和“历史复盘页面”。
5. 设置页的职责更收敛，不再继续承担工作台解释职责。
6. 前端结构开始围绕领域收口，而不是继续向单一入口文件集中。

以下情况视为未达标：

1. 页面看起来更漂亮，但任务状态仍然混乱。
2. 风险确认和错误仍然只是局部提示。
3. 日志页仍然主要靠用户自己从记录中拼凑上下文。
4. 设置页继续承载资源、配置、运行态混写。
5. 新增一个体验模块时，仍然主要依赖继续扩大总装逻辑。

---

## 17. 最终结论

当前前端后续优化的正确方向已经明确：

1. 不再继续把前端理解成一个有三四个页面的单页应用。
2. 应把前端理解成一个围绕当前任务组织的本地智能体工作系统。
3. 页面仍然保留，但页面只是信息域，不再是体验骨架。
4. 真正的体验骨架应该是 `任务状态 + 连接状态 + 全局层 + 工作台主现场 + 历史复盘层 + 配置入口`。

> 后续凡是涉及前端优化、工作台重构、状态拆分、日志体验、设置页收口的工作，默认都应先对齐本文，再进入实现。

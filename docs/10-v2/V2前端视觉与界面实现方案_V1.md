## 1. 文档目的

本文件用于解决以下问题：

1. 下一轮前端不再依赖 Stitch 提示词，而是有一份正式的实现语言规范。
2. 让实现型 AI 明确知道当前前端应该朝什么气质、什么页面结构、什么模块关系去收口。
3. 确保前端提升视觉质量时，不破坏当前已经落地的 V2 三视图产品边界。
4. 给后续继续打磨前端留出清晰的第二阶段入口，而不是一轮把全部细节做死。

本文件不是纯视觉灵感文档，也不是前端代码实现文档。  
它是“前端实现目标说明书”，目的是让另一个 AI 或工程师可以按统一口径推进。

---

## 2. 核心结论

下一轮前端的目标不是做成“更花的后台”，而是把现有三视图产品壳提升为：

> 一个安静、可长期使用、任务过程可见、带有本地工作台质感的单智能体桌面 Web 工作台。

必须坚持以下判断：

1. 这是“本地 AI 智能体工作台”，不是通用 SaaS 管理后台。
2. 这是“围绕单任务持续协作的桌面客户端”，不是聊天气泡网页。
3. 这是“有过程、有状态、有风险控制”的任务界面，不是卡片拼盘。
4. 首页仍然是第一焦点，但首页不能被日志和设置淹没。
5. 视觉可以更成熟，但不能因此把产品做成重运维控制台。

---

## 3. 本轮实现目标

### 3.1 本轮目标

本轮前端要先达到以下效果：

1. 让首页、日志、设置三页都具备统一的成熟视觉语言。
2. 让首页更像“任务工作台”，而不是“聊天区 + 右栏 + 底栏的机械拼接”。
3. 让日志页更像“调查与回放页面”，而不是简单的日志列表。
4. 让设置页更像“本地运行时控制面板”，而不是常规设置页。
5. 保留未来扩展空间，但当前不要求把所有预留能力真正实现完。

### 3.2 本轮不追求

本轮明确不追求：

1. 新增一级页面。
2. 引入复杂多智能体编排界面。
3. 引入 IDE 式文件树作为主导航。
4. 为了好看而额外引入重型前端依赖。
5. 为了展示未来能力，把首页堆成全功能驾驶舱。

## 4. 前端总体设计语言

### 4.1 总体气质

前端整体气质应定义为：

> 温和克制的技术工作台  
> 安静但不单调  
> 有秩序、有深度、有材料感  
> 更像长期使用的本地工具，而不是一次性网页模板

这套气质必须同时满足以下体验：

1. 第一眼看上去专业、稳定、不廉价。
2. 长时间使用不会刺眼，不会疲劳。
3. 用户能快速理解“现在系统正在做什么”。
4. 重点信息会自己浮出来，不依赖大面积强色。

### 4.2 禁止风格

实现时要避免以下风格倾向：

1. 通用 SaaS 仪表盘感。
2. 紫色、赛博霓虹、过强发光。
3. 过度玻璃拟态与大片毛玻璃。
4. 大量悬浮卡片拼盘。
5. 首页密集 KPI 化。
6. 过亮的科技蓝和高饱和渐变按钮。

### 4.3 推荐视觉方向

推荐采用以下方向：

1. 矿物灰、雾白、石板灰为主基底。
2. 青灰、深青绿、冷墨蓝作为少量状态强调色。
3. 背景采用轻微层次变化，而不是纯平大白底。
4. 通过边框、分区、密度和留白建立秩序，而不是靠大量阴影。
5. 让信息区块像“工作台上的功能区”，不是一堆 marketing card。

## 5. 视觉系统规范

### 5.1 色彩系统

建议实现为三层 token：

1. Primitive Tokens：原始色值。
2. Semantic Tokens：语义色，例如 `text-primary`、`surface-panel`、`border-subtle`。
3. Component Tokens：组件级色值，例如 `topbar-bg`、`task-hub-border`、`danger-pill-bg`。

推荐语义方向如下：

1. 页面底色：偏雾白、灰蓝白、冷灰白，不纯白。
2. 主面板底色：带轻微暖灰或冷灰调的实体面，不透明度不要太低。
3. 次级面板：略低对比，避免所有区块都一样重。
4. 主文字：深石墨灰，不要纯黑。
5. 次级文字：中性灰蓝。
6. 运行中：低饱和青绿色。
7. 等待确认：低饱和赭黄或琥珀色。
8. 失败/高风险：深砖红或锈红，不要荧光红。
9. 完成：偏冷蓝或灰蓝，不要成功绿泛滥。

实现要求：

1. 所有页面必须基于统一 CSS 变量或 token 文件管理。
2. 不允许在组件里散落大量硬编码颜色。
3. 颜色优先服务状态表达，不服务装饰。

### 5.2 字体系统

当前项目已有中文环境，因此建议：

1. 正文维持系统友好字体栈，优先兼顾 Windows 中文显示稳定性。
2. 英文标识与技术字段可用更干净的无衬线搭配，但不要强依赖外链字体。
3. 代码、路径、模型 ID、run id 等技术内容使用等宽字体。

排版要求：

1. 页面主标题不需要过大，更像应用标题，而不是海报标题。
2. 小标题层级清晰，但不要每块都做“营销式大标题”。
3. 英文标签可以小号大写，但必须克制使用。
4. 中文文案优先简洁、稳定、专业，不写花哨口号。

### 5.3 圆角、边框、阴影

建议基线：

1. 页面主容器圆角统一，不同组件只允许有限的 2 到 3 档圆角。
2. 边框比阴影更重要，优先用弱边框建立结构。
3. 阴影只用于抬升主容器和少量浮层，不要所有卡片都高阴影。
4. 分割线要轻，不要 UI 噪音过强。

### 5.4 动效原则

动效只做三类：

1. 页面切换的轻淡入或分区滑入。
2. 状态切换时的微弱颜色和位置反馈。
3. 底部面板展开收起的平滑过渡。

禁止：

1. 花哨加载动画。
2. 大范围弹跳。
3. 过多 hover 漂浮效果。

## 6. 信息架构与页面定位

V2 固定保留三个正式页面：

1. 首页 `Workbench`
2. 日志页 `Logs`
3. 设置页 `Settings`

下一轮前端必须围绕这三页深化，不新增一级视图。

### 6.1 首页定位

首页不是聊天页的别名，它是：

> 任务工作台

用户打开首页后，应该立即理解：

1. 当前正在做什么任务。
2. 任务现在推进到哪一步。
3. 是否需要我确认。
4. 最近做过什么动作。
5. 如需更深排查，可以去哪里看。

### 6.2 日志页定位

日志页不是“把事件流换个页面再展示一遍”，而是：

> 调查、复盘、验收、排障专用页面

### 6.3 设置页定位

设置页不是“偏好设置”，而是：

> 本地运行时控制面板

## 7. 首页 Workbench 方案

### 7.1 页面目标

首页需要形成以下感受：

1. 用户正在和一个真实可工作的本地智能体协作。
2. 主区域用于推进任务。
3. 右侧区域用于提供任务中枢信息。
4. 底部区域用于查看过程，但默认不干扰主链路。

### 7.2 页面结构

首页继续采用现有四段结构，但要做成熟化重构：

1. 顶栏 Top Bar
2. 主工作区 Main Work Area
3. 右侧任务中枢 Task Hub
4. 底部调查面板 Bottom Investigation Panel

### 7.3 顶栏设计要求

顶栏目标不是功能堆叠，而是建立全局框架感。

必须包含：

1. 产品名或当前应用名。
2. 全局运行摘要。
3. 首页 / 日志 / 设置导航。
4. 当前模型快速切换。
5. 当前模式快速切换。

建议新增但不强制后端支持的轻量信息：

1. 当前工作区名称。
2. 当前运行态小圆点。
3. 最近一次任务时间或会话状态摘要。

文案风格要求：

1. 产品副标题使用一句简短定位语，例如“本地优先的单智能体工作台”。
2. 状态摘要必须简短，例如“运行中”“等待确认”“空闲”“已完成”“失败”。
3. 不要写大段说明文字。

### 7.4 主工作区设计要求

主工作区是首页第一焦点，建议由以下模块组成：

1. 当前任务标题区
2. 会话消息流
3. 风险确认卡片
4. 输入区

要求：

1. 当前任务标题区要比现在更明确，让用户知道本轮任务的主题，而不是只看到消息列表。
2. 会话消息流要弱化“聊天软件感”，强化“任务线程感”。
3. 助手回复中，最终结论、执行摘要、风险提醒要有更清晰层级。
4. 输入区应更像“工作指令输入”，而不是社交聊天输入框。

建议文案语气：

1. 输入占位文案不使用闲聊语气。
2. 推荐使用“描述你希望智能体完成的任务”这一类工作型文案。
3. 如果无任务，可显示轻量空态，引导用户发起真实工作请求。

### 7.5 右侧 Task Hub 设计要求

右侧栏必须从“若干卡片列表”升级为“统一任务中枢”。

本质要求：

1. 它不是单纯的附属信息区。
2. 它必须回答“当前任务的控制态与上下文态”。
3. 视觉上必须比普通 sidebar 更有结构感。

推荐整合为以下 6 个固定区块：

1. `Mission`
   当前任务主题、状态、当前阶段。
2. `Next Action`
   下一步建议、当前阻塞点、是否等待确认。
3. `Runtime Lane`
   最近一次工具动作、最近一次结果摘要、执行健康状态。
4. `Repo Context`
   工作区根、仓库根、分支、dirty 状态、repo context 摘要。
5. `Risk Gate`
   当前风险级别、待确认动作、确认状态。
6. `Memory Trace`
   最近是否命中记忆、是否写入记忆、记忆类型。

实现原则：

1. 这 6 个区块允许部分为空，但结构必须固定。
2. 空状态也要能解释“暂无数据”，不能出现视觉空洞。
3. 必须让用户一眼知道哪里是最重要的当前信息。

特别要求：

1. `Repo Context` 必须保留现有 M1 已落地字段，不允许因为美化而丢失。
2. `Risk Gate` 必须明显但不惊悚。
3. `Next Action` 应成为右栏最有价值的模块之一。

### 7.6 底部调查面板设计要求

底部面板不能只是“展开后出现一堆条目”，而要更明确地表达：

> 这里是任务执行过程的观察和调查层

建议结构：

1. 面板头部：标题、展开收起按钮、当前数量摘要。
2. 标签区：`事件流`、`全局日志`。
3. 主内容区：统一时间线样式，但语义不同。

`事件流` 标签要求：

1. 更强调当前会话内的过程推进。
2. 清楚展示阶段、工具、摘要、风险、错误。

`全局日志` 标签要求：

1. 更强调稳定持久化记录。
2. 清楚展示分类、来源、工具、级别。

### 7.7 首页空状态设计

当没有运行任务时，首页不能显得像“坏了”。

首页空态应展示：

1. 一句简单的欢迎语或引导语。
2. 当前模型、当前模式、当前工作区的轻量状态。
3. 2 到 3 个高质量示例任务，不要写成教程列表。

## 8. 日志页 Logs 方案

### 8.1 页面目标

日志页要从“日志列表页”升级为：

> 有摘要、有筛查入口、有时间线主体的调查页

### 8.2 页面结构

推荐结构如下：

1. 页面头部
2. 摘要统计区
3. 筛查工具条
4. 主时间线区
5. 详情侧区或详情抽屉预留位

### 8.3 页面头部要求

页面头部应回答：

1. 这里看什么。
2. 为什么和首页底部面板不同。
3. 当前记录规模如何。

建议头部文案方向：

1. 标题使用“日志与回放”或“运行日志”。
2. 副文案强调“跨会话复盘、排障、验收观察”。

### 8.4 摘要统计区要求

保留 V2 既有统计方向，但视觉上要更稳：

1. 总记录量
2. 错误数量
3. 风险确认数量
4. 涉及工具数量

要求：

1. 统计卡片不做大屏运营看板风格。
2. 统计只做摘要，不喧宾夺主。
3. 错误和风险数量应有轻微状态强调色。

### 8.5 筛查工具条要求

即使当前后端能力未完全支持，前端也应先预留以下结构：

1. 时间范围筛选占位。
2. 类型筛选占位。
3. 级别筛选占位。
4. 搜索框占位。
5. 仅看错误/仅看确认开关占位。

注意：

1. 如果能力尚未接通，可先做只读样式占位。
2. 必须标注哪些项“即将支持”或“后续接入”。
3. 不允许伪造已生效交互。

### 8.6 主时间线区要求

日志页的主体不是表格，而是时间线式调查流。

必须展示：

1. 事件类型
2. 阶段或运行 ID
3. 摘要
4. 详情
5. 工具名
6. 风险级别
7. 错误信息
8. agent 标签

视觉要求：

1. 每条记录必须有足够的阅读层次。
2. 重点字段要容易扫读。
3. 详情默认可读，但不能淹没整个列表。

### 8.7 详情区预留要求

日志页应预留一块“详情查看位”，暂时可不做完整交互。

可预留内容：

1. 选中记录详情
2. 关联 run id
3. 关联 session id
4. 原始 metadata 摘要
5. 相关工具输出占位

这块区域当前可以是：

1. 桌面宽屏右侧详情栏
2. 或移动端抽屉

## 9. 设置页 Settings 方案

### 9.1 页面目标

设置页必须从“基础设置页”升级为：

> 本地运行时控制面板

用户应能理解：

1. 当前系统如何运行。
2. 默认模型和模式是什么。
3. 当前工作区与授权边界是什么。
4. 哪些风险策略正在生效。

### 9.2 页面结构

设置页推荐分成两层：

1. 顶部环境摘要带
2. 下方分区控制面板

分区控制面板固定为以下 6 个模块：

1. `Runtime Environment`
2. `Model Providers & Routing`
3. `Workspace Roots`
4. `Risk & Permissions`
5. `Memory Policy`
6. `Diagnostics & Utilities`

### 9.3 Runtime Environment 要求

必须显示：

1. 应用名
2. Runtime 状态
3. Runtime 版本
4. Gateway 端口
5. 当前工作区

建议预留：

1. Runtime profile
2. 当前模式
3. 当前运行负载级别

### 9.4 Model Providers & Routing 要求

模型区不只是下拉框，应更像“模型控制区”。

必须包含：

1. 当前模型
2. provider 分组或 provider 标签
3. 可用/不可用状态
4. 切换反馈

可预留但不强制本轮实现：

1. 默认模型说明
2. 备用模型说明
3. 当前 provider 健康状态

### 9.5 Workspace Roots 要求

工作区模块必须体现“当前工作上下文的边界”。

必须包含：

1. 当前工作区
2. 当前路径
3. 可切换工作区列表
4. 已记住授权列表
5. 移除授权动作

可预留：

1. 最近访问工作区
2. 当前 repo context 状态入口

### 9.6 Risk & Permissions 要求

这是设置页必须做出产品识别度的区域。

必须包含：

1. 新目录首次接触提醒
2. 风险等级展示开关
3. 已授权范围的撤销机制

建议预留但允许只做占位：

1. 高风险动作二次确认策略
2. 目录外访问提醒策略
3. 命令执行风险级别阈值
4. 写入类动作默认确认策略

### 9.7 Memory Policy 要求

当前即使能力未完全接通，页面也应预留“记忆策略”的正式位置。

可展示：

1. 记忆是否启用
2. 记忆摘要策略占位
3. 写入策略占位
4. 清理策略占位

要求：

1. 如果当前接口未提供真实字段，则以“后续接入”形式展示。
2. 不允许在首页抢占一级导航。

### 9.8 Diagnostics & Utilities 要求

设置页底部应统一收口工具型动作。

推荐包含：

1. 导出运行日志
2. 导出当前配置
3. 打开诊断信息
4. 展示扩展参数占位

要求：

1. 这些动作必须归于设置页，不上移到首页。
2. 要有工具区感，但不能像开发者调试面板硬塞在页面中间。

## 10. 文案与语言风格规范

### 10.1 总体文案原则

前端文案必须满足：

1. 专业。
2. 安静。
3. 不卖弄。
4. 不口语化过头。
5. 不写营销式口号。

### 10.2 中文文案风格

建议采用以下风格：

1. 短句。
2. 工程化表达。
3. 尽量一句话说清楚当前状态。
4. 用词稳定，前后一致。

推荐词汇：

1. 当前任务
2. 下一步
3. 仓库上下文
4. 最近动作
5. 风险确认
6. 记忆痕迹
7. 运行日志
8. 工作区
9. 运行环境
10. 授权策略

避免词汇：

1. 灵感
2. 创作
3. 魔法
4. 超级智能
5. 全能
6. AI 协作者宇宙

### 10.3 空状态文案原则

空状态文案应做到：

1. 解释为什么此处为空。
2. 告诉用户下一步能做什么。
3. 不使用“暂无数据”四个字草率收尾。

示例方向：

1. “任务开始后，这里会显示当前仓库上下文摘要。”
2. “完成一次运行后，这里会出现稳定日志记录。”
3. “当前没有待确认动作，新的高风险操作会在这里提示。”

### 10.4 按钮文案原则

按钮文案优先使用动词 + 对象：

1. 提交任务
2. 查看日志
3. 批准变更
4. 拒绝操作
5. 导出配置
6. 移除授权

避免：

1. 去看看
2. 试试看
3. 立即开始魔法

## 11. 实现技术建议

### 11.1 技术基线

下一轮前端继续使用现有技术基线：

1. React
2. TypeScript
3. Vite
4. 现有 `frontend/src/` 模块结构

不建议本轮更换：

1. 前端框架
2. 构建工具
3. 路由模式
4. 主状态组织方式

### 11.2 样式实现建议

推荐做法：

1. 在 `index.css` 或独立 token 文件中建立语义设计变量。
2. 对现有页面样式做统一升级，而不是各组件各自拼风格。
3. 先把结构、密度、留白、层级拉齐，再做局部细化。
4. 保持桌面优先，同时兼顾移动端收缩。

不建议：

1. 引入大型设计系统库。
2. 为了追求效果大规模重写交互逻辑。
3. 同时改视觉、状态逻辑、数据流三大层。

### 11.3 动效与交互建议

推荐：

1. 用轻微 transition 处理 hover、状态、展开收起。
2. 用 CSS 或轻量 React 动效完成入场和折叠。
3. 把动效重点放在“任务状态变化”和“底板展开”。

不建议：

1. 引入复杂动画库作为必要前提。

## 13. 实施顺序建议

建议实现顺序如下：

### 阶段 A：建立统一视觉基线

目标：先让整个应用“看起来像一个成熟产品”。

顺序：

1. 改 `index.css` 的 token 和基础布局。
2. 改 `TopBar.tsx` 和公共 panel 样式。
3. 让首页、日志、设置三页先统一成同一套视觉语法。

验收重点：

1. 三页风格统一。
2. 不再像三套不同来源的页面。
3. 页面已经具备“本地工作台”气质。

### 阶段 B：重构首页信息秩序

目标：把首页从现有骨架升级成真正工作台。

顺序：

1. 改主工作区层级。
2. 重构 `ContextSidebar.tsx`。
3. 优化 `BottomPanel.tsx`。

验收重点：

1. 首页第一眼能看出“当前任务”和“下一步”。
2. 右栏像任务中枢，而不是卡片集合。
3. 底部面板更像调查层。

### 阶段 C：补齐日志页和设置页成熟度

目标：让三页都达到可展示、可继续演进的基线。

顺序：

1. 重构 `LogsPanel.tsx`。
2. 重构 `SettingsPanel.tsx`。
3. 对齐空状态、占位态、异常态文案。

验收重点：

1. 日志页具备回放与排障感。
2. 设置页具备运行时控制面板感。
3. 三页都能作为之后继续细化的稳定基线。

本文件对应的是“前端成熟化第一轮验收”，不是最终终稿验收。

## 16. 给实现型 AI 的明确执行要求

如果把本文件交给另一个 AI 实现，必须明确以下要求：

1. 只在现有 `frontend/` 结构上收口，不要重做技术栈。
2. 不新增一级页面，不改产品边界。
3. 先完成视觉统一和结构优化，再考虑局部细节。
4. 允许做占位，但占位必须清楚标注，不伪造功能。
5. 不要为了视觉效果新增重型依赖。
6. 不要把页面做成通用 SaaS 后台。
7. 不要牺牲 repo context、风险确认、事件流、日志这几个核心产品语义。

最终实现出来的前端应满足一句话判断：

> 它看起来像一个本地运行的 AI 智能体工作台，安静、专业、可持续使用，并且用户能看见任务推进过程。

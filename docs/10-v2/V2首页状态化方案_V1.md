# 本地智能体 - V2 首页状态化方案 V1

更新时间：2026-04-08

状态：`当前前端首页状态化正式方案`

使用规则：

1. 本文用于确定首页产品定义，不讨论是否继续保留任务页主工作区。
2. 本文只解决首页应该如何在不同状态下承担“开始”和“恢复”职责。
3. 若与更早“首页即主聊天区”的口径冲突，以本文和 [V2前端当前执行入口_V1](D:/newwork/本地智能体/docs/10-v2/V2前端当前执行入口_V1.md) 为准。

---

## 1. 文档目的

本文只回答 5 个问题：

1. 首页在当前产品里到底是什么
2. 首页什么时候显示首用态、恢复态、阻塞态
3. 首页每种状态下，用户第一眼看到什么、第一步点哪里
4. 风险确认、日志、设置在首页中的位置怎么安排
5. 实施和验收时应以什么口径继续推进

---

## 2. 正式定义

首页不是固定摘要页，也不是固定主聊天页。

首页正式定义为：

> 系统首页。首次使用时负责快速开始，持续使用时负责恢复上下文；任务推进始终在任务页完成。

当前正式页面关系固定为：

1. 首页：开始与恢复
2. 任务页：唯一主工作区
3. 记录页：跨会话复盘
4. 设置页：运行控制与策略管理

---

## 3. 首页状态模型

首页只允许以下 3 种状态：

1. `首用态`
2. `恢复态`
3. `阻塞态`

### 3.1 状态判定优先级

判定顺序固定为：

1. 先判断是否阻塞
2. 再判断是否存在可恢复上下文
3. 否则进入首用态

### 3.2 阻塞态触发条件

满足任一条件即进入阻塞态：

1. `settings` 尚未可用且存在明确加载失败
2. `runtime_status.ok = false`
3. 当前工作区不可访问或不可用
4. 当前模型配置不可用
5. 存在需要优先处理的全局错误

### 3.3 恢复态触发条件

在不阻塞时，满足任一条件即进入恢复态：

1. 有 `currentRunId`
2. 有 `sessionId`
3. `events.length > 0`
4. 存在待确认项
5. 存在最近任务摘要

### 3.4 首用态触发条件

在不阻塞且不可恢复时进入首用态：

1. 没有历史任务
2. 没有当前运行
3. 没有待确认项
4. 没有最近事件

### 3.5 用户主动进入首用态

即使存在历史上下文，用户主动点击“新建任务”时，也允许首页显示首用态入口，但不清空历史会话数据。

## 4. 首页首用态

### 4.1 页面目标

1. 让用户第一次打开时立刻知道这里可以开始任务
2. 不要求用户先理解任务页、记录页和设置页分工
3. 首屏直接传达产品价值：本地执行、过程可见、风险可控

### 4.2 页面结构

首页首用态固定为：

1. 顶栏
2. 首用引导主区 Hero
3. 快速开始示例区
4. 当前系统状态条
5. 轻量能力说明区

### 4.3 线框文案

Hero 标题：

`今天想让本地智能体帮你完成什么？`

Hero 说明：

`它可以执行命令、处理文件、调用本地程序，并把过程、结果和风险提示清楚展示给你。`

输入区标题：

`输入任务`

输入框占位文案：

`例如：帮我检查这个项目为什么构建失败，并给出最小修改方案`

主按钮文案：

`开始任务`

主按钮说明：

`提交后会进入任务页，持续展示执行过程、风险确认和结果收口。`

快速开始区标题：

`快速开始`

示例任务按钮：

1. `修改项目文件`
2. `执行命令并排错`
3. `整理项目说明`
4. `检索本地知识`

示例任务预填文案：

1. `帮我检查当前项目里最需要修的一个问题，并做最小改动修复`
2. `帮我运行构建命令，定位失败原因并给出修复建议`
3. `根据 docs 和当前代码，说明这个项目现在做到什么程度`
4. `从本地文档中检索当前项目的正式需求和验收口径`

当前系统状态条标题：

`当前环境`

状态字段：

1. `模型`
2. `模式`
3. `工作区`
4. `Runtime`

能力说明区标题：

`你可以这样使用`

能力说明文案：

1. `本地执行：直接操作当前工作区内的文件和命令`
2. `过程可见：任务推进、工具动作和结果摘要持续可见`
3. `风险可控：高风险动作不会绕过确认`

### 4.4 用户第一步行为

首用态优先引导用户：

1. 直接输入任务
2. 点击示例任务按钮后进入任务页继续编辑或直接提交

## 5. 首页恢复态

### 5.1 页面目标

1. 让用户回来时知道上次做到哪一步
2. 让用户一眼判断当前应继续任务、处理确认，还是去复盘
3. 不把执行过程重新塞回首页

### 5.2 页面结构

首页恢复态固定为：

1. 顶栏
2. 恢复上下文主卡
3. 继续操作区
4. 系统与风险区
5. 最近活动概览区

### 5.3 线框文案

恢复上下文主卡标题：

`恢复上下文`

恢复上下文主卡字段：

1. `最近任务`
2. `当前阶段`
3. `最近摘要`
4. `当前运行`
5. `当前会话`

空值文案规则：

1. 最近任务为空：`等待继续的任务`
2. 当前阶段为空：`等待运行`
3. 最近摘要为空：`当前还没有新的运行摘要。`

继续操作区标题：

`从这里继续`

主按钮文案规则：

1. 正常情况：`继续任务主线程`
2. 有待确认项时：`处理待确认动作`
3. 有运行阻塞时：`查看阻塞原因`

次按钮文案：

`查看记录页复盘`

辅助说明文案规则：

1. 正常恢复态：`进入任务页继续主线程，或去记录页查看历史复盘。`
2. 有待确认项：`当前任务停在确认节点，处理后才能继续推进。`
3. 有运行阻塞：`建议先解决当前阻塞，再继续后续任务。`

系统与风险区标题：

`系统与风险`

字段：

1. `当前判断`
2. `连接`
3. `模式`
4. `工作区`
5. `会话事件`

当前判断文案规则：

1. 正常：`当前没有新的风险或阻塞。`
2. 待确认：显示确认动作摘要
3. Runtime 不可达：`运行时不可达，新的任务执行会失败。`

最近活动概览区标题：

`最近活动`

最多展示 3 项：

1. 最近工具动作
2. 最近验证结果
3. 最近记忆痕迹

无最近活动文案：

`当前还没有可恢复的活动记录。`

### 5.4 用户第一步行为

恢复态优先引导用户：

1. 点击 `继续任务主线程`
2. 若存在待确认项，点击 `处理待确认动作`
3. 若需要理解历史过程，点击 `查看记录页复盘`

## 6. 首页阻塞态

### 6.1 页面目标

1. 先告诉用户为什么现在不能继续
2. 再明确告诉用户下一步该做什么
3. 不让用户一打开就面对大段技术错误文本

### 6.2 页面结构

首页阻塞态固定为：

1. 顶栏
2. 阻塞提示主卡
3. 推荐动作区
4. 诊断补充区

### 6.3 线框文案

阻塞提示主卡标题规则：

1. Runtime 不可达：`运行时当前不可达`
2. 设置加载失败：`设置当前未成功加载`
3. 工作区异常：`当前工作区不可访问`
4. 模型异常：`当前模型配置不可用`

阻塞提示主卡正文规则：

1. Runtime：`系统当前无法建立执行链路，新任务会提交失败或无法继续。`
2. 设置：`当前基础配置没有正常加载，建议先检查设置页和运行环境。`
3. 工作区：`当前工作区路径不可访问，继续任务可能失败或产生错误结果。`
4. 模型：`当前模型不可用于继续任务，建议先切换到可用模型。`

推荐动作区标题：

`建议先做这一步`

主按钮文案规则：

1. Runtime 问题：`重新连接`
2. 设置问题：`前往设置检查`
3. 工作区问题：`前往设置检查工作区`
4. 模型问题：`前往设置切换模型`

次按钮文案：

`查看记录页最近错误`

诊断补充区标题：

`更多信息`

诊断补充区字段：

1. `连接状态`
2. `Runtime`
3. `模型`
4. `工作区`

### 6.4 用户第一步行为

阻塞态优先引导用户：

1. 点击主按钮去修复问题
2. 如仍不清楚，再去记录页查看最近错误

---

## 7. 风险确认横幅

### 7.1 正式规则

风险确认不在首页正文中展开完整确认卡。

首页统一只展示一条高优先级确认横幅。

### 7.2 触发条件

满足以下条件时显示：

1. 当前存在 `confirmation`

### 7.3 横幅文案

标题：

`当前有待处理确认`

正文：

显示当前确认动作摘要

主按钮：

`前往任务页处理`

辅助说明：

`确认前不会继续执行当前高风险动作。`

### 7.4 跳转规则

点击后：

1. 进入任务页
2. 焦点落在当前确认区域

---

## 8. 页面跳转规则

首页状态化后，页面跳转固定为：

1. 首页输入任务：进入任务页，并提交任务
2. 首页点击示例任务：预填任务内容，并进入任务页
3. 首页点击继续任务主线程：进入任务页，保持当前会话焦点
4. 首页点击处理待确认动作：进入任务页，并聚焦确认区域
5. 首页点击查看记录页复盘：进入记录页
6. 首页点击设置：进入设置页

---

## 9. 与其他页面的职责边界

### 9.1 首页

首页只负责：

1. 快速开始
2. 恢复上下文
3. 暴露阻塞与待确认入口

首页不负责：

1. 主线程任务推进
2. 当前会话完整过程展开
3. 跨会话日志正文
4. 详细设置管理

### 9.2 任务页

任务页继续固定为唯一主工作区，负责：

1. 输入任务
2. 结果推进
3. 确认流
4. 当前会话调查

### 9.3 记录页

记录页继续负责：

1. 跨会话日志时间线
2. 记录详情阅读
3. 风险、错误、上下文、验证信息复盘

### 9.4 设置页

设置页继续负责：

1. 运行环境
2. 模型与模式
3. 工作区与授权
4. 风险与权限
5. 资源与诊断

## 10. 实施范围

本轮首页状态化实施只允许覆盖：

1. 首页状态判断逻辑
2. 首页首用态、恢复态、阻塞态的界面表达
3. 首页与任务页、记录页、设置页之间的跳转动作
4. 顶栏中的首页状态语义和高频入口调整

---

## 11. 非目标

本轮明确不做：

1. 把首页重新改成主聊天页
2. 新增一级页面
3. 把完整确认卡搬回首页
4. 把全局日志搬回首页正文
5. 新增新的顶层 API
6. 推翻任务页主工作区结构
7. 引入新的样式体系或状态管理体系

## 13. 验收标准

首页状态化至少必须满足：

1. 首页能稳定区分首用态、恢复态、阻塞态
2. 首用态下用户一眼能看懂怎么开始任务
3. 恢复态下用户一眼能看懂怎么继续当前任务
4. 阻塞态下用户一眼能看懂为什么不能继续以及该点哪里
5. 待确认项在首页以高优先级横幅形式出现
6. 任务页仍然是唯一主工作区
7. 记录页和设置页职责没有被首页侵蚀

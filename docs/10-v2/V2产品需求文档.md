## 1. 文档目的

本文件用于定义本项目 V2 的正式产品形态，解决以下问题：

1. 当前项目在产品层到底是什么
2. 首页、日志页、设置页分别承担什么职责
3. 模型、工作区、记忆、Skills 在 V2 中分别处于什么可见性层级
4. 前端工作台状态模型应该如何组织
5. 后续实现和验收应以什么口径继续推进

本文件替代 V1 中与前端产品形态直接冲突的界面定义，但不推翻 V1 的核心技术边界。

---

## 2. V2 产品定义

V2 将本项目正式定义为：

> 一个本地优先、桌面启动、围绕单智能体持续协作的桌面 AI 客户端。

它不是：

1. 普通聊天网页
2. 复杂运维控制台
3. 自动化调试工作台
4. 多智能体编排界面

它强调的体验是：

1. 首页安静
2. 任务过程可见
3. 风险确认可控
4. 全局设置不干扰聊天主链路
5. 有本地原生工作台感，而不是网页拼盘感

---

## 3. V2 正式范围

### 3.1 正式纳入范围

V2 正式纳入以下产品模块：

1. 首页工作台
2. 日志视图
3. 设置视图
4. 顶栏中的模型快速切换
5. 顶栏中的模式快速切换
6. 事件流与全局日志的分层展示
7. 风险确认与任务上下文侧栏

### 3.2 保留但不前台化的能力

以下能力在 V2 中继续存在，但不升级为前台主模块：

1. Skills 分类与路由
2. 记忆分层与按需召回
3. 能力组选择
4. 后续弱多代理兼容字段

### 3.3 明确不进入 V2 主形态

以下内容不属于 V2 正式交付形态：

1. 独立 Skills 管理页
2. 独立记忆管理页
3. 独立模型管理工作区
4. 浏览器自动化界面
5. 桌面自动化界面
6. 语音入口
7. 多智能体任务编排页
8. 自动 fallback 和测速策略页

---

## 4. 信息架构

V2 的应用级导航固定为三个正式入口：

1. 首页
2. 日志
3. 设置

导航原则：

1. 采用轻量应用导航
2. 不采用 IDE 式对象树
3. 不让导航承担过程展示职责
4. 不把工作区、Skills、记忆做成一级导航

---

## 5. 首页工作台

首页采用以下固定结构：

1. 顶部栏 Top Bar
2. 主聊天区 Main Chat
3. 右侧上下文栏 Context Sidebar
4. 底部折叠面板 Bottom Panel

### 5.1 顶部栏

顶部栏负责全局信息和全局切换，只允许出现：

1. 产品名
2. 当前状态摘要
3. 首页 / 日志 / 设置导航
4. 当前模型快速切换
5. 当前模式快速切换

顶部栏不承载：

1. 工作区详细管理
2. 日志正文
3. 复杂系统设置
4. 任务过程流水

### 5.2 主聊天区

主聊天区是首页第一焦点。

职责：

1. 展示会话线程
2. 承载输入框
3. 承载最终结果收口
4. 承载确认卡片

不承载：

1. 全局设置块
2. 全局日志块
3. 持续固定的大状态卡片

### 5.3 右侧上下文栏

右侧上下文栏用于回答用户此刻最关心的问题：

1. 当前任务是什么
2. 现在进行到哪一步
3. 下一步是什么
4. 最近一次工具动作是什么
5. 当前是否存在待确认风险
6. 是否有记忆相关痕迹

右侧栏中的工作区信息只做轻量显示，不作为主对象强调。

### 5.4 底部折叠面板

底部面板默认折叠，仅在用户需要复盘、观察过程或排障时展开。

V2 正式提供两个标签：

1. 事件流
2. 全局日志

原则：

1. 首页默认不被过程信息挤满
2. 事件流与全局日志语义分离
3. 两者均服务于复盘、观察与排障

---

## 6. 日志视图

日志视图是独立页面，不再作为首页的附属拼块。

定位：

1. 跨会话复盘
2. 排障
3. 验收观察

日志视图中应突出：

1. 总记录量
2. 错误数量
3. 风险确认数量
4. 涉及工具数量
5. 最近记录流

日志页不承担：

1. 模型切换
2. 工作区管理
3. 聊天主链路输入

---

## 7. 设置视图

设置视图是系统配置和次级对象管理入口。

V2 固定分为四个区块：

1. 运行环境
2. 模型设置
3. 工作区设置
4. 风险与授权策略

### 7.1 运行环境

用于展示当前系统基础状态：

1. 应用名
2. Runtime 状态
3. Gateway 端口
4. Runtime 端口

### 7.2 模型设置

模型区块负责：

1. 展示当前模型
2. 展示按 provider 分组的可选模型
3. 展示不可用模型状态
4. 明确提示不可用模型不会被后端接受

V2 不要求提供独立模型工作区。

### 7.3 工作区设置

工作区在 V2 中属于设置对象，而不是首页一级主状态。

职责：

1. 切换当前工作区
2. 显示当前工作区路径
3. 管理已记住的目录授权

### 7.4 风险与授权策略

风险与授权区块负责：

1. 新目录首次接触是否提醒
2. 确认卡片是否显示风险等级
3. 已记住授权的撤销

---

## 8. 模型、工作区、Skills、记忆的定位

### 8.1 模型

模型属于高频全局对象，因此：

1. 顶栏允许快速切换
2. 设置页提供完整说明和状态展示
3. 不单独拆成模型管理工作区

### 8.2 工作区

工作区属于系统上下文对象，但不是首页主角，因此：

1. 首页只轻量显示
2. 设置页负责切换和管理
3. 目录授权策略归设置页

### 8.3 Skills

Skills 在 V2 中是后台能力体系。

要求：

1. 继续保留分类、注册、路由方向
2. 仅通过事件、日志、结果摘要体现调用痕迹
3. 不提供前台独立管理界面

### 8.4 记忆

记忆在 V2 中采用“轻可见，非主入口”策略。

表现方式：

1. 右侧栏显示最近记忆痕迹
2. 事件流与日志流显示记忆写入事件
3. 不提供记忆后台管理页

---

## 9. 前端状态模型

V2 前端必须显式维护以下状态：

1. 当前视图：`home | logs | settings`
2. 当前会话：`session_id`
3. 当前运行：`run_id`
4. 当前运行状态：
   - `idle`
   - `running`
   - `waiting_confirmation`
   - `failed`
   - `finished`
5. 底部面板展开状态
6. 底部面板当前标签
7. 当前待确认对象
8. 右侧上下文摘要对象

这些状态用于保证：

1. 首页信息分层清晰
2. 日志与事件不再抢主聊天区
3. 风险确认固定出现在正确位置

---

## 10. 合同与接口原则

V2 继续沿用既有顶层 API：

1. `POST /api/v1/chat/run`
2. `POST /api/v1/chat/confirm`
3. `GET /api/v1/events/stream`
4. `GET /api/v1/settings`
5. `POST /api/v1/settings`
6. `GET /api/v1/logs`

V2 不新增必须落地的新顶层 endpoint。

V2 对事件和日志合同的要求是：

1. 事件流继续作为首页过程可见性的核心来源
2. 日志继续作为跨会话稳定记录
3. 在现有合同上补充最小元数据，而不是扩接口面

当前约定的关键 metadata 包括：

1. `task_title`
2. `next_step`
3. `tool_name`
4. `result_summary`
5. `risk_level`
6. `memory_kind`

同时允许以下未来兼容字段存在但不前台化：

1. `agent_id`
2. `agent_label`

## 11. V1 继承关系

V2 与 V1 的关系如下：

继续继承：

1. 本地优先
2. 单智能体优先
3. Go 控制面 + Rust 运行时 + React 前端
4. 事件流、日志、确认机制
5. 工作区权限边界

明确作废的界面口径：

1. 首页只做聊天 / 日志 / 设置三块并排卡片
2. 聊天页与设置页、日志页长期并列展示

今后若出现 V1 与 V2 的前端产品定义冲突，以本文件为准。

---

## 12. V2 验收标准

V2 至少应满足以下验收条件：

1. 默认进入首页工作台，而不是卡片拼盘页
2. 聊天主视图始终是首页第一焦点
3. 右侧栏持续显示当前任务、状态、下一步、风险和记忆痕迹
4. 底部面板可展开查看事件流与全局日志
5. 日志视图能服务复盘与排障
6. 设置视图能完成模式、模型、工作区和风险策略管理
7. 不新增 Skills 页、记忆页、模型工作区、多智能体页

---

## 13. 后续建议

V2 基线完成后，后续优先建议为：

1. 继续打磨日志视图的筛选与复盘体验
2. 继续打磨设置页中的模型状态说明
3. 将 V2 页面和合同变化同步回写到 API/验收文档
4. 如需扩展 Skills 与记忆管理，另起 V3 规划，不直接塞入当前首页

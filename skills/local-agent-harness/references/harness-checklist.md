# Harness 清单

按这个顺序检查，不要跳着看。顺序本身就是约束。

## 1. 目标边界

先确认这次任务是在补哪一层壳，而不是只在补提示词：

1. `主循环`
2. `上下文装配`
3. `工具协议`
4. `权限与确认`
5. `验证闭环`
6. `记忆与知识路由`
7. `artifact 外置`
8. `事件与合同`
9. `长任务交接`

如果一句话说不清“这次在补哪层壳”，先不要动实现。

## 2. 主循环

默认检查这 6 段：

1. `Analyze`
2. `Plan`
3. `Execute`
4. `Observe`
5. `Verify`
6. `Finish`

要点：

1. 阶段边界必须可见，不能只剩一坨隐式流程。
2. `失败`、`等待确认`、`验证不通过` 都必须回到主线，而不是在侧边临时打补丁。
3. 每一阶段都要产生稳定 metadata，方便事件回流和审计。

## 3. 上下文装配

上下文的目标不是“尽量多”，而是“足够完成下一步”。

优先使用：

1. 当前任务输入
2. 当前模式和工作区
3. 相关 repo context
4. memory digest
5. 最近相关工具摘要
6. 必要的知识摘要

避免：

1. 全量记忆注入
2. 把整份长文档塞进 prompt
3. 把大段日志原文直接回灌给模型
4. 为了保险把所有规则一次性全塞进去

判定标准：

1. 模型拿到的是“下一步决策所需最小包”
2. 非当前任务必需的材料应通过索引或引用暴露，而不是直接展开

## 4. 工具协议

每个工具至少要回答下面这些问题：

1. 叫什么
2. 属于什么类别
3. 风险级别是什么
4. 输入长什么样
5. 输出是摘要、结构化结果，还是 artifact
6. 是否需要确认

输出统一关注：

1. `summary`
2. `detail_preview`
3. `artifact_path`
4. `error_code`
5. `retryable`

如果工具结果没有统一结构，模型就会被迫反复猜测工具语义。

## 5. 权限与确认

高风险动作默认不直接执行。

要点：

1. 风险判定应发生在执行前，而不是执行后补记录。
2. 确认请求要能恢复原主循环，而不是靠前端重新发起整轮任务。
3. 等待确认的原因要在事件流里可见。

重点检查：

1. 模式与权限策略是否一一对应
2. 旁路执行是否还存在
3. 确认结果是否能无损回到原状态

## 6. 验证闭环

默认要求独立验证，不接受“生成者自己说完成了”作为终点。

至少补一类证据：

1. 自动化测试
2. 合同校验
3. 运行时事件观测
4. 人工确认点
5. 回归脚本

优先思路：

1. 先定义“什么叫通过”
2. 再决定如何生成
3. 验不过就回到修复，而不是直接收尾

## 7. 记忆与知识路由

记忆服务复用，不服务堆料。

规则：

1. 先读相关摘要，不读全量
2. 先判断本轮是否真的需要记忆
3. 本轮产物只有在后续复用价值明确时才沉淀
4. 长期知识与短期工作记忆要分开

反模式：

1. 用“多塞点记忆”代替上下文筛选
2. 把知识库当成默认 prompt 附件
3. 让 planner 无差别读取所有记忆分支

## 8. Artifact 外置

大输出不要直接污染主链路。

适合外置的内容：

1. 大段日志
2. 完整工具输出
3. 长摘要
4. 中间产物
5. 验证报告

主链路只保留：

1. 摘要
2. preview
3. artifact path

## 9. 事件与合同

事件首先服务可恢复性和可审计性。

检查点：

1. 阶段事件是否稳定
2. 工具事件是否稳定
3. 风险确认事件是否稳定
4. 最终结果事件是否稳定
5. 字段是否优先结构化，而不是字符串拼接

如果前端、Go、Rust 三端对同一事件理解不一致，这层壳就不成立。

## 10. 长任务与交接

长任务最容易在两处失稳：

1. 上下文变脏后继续硬跑
2. 生成者临近上下文极限时草率收尾

默认做法：

1. 定义 handoff artifact
2. 在关键节点压缩当前状态
3. 需要时重建上下文再继续，不强行一路带着旧上下文跑到底
4. 把“当前计划、已完成项、未决风险、下一步”写成交接包

## 11. 项目专属禁止项

针对这个仓库，默认不做这些事：

1. 为了看起来先进而提前引入复杂多智能体
2. 为了省事把高频运行时逻辑塞回脚本语言
3. 为了快把 Go / Rust 边界重新打乱
4. 为了保险继续扩大 prompt，而不是改上下文装配
5. 为了完成任务跳过 verify

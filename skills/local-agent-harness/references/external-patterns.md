# 外部模式信号

只保留对这个仓库最有用的外部结论，不做长篇转载。

## 1. OpenAI 的信号

近期官方对 `Harness Engineering` 的强调，核心不是让模型更会“说”，而是让系统更会：

1. 装配上下文
2. 组织工具
3. 执行验证
4. 控制状态流转
5. 让长任务在有限上下文里继续稳定推进

对这个项目最有价值的落点：

1. 提示词不是主工程面，运行时外壳才是主工程面
2. 验证和恢复机制要写进系统，不要寄希望于模型自律
3. 工程师的工作正在转向“设计约束与执行环境”

## 2. Anthropic 的信号

Anthropic 关于长任务 harness 的公开讨论里，最值得吸收的是两件事：

1. `Progressive disclosure`
   先给索引，再按需加载，不要一次性喂全量文档。

2. `独立 evaluator`
   生成者和评估者最好在角色或阶段上分离，否则模型容易把“看起来差不多”误判成“已经完成”。

对这个项目的直接启发：

1. `docs/README.md` 这种入口导航是有价值的，应继续保持“先入口、后深入”
2. verify 不应只是最后一句“请检查”，而应成为主循环阶段
3. 长任务需要 handoff artifact，否则上下文一长就会开始乱

## 3. 行业常见失败模式

做本地智能体时最常见的失败，不是模型不会写代码，而是外壳失稳：

1. 上下文越塞越多，最后噪声压过信号
2. 工具结果没有统一结构，模型靠猜
3. 高风险动作没有前置闸门
4. 事件只对人类可读，对系统不可恢复
5. 长任务没有交接包，重试时只能重新摸索
6. 生成和验证由同一个主体完成，导致过度乐观

## 4. 对本仓库的效率建议

如果目标是提效，而不是只增加复杂度，优先做这些：

1. 强化按需上下文装配，减少无关文档进入模型
2. 强化 artifact 外置，减少事件流和 prompt 被长文本污染
3. 强化 verify 和回归入口，把“完成”从主观判断改成证据判断
4. 强化确认主线恢复能力，避免一次确认变成一次重跑
5. 强化交接包格式，支持长任务续跑

## 5. 不要误学的地方

不要把 harness 理解成：

1. 继续堆更多系统提示词
2. 提前上复杂多智能体
3. 为了显得智能而扩很多不稳定旁支
4. 把所有知识和记忆默认注入

真正的方向是：

1. 让系统默认更稳
2. 让失败更可恢复
3. 让验证更独立
4. 让长任务更可交接

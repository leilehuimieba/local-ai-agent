# Harness Engineering

## 概览

把注意力从“模型这次怎么说”转到“系统默认怎么让它更稳”。
先判断失败是哪个壳薄了，再决定要补提示词、补协议、补上下文还是补验证。

## 定位说明

这份 skill 是用来设计、诊断、评审和收口 Harness 的方法包，不等同于系统运行时已经具备完整 Harness 能力。

它解决的是“哪一层壳薄了、应先补哪里、怎样用最小路径补强”的问题，不直接代表该层已经在代码、状态、合同、观测或恢复机制上完整落地。

默认使用场景是增量治理，而不是一次性追求理想化终态。使用这份 skill 时，应把“识别问题并给出收口路径”和“该能力已工程化落地”明确区分开。

## 何时使用

出现下面任一信号时，优先用这份 skill：

1. 模型能做对一部分，但整体不稳定。
2. 长任务前半程正常，后半程开始跑偏、偷懒或草率收尾。
3. 工具很多，但调用结果没有统一结构。
4. 同一个 agent 既生成又自评，完成判断明显偏乐观。
5. 上下文越塞越多，结果反而变差。
6. 需要把 agent 工作流从 demo 变成可恢复、可验证、可审计的系统。

## 调用边界

### 什么时候优先触发通用版

优先在下面这些场景触发 `harness-engineering`：

1. 还没有确定具体是哪一层设计失稳。
2. 当前项目不是这个仓库，或者仓库结构还不熟。
3. 你想先做跨项目通用的 agent 方案设计或评审。
4. 你怀疑问题出在上下文、验证、工具或长任务机制，但还没落到具体模块。

### 什么时候切到项目版

如果当前任务已经明确发生在某个具体仓库，并且需要落到仓库文档、模块、合同或实现顺序上，就切到项目专用 skill。

### 两个 skill 连用时的顺序

推荐顺序：

1. 先用 `harness-engineering` 判断失稳层和最小补强方向。
2. 再用项目专用 skill，把通用判断映射到当前仓库的具体文档、模块和验证入口。
3. 如果项目版收口后仍然发现问题定义不清，再回通用版重新诊断，而不是在项目版里硬猜。

## 与六层架构的关系

如果采用“上下文边界、工具系统、执行编排、记忆与状态、评估与观测、约束校验与恢复”这类六层口径，这份 skill 与其不冲突，只是拆分粒度更偏工程收口。

本 skill 会把六层进一步展开为更易落地的检查项，例如主循环、上下文装配、工具合同、权限确认、验证闭环、artifact 外置、记忆与知识路由、事件合同、长任务交接。

因此，使用本 skill 时，不要把“检查项更多更细”理解成“架构层级更多”；它只是把高层治理结构拆成更容易定位和补强的工程落点。

## 使用流程

1. 先读 `references/diagnostic-signals.md`，判断当前问题更像哪种壳失稳。
2. 再读 `references/core-checklist.md`，按固定顺序补主循环、上下文、工具、权限、验证、artifact、记忆、交接。
3. 需要做方案设计时，再读 `references/design-patterns.md`。
4. 需要评审已有实现时，再读 `references/review-template.md`。

## 默认原则

1. 把模型视为系统组件，不把模型本身当成系统。
2. 默认认为可靠性来自外壳设计，而不是单次提示词优化。
3. 默认先补稳定的主循环和验证闭环，再补高级能力。
4. 默认按需提供上下文，不做无差别堆料。
5. 默认让工具输出结构化，减少模型靠猜。
6. 默认让高风险动作进入确认主线。
7. 默认让长任务支持 handoff artifact 和上下文重建。
8. 默认让“完成”依赖证据，而不是依赖生成者自述。

## 边界提醒

`Verify` 不等于完整的 `评估与观测`。如果问题涉及日志、指标、归因、回归趋势、长期质量反馈，应在方案中单独显式说明，而不要默认被 verify 覆盖。

`确认恢复` 或 `handoff` 不等于完整的 `失败恢复机制`。如果问题涉及重试策略、回滚设计、不可恢复错误分类、资源退避或超时恢复，也应单独显式设计，而不要默认被现有交接或确认主线覆盖。

评审时应区分三种状态：`已有骨架`、`正在收口`、`已形成稳定默认能力`，避免把阶段性能力写成终态能力。

## 输出要求

使用这个 skill 产出方案、评审或修复建议时，优先给出：

1. `失稳层判断`
   当前更像是哪一层 harness 失稳。

2. `最小补强路径`
   先补哪一层，哪些复杂度明确后置。

3. `验证方式`
   用什么证据证明这次补强真的有效。
